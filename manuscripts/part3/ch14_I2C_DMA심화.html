Ch14. I2C DMA 심화

SHT31 온습도 센서를 DMA 비동기 방식으로 전환하고, UART·SPI·I2C 세 가지 DMA를 동시에 운용하는 실무 아키텍처를 완성합니다.

누적 기능: SHT31 폴링 → DMA 비동기 고속화 + 3종 DMA 동시 운용 (v1.4)

이 챕터의 아키텍처 위치

프로젝트 v1.3까지 우리는 세 가지 Driver를 구축했습니다. UART Driver(Ch06), SPI Driver(Ch12), I2C Driver(Ch11). 그런데 I2C Driver만 아직 폴링(Polling) 방식입니다. 이번 챕터에서는 i2c_driver.c를 DMA 모드로 업그레이드하여 시스템의 모든 주요 통신 드라이버가 DMA로 동작하는 상태를 완성합니다.

인터페이스 설계

이 챕터에서 업그레이드되는 I2C DMA Driver의 공개 API입니다. 상위 레이어(sht31_driver.c, sensor_service.c)는 아래 인터페이스만 사용하면 됩니다. 내부에서 DMA를 쓰는지 폴링을 쓰는지 알 필요가 없습니다. 이것이 레이어드 아키텍처의 핵심 이점입니다. 

/* ch14_i2c_dma_driver.h — 공개 인터페이스 */

/* I2C DMA 드라이버 초기화 */
HAL_StatusTypeDef I2cDma_Init(I2cDmaHandle *drv, I2C_HandleTypeDef *hi2c);

/* 비동기 송신 — 즉시 반환, 완료는 tx_cb 콜백으로 통보 */
HAL_StatusTypeDef I2cDma_MasterTransmit(I2cDmaHandle  *drv,
                                         uint16_t       dev_addr,
                                         uint8_t       *p_data,
                                         uint16_t       size,
                                         I2cDmaTxCpltCb tx_cb);

/* 비동기 수신 — 즉시 반환, 완료는 rx_cb 콜백으로 통보 */
HAL_StatusTypeDef I2cDma_MasterReceive(I2cDmaHandle  *drv,
                                        uint16_t       dev_addr,
                                        uint8_t       *p_data,
                                        uint16_t       size,
                                        I2cDmaRxCpltCb rx_cb);

/* SHT31 드라이버 — DMA 비동기 인터페이스 */
HAL_StatusTypeDef SHT31_StartMeasurement(void);    /* 측정 시작 (즉시 반환) */
bool              SHT31_GetData(float *temp, float *humi);  /* 데이터 읽기 */

학습 목표

이 챕터를 완료하면 다음을 할 수 있습니다.

  • [이해] I2C 폴링 방식과 DMA 방식의 CPU 점유율 차이를 설명하고, DMA 전환이 필요한 시나리오를 판단할 수 있다.
  • [적용] HAL_I2C_Master_Transmit_DMA / Receive_DMA를 사용하여 SHT31 온습도 센서를 비동기 방식으로 구동하는 드라이버를 작성할 수 있다.

  • [창조] 콜백 체인 패턴으로 Repeated Start 미지원 문제를 우회하고, 연속 측정 루프를 설계할 수 있다.

1절. I2C DMA 전환 이유와 아키텍처

Ch11에서 구현한 SHT31 드라이버는 HAL_I2C_Master_Transmit과 HAL_I2C_Master_Receive(폴링)을 사용했습니다. 이 방식은 코드가 단순하고 이해하기 쉽지만, 치명적인 약점이 있습니다. I2C 전송이 완료될 때까지 CPU가 아무 일도 하지 못하고 기다려야 한다는 점입니다.

비유로 이해하기 — 커피숍 직원의 두 가지 방식
폴링 방식은 커피숍 직원이 에스프레소 머신 앞에 서서 추출이 끝날 때까지 가만히 기다리는 것과 같습니다. 머신이 돌아가는 25초 동안 직원은 주문도 받지 못하고, 컵도 준비하지 못합니다.

DMA 방식은 직원이 머신을 켜놓고 추출되는 동안 다른 손님 주문을 받고, 컵을 준비하는 것과 같습니다. 머신이 완료 신호(콜백)를 울리면 그때 커피를 가져갑니다.
※ 이 비유의 한계: 실제 DMA는 별도의 하드웨어 회로가 독립적으로 동작하므로, 직원이 두 명인 것에 더 가깝습니다. 하지만 비동기 완료 통보라는 핵심 개념은 정확합니다.

SHT31 온습도 측정의 폴링 vs DMA 성능을 구체적으로 비교해 봅시다. SHT31의 고정밀 단발 측정 시퀀스는 다음과 같습니다:

  • I2C TX (CMD 2바이트 전송): 약 3ms
  • SHT31 내부 측정 대기: 약 15ms (HAL_Delay 또는 타이머)
  • I2C RX (데이터 6바이트 수신): 약 3ms

폴링 방식은 500ms마다 21ms를 CPU가 I2C 작업으로 블로킹되므로 약 4.2%의 CPU 점유율을 가집니다. 반면 DMA 방식은 콜백 처리에 ~1ms만 사용하여 CPU 점유율이 0.2% 수준으로 줄어듭니다. 이 차이가 중요한 이유는 프로젝트 v1.4에서는 LCD 화면 갱신(SPI DMA), PC 로그 출력(UART DMA), 센서 측정(I2C DMA)이 동시에 돌아가야 하기 때문입니다.

💡 실무 팁 — DMA 전환의 손익 계산
DMA 전환은 항상 이득이 아닙니다. 코드 복잡도가 크게 증가합니다. 전송 시간이 1ms 이상이고, 동시에 수행할 다른 작업이 있을 때만 DMA 전환을 고려하세요. 1ms 이하의 짧은 전송(예: 단순 GPIO 읽기)은 폴링이 더 간단하고 안정적입니다.
❓ 수강생 단골 질문
Q. RTOS를 쓰면 폴링 방식도 괜찮다고 들었는데, 그럼 DMA가 필요 없는 건가요?
A. RTOS에서는 I2C 태스크가 블로킹될 때 스케줄러가 다른 태스크를 실행시켜 CPU 낭비를 줄입니다. 하지만 DMA는 CPU 자체를 해방시키므로 RTOS 환경에서도 DMA가 더 효율적입니다. 특히 실시간 응답이 중요한 인터럽트 서비스나 타임-크리티컬 처리에서 DMA의 이점이 더욱 커집니다.

2절. HAL I2C DMA API 및 콜백 구조

STM32 HAL은 I2C DMA 전송을 위한 두 가지 함수를 제공합니다. 이름은 폴링 버전과 거의 같고 끝에 _DMA만 붙습니다. 하지만 동작 방식은 근본적으로 다릅니다. 

/* ===== HAL I2C DMA API 비교 ===== */

/* [폴링] TX — 완료될 때까지 대기 후 반환 */
HAL_StatusTypeDef HAL_I2C_Master_Transmit(I2C_HandleTypeDef *hi2c,
                                           uint16_t DevAddress,
                                           uint8_t  *pData,
                                           uint16_t Size,
                                           uint32_t Timeout);  /* ← 타임아웃 있음 */

/* [DMA] TX — DMA 설정 후 즉시 반환, 완료는 콜백으로 통보 */
HAL_StatusTypeDef HAL_I2C_Master_Transmit_DMA(I2C_HandleTypeDef *hi2c,
                                               uint16_t DevAddress,
                                               uint8_t  *pData,
                                               uint16_t Size);  /* ← 타임아웃 없음 */

/* [폴링] RX */
HAL_StatusTypeDef HAL_I2C_Master_Receive(I2C_HandleTypeDef *hi2c,
                                          uint16_t DevAddress,
                                          uint8_t  *pData,
                                          uint16_t Size,
                                          uint32_t Timeout);

/* [DMA] RX */
HAL_StatusTypeDef HAL_I2C_Master_Receive_DMA(I2C_HandleTypeDef *hi2c,
                                              uint16_t DevAddress,
                                              uint8_t  *pData,
                                              uint16_t Size);

DMA 버전에서 중요한 변화가 두 가지 있습니다. 첫째, Timeout 파라미터가 없습니다. DMA는 비동기이므로 함수 안에서 대기하지 않기 때문입니다. 둘째, 완료 여부를 콜백으로 알립니다. DMA 전송이 완료되면 DMA 인터럽트가 발생하고, HAL이 이를 처리하여 사용자가 구현한 콜백 함수를 호출합니다. 

/* ===== HAL I2C DMA 완료 콜백 (사용자가 구현) ===== */

/**
 * @brief I2C Master TX DMA 완료 — HAL이 자동 호출
 * @note  DMA 인터럽트 컨텍스트에서 실행됩니다 (ISR!)
 *        실행 시간을 최대한 짧게 유지하세요.
 */
void HAL_I2C_MasterTxCpltCallback(I2C_HandleTypeDef *hi2c)
{
    if (hi2c->Instance == I2C1) {
        /* TX 완료 처리 — 플래그 설정 또는 다음 동작 트리거 */
        LOG_D("I2C1 TX 완료");
        /* → 여기서 다음 RX를 시작하거나 플래그를 세웁니다 */
    }
}

/**
 * @brief I2C Master RX DMA 완료 — HAL이 자동 호출
 */
void HAL_I2C_MasterRxCpltCallback(I2C_HandleTypeDef *hi2c)
{
    if (hi2c->Instance == I2C1) {
        LOG_D("I2C1 RX 완료 — 데이터 준비됨");
        /* → 여기서 수신된 데이터를 파싱하거나 플래그를 세웁니다 */
    }
}

/**
 * @brief I2C 오류 — HAL이 자동 호출
 */
void HAL_I2C_ErrorCallback(I2C_HandleTypeDef *hi2c)
{
    if (hi2c->Instance == I2C1) {
        LOG_E("I2C1 DMA 오류: ErrorCode=0x%08lX", hi2c->ErrorCode);
        /* → 오류 복구 처리 */
    }
}
💡 실무 팁 — ISR(인터럽트 서비스 루틴) 콜백의 황금 원칙
HAL_I2C_MasterTxCpltCallback은 DMA 인터럽트 컨텍스트(ISR)에서 호출됩니다. ISR은 CPU가 현재 하던 일을 잠시 멈추고 실행하는 "긴급 처리 구간"입니다. 여기서 HAL_Delay()나 float 연산처럼 시간이 오래 걸리는 작업을 하면, 다른 인터럽트(UART, SPI 등)가 지연되어 시스템 전체가 불안정해집니다.

황금 원칙: ISR에서는 플래그 세팅 + 상태 전이만. 실제 처리는 메인 루프에서.

CubeMX I2C DMA 설정 방법

CubeMX에서 I2C DMA를 활성화하는 절차입니다. 이 단계를 빠뜨리면 HAL_I2C_Master_Transmit_DMA를 호출해도 아무 일이 일어나지 않습니다.

  1. CubeMX → Connectivity → I2C1 선택
  2. 하단 DMA Settings 탭 클릭
  3. Add 버튼 → I2C1_TX 선택 → Direction: Memory to Peripheral
  4. 다시 Add → I2C1_RX 선택 → Direction: Peripheral to Memory
  5. I2C1_TX: DMA1 Stream7, Channel 1, Priority: Low 설정
  6. I2C1_RX: DMA1 Stream0, Channel 1, Priority: Low 설정
  7. NVIC Settings 탭 → DMA1 Stream7 / Stream0 글로벌 인터럽트 활성화
❓ 수강생 단골 질문
Q. CubeMX에서 DMA를 설정했는데 HAL_I2C_Master_Transmit_DMA가 HAL_ERROR를 반환합니다.
A. 가장 흔한 원인은 I2C1의 DMA Request가 추가되지 않은 경우입니다. CubeMX DMA Settings 탭에서 "Add" 버튼으로 I2C1_TX와 I2C1_RX를 명시적으로 추가해야 합니다. 또한 NVIC Settings에서 해당 DMA 스트림의 글로벌 인터럽트가 활성화되었는지 확인하세요.

3절. Repeated Start(반복 시작) 미지원 문제 해결

이번 절에서는 이 챕터에서 가장 어려운 개념인 Repeated Start 미지원 우회 패턴을 다룹니다. 처음 보면 콜백 체인 흐름이 복잡해 보이지만, 단계별로 따라가면 반드시 이해할 수 있습니다. 그리고 한 번 익히면 I2C뿐 아니라 여러 비동기 드라이버에서 반복해서 쓸 수 있는 강력한 패턴입니다.

I2C DMA를 사용하다 보면 폴링 방식에서는 없던 문제가 하나 나타납니다. 바로 Repeated Start(반복 시작, RS) 미지원 문제입니다. 먼저 Repeated Start가 무엇인지부터 알아봅시다.

비유로 이해하기 — 택시 환승 vs 새 택시 잡기
일반 I2C 트랜잭션은 "START → 데이터 전송 → STOP"입니다. STOP은 버스를 비우는 신호입니다. 버스에서 내린 후 새로 타는 것과 같습니다.

Repeated Start는 STOP 없이 바로 다음 START를 발행하는 것입니다. 택시에서 내리지 않고 목적지를 바꾸는 것과 같습니다. 버스 점유권을 유지한 채로 방향(Read/Write)을 바꿀 수 있어서, 다른 마스터가 끼어들 틈이 없습니다.
※ 이 비유의 한계: 단일 마스터 시스템에서는 Repeated Start와 Stop+Start의 차이가 실질적으로 없는 경우가 많습니다.

SHT31 온습도 측정 시 I2C 폴링 HAL은 내부적으로 Repeated Start를 사용합니다. 하지만 HAL_I2C_Master_Transmit_DMA / Receive_DMA는 이 기능을 지원하지 않습니다. 매 트랜잭션마다 STOP 후 새 START를 발행합니다.

다행히 SHT31은 Repeated Start 방식과 Stop+Start 방식 모두를 허용합니다. 따라서 우리는 콜백 체인 패턴으로 이 문제를 우회할 수 있습니다. TX 완료 콜백에서 15ms 타이머를 시작하고, 타이머 만료 후 RX를 시작하는 방식입니다. 

/* ===== 콜백 체인 패턴 구현 ===== */

/* 단계 1: 측정 명령 전송 시작 */
HAL_StatusTypeDef SHT31_StartMeasurement(void)
{
    uint8_t cmd[2] = {0x24, 0x00};  /* 고정밀 단발 측정 명령 */

    g_state = SHT31_DMA_CMD_TX;

    LOG_D("SHT31_StartMeasurement: CMD 전송 시작");

    /* TX DMA 시작 → 완료 시 on_tx_cplt 호출 */
    return I2cDma_MasterTransmit(&g_i2c_drv, SHT31_I2C_ADDR,
                                  cmd, 2, on_tx_cplt);
}

/* 단계 2: TX 완료 → 15ms 타이머 시작 (ISR 컨텍스트) */
static void on_tx_cplt(void)
{
    g_state = SHT31_DMA_WAIT_MEAS;

    /* TIM6 원샷 타이머 시작 (15ms 후 PeriodElapsedCallback 호출) */
    HAL_TIM_Base_Start_IT(&htim6);

    LOG_D("SHT31 on_tx_cplt: 15ms 측정 대기 시작");
}

/* 단계 3: 타이머 만료 → 데이터 수신 시작 */
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance == TIM6) {
        HAL_TIM_Base_Stop_IT(htim);  /* 원샷: 타이머 중지 */

        g_state = SHT31_DMA_DATA_RX;

        /* RX DMA 시작 → 완료 시 on_rx_cplt 호출 */
        I2cDma_MasterReceive(&g_i2c_drv, SHT31_I2C_ADDR,
                              g_rx_buf, 6, on_rx_cplt);

        LOG_D("SHT31 타이머 완료: 데이터 수신 시작");
    }
}

/* 단계 4: RX 완료 → 데이터 파싱 (ISR 컨텍스트) */
static void on_rx_cplt(void)
{
    sht31_parse_raw(g_rx_buf);   /* CRC 검증 + 온습도 계산 */

    g_state      = SHT31_DMA_IDLE;
    g_data_ready = true;

    LOG_I("SHT31 측정 완료: 온도=%.1f°C, 습도=%.1f%%",
          g_temperature, g_humidity);
}
💡 실무 팁 — HAL_BUSY 오류 대처법
TX 완료 콜백에서 바로 RX를 호출하면 간헐적으로 HAL_BUSY가 반환될 수 있습니다. 이는 HAL 내부 상태가 READY로 전환되기 전에 호출되는 타이밍 문제입니다. TIM6 타이머(1ms 이상)를 사이에 두는 패턴은 이 문제를 자연스럽게 해결하면서 SHT31 측정 대기 시간도 처리합니다. 일석이조입니다.
❓ 수강생 단골 질문
Q. Repeated Start가 필요한 다른 I2C 센서에서는 어떻게 해야 하나요? SHT31처럼 Stop+Start를 허용하지 않는 경우도 있나요?
A. 일부 오래된 I2C 장치는 데이터시트에 "Repeated Start 필수"라고 명시합니다. 이런 경우 HAL DMA 대신 HAL_I2C_Master_Sequential_Transmit/Receive_IT(인터럽트 모드)를 사용하거나, 레지스터 직접 조작 방식을 써야 합니다. 하지만 현대 센서 대부분은 Stop+Start를 허용합니다. 데이터시트의 "Timing Diagram" 섹션에서 Sr(Repeated Start) 표기를 확인하세요.
📌 강사 꿀팁
로직 애널라이저가 있다면 폴링 모드와 DMA 모드 각각의 I2C 버스 파형을 캡처해서 비교해 보세요. 폴링 모드에서는 CMD 전송 → REPEATED START → 데이터 수신이 하나의 연속된 파형으로 보이고, DMA 모드에서는 CMD 전송 → STOP → (15ms 간격) → START → 데이터 수신으로 보입니다. 파형을 직접 보면 Repeated Start의 개념이 확실히 이해됩니다. 로직 애널라이저가 없다면 I2C 핀에 LED를 달아 SCL 활성/비활성 구간을 시각화하는 것도 방법입니다.

4절. 다중 DMA 동시 운용 및 우선순위 설정

프로젝트 v1.4에서는 세 가지 DMA가 동시에 동작합니다. UART DMA(로그 출력), SPI DMA(LCD 화면 갱신), I2C DMA(온습도 측정). 이 세 가지를 동시에 돌리려면 DMA 스트림 충돌(Stream Conflict)이 없어야 하고, 우선순위(Priority)를 올바르게 설정해야 합니다.

비유로 이해하기 — 고속도로 차선 배치
STM32F411의 DMA1에는 8개의 스트림(Stream 0~7)이 있습니다. 각 스트림은 고속도로의 차선과 같습니다. 한 차선에는 차 한 대만 달릴 수 있습니다. I2C1 TX와 UART2 TX가 모두 "DMA1 Stream6"을 요청한다면, 두 차가 같은 차선으로 진입하려는 충돌이 발생합니다. 이런 경우 하나를 다른 차선(스트림)으로 옮겨야 합니다.
※ 이 비유의 한계: 스트림과 채널은 2차원 구조입니다. 같은 스트림이라도 채널이 다르면 동시 사용 불가입니다. 한 스트림에는 한 번에 하나의 채널만 활성화됩니다.

스트림 충돌 분석 및 해결

v1.3 상태에서 I2C DMA를 추가할 때 주의해야 할 충돌이 단 하나 있습니다. CubeMX가 경고를 표시하므로 경고를 무시하지 않으면 실수할 일이 없습니다. 기본 CubeMX 설정에서 I2C1 TX는 DMA1 Stream6 Channel 1을 사용합니다. 그런데 UART2 TX(Ch06에서 설정)도 DMA1 Stream6 Channel 4를 사용합니다. 채널은 다르지만 스트림이 같으므로, 동시에 두 DMA를 요청하면 충돌이 발생합니다. 해결책은 간단합니다 — I2C1 TX를 Stream7으로 옮기면 됩니다. 

/* ===== v1.4 최종 DMA 스트림 할당 (CubeMX 설정 기준) ===== */

/*
 * [변경 필요] I2C1 TX 기본 설정 (충돌 발생!)
 *   DMA1 Stream6 Channel 1  ← UART2 TX(Stream6 Ch4)와 스트림 충돌
 *
 * [올바른 설정] I2C1 TX 재배치
 *   DMA1 Stream7 Channel 1  ← 독립 스트림, 충돌 없음
 *
 * UART2 TX:  DMA1 Stream6 Channel 4 — Priority: Medium (기존 유지)
 * UART2 RX:  DMA1 Stream5 Channel 4 — Priority: Medium (기존 유지)
 * SPI1  TX:  DMA2 Stream3 Channel 3 — Priority: Very High (기존 유지)
 * I2C1  TX:  DMA1 Stream7 Channel 1 — Priority: Low (재배치!)
 * I2C1  RX:  DMA1 Stream0 Channel 1 — Priority: Low
 *
 * 참고: SPI1은 DMA2 사용 → DMA1과 완전 독립 → 충돌 없음
 */

/* NVIC 우선순위 (숫자가 작을수록 높은 우선순위) */
/* SPI DMA = 5, UART DMA = 6, I2C DMA = 7 */
/* 이유: LCD 화면 끊김이 가장 나쁜 사용자 경험이므로 SPI 최우선 */

void multi_dma_nvic_config_example(void)
{
    /* SPI1 TX DMA — 화면 갱신 최우선 */
    HAL_NVIC_SetPriority(DMA2_Stream3_IRQn, 5, 0);
    HAL_NVIC_EnableIRQ(DMA2_Stream3_IRQn);

    /* UART2 TX/RX DMA — 로그 출력 중간 우선순위 */
    HAL_NVIC_SetPriority(DMA1_Stream6_IRQn, 6, 0);
    HAL_NVIC_EnableIRQ(DMA1_Stream6_IRQn);
    HAL_NVIC_SetPriority(DMA1_Stream5_IRQn, 6, 0);
    HAL_NVIC_EnableIRQ(DMA1_Stream5_IRQn);

    /* I2C1 TX/RX DMA — 센서 측정, 낮은 우선순위 */
    HAL_NVIC_SetPriority(DMA1_Stream7_IRQn, 7, 0);
    HAL_NVIC_EnableIRQ(DMA1_Stream7_IRQn);
    HAL_NVIC_SetPriority(DMA1_Stream0_IRQn, 7, 0);
    HAL_NVIC_EnableIRQ(DMA1_Stream0_IRQn);

    LOG_I("다중 DMA NVIC 우선순위 설정 완료");
}
💡 실무 팁 — DMA 스트림 배치 확인하기
STM32F411 Reference Manual(RM0383) Table 28(DMA1 requests)과 Table 29(DMA2 requests)를 반드시 확인하세요. CubeMX가 자동으로 설정하지만, 스트림 충돌 경고를 무시하면 런타임에서 예측 불가능한 오류가 발생합니다. 개발 초기에 MultiDma_Diagnose() 같은 진단 함수를 만들어 스트림 설정을 한 번씩 로그로 출력하는 습관을 들이세요.
🎯 면접 포인트
"STM32에서 두 개의 DMA 채널이 같은 스트림을 사용할 때 어떤 문제가 발생하고, 어떻게 해결하나요?"

핵심 답변 키워드: 스트림 충돌, 동시 DMA 요청 불가, CubeMX 스트림 재배치, RM0383 Table 28/29 참조
🔍 스스로 점검
  • DMA 스트림 우선순위(DMA_PRIORITY_*)와 NVIC 인터럽트 우선순위는 어떤 차이가 있나요?
  • SPI1이 DMA2를 사용하는 이유가 무엇인지 RM0383 Table 29에서 확인해 보세요.

5절. 연속 측정 루프 + 실무 검증

지금까지 배운 내용을 모두 통합하여 SHT31 온습도 연속 측정 루프를 완성합니다. 연속 측정이란 한 번 측정이 완료되면 자동으로 다음 측정을 시작하는 것입니다. TIM2 500ms 주기 인터럽트가 측정 요청을 트리거하고, 콜백 체인이 나머지를 자동으로 처리합니다. 

/* ===== 메인 루프 통합 예제 — main.c ===== */

/* 전역 변수 — 모듈 간 데이터 공유 */
static float     g_temperature = 0.0f;
static float     g_humidity    = 0.0f;
static bool      g_sensor_ready = false;

/* TIM2 500ms 주기 인터럽트 — 측정 트리거 */
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance == TIM2) {
        /* 이전 측정이 완료된 경우에만 새 측정 시작 */
        if (SHT31_StartMeasurement() == HAL_BUSY) {
            LOG_W("SHT31: 이전 측정 미완료 — 이번 주기 건너뜀");
        }
    }

    if (htim->Instance == TIM6) {
        /* SHT31 15ms 측정 대기 타이머 만료 */
        SHT31_DMA_OnTimerElapsed();
    }
}

/* 메인 루프 — 비블로킹 처리 */
void app_main_loop(void)
{
    float temp, humi;

    /* 센서 데이터 준비 확인 (폴링이지만 짧음) */
    if (SHT31_GetData(&temp, &humi)) {
        g_temperature = temp;
        g_humidity    = humi;
        g_sensor_ready = true;

        LOG_I("온습도 갱신: %.1f°C, %.1f%%", temp, humi);

        /* UI 서비스 갱신 요청 */
        UI_UpdateSensorData(temp, humi);
    }

    /* LCD, CLI 등 다른 작업은 여기서 계속 수행 */
    /* SPI DMA, UART DMA가 동시에 동작 중 */
}

💡 실무 팁 — 재진입(Re-entrant, 재진입 문제) 예방
재진입 문제란 함수가 실행 완료 전에 다시 호출되는 상황입니다. 예를 들어 on_rx_cplt()에서 새 DMA를 시작했는데, 직전 DMA TC 인터럽트가 아직 처리 중이라면 HAL 내부 상태가 꼬일 수 있습니다. 연속 측정 루프에서 주의할 점은 이 콜백 재진입입니다. I2cDmaState FSM으로 상태를 체크하고, IDLE이 아닌 상태에서 새 요청이 들어오면 HAL_BUSY로 거부하는 패턴이 이 문제를 방지합니다. TIM2 500ms 주기가 SHT31 전체 측정 시간(~21ms)보다 훨씬 길므로, 정상 동작 시 재진입은 발생하지 않습니다.
❓ 수강생 단골 질문
Q. 콜백이 전혀 불리지 않습니다. 로그가 "CMD 전송 시작" 이후로 멈춥니다.
A. 이 증상의 가장 흔한 원인 세 가지입니다. ① DMA 인터럽트(DMA1_Stream7_IRQn)가 NVIC에서 활성화되지 않은 경우, ② HAL_I2C_MasterTxCpltCallback 함수명이 정확히 일치하지 않는 경우(오타), ③ CubeMX에서 I2C1 DMA Request가 추가되지 않은 경우. CubeMX 설정을 재확인하고 NVIC 설정 탭에서 DMA 인터럽트 체크박스를 확인하세요.
🔍 스스로 점검
  • 연속 측정 루프에서 TIM2 주기를 10ms로 줄이면 어떻게 될까요? SHT31 전체 측정 시간(약 21ms)과 비교해 생각해 보세요.
  • g_data_ready 플래그를 volatile로 선언해야 할까요? 메인 루프와 ISR이 공유하는 변수라는 점을 고려하세요.
📌 강사 꿀팁 — 측정 횟수로 동작 검증
SHT31_GetMeasCount()가 반환하는 누적 측정 횟수를 5초마다 로그로 출력해 보세요. 정상이라면 500ms 주기이므로 5초에 약 10회 측정이 이루어져야 합니다. 만약 0이면 콜백이 전혀 불리지 않는 것이고, 예상보다 적으면 HAL_BUSY로 일부 측정이 건너뛰어진 것입니다. 이 숫자 하나로 시스템 상태를 빠르게 진단할 수 있습니다.

실습: SHT31 폴링 → DMA 비동기 전환 및 3종 DMA 동시 운용

목표: Ch11에서 구현한 SHT31 폴링 드라이버를 DMA 비동기 방식으로 업그레이드하고, UART/SPI/I2C DMA가 동시에 동작함을 LOG로 확인합니다.

환경: STM32CubeIDE, NUCLEO-F411RE, SHT31 모듈(I2C1), ILI9341 TFT LCD(SPI1)

Step 1. CubeMX DMA 설정 업데이트

  1. CubeMX 프로젝트 열기 (v1.3 기준)
  2. Connectivity → I2C1 → DMA Settings 탭
  3. Add → I2C1_TX: DMA1 Stream7 Ch1, Priority Low
  4. Add → I2C1_RX: DMA1 Stream0 Ch1, Priority Low
  5. NVIC Settings: DMA1_Stream7 / DMA1_Stream0 글로벌 인터럽트 활성화
  6. Timers → TIM6: Basic 타이머, PSC=839, ARR=1499 설정
    계산: TIM6 클럭 = 84MHz, 타이머 클럭 = 84MHz/(839+1) = 100kHz, 주기 = (1499+1) × (1/100kHz) = 15ms
  7. GENERATE CODE

Step 2. 코드 적용

  1. ch14_i2c_dma_driver.c/h → 프로젝트 Drivers 폴더에 추가
  2. ch14_sht31_dma.c/h → 프로젝트 Drivers 폴더에 추가
  3. main.c에서 SHT31_Read() 폴링 호출 → SHT31_StartMeasurement() 비동기 호출로 교체
  4. TIM2 500ms 주기 인터럽트에서 SHT31_StartMeasurement() 호출
  5. TIM6 인터럽트에서 SHT31_DMA_OnTimerElapsed() 호출

Step 3. 동작 확인

예상 결과: 500ms마다 온습도가 갱신되고, LCD 화면이 끊김 없이 갱신됩니다.

[INFO ] I2C1  TX: DMA1 S7 Ch1 Priority=0 (기대값: DMA_PRIORITY_LOW=0)

💡 실습이 안 될 때 — 체크리스트
이것은 DMA를 처음 사용하는 개발자 대부분이 겪는 문제입니다. 아래 체크리스트를 순서대로 확인하면 반드시 해결됩니다.

증상: 콜백이 불리지 않음
→ CubeMX DMA Settings에서 I2C1_TX/RX Add 여부 재확인
→ NVIC에서 DMA1_Stream7_IRQn / DMA1_Stream0_IRQn 활성화 여부 확인
→ HAL_I2C_MasterTxCpltCallback 함수명 오타 확인 (대소문자 포함)
→ MX_I2C1_Init()에서 __HAL_LINKDMA(&hi2c1, hdmarx, hdma_i2c1_rx) 코드 존재 여부 확인 (DMA Request 미추가 시 이 줄이 없음)

증상: HAL_BUSY 반환
→ 이전 측정이 완료되지 않았음. SHT31_DMA_STATE 확인용 LOG_D 추가
→ TIM2 주기가 너무 짧지 않은지 확인 (최소 25ms 이상 권장)

이것은 흔한 실수입니다. CubeMX 재설정 후 코드 재생성을 먼저 시도하세요.

핵심 정리

  • I2C DMA(HAL_I2C_Master_Transmit/Receive_DMA)는 비동기 방식으로 CPU를 해방시켜 동시 작업이 가능하게 한다.
  • HAL I2C DMA는 Repeated Start를 지원하지 않으므로, TX 완료 콜백에서 타이머(15ms)를 거쳐 RX를 시작하는 콜백 체인 패턴으로 우회한다.
  • STM32F411에서 I2C1 TX(DMA1 Stream6 Ch1)와 UART2 TX(DMA1 Stream6 Ch4)는 스트림 충돌이 발생하므로, I2C1 TX를 Stream7 Ch1으로 재배치해야 한다.
  • SPI1은 DMA2를 사용하므로 DMA1의 I2C/UART와 충돌 없이 동시 운용 가능하다.
  • DMA 우선순위는 SPI(LCD) > UART(로그) > I2C(센서) 순으로 설정하여 화면 끊김을 방지한다.
  • 콜백 재진입 문제는 드라이버 내부 FSM(I2cDmaState)으로 상태를 관리하여 예방한다.

아키텍처 업데이트

이 챕터 완료 후 i2c_driver.c가 DMA 모드로 업그레이드되었습니다. 이제 프로젝트의 모든 주요 통신 드라이버(UART, SPI, I2C)가 DMA로 동작하여 CPU는 센서·화면·통신 처리를 동시에 수행할 수 있는 여유를 가집니다.

연습문제

기억 1. HAL_I2C_Master_Transmit(폴링)과 HAL_I2C_Master_Transmit_DMA의 함수 시그니처에서 다른 파라미터가 무엇인지 쓰고, 그 이유를 설명하세요.
이해 2. STM32F411에서 I2C1 TX를 DMA1 Stream6 Channel 1에 배치하고 UART2 TX를 DMA1 Stream6 Channel 4에 배치할 때, 왜 동시 DMA 전송이 불가능한지 설명하세요. "스트림"과 "채널"의 차이를 포함하여 서술하세요.
적용 3. 다음 코드에서 HAL_BUSY가 간헐적으로 반환되는 이유를 찾고, 수정 방법을 제안하세요. 
void HAL_I2C_MasterTxCpltCallback(I2C_HandleTypeDef *hi2c) {
    /* TX 완료 즉시 RX 시작 시도 */
    HAL_I2C_Master_Receive_DMA(hi2c, 0x88, g_buf, 6);
}
분석 4. SHT31의 연속 측정 루프에서 측정 주기를 500ms에서 100ms로 단축했더니 LOG에 "이전 측정 미완료 — 이번 주기 건너뜀"이 빈번하게 출력됩니다. 이 현상의 원인을 분석하고, 100ms 주기에서도 정상 동작하도록 수정하는 방법을 설명하세요. (힌트: SHT31 측정 시간과 I2C 전송 시간의 합산을 고려하세요.)
