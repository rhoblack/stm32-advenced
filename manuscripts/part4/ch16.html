Ch16. PC 대시보드 (Python + pyserial)

MCU가 직접 "말을 걸어오는" 바이너리 프레임 프로토콜을 설계하고, Python 대시보드가 실시간으로 온습도·시각·모터 상태를 화면에 그린다.

누적 기능: 실시간 데이터 시각화 대시보드 (마일스톤 ② 달성)

이 챕터의 아키텍처 위치

Ch15(CLI)까지 우리는 텍스트 기반 명령 채널을 완성했습니다. Ch16에서는 동일한 UART 라인 위에 바이너리 프레임 채널을 추가합니다. 새로 추가되는 protocol_app.c는 App 레이어에 위치하며, Driver 레이어의 uart_driver.c를 재사용합니다.

인터페이스 설계

protocol_app.h가 상위(main 루프)에 노출하는 핵심 API입니다. UART 버스 점유 여부는 내부에서 관리하므로 호출자는 반환 코드만 확인하면 됩니다.

1. MCU-PC 통신 프로토콜 설계 원칙

Ch15까지 우리는 UART를 텍스트로 사용했습니다. time get을 입력하면 12:34:56\r\n이 돌아오는 방식입니다. 이 방식은 사람이 터미널에서 읽기 편하지만, Python 프로그램이 자동으로 파싱하기에는 불편합니다. 줄 끝 문자 처리, 숫자 문자열 변환, 오류 시 재동기 방법 등을 모두 직접 구현해야 하기 때문입니다.

비유로 이해하기: 수도 계량기의 두 얼굴
현장 기술자는 계량기 옆 디스플레이(텍스트 CLI)를 눈으로 읽습니다. 반면 원격 검침 센터는 미리 약속된 짧은 전파 코드(바이너리 프레임)를 수신합니다. 같은 물 소비량 데이터라도 용도에 따라 두 가지 표현이 공존합니다.
※ 이 비유의 한계: 실제 계량기는 단방향 전송이지만, 우리 시스템은 PC가 MCU에게 명령도 보낼 수 있는 양방향입니다.

1.1 텍스트 vs 바이너리 프로토콜 비교

임베디드 시스템에서 통신 프로토콜을 선택할 때는 세 가지 기준을 비교합니다: 파싱 복잡도, 오류 감지 능력, 대역폭 효율.

항목 텍스트 (CLI) 바이너리 프레임
사람 가독성 높음 — 터미널에서 직접 확인 낮음 — 헥스 뷰어 필요
파싱 복잡도 높음 — 공백/줄바꿈 처리 필요 낮음 — 고정 오프셋 직접 접근
오류 감지 없음 (별도 구현 필요) CRC-16으로 강력히 보호
대역폭 효율 낮음 — 숫자도 문자열로 전송 높음 — float 4바이트 그대로 전송
재동기 줄 끝 문자 기준 — 취약 STX/ETX 스캔 — 강건

1.2 프레임 설계 원칙 3가지

좋은 바이너리 프레임 설계는 세 가지 원칙을 따릅니다.

원칙 1 — 재동기 가능성: UART 노이즈로 바이트가 깨지면 수신 측이 다음 유효 프레임의 시작을 찾아야 합니다. STX(0xAA)와 ETX(0x55) 쌍으로 프레임 경계를 명시합니다. STX 값(0xAA)이 페이로드에 우연히 등장할 경우를 대비해 LEN 필드로 페이로드 끝 위치를 미리 알 수 있도록 설계합니다.

원칙 2 — 타입 구분: 하나의 UART 채널로 온도, 시각, 모터 상태 등 여러 종류의 데이터를 보냅니다. TYPE 필드(1바이트)로 수신 측이 페이로드를 어떻게 해석할지 결정합니다. 마치 HTTP의 Content-Type 헤더와 같은 역할입니다.

원칙 3 — 무결성 검증: UART 전송 중 비트 오류가 발생해도 수신 측이 감지할 수 있어야 합니다. CRC-16(16비트 순환 중복 검사)은 단순 합산 체크섬보다 훨씬 강한 오류 감지 능력을 제공합니다.

💡 실무 팁: 프레임 설계는 확장성을 먼저 생각하라
TYPE 필드를 1바이트로 설계하면 256가지 메시지 타입을 지원합니다. 현재 5개만 사용해도 나중에 디버그 로그 타입(0x10), 펌웨어 업데이트 타입(0xF0) 등을 추가할 여지를 남겨둡니다. 실무에서는 0x00(예약)과 0xFF(예약)는 미리 제외해 두는 관행이 있습니다.
❓ 수강생 단골 질문
Q. CLI도 있고 대시보드도 있으면 UART가 두 개 필요한가요?
A. 아닙니다. 같은 UART2(USB-Serial) 하나를 씁니다. 다만 CLI는 사람이 터미널을 켜서 직접 타이핑할 때 사용하고, 대시보드는 Python 프로그램이 자동으로 수신할 때 사용합니다. 두 프로그램을 동시에 열면 포트 충돌이 발생하므로, 실제 운용 시 둘 중 하나만 포트를 열어야 합니다.

2. 시리얼 프레임 구조와 CRC-16 구현

이 절에서는 실제 바이트 수준에서 프레임이 어떻게 생겼는지, 그리고 CRC-16이 무엇인지 살펴봅니다. CRC 수학 이론은 깊게 다루지 않습니다 — 계산 함수는 제공되며, "어떻게 쓰는가"에 집중합니다.

2.1 프레임 구조 상세

우리가 설계한 프레임의 바이트 구조는 다음과 같습니다.

/*
 * 시리얼 프레임 구조 (총 6 + n 바이트)
 * ┌──────┬──────┬──────┬─────────────┬─────────────┬──────┐
 * │ STX  │ TYPE │ LEN  │   PAYLOAD   │   CRC-16    │ ETX  │
 * │ 0xAA │  1B  │  1B  │   0~32 B    │    2 B      │ 0x55 │
 * └──────┴──────┴──────┴─────────────┴─────────────┴──────┘
 *           ↑─────────────────────────────↑
 *           CRC 계산 범위: TYPE + LEN + PAYLOAD
 */

각 필드의 역할을 정리하면:

  • STX (0xAA, 1바이트): 프레임 시작 마커. 수신 측이 이 값을 만날 때까지 바이트를 버립니다.
  • TYPE (1바이트): 페이로드 해석 방법. 0x01=센서, 0x02=시각, 0x03=모터, 0x04=시스템, 0x05=ACK.
  • LEN (1바이트): 페이로드 바이트 수 (0~32). 수신 측이 페이로드 끝 위치를 계산합니다.
  • PAYLOAD (0~32바이트): 실제 데이터. 타입별로 구조가 다릅니다.
  • CRC-16 (2바이트, 빅 엔디언): TYPE+LEN+PAYLOAD의 CRC. 상위 바이트 먼저 전송.
  • ETX (0x55, 1바이트): 프레임 종료 마커.

2.2 타입별 페이로드 구조

/* TYPE 0x01 — SENSOR_DATA (8바이트) */
/* float temp(4B) + float humi(4B) — IEEE 754, 리틀 엔디언 */
/* 예: 25.3°C = 0xC1CCCC41 → bytes[0..3] = {0x41, 0xCC, 0xCC, 0xC1} */

/* TYPE 0x02 — TIME_DATA (3바이트) */
/* uint8_t hour(1B) + uint8_t min(1B) + uint8_t sec(1B) */
/* 예: 14:30:05 → {0x0E, 0x1E, 0x05} */

/* TYPE 0x03 — MOTOR_STATUS (3바이트) */
/* uint16_t angle(2B, 리틀 엔디언) + uint8_t state(1B) */
/* 예: 2048 하프스텝, MOVING(1) → {0x00, 0x08, 0x01} */

/* TYPE 0x04 — SYSTEM_STATUS (6바이트) */
/* uint16_t version(2B) + uint32_t uptime_sec(4B) */
/* 예: v1.6, 업타임 120초 → {0x06, 0x01, 0x78, 0x00, 0x00, 0x00} */

2.3 CRC-16/CCITT란?

CRC(Cyclic Redundancy Check, 순환 중복 검사)는 데이터의 비트 오류를 감지하는 수학적 방법입니다. Ch11에서 SHT31 센서 데이터 검증에 CRC-8(8비트)을 사용한 것을 기억하시나요? 이번에는 통신 프레임 보호용으로 더 강력한 CRC-16(16비트)을 사용합니다. 다항식이 다를 뿐 기본 원리는 동일합니다.

비유: 계산기 영수증의 합계 금액
마트 영수증 하단의 "합계" 금액은 위 항목을 전부 더한 결과입니다. 만약 누군가 영수증 중간 항목의 숫자를 바꿔치기했다면 합계가 맞지 않아 즉시 위변조를 알 수 있습니다. CRC는 이보다 훨씬 강력한 "합계 계산법"으로, 단순 덧셈 대신 다항식 나눗셈을 사용합니다.
※ 이 비유의 한계: 합계는 덧셈 순서 바꿈을 감지 못하지만, CRC는 바이트 순서 변경도 높은 확률로 감지합니다.

CRC-16/CCITT는 다항식 0x1021, 초기값 0xFFFF을 사용합니다. 아래 코드는 code_examples/ch16_protocol_app.c에 그대로 제공됩니다. 외울 필요 없습니다 — 복사해서 사용하세요. 동작 원리 이해를 원하는 분만 코드를 분석하세요. 

/* protocol_app.c — CRC-16/CCITT 구현 */

uint16_t Protocol_CalcCRC16(const uint8_t *data, uint16_t len)
{
    uint16_t crc = 0xFFFFU;  /* 초기값 */
    uint16_t i;
    uint8_t  j;

    for (i = 0; i < len; i++)
    {
        crc ^= ((uint16_t)data[i] << 8);  /* 데이터 바이트를 상위로 XOR */
        for (j = 0; j < 8; j++)
        {
            if (crc & 0x8000U)
            {
                crc = (uint16_t)((crc << 1) ^ 0x1021U);  /* 다항식 XOR */
            }
            else
            {
                crc <<= 1;
            }
        }
    }
    LOG_D("CRC16 계산 완료: len=%d 결과=0x%04X", len, crc);
    return crc;
}

이 함수는 data 버퍼의 len 바이트에 대해 CRC를 계산합니다. 중요: CRC 계산 범위는 STX와 ETX를 제외한 TYPE+LEN+PAYLOAD입니다. STX와 ETX는 항상 고정값(0xAA, 0x55)이므로 CRC에 포함해도 무의미하며, 관례적으로 제외합니다.

🎯 면접 포인트
"체크섬(Checksum)과 CRC의 차이를 설명하세요."
핵심 키워드: 체크섬은 단순 합산 → 바이트 순서 바뀌어도 값 동일 (감지 불가). CRC는 다항식 나눗셈 → 1비트 오류, 2비트 오류, 버스트 오류 (최대 16비트)를 높은 확률로 감지. CRC-16은 임베디드 통신에서 표준적으로 사용.
💡 실무 팁: CRC 테이블 방식으로 속도 향상
위 코드는 비트 단위 루프(O(8n))입니다. 데이터량이 많다면 256개 항목의 사전 계산 테이블을 사용하는 방식(O(n))이 3~8배 빠릅니다. NUCLEO-F411RE의 속도(84MHz)에서는 위 방식으로도 충분하지만, 1Mbps 이상 고속 통신에서는 테이블 방식을 고려하세요.

3. protocol_app.c 직렬화 구현

이제 실제 MCU 코드를 작성합니다. 이 절이 Ch16의 학습 핵심입니다 — Python 코드는 이미 완성되어 있지만, MCU 측 프레임 직렬화(Serialization)는 직접 구현해야 합니다.

3.1 프레임 조립 함수 설계

모든 송신 함수가 공통으로 사용하는 내부 함수 build_frame()을 먼저 설계합니다. 이 함수는 타입, 페이로드를 받아 완성된 프레임 바이트 배열을 채우고 총 길이를 반환합니다. 

/* protocol_app.c — 내부 프레임 조립 함수 */

static uint16_t build_frame(Proto_TypeDef type,
                             const uint8_t *payload, uint8_t payload_len,
                             uint8_t *out_frame)
{
    uint16_t crc;
    uint16_t idx = 0;

    /* 1단계: STX */
    out_frame[idx++] = PROTO_STX;  /* 0xAA */

    /* 2단계: TYPE */
    out_frame[idx++] = (uint8_t)type;

    /* 3단계: LEN */
    out_frame[idx++] = payload_len;

    /* 4단계: PAYLOAD 복사 */
    if (payload != NULL && payload_len > 0)
    {
        memcpy(&out_frame[idx], payload, payload_len);
        idx += payload_len;
    }

    /* 5단계: CRC-16 계산 및 삽입 (범위: TYPE + LEN + PAYLOAD) */
    crc = Protocol_CalcCRC16(&out_frame[1], 2U + payload_len);
    out_frame[idx++] = (uint8_t)((crc >> 8) & 0xFFU);  /* 상위 바이트 먼저 */
    out_frame[idx++] = (uint8_t)(crc & 0xFFU);

    /* 6단계: ETX */
    out_frame[idx++] = PROTO_ETX;  /* 0x55 */

    LOG_D("build_frame: type=0x%02X len=%d total=%d CRC=0x%04X",
          (uint8_t)type, payload_len, idx, crc);

    return idx;  /* 완성된 프레임 총 바이트 수 */
}
📌 강사 꿀팁
idx 변수를 배열 인덱스와 "완성 길이" 두 용도로 동시에 쓰는 패턴은 임베디드 직렬화 코드에서 매우 흔합니다. 함수 마지막의 return idx가 총 길이를 자연스럽게 반환합니다. 수강생들에게 "idx 값을 단계마다 추적해 보라"고 하면 프레임 구조가 즉시 이해됩니다.

3.2 센서 데이터 전송 구현

float 값을 바이트 배열로 변환할 때 memcpy를 사용합니다. 캐스팅((uint8_t *)&temp)도 가능하지만, 엄밀한 C 표준 준수를 위해 memcpy가 선호됩니다. 

/* protocol_app.c — 센서 데이터 전송 */

HAL_StatusTypeDef Protocol_SendSensorData(float temp, float humi)
{
    uint8_t  payload[8];
    uint16_t frame_len;

    /* float → 4바이트 직렬화 (리틀 엔디언, IEEE 754) */
    memcpy(&payload[0], &temp, sizeof(float));
    memcpy(&payload[4], &humi, sizeof(float));

    LOG_D("Protocol_SendSensorData: temp=%.1f humi=%.1f", temp, humi);

    frame_len = build_frame(PROTO_TYPE_SENSOR_DATA,
                            payload, 8, s_tx_frame);

    /* TX 버스 점유 확인 */
    if (UART_IsTxBusy())
    {
        LOG_W("Protocol_SendSensorData: TX 버스 중 — 이번 전송 건너뜀");
        return HAL_BUSY;
    }

    s_frame_count++;
    LOG_I("Protocol_SendSensorData: 프레임 #%lu 전송 (len=%d)",
          s_frame_count, frame_len);
    return UART_Send(s_tx_frame, frame_len);
}
❓ 수강생 단골 질문
Q. HAL_BUSY를 반환하면 그 데이터는 그냥 버려지는 건가요?
A. 맞습니다. 대시보드용 센서 데이터는 1초마다 갱신되므로, 이번 전송이 실패해도 1초 후에 다시 최신 데이터가 전송됩니다. 만약 "절대 유실 불가" 데이터라면 큐(Queue)를 설계해야 합니다 — 이 챕터에서는 단순함을 위해 큐 없이 구현합니다.

3.3 메인 루프 통합

main.c의 메인 루프에서 1초 타이머 플래그를 보고 각 데이터를 순서대로 전송합니다. 모든 전송을 동시에 하면 TX 버스 충돌이 발생하므로, 플래그 방식으로 분산 전송합니다. 

/* main.c — 메인 루프 통합 예시 */

/* 전역 플래그 (TIM 인터럽트에서 설정) */
volatile uint8_t g_tick_1s = 0;
volatile uint8_t g_tick_10s = 0;

void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    static uint32_t cnt = 0;
    if (htim->Instance == TIM3)
    {
        cnt++;
        g_tick_1s = 1;
        if (cnt % 10 == 0) { g_tick_10s = 1; }
    }
}

/* 메인 루프 */
while (1)
{
    if (g_tick_1s)
    {
        g_tick_1s = 0;
        float temp, humi;
        RTC_TimeTypeDef t;

        /* 센서 데이터 전송 */
        Sensor_GetData(&temp, &humi);
        Protocol_SendSensorData(temp, humi);

        /* 시각 데이터 전송 (약간 지연해서 TX 충돌 방지) */
        HAL_Delay(20);
        HAL_RTC_GetTime(&hrtc, &t, RTC_FORMAT_BIN);
        Protocol_SendTimeData(t.Hours, t.Minutes, t.Seconds);

        /* 모터 상태 전송 */
        HAL_Delay(20);
        Protocol_SendMotorStatus(Stepper_GetPosition(), Stepper_GetState());
    }

    if (g_tick_10s)
    {
        g_tick_10s = 0;
        Protocol_SendSystemStatus();
        LOG_I("시스템 상태 프레임 전송 완료");
    }
}
💡 주의: 인터럽트 컨텍스트에서 호출 금지
Protocol_Send*() 함수들은 내부적으로 정적 버퍼(s_tx_frame)를 공유합니다. 인터럽트 핸들러(ISR)에서 이 함수를 호출하면 메인 루프의 전송 중에 버퍼가 덮어써져 프레임이 손상됩니다. 반드시 메인 루프(또는 RTOS 태스크)에서만 호출하세요. 인터럽트에서 호출이 필요한 경우 Critical Section(PRIMASK)으로 보호해야 합니다.
💡 실무 팁: HAL_Delay 대신 비동기 스케줄러
위 코드의 HAL_Delay(20)은 CPU를 블로킹합니다. 이 코드는 실습용 단순화입니다. 실무에서는 "이전 TX 완료 콜백에서 다음 전송을 트리거"하는 비동기 체인 방식을 사용합니다. 이 패턴은 Ch17(아키텍처 심화)에서 리팩토링 예제로 다룹니다.

4. pyserial 수신 스레드 코드 분석

이 절은 코드 분석 수준입니다. Python 코드를 직접 작성하지 않아도 됩니다. 완성된 ch16_dashboard.py를 읽으면서 어떻게 동작하는지 이해합니다.

❓ 수강생 단골 질문
Q. 스레드(Thread)를 처음 보는데 어렵지 않나요?
A. 이 절에서 스레드를 직접 구현하지 않습니다. "왜 스레드가 필요한가"와 "코드 흐름이 어떻게 되는가"만 이해하면 됩니다. 스레드는 "동시에 두 가지를 하는 방법"입니다 — 하나는 시리얼 데이터를 계속 받는 일, 다른 하나는 화면을 그리는 일입니다.

4.1 왜 수신에 스레드가 필요한가?

GUI 프로그램(PyQt5)의 메인 스레드는 화면 그리기와 버튼 클릭 처리를 담당합니다. 만약 메인 스레드에서 serial.read()를 호출하면 데이터가 올 때까지 메인 스레드가 멈추고 화면이 얼어붙습니다. 해결책은 시리얼 수신을 별도 스레드(QThread)로 분리하는 것입니다.

비유: 편의점 직원 2명
한 직원은 계산대 앞에서 손님을 받고(GUI 메인 스레드), 다른 직원은 뒤쪽 창고에서 택배를 받습니다(SerialReader 스레드). 택배가 와도 계산대 줄은 멈추지 않습니다.
※ 이 비유의 한계: 스레드는 실제 CPU 코어가 2개일 때만 진정한 병렬 실행이 됩니다. 단일 코어에서는 빠른 컨텍스트 전환으로 "동시처럼" 보이는 방식입니다.

4.2 SerialReader 스레드 흐름

# ch16_dashboard.py — SerialReader 핵심 구조 (설명용 발췌)

class SerialReader(QThread):
    frame_received = pyqtSignal(dict)   # 완성 프레임을 GUI 스레드로 전달

    def run(self):
        """스레드 시작 시 자동 호출 — 시리얼 수신 루프"""
        with serial.Serial(self._port, self._baud, timeout=0.1) as ser:
            while self._running:
                data = ser.read(64)       # 최대 64바이트씩 읽기
                for byte in data:
                    frame = self._parser.feed(byte)  # 1바이트씩 파서에 전달
                    if frame is not None:
                        self.frame_received.emit(frame)  # 시그널 발행

핵심 흐름: ser.read() → FrameParser.feed() → 완성 시 시그널 발행. 시그널-슬롯 메커니즘이 스레드 간 안전한 데이터 전달을 보장합니다.

4.3 FrameParser 상태 머신

Python의 FrameParser는 MCU 측의 build_frame()과 정확히 역방향 동작입니다. 5개 상태를 가진 FSM(유한 상태 머신)으로 구현되었습니다. 

# ch16_dashboard.py — FrameParser 상태 전이 요약

# STATE_HUNT_STX  → STX(0xAA) 수신 시 STATE_HEADER로
# STATE_HEADER    → TYPE + LEN 2바이트 수집 후 STATE_PAYLOAD/CRC로
# STATE_PAYLOAD   → LEN 바이트 수집 완료 시 STATE_CRC로
# STATE_CRC       → CRC 2바이트 수집 후 STATE_ETX로
# STATE_ETX       → ETX(0x55) 확인 + CRC 검증 → 완성 프레임 반환
💡 실무 팁: Python FrameParser는 MCU FSM의 거울
MCU의 build_frame()과 Python의 FrameParser.feed()는 동일한 프레임 정의를 공유합니다. 프로토콜을 변경할 때 양쪽을 동시에 수정하지 않으면 통신이 깨집니다. 실무에서는 공통 정의를 하나의 파일(예: protocol_spec.md)에 문서화하고 두 팀(임베디드/SW)이 참조하는 방식을 씁니다.

  • FrameParser의 STATE_HUNT_STX 상태가 존재하는 이유는? (재동기 목적)

핵심 정리

  • 바이너리 프레임 프로토콜은 텍스트 CLI보다 파싱이 단순하고, CRC-16으로 무결성을 강력히 보호하며, 대역폭 효율이 높습니다.
  • 프레임 구조 [STX][TYPE][LEN][PAYLOAD][CRC16][ETX]에서 LEN 필드는 수신 측이 페이로드 끝을 알고 재동기하는 핵심 역할을 합니다.
  • MCU 측 build_frame()은 6단계(STX → TYPE → LEN → PAYLOAD → CRC → ETX)로 프레임을 조립합니다. CRC 계산 범위는 TYPE+LEN+PAYLOAD입니다.
  • float 직렬화는 memcpy(&payload[0], &temp, sizeof(float))로 플랫폼 독립적으로 처리합니다 (캐스팅 대신 memcpy 권장).
  • Python의 FrameParser는 MCU build_frame()의 역함수입니다. 5개 상태 FSM으로 바이트 스트림에서 프레임을 복원합니다.
  • GUI와 시리얼 수신을 QThread로 분리하지 않으면 화면이 얼어붙습니다. 시그널-슬롯 메커니즘이 스레드 간 안전한 데이터 전달을 보장합니다.
