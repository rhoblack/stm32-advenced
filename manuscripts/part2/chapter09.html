Ch09. 내부 RTC 활용

경과 시간(스톱워치)을 넘어 "지금 몇 시인가" — 실시간 시계와 알람 FSM으로 스마트 시계의 심장을 완성한다

누적 기능: RTC 실시간 시각 + Alarm FSM (alarm_service + rtc_driver 추가, v0.8 → v0.9)

§9.0 "스톱워치의 한계 — 진짜 시계를 만들자"

여기까지의 여정을 돌아봅시다. Ch08에서 Stopwatch FSM을 완성하고, Service 레이어를 처음 구현했습니다. 버튼을 누르면 IDLE → RUNNING → PAUSED → IDLE. v0.8의 stopwatch_service — 여러분이 직접 설계한 첫 FSM 서비스입니다.

그런데 한 가지 근본적인 한계가 있습니다. 스톱워치는 "얼마나 지났나"(경과 시간, elapsed time)만 알 수 있습니다. "지금 몇 시인지"는 모릅니다. 핸드폰으로 비유하면, 타이머 앱은 있는데 시계 앱이 없는 셈입니다. 진짜 스마트 시계를 만들려면 "지금 몇 시인가"(절대 시각, wall-clock time)를 알아야 합니다.

이 챕터가 끝나면:

  • RTC(Real-Time Clock)로 시:분:초, 년:월:일을 읽고 씁니다.
  • AlarmA를 설정하여 특정 시각에 인터럽트를 발생시킵니다.
  • Ch08 Stopwatch FSM과 동일한 패턴으로 Alarm FSM을 설계·구현합니다.
  • 아키텍처를 v0.8 → v0.9로 업그레이드합니다 — alarm_service + rtc_driver 추가.

특히 중요한 것은 패턴의 반복입니다. Ch08에서 FSM을 처음 구현했다면, 이 챕터에서 두 번째로 동일한 패턴을 적용합니다. "이 구조, 전에 봤는데?" — 그 느낌이 들면 성공입니다. FSM 패턴이 체화되기 시작했다는 뜻입니다.

§9.1 RTC 아키텍처 — "MCU 안의 시계"

RTC는 Real-Time Clock(실시간 시계)의 약자입니다. 이름에 "Real-Time"이 있지만, RTOS의 "Real-Time"과는 완전히 다른 의미입니다. RTC의 Real-Time은 "실제 시각(wall-clock time)" — 사람이 사용하는 시:분:초, 년:월:일을 뜻합니다. RTOS의 Real-Time은 "시간 제약 보장(deadline guarantee)"입니다. 전혀 다른 개념이니 혼동하지 마세요.

STM32F411의 RTC는 MCU 내부에 있는 독립적인 시계 블록입니다. 세 가지 핵심 특성을 이해하면 됩니다: 백업 도메인, BCD 레지스터, 프리스케일러.

백업 도메인 — "비상 배터리가 있는 시계"

비유로 이해하기
집에 벽걸이 시계가 있습니다. 정전이 되어도 건전지로 계속 돌아갑니다. MCU의 RTC도 같습니다. 메인 전원(VDD)이 꺼져도 VBAT 핀에 코인셀 배터리를 연결하면, RTC만 독립적으로 시간을 유지합니다. 이 독립 영역이 백업 도메인(Backup Domain)입니다.
※ 이 비유의 한계: NUCLEO-F411RE에서는 VBAT가 VDD에 연결되어 있어, 전원을 끄면 시간이 소실됩니다. 코인셀 배터리를 별도로 연결해야 백업이 동작합니다. 실습에서는 매번 시간을 설정해야 합니다.

백업 도메인에 포함된 것:

  • RTC 하드웨어 — 캘린더, 알람, Wake-up 타이머
  • 백업 레지스터 20개 — 각 32비트, 총 80바이트 (전원 차단에도 유지)
  • LSE 오실레이터 — 32.768kHz 외부 크리스탈 (연결 시)

BCD 레지스터 — "전광판처럼 자릿수별 저장"

비유로 이해하기
지하철 전광판을 떠올리세요. 각 자릿수마다 0~9 숫자가 독립적으로 표시됩니다. BCD(Binary-Coded Decimal)도 같습니다. 23시를 저장할 때, "2"와 "3"을 각각 4비트(니블)로 나눠 담습니다. 결과: 0x23. 이것은 십진수 35(=0x23)가 아니라, "23"이라는 10진수를 그대로 16진 표기한 것입니다.
※ 이 비유의 한계: BCD는 각 니블이 0~9만 유효합니다. 0xA~0xF가 나오면 잘못된 값입니다. 전광판은 물리적으로 0~9만 표시하므로 이 제약이 자연스럽지만, 메모리에서는 A~F가 저장될 수 있어 주의가 필요합니다.

RTC가 왜 BCD를 사용할까요? 두 가지 이유입니다:

  1. 디스플레이 친화적: 7-세그먼트나 LCD에 시간을 표시할 때, BCD 값을 그대로 보내면 됩니다. 별도 변환이 필요 없습니다.
  2. 사람이 읽기 쉬움: 디버거에서 RTC_TR 레지스터를 보면 0x145930 → "14시 59분 30초"로 바로 읽힙니다.

BCD ↔ Binary 변환은 간단합니다: 

/* BCD → Binary: (상위 니블 × 10) + 하위 니블 */
uint8_t bcd_to_bin(uint8_t bcd) {
    return ((bcd >> 4) * 10) + (bcd & 0x0F);
}

/* Binary → BCD: (십의 자리 << 4) | 일의 자리 */
uint8_t bin_to_bcd(uint8_t bin) {
    return ((bin / 10) << 4) | (bin % 10);
}

/* 실무에서는 HAL 매크로 사용: */
/* RTC_ByteToBcd2(value), RTC_Bcd2ToByte(value) */

프리스케일러 — "32,768Hz를 1Hz로"

RTC 클럭(LSE 32,768Hz 또는 LSI ~32kHz)을 1초 단위로 만들기 위해 두 단계 분주를 합니다:

  • 비동기 프리스케일러 (PREDIV_A, 7비트): 기본값 127 → 128분주
  • 동기 프리스케일러 (PREDIV_S, 15비트): 기본값 255 → 256분주

결과: 32,768 / (128 × 256) = 1Hz (ck_spre)

Ch07의 TIM에서 PSC/ARR로 주기를 계산한 것과 같은 원리입니다. TIM은 84MHz를 분주했다면, RTC는 32kHz를 분주합니다. 목표 주파수가 다를 뿐, "프리스케일러로 원하는 주기를 만든다"는 개념은 동일합니다.

❓ 수강생 단골 질문
Q. BCD가 왜 필요한가요? Binary로 하면 안 되나요?
A. HAL_RTC_GetTime()이 BCD → Binary 변환을 자동으로 해주므로, 실무에서 직접 BCD 계산할 일은 거의 없습니다. 하지만 디버거에서 RTC_TR 레지스터를 직접 볼 때 BCD 값이 표시됩니다. "0x23이 왜 35가 아니라 23이지?"를 이해해야 합니다. 또한 7-세그먼트 디스플레이 같은 BCD 출력 장치와 연동할 때 변환 없이 바로 쓸 수 있습니다.
🎯 면접 포인트
Q. "RTC 백업 도메인의 목적과 구성 요소를 설명하세요."
A. 정전이나 시스템 리셋이 발생해도 시간 정보와 중요 설정값을 유지하기 위한 독립 전원 영역입니다. 구성: RTC 하드웨어, 백업 레지스터 20개, LSE 오실레이터. 전원: VDD 또는 VBAT (자동 전환). 리셋(시스템/전원/Standby)에 영향받지 않습니다.
💡 실무 팁
NUCLEO-F411RE에서 VBAT 핀은 VDD에 연결되어 있어 별도 백업 전원이 없습니다. 전원을 끄면 RTC 시간이 소실됩니다. 제품 설계 시에는 VBAT 핀에 CR2032 코인셀 배터리를 연결하여 RTC를 독립 유지하세요. RTC 소비 전류는 약 1μA로, 코인셀 하나로 수 년 유지됩니다.

§9.2 클럭 소스 선택과 CubeMX 설정

RTC의 정확도는 클럭 소스에 의해 결정됩니다. STM32F411은 두 가지 저속 클럭을 제공합니다: LSE(외부 크리스탈)와 LSI(내부 RC).

비유로 이해하기
석영(quartz) 시계와 기계식 시계의 차이입니다. 석영 시계(LSE)는 월 수 초 오차. 기계식 시계(LSI)는 월 수 분 오차. 정확도가 중요하면 석영, 아니면 기계식으로 충분합니다.
※ 이 비유의 한계: LSI는 RC 오실레이터이므로 기계식보다는 "저가형 전자시계"에 가깝습니다. 온도에 따른 주파수 편차도 있습니다.
항목LSE (32.768kHz)LSI (~32kHz)
정확도±20ppm (월 ~1초)±5% (하루 ~72분)
주파수정확히 32,768Hz약 32,000Hz (변동)
외부 부품크리스탈 필요없음 (내장)
NUCLEO 기본미장착 (SB48/SB49 OFF)즉시 사용 가능
실무 판단양산 제품 → 필수프로토타입/교육 → OK

NUCLEO-F411RE LSE 상태 확인

NUCLEO-F411RE는 기본적으로 LSE 크리스탈이 장착되어 있지 않습니다. 보드 뒷면의 솔더 브릿지 SB48, SB49가 열려 있습니다. LSE를 사용하려면:

  1. SB48, SB49에 납땜으로 브릿지 연결
  2. 32.768kHz 크리스탈 장착
  3. CubeMX에서 RCC → LSE를 "Crystal/Ceramic Resonator"로 변경

오늘 실습에서는 LSI를 사용합니다. 추가 하드웨어가 필요 없고, 교육 목적으로는 충분합니다. 하루 72분 오차가 있지만, 실습 시간(4시간) 동안에는 약 12분 오차 — 동작 확인에 지장 없습니다.

CubeMX 설정 순서

  1. RCC 설정: Pinout & Configuration → System Core → RCC → Low Speed Clock = "LSI" 선택
  2. 클럭 소스 확인: Device Configuration → RTC → Clock Source가 "LSI" 또는 "LSE"인지 확인 (의도한 소스와 일치해야 합니다)
  3. RTC 활성화: Timers → RTC → "Activate Clock Source" 체크, "Activate Calendar" 체크
  4. Alarm 활성화: RTC → Alarm A → "Internal Alarm A" 선택
  5. NVIC 설정: NVIC → "RTC Alarm interrupt through EXTI line 17" 체크
  6. 초기 시간: RTC → Calendar → Hours/Minutes/Seconds 설정
❓ 수강생 단골 질문
Q. LSI로 시계를 만들면 하루에 얼마나 틀려요?
A. LSI는 ±5% 편차입니다. 최악의 경우 32kHz × 5% = 1,600Hz 오차. 하루 86,400초 × 5% = 4,320초 = 약 72분. 실습에서는 "시간이 빠르게/느리게 흘러간다"는 느낌이 들 수 있지만, 동작 자체에는 문제없습니다. 제품에서는 반드시 LSE를 사용합니다.

§9.3 시간 읽기와 쓰기 — rtc_driver 설계

CubeMX에서 RTC를 활성화했다면, 이제 시간을 읽고 써봅시다. 그 전에 중요한 경고가 하나 있습니다.

HAL_RTC_GetTime() 호출 후 반드시 HAL_RTC_GetDate()를 호출해야 합니다!

이것은 RM0383 Reference Manual에 명시된 동작입니다. RTC의 Shadow Register(그림자 레지스터)는 GetTime을 호출하면 잠금(lock)되어 값이 고정됩니다. GetDate를 호출해야 잠금이 해제되어 다음 읽기에서 새로운 값이 갱신됩니다. GetDate를 빠뜨리면 시간이 멈춘 것처럼 보입니다 — 이것은 100% 모든 수강생이 처음에 빠지는 함정입니다. 일부러 경험해 보세요!

기본 시간 읽기

/* ch09_02_rtc_get_time.c — 시간 읽기 기본 패턴 */

void rtc_read_time(void)
{
    RTC_TimeTypeDef time;
    RTC_DateTypeDef date;

    /* ⚠️ 반드시 GetTime → GetDate 순서! 쌍으로 호출! */
    HAL_RTC_GetTime(&hrtc, &time, RTC_FORMAT_BIN);
    HAL_RTC_GetDate(&hrtc, &date, RTC_FORMAT_BIN);  /* 생략 금지! */

    LOG_I("현재 시각: %02d:%02d:%02d", time.Hours, time.Minutes, time.Seconds);
    LOG_D("현재 날짜: 20%02d-%02d-%02d", date.Year, date.Month, date.Date);
}

시간 설정

/* ch09_03_rtc_set_time.c — 시간 설정 */

void rtc_set_time_example(void)
{
    RTC_TimeTypeDef time = {0};
    time.Hours   = 14;
    time.Minutes = 30;
    time.Seconds = 0;
    time.DayLightSaving = RTC_DAYLIGHTSAVING_NONE;
    time.StoreOperation = RTC_STOREOPERATION_RESET;

    if (HAL_RTC_SetTime(&hrtc, &time, RTC_FORMAT_BIN) != HAL_OK) {
        LOG_E("RTC 시간 설정 실패!");
        return;
    }

    RTC_DateTypeDef date = {0};
    date.Year    = 26;   /* 2026년 */
    date.Month   = RTC_MONTH_MARCH;
    date.Date    = 17;
    date.WeekDay = RTC_WEEKDAY_TUESDAY;

    if (HAL_RTC_SetDate(&hrtc, &date, RTC_FORMAT_BIN) != HAL_OK) {
        LOG_E("RTC 날짜 설정 실패!");
        return;
    }

    LOG_I("RTC 설정: 14:30:00, 2026-03-17");
}

rtc_driver 설계 — tim_driver 패턴 재활용

Ch07에서 tim_driver를 설계할 때 init → start → stop → set 패턴을 사용했습니다. rtc_driver도 같은 패턴입니다: init → get/set → alarm → wakeup. Driver는 하드웨어 추상화만 담당하고, 비즈니스 로직(알람 FSM)은 Service 레이어에 둡니다. 

/* rtc_driver.h — 공개 인터페이스 (Driver 레이어) */

typedef enum { RTC_OK = 0, RTC_ERROR = 1 } rtc_status_t;

typedef struct {
    uint8_t hours, minutes, seconds;
} rtc_time_t;

typedef struct {
    uint8_t year, month, date, weekday;
} rtc_date_t;

typedef void (*rtc_alarm_callback_t)(void);

/* 초기화 */
rtc_status_t rtc_driver_init(void);

/* 시간/날짜 */
rtc_status_t rtc_get_time(rtc_time_t *time);
rtc_status_t rtc_set_time(const rtc_time_t *time);

/* 알람 — 상위 레이어(alarm_service)에서 사용 */
rtc_status_t rtc_set_alarm_a(const rtc_time_t *time, uint32_t mask);
rtc_status_t rtc_deactivate_alarm_a(void);
void         rtc_register_alarm_callback(rtc_alarm_callback_t cb);

rtc_driver의 구현은 ch09_06_rtc_driver.c에 있습니다. 핵심은 HAL 콜백을 Driver 콜백으로 전달하는 패턴입니다: 

/* rtc_driver.c — HAL 콜백 → Driver 콜백 전달 */
static rtc_alarm_callback_t s_alarm_cb = NULL;

void rtc_register_alarm_callback(rtc_alarm_callback_t cb)
{
    s_alarm_cb = cb;
    LOG_D("알람 콜백 등록");
}

/* HAL이 호출하는 콜백 → 등록된 Service 콜백으로 전달 */
void HAL_RTC_AlarmAEventCallback(RTC_HandleTypeDef *hrtc_cb)
{
    if (s_alarm_cb != NULL) {
        s_alarm_cb();   /* alarm_service의 콜백 호출 */
    }
}
💡 실무 팁
HAL_RTC_GetTime()에서 RTC_FORMAT_BIN을 사용하면 HAL이 BCD → Binary 변환을 자동으로 해줍니다. RTC_FORMAT_BCD를 사용하면 원본 BCD 값을 얻습니다. 디버깅 시 레지스터 뷰에서 BCD 값이 보이므로, 두 형식의 차이를 이해해두면 좋습니다.
📌 강사 꿀팁
BCD → Binary 변환은 칠판에서 1~2회만 연습하고 빠르게 넘어가세요. HAL이 자동 변환해주므로 15분 이상 투자하면 시간이 부족합니다. "디버거에서 레지스터 읽을 때만 BCD를 알면 됩니다"로 마무리.
❓ 수강생 단골 질문
Q. RTC 시간이 자꾸 틀려요. 보정할 수 있나요?
A. LSI를 사용하면 ±5% 편차는 정상입니다. 해결 방법 3가지: ① LSE 크리스탈로 교체(근본 해결) ② STM32의 Smooth Calibration 레지스터 사용(CALR, ±488ppm 보정) ③ 외부 시간 소스(NTP, GPS)와 주기적 동기화. 우리 실습에서는 LSI를 사용하되 "정확도 한계를 이해하는 것"이 목표입니다.

§9.4 RTC 알람 — AlarmA/AlarmB 인터럽트

시간을 읽고 쓸 수 있게 되었으니, 이제 특정 시각에 이벤트를 발생시킵시다. 이것이 RTC 알람입니다. 스마트폰의 알람 앱과 정확히 같은 원리입니다 — "7시 30분에 울려!"

STM32F411의 RTC는 AlarmA와 AlarmB, 두 개의 독립 알람을 제공합니다. 각각 시:분:초와 요일/날짜를 비교하여 매칭 시 인터럽트를 발생시킵니다.

알람 마스크 — "스마트폰 반복 설정"

비유로 이해하기
스마트폰에서 알람을 설정할 때 "매일 반복"을 선택하면 날짜를 무시합니다. RTC 알람의 마스크도 같은 원리입니다. 마스크를 켜면 "이 필드는 무시(=항상 매칭)"가 됩니다. 마스크를 많이 켤수록 알람이 더 자주 울립니다.
※ 이 비유의 한계: 스마트폰은 "반복 요일"을 선택하지만, RTC 마스크는 시/분/초/요일 각각을 독립적으로 마스킹합니다. 더 유연하지만 더 복잡합니다.
마스크 설정동작트리거 빈도
모두 OFF정확히 해당 시:분:초에만하루 1회
날짜/요일 마스크 ON매일 해당 시:분:초에하루 1회 (매일 반복)
시+날짜 마스크 ON매시간 해당 분:초에하루 24회
시+분+날짜 마스크 ON매분 해당 초에하루 1,440회
모두 ON매초마다하루 86,400회

AlarmA 설정 코드

/* ch09_04_rtc_alarm_setup.c — AlarmA 설정 */

void rtc_alarm_set_daily(void)
{
    RTC_AlarmTypeDef alarm = {0};

    alarm.AlarmTime.Hours   = 7;
    alarm.AlarmTime.Minutes = 30;
    alarm.AlarmTime.Seconds = 0;

    /* 마스크: 날짜/요일만 마스크 → 매일 07:30:00에 트리거 */
    alarm.AlarmMask = RTC_ALARMMASK_DATEWEEKDAY;

    alarm.AlarmDateWeekDaySel = RTC_ALARMDATEWEEKDAYSEL_DATE;
    alarm.AlarmDateWeekDay    = 1;   /* 마스크되어 무시됨 */
    alarm.Alarm = RTC_ALARM_A;

    if (HAL_RTC_SetAlarm_IT(&hrtc, &alarm, RTC_FORMAT_BIN) != HAL_OK) {
        LOG_E("AlarmA 설정 실패!");
        return;
    }

    LOG_I("AlarmA 설정: 매일 07:30:00");
}

ISR → 플래그 → main loop (Ch08 패턴 동일!)

/* 알람 콜백 — ISR에서는 플래그만! (학습용 단독 예제) */
/* ⚠️ 최종 프로젝트에서는 rtc_driver(§9.6)의 콜백을 사용하세요.
 *    HAL 콜백은 프로젝트당 하나만 정의 가능합니다.
 *    ch09_06(rtc_driver) + ch09_07(alarm_service)만 포함하세요. */
static volatile uint8_t alarm_flag = 0;

void HAL_RTC_AlarmAEventCallback(RTC_HandleTypeDef *hrtc_cb)
{
    alarm_flag = 1;   /* ISR에서는 이것이 전부! */
}

/* main loop에서 처리 */
void alarm_process(void)
{
    if (alarm_flag) {
        alarm_flag = 0;
        LOG_I("*** AlarmA 트리거! ***");
        HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);  /* LED 토글 */
    }
}

Ch08에서 배운 패턴 그대로입니다: EXTI 콜백 → 플래그 → main loop가 RTC 콜백 → 플래그 → main loop로 바뀐 것뿐입니다. 이벤트 소스만 바뀌었을 뿐, 처리 구조는 동일합니다.

❓ 수강생 단골 질문
Q. AlarmA와 AlarmB는 뭐가 다른가요?
A. 기능적으로 거의 동일합니다. 둘 다 독립적으로 시:분:초 매칭 + 인터럽트를 지원합니다. 차이는 별도의 ISR을 가진다는 것입니다. 우리 프로젝트에서는 AlarmA를 사용자 알람(alarm_service)으로, AlarmB는 Ch11에서 온도 알람으로 활용할 예정입니다.
🎯 면접 포인트
Q. "RTC 알람의 마스크 비트 역할을 설명하시오."
A. 마스크 비트는 알람 비교 시 특정 필드를 무시하는 설정입니다. 마스크 ON = "이 필드는 항상 매칭" → 알람이 더 자주 트리거됩니다. 예: 시 마스크 ON + 분=30, 초=00 → "매시간 30분 정각에 트리거". 마스크를 조합하여 매일/매시/매분 반복 알람을 구현합니다.

§9.5 Wake-up 타이머와 백업 레지스터

알람은 "몇 시에" 트리거됩니다. 그런데 "몇 초마다" 주기적으로 트리거가 필요할 때는 어떻게 할까요? TIM 인터럽트를 쓸 수도 있지만, RTC에는 전용 기능이 있습니다 — Wake-up Timer.

비유로 이해하기
알람 시계는 "오전 7시에 깨워줘" — 특정 시각. 주방 타이머는 "5분 후에 알려줘" — 일정 주기. Alarm은 알람 시계, Wake-up Timer는 주방 타이머입니다.
※ 이 비유의 한계: Wake-up Timer는 한 번만 울리는 주방 타이머와 달리, 자동으로 반복 트리거됩니다. "5분마다 계속 울리는 반복 타이머"에 더 가깝습니다.

Wake-up Timer vs TIM 인터럽트

"둘 다 주기적 인터럽트인데 뭐가 다른가요?" 핵심 차이는 전력입니다.

항목TIM 인터럽트 (Ch07)Wake-up Timer
클럭 소스APB (84MHz)RTC (32kHz)
Stop 모드동작 불가 (APB 정지)동작 가능!
정밀도μs 단위 가능ms~초 단위
소비 전류높음 (APB 클럭 필요)매우 낮음 (~1μA)
사용처PWM, 정밀 타이밍저전력 주기 깨우기

Wake-up Timer 설정

/* ch09_05_wakeup_timer.c — 5초 주기 Wake-up */

void rtc_wakeup_start(uint16_t seconds)
{
    /* 입력 검증: 0이면 언더플로우 (0 - 1 = 0xFFFF) */
    if (seconds == 0) {
        LOG_W("Wake-up 주기 0초 — 무시");
        return;
    }

    /* 기존 타이머 해제 */
    HAL_RTCEx_DeactivateWakeUpTimer(&hrtc);

    /* 1Hz(ck_spre) 기반, seconds초 주기 */
    if (HAL_RTCEx_SetWakeUpTimer_IT(
            &hrtc,
            (uint16_t)(seconds - 1),            /* 카운트다운 값 (안전) */
            RTC_WAKEUPCLOCK_CK_SPRE_16BITS     /* 1초 단위 */
        ) != HAL_OK) {
        LOG_E("Wake-up 타이머 설정 실패!");
        return;
    }

    LOG_I("Wake-up 타이머: %d초 주기", seconds);
}

/* 콜백 — ISR에서는 카운트만! */
void HAL_RTCEx_WakeUpTimerEventCallback(RTC_HandleTypeDef *hrtc_cb)
{
    wakeup_count++;
}

백업 레지스터 — "냉장고 메모지"

백업 도메인에는 20개의 32비트 백업 레지스터(BKP_DR0~19)가 있습니다. VBAT가 유지되는 한 전원을 꺼도, 리셋해도 값이 보존됩니다. 냉장고 메모지처럼 "전원이 꺼져도 붙어있는 메모"입니다. 

/* 백업 레지스터 활용: 리셋 횟수 카운터 */

#define RTC_MAGIC_NUMBER  0xA5A5

void rtc_init_with_backup(void)
{
    uint32_t magic = HAL_RTCEx_BKUPRead(&hrtc, RTC_BKP_DR0);

    if (magic == RTC_MAGIC_NUMBER) {
        LOG_I("RTC 이미 설정됨 — 시간 유지");
    } else {
        LOG_W("RTC 최초 설정 — 시간 초기화");
        rtc_set_time_example();

        /* 매직 넘버 기록 → 다음 리셋 시 재설정 건너뛰기 */
        HAL_RTCEx_BKUPWrite(&hrtc, RTC_BKP_DR0, RTC_MAGIC_NUMBER);
    }

    /* 리셋 카운터 */
    uint32_t count = HAL_RTCEx_BKUPRead(&hrtc, RTC_BKP_DR1);
    count++;
    HAL_RTCEx_BKUPWrite(&hrtc, RTC_BKP_DR1, count);
    LOG_I("리셋 횟수: %lu", count);
}
❓ 수강생 단골 질문
Q. Wake-up Timer와 AlarmA 중 언제 뭘 쓰나요?
A. Alarm: "매일 7시 알람", "매시 정각 로그" → 특정 시각 기반. Wake-up: "10초마다 센서 읽기", "1분마다 데이터 전송" → 주기 기반. 우리 프로젝트에서는 Alarm으로 사용자 알람을, Wake-up으로 주기적 센서 체크를 구현합니다.
💡 실무 팁
백업 레지스터 실무 활용: ① 시스템 리셋 원인 기록 ② 부팅 카운터 ③ 마지막 에러 코드 저장 ④ DFU(Firmware Update) 부트로더 진입 플래그. 20개 × 32비트 = 80바이트로 용량이 적으니, 꼭 필요한 정보만 저장하세요.
❓ 수강생 단골 질문
Q. Wake-up Timer는 TIM 인터럽트와 뭐가 다른가요?
A. 기능은 비슷합니다. 차이는 전력입니다. TIM은 84MHz APB 클럭이 필요해서 Stop 모드에서 동작하지 않습니다. Wake-up Timer는 32kHz RTC 클럭만으로 동작하므로 Stop 모드에서도 MCU를 깨울 수 있습니다. 배터리 IoT 디바이스에서 "30분마다 깨워서 센서 읽기 → 전송 → 다시 Sleep" 패턴의 핵심이 RTC Wake-up입니다.

§9.6 Alarm FSM 설계 — "Ch08 패턴을 다시 한번"

이 챕터의 핵심 성취 구간에 왔습니다. Ch08에서 Stopwatch FSM을 설계하고 구현했습니다. 이제 동일한 패턴으로 Alarm FSM을 설계합니다. "같은 패턴, 다른 도메인" — FSM의 강점을 체감하는 순간입니다.

Ch08 Stopwatch FSM 복습 (2문장)

Stopwatch FSM: IDLE → RUNNING → PAUSED → IDLE. 3개 상태, 이벤트 소스는 버튼(BTN_SHORT, BTN_LONG). ISR에서 플래그 세팅 → main loop에서 stopwatch_handle_event() 호출 → switch-case로 전이 처리.

Alarm FSM: 상태 정의

핸드폰 알람 앱의 동작을 관찰해 봅시다: 알람 설정 → 대기 → 시간 되면 울림 → 사용자가 끔. 4개 상태가 필요합니다:

상태enum 값설명LED 상태
IDLEALARM_STATE_IDLE알람 미설정OFF
ARMEDALARM_STATE_ARMED알람 대기 중OFF
TRIGGEREDALARM_STATE_TRIGGERED알람 울리는 중!ON (점멸)
ACKNOWLEDGEDALARM_STATE_ACKNOWLEDGED사용자 확인 완료OFF

Stopwatch는 3상태, Alarm은 4상태입니다. ACKNOWLEDGED(확인 완료)가 추가되었습니다. "왜 TRIGGERED에서 바로 IDLE로 안 가나요?" — 알람이 울린 사실을 기록하고, 다음 동작(재설정 또는 자동 복귀)을 결정하기 위해서입니다.

❓ 수강생 단골 질문
Q. 왜 ACKNOWLEDGED 상태가 따로 필요한가요? TRIGGERED에서 바로 IDLE로 가면 안 되나요?
A. 두 가지 이유입니다. ① 알람이 울린 사실을 기록합니다(마지막 알람 시각 저장, 로그 기록). ② 사용자에게 "다시 설정할 건지, 끝낼 건지" 선택권을 줍니다. 스마트폰에서도 알람을 끄면 "내일 같은 시각?" 같은 옵션이 나오죠. 그 상태가 ACKNOWLEDGED입니다.

§9.7 Alarm FSM 구현 — "패턴 반복, 자신감 확보"

설계가 완료되었으니 코드로 옮깁시다. Ch08 stopwatch_service.c를 열고, 구조를 복사한 다음, 상태와 이벤트 이름만 바꾸면 됩니다. 이것이 FSM 패턴의 핵심 — 한 번 익히면 어떤 도메인에든 적용 가능합니다.

alarm_service.h — 인터페이스

/* ch09_07_alarm_service.h — Service 레이어 공개 API */

typedef enum {
    ALARM_STATE_IDLE         = 0,   /* 알람 미설정 */
    ALARM_STATE_ARMED        = 1,   /* 알람 대기 중 */
    ALARM_STATE_TRIGGERED    = 2,   /* 알람 울리는 중! */
    ALARM_STATE_ACKNOWLEDGED = 3    /* 사용자 확인 완료 */
} alarm_state_t;

typedef enum {
    ALARM_EVT_SET_ALARM   = 0,    /* 알람 설정 요청 */
    ALARM_EVT_CANCEL      = 1,    /* 알람 취소 */
    ALARM_EVT_ALARM_MATCH = 2,    /* RTC 시각 매칭 (ISR) */
    ALARM_EVT_ACK         = 3,    /* 사용자 확인 (버튼) */
    ALARM_EVT_TIMEOUT     = 4     /* 자동 타임아웃 */
} alarm_event_t;

alarm_state_t alarm_service_get_state(void);

stopwatch_service.h와 비교해 보세요. sw_state_t → alarm_state_t, sw_event_t → alarm_event_t. 함수 이름도 stopwatch_* → alarm_service_*. 구조가 완전히 동일합니다.

alarm_service.c — FSM 구현 (핵심부)

/* ch09_07_alarm_service.c — Alarm FSM 핵심 */

#include "alarm_service.h"
#include "rtc_driver.h"
#include "log.h"

static volatile alarm_state_t s_state = ALARM_STATE_IDLE;
static volatile uint8_t s_alarm_flag  = 0;
static rtc_time_t s_alarm_time = {0};
static uint32_t s_trigger_tick = 0;

#define ALARM_ACK_TIMEOUT_MS  30000   /* 30초 후 자동 타임아웃 (TRIGGERED → ACKNOWLEDGED) */
#define ALARM_IDLE_TIMEOUT_MS 2000    /* 2초 후 ACKNOWLEDGED → IDLE 자동 복귀 */

static const char * const STATE_NAMES[] = {
    "IDLE", "ARMED", "TRIGGERED", "ACKNOWLEDGED"
};

/* ISR 콜백: 플래그만 설정 (Ch08 패턴 동일!) */
static void alarm_isr_callback(void)
{
    s_alarm_flag = 1;
}

void alarm_service_init(void)
{
    s_state = ALARM_STATE_IDLE;
    s_alarm_flag = 0;
    rtc_register_alarm_callback(alarm_isr_callback);
    LOG_I("alarm_service 초기화 완료");
}

void alarm_service_handle_event(alarm_event_t evt)
{
    alarm_state_t prev = s_state;

    switch (s_state) {

    case ALARM_STATE_IDLE:
        if (evt == ALARM_EVT_SET_ALARM) {
            rtc_set_alarm_a(&s_alarm_time, RTC_ALARMMASK_DATEWEEKDAY);
            s_state = ALARM_STATE_ARMED;
            LOG_I("알람 설정: %02d:%02d:%02d",
                  s_alarm_time.hours, s_alarm_time.minutes, s_alarm_time.seconds);
        }
        break;

    case ALARM_STATE_ARMED:
        if (evt == ALARM_EVT_ALARM_MATCH) {
            s_trigger_tick = HAL_GetTick();
            HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET);
            s_state = ALARM_STATE_TRIGGERED;
            LOG_I("*** 알람 울림! ***");
        } else if (evt == ALARM_EVT_CANCEL) {
            rtc_deactivate_alarm_a();
            s_state = ALARM_STATE_IDLE;
            LOG_I("알람 취소");
        }
        break;

    case ALARM_STATE_TRIGGERED:
        if (evt == ALARM_EVT_ACK || evt == ALARM_EVT_TIMEOUT) {
            HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET);
            s_state = ALARM_STATE_ACKNOWLEDGED;
            LOG_I("알람 확인 완료");
        }
        break;

    case ALARM_STATE_ACKNOWLEDGED:
        if (evt == ALARM_EVT_SET_ALARM) {
            rtc_set_alarm_a(&s_alarm_time, RTC_ALARMMASK_DATEWEEKDAY);
            s_state = ALARM_STATE_ARMED;
        } else if (evt == ALARM_EVT_TIMEOUT) {
            s_state = ALARM_STATE_IDLE;
            LOG_D("자동 복귀: ACKNOWLEDGED → IDLE");
        }
        break;
    }

    if (prev != s_state) {
        LOG_D("FSM: %s → %s", STATE_NAMES[prev], STATE_NAMES[s_state]);
    }
}

/* main loop에서 매 사이클 호출 */
void alarm_service_process(void)
{
    /* ISR 플래그 → FSM 이벤트 */
    if (s_alarm_flag) {
        s_alarm_flag = 0;
        alarm_service_handle_event(ALARM_EVT_ALARM_MATCH);
    }

    /* TRIGGERED 타임아웃 체크 */
    if (s_state == ALARM_STATE_TRIGGERED) {
        if ((HAL_GetTick() - s_trigger_tick) > ALARM_ACK_TIMEOUT_MS) {
            alarm_service_handle_event(ALARM_EVT_TIMEOUT);
        }
    }
}

🎯 면접 포인트
Q. "FSM 패턴을 다른 서비스에 적용한 경험을 설명하시오."
A. Stopwatch FSM(IDLE/RUNNING/PAUSED)과 Alarm FSM(IDLE/ARMED/TRIGGERED/ACKNOWLEDGED)을 동일한 switch-case 패턴으로 구현했습니다. 상태·이벤트·전이 테이블만 변경하고 구조는 동일합니다. FSM 패턴의 장점: ① 상태를 명시적으로 관리하여 버그 감소 ② 새 상태/이벤트 추가가 용이 ③ 상태 전이표로 설계 검증 가능.

실습: RTC 시각 표시 + Alarm FSM

목표: RTC로 시각을 읽고, Alarm FSM으로 알람을 설정·트리거·확인하는 전체 과정을 구현합니다.

환경: STM32CubeIDE, NUCLEO-F411RE, UART 시리얼 모니터

Step 1. CubeMX 설정

  1. RCC → Low Speed Clock: LSI
  2. RTC → Activate Clock Source: 체크
  3. RTC → Activate Calendar: 체크
  4. RTC → Alarm A → Internal Alarm A: 체크
  5. NVIC → RTC Alarm interrupt through EXTI line 17: Enable
  6. RTC → Calendar → 초기 시간 설정 (예: 14:30:00)

Step 2. 코드 작성

  1. rtc_driver.h/c 파일 추가 (ch09_06 참고)
  2. alarm_service.h/c 파일 추가 (ch09_07 참고)
  3. main.c에 초기화 + 슈퍼루프 통합 (§9.8 참고)

Step 3. 동작 확인

예상 결과:

  1. 시리얼 모니터에 1초마다 현재 시각 출력
  2. 10초 후 "*** 알람 울림! ***" + LED ON
  3. 버튼 B1 누르면 "알람 확인 완료" + LED OFF

로그 출력 예시:

[INFO ] v0.9 시스템 시작 — RTC + Alarm + Stopwatch
[INFO ] rtc_driver 초기화
[INFO ] alarm_service 초기화 완료
[INFO ] AlarmA 설정: 14:30:10 (마스크=0x80)
[DEBUG] FSM: IDLE → ARMED

[DEBUG] FSM: ARMED → TRIGGERED

💡 실습이 안 될 때
시간이 안 변해요: HAL_RTC_GetDate() 호출을 빠뜨렸는지 확인하세요. GetTime 후 반드시 GetDate를 호출해야 합니다. 알람이 안 울려요: NVIC에서 RTC Alarm 인터럽트를 활성화했는지 확인하세요. CubeMX에서 EXTI line 17을 Enable해야 합니다. LSI 오차가 심해요: 정상입니다. LSI는 ±5% 편차가 있습니다. 10초 후 알람이 9초 또는 11초에 울릴 수 있습니다.

핵심 정리

  • RTC는 MCU 내부의 독립 시계 — 백업 도메인으로 전원 차단에도 시각 유지(VBAT 필요).
  • BCD 레지스터로 시:분:초를 "전광판처럼" 자릿수별 저장 — HAL이 자동 변환하지만 디버거에서 BCD를 읽을 줄 알아야 한다.
  • LSE(정밀, 외부 크리스탈) vs LSI(편리, 내장, ±5% 오차) — 용도에 따라 선택.
  • Alarm(특정 시각) vs Wake-up Timer(주기적) — 마스크 비트로 반복 패턴 설정.
  • Alarm FSM(IDLE/ARMED/TRIGGERED/ACKNOWLEDGED) — Ch08 Stopwatch FSM과 동일 패턴, 이벤트 소스만 다르다.
  • GetTime + GetDate 쌍 호출 필수 — Shadow Register 잠금 해제를 위해 빠뜨리지 말 것.
  • rtc_driver(Driver) + alarm_service(Service) — tim_driver/stopwatch_service와 동일한 레이어드 패턴.
