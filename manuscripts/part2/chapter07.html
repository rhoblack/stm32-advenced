Ch07. TIM 기초 (General Purpose Timer)

시간을 정밀하게 다스리다 — 타이머 인터럽트로 정확한 주기를 만들고, PWM으로 밝기와 속도를 제어한다

누적 기능: 정밀 주기 타이머 + PWM 출력 (tim_driver 추가)

§7.0 "시간을 제어하는 자, 임베디드를 지배한다"

여기까지 온 여러분의 성과를 확인합시다. Ch01에서 GPIO/EXTI, Ch02에서 로그, Ch03에서 4계층 아키텍처, Ch04~Ch06에서 UART와 DMA 링 버퍼까지 완성했습니다. 6개 챕터, Part 0~1 완주. 이제 Part 2의 첫 문을 엽니다.

지금까지 여러분의 코드에서 "시간"을 다루는 방법은 하나뿐이었습니다. HAL_Delay(500) — 500밀리초 동안 CPU를 멈추고 기다리는 것. Ch01에서 LED를 깜빡일 때 이 함수를 사용했죠. 당시에는 충분했지만, 다시 생각해 봅시다.

HAL_Delay()는 내부적으로 SysTick 카운터를 폴링합니다. 500ms 동안 CPU는 while 루프 안에서 카운터 값을 확인하며 아무것도 하지 않습니다. UART 수신도 못 하고, 센서 데이터도 못 읽고, 화면 갱신도 못 합니다. Ch04~06에서 어렵게 DMA로 CPU를 해방했는데, HAL_Delay() 한 줄이면 다시 감옥입니다.

이 챕터에서 우리는 타이머(Timer)를 배웁니다. 타이머는 CPU와 독립적으로 동작하는 하드웨어 카운터입니다. "500ms 뒤에 알려줘"라고 설정하면, CPU는 다른 일을 하다가 정확히 500ms 후에 인터럽트를 받습니다. 게다가 PWM(Pulse Width Modulation, 펄스 폭 변조)이라는 기법으로 LED의 밝기, 모터의 속도, 부저의 음높이까지 제어할 수 있습니다.

이 챕터가 끝나면:

  • HAL_Delay()를 쓰지 않고 정확한 500ms 주기로 LED를 토글합니다.
  • PWM으로 LED 밝기를 25%, 50%, 75%, 100%로 부드럽게 조절합니다.
  • Ch04의 uart_driver와 동일한 설계 패턴으로 tim_driver.h/c를 완성합니다.
📌 강사 꿀팁
도입부에서 HAL_Delay(500)을 넣은 main 루프를 보여주고, "이 500ms 동안 UART 수신이 되나요?"라고 질문하세요. 학생들이 "안 되죠…"라고 답하면, "오늘 이 문제를 해결합니다"라고 자연스럽게 연결됩니다.

§7.1 타이머란 무엇인가 — 메트로놈의 원리

피아노를 배울 때 메트로놈이라는 도구를 씁니다. "딱, 딱, 딱" — 일정한 간격으로 소리를 내서 박자를 맞춰주는 장치입니다. 메트로놈의 속도를 빠르게 하면 간격이 짧아지고, 느리게 하면 간격이 길어집니다. STM32의 타이머(Timer)가 바로 이 메트로놈입니다.

비유로 이해하기
메트로놈을 떠올리세요. 메트로놈에는 추가 달린 막대가 있고, 추의 위치를 조절하면 속도가 바뀝니다. 타이머에서는 PSC(분주기)가 추의 위치, ARR(자동 재적재)이 한 주기의 끝 지점입니다. 메트로놈이 "딱!" 소리를 낼 때마다 인터럽트가 발생합니다 — 이것이 UEV(Update Event)입니다.
※ 이 비유의 한계: 실제 메트로놈은 소리만 내지만, STM32 타이머는 PWM 출력, 캡처, 비교 등 다양한 기능을 수행합니다.

타이머의 본질: 하드웨어 카운터

타이머의 정체는 간단합니다. 클럭 신호에 맞춰 1씩 증가하는 카운터입니다. 0부터 시작해서 설정한 상한값(ARR)에 도달하면, 0으로 돌아가고 "도달했다!"는 이벤트(UEV)를 발생시킵니다. 이 동작이 하드웨어로 구현되어 있으므로, CPU가 개입할 필요가 없습니다.

이것이 HAL_Delay()와의 결정적 차이입니다. HAL_Delay()는 CPU가 직접 카운터를 확인하며 기다리지만, 타이머는 하드웨어가 알아서 세고, 끝나면 인터럽트로 알려줍니다.

STM32F411RE 타이머 계층

STM32 타이머는 기능에 따라 세 가지 등급으로 나뉩니다.

STM32F411RE에는 Basic 타이머(TIM6/TIM7)가 없습니다. F4 상위 모델(F407, F429)에는 있지만, F411은 저전력 라인이라 생략되었습니다. "Basic 타이머로 DAC 트리거를 하고 싶다"면 F411에서는 GP 타이머(TIM2~5)를 대신 사용해야 합니다. 데이터시트의 타이머 테이블을 반드시 확인하는 습관을 들이세요.

이 챕터의 주인공: TIM2

이 챕터에서는 TIM2를 사용합니다. 선택 이유:

  • 32비트 카운터: 최대 4,294,967,295까지 셀 수 있어 매우 긴 주기 설정 가능
  • PWM 4채널 지원: CH1~CH4까지 독립적인 PWM 출력 가능
  • NUCLEO 핀 배치: PA0(TIM2_CH1)이 Arduino 커넥터 A0에 연결되어 접근 편리

타이머 클럭 — "메트로놈의 전원"

타이머가 카운트하려면 클럭 신호가 필요합니다. TIM2~TIM5는 APB1 버스에 연결되어 있습니다. NUCLEO-F411RE의 기본 설정에서 APB1 타이머 클럭은 84MHz입니다.

84MHz — 1초에 8천4백만 번 카운트. 이대로 사용하면 너무 빠릅니다. 카운터가 ARR에 도달하는 시간이 마이크로초 단위가 되어버립니다. 그래서 PSC(Prescaler, 분주기)가 필요합니다 — 다음 절에서 상세히 다룹니다.

❓ 수강생 단골 질문
Q. SysTick도 타이머인데, TIM2와 뭐가 다른가요?
A. SysTick은 ARM Cortex-M 코어에 내장된 24비트 다운 카운터로, OS 틱이나 HAL_Delay()에 사용됩니다. 기능이 매우 제한적입니다 — 카운트와 인터럽트만 가능합니다. 반면 TIM2는 STM32 주변장치로, PWM 출력, 캡처, 비교, DMA 트리거 등 다양한 기능을 가집니다. SysTick은 "시스템 시계", TIM2는 "만능 스톱워치"라고 생각하세요.
🎯 면접 포인트
Q. "STM32의 타이머 종류와 차이를 설명하라"
A. Basic(카운터만), General Purpose(PWM+캡처+비교), Advanced(Complementary+Dead-time) 3단계. F411에는 Basic이 없고 GP(TIM2~5) + Advanced(TIM1)만 있다. SysTick은 Cortex-M 코어 내장 타이머로 별도 구분해야 한다.

§7.2 PSC · ARR · CCR — 주기를 만드는 세 개의 톱니바퀴

84MHz 클럭을 그대로 쓰면 카운터가 너무 빨리 돕니다. 우리가 원하는 것은 "500ms마다 한 번" 같은 인간 시간 단위의 이벤트입니다. 이것을 만드는 핵심 레지스터가 세 개 있습니다: PSC(Prescaler, 분주기), ARR(Auto-Reload Register, 자동 재적재), 그리고 CCR(Capture/Compare Register, 캡처 비교).

비유로 이해하기
자전거의 기어 시스템을 떠올리세요. 페달(클럭)을 같은 속도로 밟아도, 기어비를 바꾸면 바퀴(카운터)의 회전 속도가 달라집니다. PSC는 큰 기어(앞 기어) — 페달 회전을 몇 분의 1로 줄입니다. ARR은 바퀴 둘레 — 바퀴가 한 바퀴 돌면(=카운터가 ARR에 도달하면) "1주기 완료!"입니다. CCR은 바퀴의 특정 지점에 붙인 표시 — 그 지점을 지날 때 PWM 출력이 바뀝니다.
※ 이 비유의 한계: 실제 자전거 기어는 연속적이지만, PSC/ARR은 정수 값만 가능합니다. 또한 CCR은 여러 개(CH1~CH4)를 동시에 사용할 수 있지만, 바퀴의 표시는 보통 하나입니다.

PSC (Prescaler) — 클럭을 느리게 하는 분주기

84MHz 클럭이 타이머에 들어오면, PSC가 먼저 속도를 줄입니다. 분주 공식: 

/* 카운터 클럭 주파수 */
f_cnt = f_clk / (PSC + 1)

왜 PSC + 1일까요? PSC = 0이면 분주하지 않습니다(÷1), PSC = 1이면 2분주(÷2)입니다. 레지스터 값이 0부터 시작하므로 "+1"이 붙습니다.

예를 들어 PSC = 8399이면: 

f_cnt = 84,000,000 / (8399 + 1) = 84,000,000 / 8400 = 10,000 Hz = 10kHz

84MHz가 10kHz로 줄었습니다. 이제 카운터는 1초에 10,000번 증가합니다 — 0.1ms(100us)마다 1씩.

ARR (Auto-Reload Register) — 카운터의 상한선

카운터는 0부터 시작해서 1씩 증가합니다. ARR 값에 도달하면 0으로 리셋되고, UEV(Update Event)가 발생합니다. 이 UEV가 바로 "한 주기 완료"를 알리는 신호입니다. 

/* 주기 (초) */
T = (ARR + 1) / f_cnt

/* 위의 두 공식을 합치면 — 마스터 공식 */
T = (PSC + 1) × (ARR + 1) / f_clk

static const tim_calc_entry_t calc_table[] = {
    { 8399,   9999, "1초 (1 Hz)"       },
    { 8399,   4999, "500ms (2 Hz)"     },
    { 8399,    999, "100ms (10 Hz)"    },
    {   83,   9999, "10ms (100 Hz)"    },
    {   83,    999, "1ms (1 kHz)"      },
};

void tim_period_calc_demo(void)
{
    const uint32_t f_clk = 84000000;  /* 84 MHz */

    for (uint32_t i = 0; i < count; i++) {
        uint64_t period_us = (uint64_t)(psc + 1) * (arr + 1)
                             * 1000000 / f_clk;
        LOG_I("[%lu] PSC=%5lu, ARR=%5lu → T=%lu us (%s)",
              i, psc, arr, (uint32_t)period_us,
              calc_table[i].desc);

}

CCR (Capture/Compare Register) — PWM의 핵심

CCR은 §7.4에서 본격적으로 다루지만, 여기서 맛보기를 합시다. CCR은 카운터가 특정 값에 도달했을 때 출력 핀의 상태를 바꾸는 기준점입니다. 카운터가 0에서 ARR까지 올라가는 동안:

  • CNT < CCR → 출력 High
  • CNT ≥ CCR → 출력 Low

이것이 PWM(Pulse Width Modulation)의 기본 원리입니다. CCR 값을 바꾸면 High 구간의 비율(=Duty Cycle)이 바뀝니다.

📌 강사 꿀팁
PSC/ARR 계산이 처음인 학생에게는 "역방향 계산법"을 가르치세요. 1단계: 원하는 주기(예: 500ms = 0.5s) 결정. 2단계: PSC로 "편한 카운터 속도" 만들기 (예: 10kHz → PSC = 8399). 3단계: ARR = 주기 × f_cnt - 1 (예: 0.5 × 10000 - 1 = 4999). 항상 PSC를 먼저 결정하고 ARR을 계산하는 순서로 안내하세요.
❓ 수강생 단골 질문
Q. PSC와 ARR을 어떻게 나눠야 하나요? 같은 주기인데 조합이 여러 개잖아요.
A. 원칙이 있습니다. PWM을 함께 사용한다면 ARR을 크게 잡으세요 — ARR이 클수록 Duty 해상도(=CCR 설정 단위)가 세밀해집니다. 단순 주기 인터럽트만이라면 PSC를 크게 잡아 카운터 속도를 늦추는 것이 에너지 효율적입니다. 실무에서는 대부분 PSC = 83 (1MHz 카운터)이나 PSC = 8399 (10kHz 카운터)을 표준으로 사용합니다.

§7.3 주기 타이머 인터럽트 — LED 깜빡이기의 정석

공식을 알았으니 실전입니다. §7.1에서 "HAL_Delay()는 감옥"이라고 했죠. 이제 타이머 인터럽트로 CPU를 해방하면서 정확한 500ms 주기로 LED를 토글합니다.

UEV (Update Event) — "띵!" 소리

카운터가 ARR에 도달하면 발생하는 이벤트를 UEV(Update Event)라고 합니다. 메트로놈의 "딱!" 소리와 같습니다. NVIC에서 TIM2 인터럽트를 활성화하면, 이 UEV가 ISR(인터럽트 서비스 루틴)을 호출합니다.

CubeMX 설정

CubeMX에서 TIM2를 설정하는 단계입니다.

  1. Timers → TIM2: Clock Source = "Internal Clock"
  2. Parameter Settings:
    • Prescaler (PSC) = 8399
    • Counter Period (ARR) = 4999
    • Counter Mode = Up
    • auto-reload preload = Enable
  3. NVIC Settings: "TIM2 global interrupt" → Enable
  4. GPIO: PA5는 이미 LED(LD2)로 설정되어 있음
💡 실무 팁
auto-reload preload를 Enable로 설정하세요. 이 옵션을 켜면 ARR 값 변경이 다음 UEV 시점에 적용됩니다. 끄면(Disable) 즉시 적용되어, 타이머 동작 중 ARR을 바꿀 때 예상치 못한 짧은/긴 주기가 한 번 발생할 수 있습니다.

코드 구현

/* ch07_02_tim_interrupt_led.c — TIM2 500ms 인터럽트 LED 토글 */

#include "main.h"
#include "log.h"

extern TIM_HandleTypeDef htim2;

/* TIM2 인터럽트 시작 */
void tim_interrupt_led_start(void)
{
    /* TIM2 인터럽트 모드 시작 */
    HAL_TIM_Base_Start_IT(&htim2);
    LOG_I("TIM2 인터럽트 LED 토글 시작 (500ms 주기)");
}

/* UEV 콜백 — HAL이 자동 호출 */
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance != TIM2) {
        return;  /* 다른 타이머는 무시 */
    }

    /* LED 토글 (PA5 = NUCLEO LD2) */
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
    LOG_D("TIM2 UEV: LED 토글");
}

{
    /* 타이머가 500ms마다 알아서 LED를 토글함 */

❓ 수강생 단골 질문
Q. HAL_TIM_Base_Start_IT()와 HAL_TIM_Base_Start()는 뭐가 다른가요?
A. _IT 접미사는 인터럽트 모드를 의미합니다. HAL_TIM_Base_Start()는 타이머를 시작하지만 UEV 인터럽트를 활성화하지 않습니다. 카운터만 돌아가고 콜백이 호출되지 않습니다. PWM만 사용하고 주기 콜백이 필요 없다면 _IT 없이 시작해도 됩니다.
❓ 수강생 단골 질문
Q. 콜백 안에서 LOG_D()를 호출해도 안전한가요? ISR 안이잖아요.
A. Ch06에서 만든 링 버퍼 기반 비동기 로그를 사용한다면 안전합니다. 링 버퍼에 문자열을 넣기만 하고, 실제 UART 전송은 메인 루프에서 DMA로 처리됩니다. 단, printf 직접 호출(블로킹)은 ISR에서 절대 사용하지 마세요 — UART 전송 중 인터럽트가 중첩되어 시스템이 멈출 수 있습니다.
🎯 면접 포인트
Q. "임베디드에서 HAL_Delay()를 사용하면 안 되는 이유를 설명하라"
A. HAL_Delay는 내부적으로 SysTick 폴링 방식의 블로킹 호출이다. 대기 시간 동안 CPU가 다른 작업을 수행할 수 없어 실시간 응답성이 저하된다. 타이머 인터럽트를 사용하면 CPU가 자유롭게 다른 작업을 수행하다가 정확한 시점에 콜백을 받을 수 있다.
💡 LED가 깜빡이지 않으면?
  1. CubeMX 설정 확인: TIM2 → NVIC → "TIM2 global interrupt" Enable 되어 있는지 확인하세요. 가장 흔한 실수입니다.
  2. 코드 확인: HAL_TIM_Base_Start_IT()에서 _IT 접미사를 빠뜨리지 않았나요? HAL_TIM_Base_Start()는 인터럽트를 활성화하지 않습니다.
  3. PSC/ARR 값 확인: LOG_D로 실제 설정된 PSC/ARR 값을 출력하여 계산 표와 대조하세요.
  4. 콜백 함수명: HAL_TIM_PeriodElapsedCallback — 오타가 있으면 HAL이 기본(빈) 구현을 호출합니다. 빌드 에러 없이 동작하지 않는 가장 찾기 어려운 버그입니다.

§7.4 PWM 기초 — 빛의 밝기를 숫자로 제어하다

타이머로 주기적 인터럽트를 만드는 것은 배웠습니다. 이제 타이머의 또 다른 강력한 기능, PWM(Pulse Width Modulation, 펄스 폭 변조)을 배웁니다. PWM을 사용하면 디지털 출력으로 아날로그적인 제어가 가능합니다 — LED의 밝기, 모터의 속도, 부저의 음높이를 숫자 하나로 바꿀 수 있습니다.

비유로 이해하기
수도꼭지를 생각하세요. 수도꼭지를 완전히 열면 물이 최대로 나옵니다(100%). 반만 열면 절반(50%). 아주 살짝 열면 물방울(10%). PWM도 마찬가지입니다 — 한 주기 동안 전압이 High인 비율을 Duty Cycle(듀티 사이클)이라고 합니다. Duty 100%면 항상 켜짐, 50%면 절반만 켜짐, 0%면 항상 꺼짐입니다. LED에 50% PWM을 걸면 사람 눈에는 절반 밝기로 보입니다 — 너무 빠른 깜빡임을 눈이 구분하지 못하기 때문입니다.
※ 이 비유의 한계: 수도꼭지는 실제로 유량을 연속적으로 조절하지만, PWM은 켜짐/꺼짐을 빠르게 반복하는 디지털 방식입니다. 결과적으로 "평균"은 아날로그처럼 보이지만 순간순간은 항상 0V 또는 3.3V입니다.

Edge-Aligned PWM 동작 원리

STM32의 기본 PWM 모드는 Edge-Aligned (엣지 정렬)입니다. 카운터가 0부터 ARR까지 올라가는 동안, CCR과 비교하여 출력을 결정합니다.

  • PWM Mode 1: CNT < CCR → High, CNT ≥ CCR → Low
  • PWM Mode 2: 반대 (CNT < CCR → Low)

Duty Cycle 계산

/* Duty 계산 공식 */
Duty(%) = CCR / (ARR + 1) × 100

/* 역산: 원하는 Duty에서 CCR 구하기 */
CCR = (ARR + 1) × Duty(%) / 100

예를 들어 ARR = 999 (주기 1000 카운트)이면:

DutyCCR 값High 시간
25%250250/10000 = 25us (1kHz 기준)
50%500500/10000 = 50us
75%750750/10000 = 75us
100%1000전체 주기 High

CubeMX 설정

  1. Timers → TIM2 → Channel1: "PWM Generation CH1" 선택
  2. Parameter Settings:
    • PSC = 83 (카운터 1MHz)
    • ARR = 999 (PWM 주파수 1kHz)
    • Pulse (초기 CCR) = 500 (50% 듀티)
    • Mode = PWM mode 1
  3. GPIO: PA0 → TIM2_CH1 (자동 할당됨, NUCLEO 커넥터 A0)

코드 구현

/* ch07_03_pwm_led_brightness.c — PWM 4단계 밝기 제어 (발췌) */

extern TIM_HandleTypeDef htim2;

void pwm_led_brightness_start(void)
{
    HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_1);
    LOG_I("PWM LED 밝기 제어 시작 (PA0, 1kHz)");
}

void pwm_led_set_duty(uint8_t percent)
{
    if (percent > 100) {
        LOG_W("듀티 범위 초과: %d%% → 100%%로 제한", percent);
        percent = 100;
    }

    uint32_t arr = __HAL_TIM_GET_AUTORELOAD(&htim2);
    uint32_t ccr = (arr + 1) * percent / 100;

    __HAL_TIM_SET_COMPARE(&htim2, TIM_CHANNEL_1, ccr);
    LOG_D("PWM 듀티 변경: %d%% (CCR=%lu)", percent, ccr);
}

void pwm_led_brightness_demo(void)
{
    const uint8_t levels[] = { 25, 50, 75, 100 };
    pwm_led_brightness_start();

    for (uint32_t i = 0; i < 4; i++) {
        pwm_led_set_duty(levels[i]);
        LOG_I("밝기 단계 %lu: %d%%", i + 1, levels[i]);
        HAL_Delay(2000);  /* 2초 유지 */
    }
}

💡 실무 팁
PA0은 TIM2_CH1과 TIM5_CH1이 겹칩니다. CubeMX에서 어떤 타이머에 할당했는지 반드시 확인하세요. 또한 PA2/PA3는 USART2 TX/RX와 겹치므로, UART를 사용 중이라면 TIM2 CH3/CH4는 사용할 수 없습니다. 핀 충돌은 임베디드에서 가장 흔한 실수 중 하나입니다.
💡 실무 팁
PA5(LD2)는 TIM2_CH1의 대체 핀입니다. NUCLEO 보드의 내장 LED로 PWM 테스트를 하고 싶다면 PA5를 TIM2_CH1에 매핑할 수 있습니다. 단, PA0에 이미 TIM2_CH1을 할당했다면 충돌합니다 — 둘 중 하나만 선택하세요.
❓ 수강생 단골 질문
Q. PWM 주파수는 어떻게 정하나요? 1kHz면 충분한가요?
A. 용도에 따라 다릅니다. LED 밝기 제어: 500Hz 이상이면 눈에 깜빡임이 보이지 않습니다. 1kHz면 충분합니다. DC 모터 속도 제어: 20~30kHz가 일반적입니다. 가청 주파수(~20kHz) 이상으로 설정해야 "윙윙" 소리가 나지 않습니다. 서보 모터: 정확히 50Hz (주기 20ms)를 요구합니다. 주파수가 너무 높으면 스위칭 손실이 커지고, 너무 낮으면 효과(밝기, 속도)가 고르지 않습니다.
🎯 면접 포인트
Q. "PWM으로 LED 밝기를 제어하는 원리를 설명하라"
A. PWM은 디지털 출력을 빠르게 켜고/끄는 방식이다. 사람의 눈은 약 60Hz 이상의 깜빡임을 구분하지 못하므로, 1kHz PWM은 "평균 밝기"로 인식된다. Duty Cycle 50%이면 전체 주기의 절반만 켜지므로 밝기도 약 50%가 된다. 핵심은 CCR/ARR 비율이 Duty를 결정한다는 것이다.
🔍 스스로 점검
  • ARR = 999일 때 75% 듀티에 해당하는 CCR 값은?
  • PWM Mode 1과 Mode 2의 차이를 설명할 수 있나요?
  • PWM 주파수 1kHz와 20kHz는 각각 어떤 용도에 적합한가요?

§7.5 tim_driver 구현 — uart_driver 패턴으로 만드는 타이머 드라이버

§7.3과 §7.4에서 타이머 인터럽트와 PWM을 직접 HAL API로 다뤘습니다. 이제 이 코드를 tim_driver.h/c로 캡슐화합니다. Ch04에서 만든 uart_driver와 정확히 같은 설계 패턴을 따릅니다.

왜 캡슐화해야 할까요? §7.3~7.4 코드에서는 htim2, HAL_TIM_Base_Start_IT(), __HAL_TIM_SET_COMPARE() 같은 HAL 상세가 그대로 노출되어 있습니다. 만약 TIM2 대신 TIM3을 사용하고 싶다면? 또는 향후 Ch08에서 Advanced TIM1으로 확장한다면? HAL 함수를 직접 호출하는 코드가 App 전체에 흩어져 있으면, 모두 수정해야 합니다.

uart_driver가 보여준 해결책을 그대로 적용합니다: App은 tim_driver.h만 include하고, HAL 헤더는 tim_driver.c 안에서만 참조.

tim_driver.h — 공개 인터페이스

/* ch07_04_tim_driver.h — TIM Driver 공개 인터페이스 */

#ifndef TIM_DRIVER_H
#define TIM_DRIVER_H

#include <stdint.h>

/* 반환 코드 (uart_driver.h와 동일 패턴) */
typedef enum {
    TIM_OK    = 0,   /* 성공 */
    TIM_ERROR = 1,   /* 일반 오류 */
    TIM_BUSY  = 2,   /* 동작 중 */
    TIM_PARAM = 3    /* 파라미터 오류 */
} tim_status_t;

/* 초기화 */
void tim_driver_init(void);

/* 주기 타이머 */
tim_status_t tim_start(void);
tim_status_t tim_stop(void);

/* PWM */
tim_status_t pwm_start(void);
tim_status_t pwm_stop(void);

/* 콜백 등록 */

#endif /* TIM_DRIVER_H */

tim_driver.c — 내부 구현 (핵심 발췌)

/* ch07_04_tim_driver.c — TIM Driver 구현 (발췌) */

#include "tim_driver.h"
#include "main.h"       /* HAL, GPIO 정의 */
#include "log.h"        /* LOG_D/I/W/E */

extern TIM_HandleTypeDef htim2;

static volatile tim_callback_t s_uev_callback = NULL;
#define TIM2_CLK_HZ     84000000U

void tim_driver_init(void)
{
    s_uev_callback = NULL;
    LOG_I("tim_driver 초기화 완료 (TIM2, APB1=%luHz)",
          TIM2_CLK_HZ);
}

tim_status_t tim_start(void)
{
    HAL_StatusTypeDef hal_st;
    hal_st = HAL_TIM_Base_Start_IT(&htim2);
    if (hal_st != HAL_OK) {
        LOG_E("tim_start 실패: hal=%d", hal_st);
        return TIM_ERROR;
    }
    LOG_I("TIM2 인터럽트 시작");
    return TIM_OK;
}

tim_status_t tim_set_period_ms(uint32_t period_ms)
{
    if (period_ms == 0 || period_ms > 60000) {
        LOG_W("범위 초과: %lu ms", period_ms);
        return TIM_PARAM;
    }

    uint32_t psc, arr;
    /* 10kHz 카운터 (PSC=8399, 16비트 범위 안전) */
    psc = (TIM2_CLK_HZ / 10000) - 1;  /* 8399 */
    arr = period_ms * 10 - 1;

    __HAL_TIM_SET_PRESCALER(&htim2, psc);
    __HAL_TIM_SET_AUTORELOAD(&htim2, arr);
    __HAL_TIM_GENERATE_SOFTWARE_UPDATE_EVENT(&htim2);
    __HAL_TIM_CLEAR_FLAG(&htim2, TIM_FLAG_UPDATE);

    LOG_D("주기 변경: %lu ms (PSC=%lu, ARR=%lu)",
          period_ms, psc, arr);
    return TIM_OK;
}

tim_status_t pwm_set_duty(uint8_t percent)
{
    if (percent > 100) {
        LOG_W("듀티 범위 초과: %d%%", percent);
        return TIM_PARAM;
    }

    uint32_t arr = __HAL_TIM_GET_AUTORELOAD(&htim2);
    uint32_t ccr = (arr + 1) * percent / 100;
    __HAL_TIM_SET_COMPARE(&htim2, TIM_CHANNEL_1, ccr);

    LOG_D("PWM 듀티: %d%% (CCR=%lu)", percent, ccr);
    return TIM_OK;
}

/* UEV 콜백 → 사용자 등록 함수 호출 */
void HAL_TIM_PeriodElapsedCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance != TIM2) return;
    if (s_uev_callback != NULL) s_uev_callback();
}

App에서 사용하기

/* main.c — tim_driver를 사용하는 App 코드 */

#include "tim_driver.h"

/* LED 토글 콜백 */
static void on_timer_tick(void)
{
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
}

int main(void)
{
    /* ... HAL_Init, SystemClock, MX 초기화 ... */

    tim_driver_init();
    tim_register_callback(on_timer_tick);
    tim_set_period_ms(500);
    tim_start();

    pwm_set_duty(50);
    pwm_start();

    while (1) {
        /* CPU 자유! UART, 센서 등 다른 작업 */
    }
}
💡 실무 팁
tim_set_period_ms()에서 __HAL_TIM_GENERATE_SOFTWARE_UPDATE_EVENT()를 호출하는 이유: PSC와 ARR은 쉐도우 레지스터에 먼저 쓰여지고, 다음 UEV에서 실제 레지스터로 복사됩니다. 소프트웨어 UEV(UG 비트 세트)를 발생시키면 즉시 반영됩니다. 중요: UG 세트 시 UIF(Update Interrupt Flag)도 함께 세트되어 의도치 않은 콜백이 1회 호출될 수 있습니다. 반드시 __HAL_TIM_CLEAR_FLAG(&htim2, TIM_FLAG_UPDATE)로 UIF를 클리어하세요.
❓ 수강생 단골 질문
Q. 왜 콜백을 함수 포인터로 등록하나요? 그냥 tim_driver.c 안에서 LED 토글하면 안 되나요?
A. 드라이버는 "무엇을 할지" 모르는 것이 올바른 설계입니다. tim_driver는 "정해진 주기마다 콜백을 호출한다"만 담당합니다. LED를 토글할지, 센서를 읽을지, 모터를 움직일지는 App이 결정합니다. 이것이 Ch03에서 배운 계층 간 역할 분리의 실천입니다. Ch10에서 RTC와 스텝 모터를 연동할 때도 이 콜백 패턴이 그대로 사용됩니다.
🎯 면접 포인트
Q. "HAL 함수를 직접 호출하는 것과 Driver 레이어로 래핑하는 것의 차이를 설명하라"
A. 직접 호출하면 HAL 의존성이 App 전체에 퍼진다. 타이머 교체(TIM2→TIM3), HAL 버전 변경, 플랫폼 이식 시 모든 호출부를 수정해야 한다. Driver로 래핑하면 변경 지점이 .c 파일 하나로 격리된다. "인터페이스 불변, 구현만 교체" — Ch06에서 uart_driver의 DMA 전환으로 실증한 원칙과 동일하다.

§7.6 누적 통합 + 아키텍처 업데이트

개별 기능을 모두 배웠으니, 이제 이전 챕터와 통합합니다. 이것이 누적 성장형 프로젝트의 핵심입니다 — 새 기능을 기존 코드에 자연스럽게 합칩니다.

통합 실습: 버튼으로 TIM 주기 + PWM 밝기 변경

Ch01의 버튼 EXTI + Ch07의 tim_driver를 결합합니다. 버튼을 누를 때마다 LED 깜빡임 속도와 PWM 밝기를 순환시킵니다. 

/* main.c — 누적 통합 예제 (Ch01 EXTI + Ch07 TIM) */

#include "tim_driver.h"
#include "uart_driver.h"
#include "log.h"

/* 상태 */
static uint8_t s_mode = 0;

/* 모드별 설정 */
typedef struct {
    uint32_t period_ms;
    uint8_t  duty;
    const char *desc;
} mode_entry_t;

static const mode_entry_t modes[] = {
    { 1000, 25,  "느린 깜빡 + 25% 밝기" },
    {  500, 50,  "보통 깜빡 + 50% 밝기" },
    {  200, 75,  "빠른 깜빡 + 75% 밝기" },
    {  100, 100, "매우 빠른 + 100% 밝기" },
};

#define MODE_COUNT  (sizeof(modes) / sizeof(modes[0]))

/* LED 토글 콜백 */
static void on_timer_tick(void)
{
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
}

/* 버튼 EXTI 콜백 (Ch01) */
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin)
{
    if (GPIO_Pin != GPIO_PIN_13) return;  /* PC13 = USER BTN */

    s_mode = (s_mode + 1) % MODE_COUNT;
    tim_set_period_ms(modes[s_mode].period_ms);
    pwm_set_duty(modes[s_mode].duty);

    LOG_I("모드 변경 → [%d] %s", s_mode, modes[s_mode].desc);
}

int main(void)
{
    /* ... HAL_Init, SystemClock, MX 초기화 ... */

    uart_driver_init();  /* Ch04/06 */
    tim_driver_init();   /* Ch07 */

    tim_register_callback(on_timer_tick);
    tim_set_period_ms(modes[0].period_ms);
    tim_start();

    pwm_set_duty(modes[0].duty);
    pwm_start();

    LOG_I("=== 통합 데모 시작 (버튼으로 모드 전환) ===");

    while (1) {
        /* CPU 자유 — UART 수신 처리 등 */
        if (uart_available()) {
            uint8_t ch = uart_read_byte();
            LOG_D("UART 수신: 0x%02X", ch);
        }
    }
}
📌 강사 꿀팁
이 통합 데모에서 강조할 포인트: 1) while(1) 안에 HAL_Delay()가 하나도 없습니다. 2) LED 토글, UART 수신, 버튼 감지가 모두 인터럽트 기반입니다. 3) main 루프의 CPU 점유율은 사실상 0%입니다. 이것이 "타이머를 쓰는 진짜 이유"입니다.

핵심 정리

  • 타이머는 CPU 독립 하드웨어 카운터. HAL_Delay 대신 타이머 인터럽트를 사용하면 CPU를 해방시킬 수 있다.
  • 마스터 공식: T = (PSC+1) × (ARR+1) / f_clk. PSC로 속도를 줄이고, ARR로 상한을 정한다.
  • UEV(Update Event): 카운터가 ARR에 도달하면 발생. HAL_TIM_PeriodElapsedCallback()으로 처리.
  • PWM: CCR/ARR 비율 = Duty Cycle. HAL_TIM_PWM_Start() + __HAL_TIM_SET_COMPARE()로 제어.
  • tim_driver: uart_driver와 동일한 패턴으로 HAL을 캡슐화. App은 tim_driver.h만 참조한다.
  • F411RE 특수: Basic 타이머 없음. TIM2/5(32비트), TIM3/4(16비트), TIM1(Advanced)만 존재.

연습문제

기억 1. STM32F411RE에서 사용 가능한 타이머를 모두 열거하고, 각각의 비트 수와 버스를 적으시오.
이해 2. PSC = 41999, ARR = 1999일 때 TIM2의 인터럽트 주기를 계산하시오. (f_clk = 84MHz)
적용 3. 200ms 주기로 LED를 토글하는 TIM3 설정값(PSC, ARR)을 구하고, CubeMX 설정 순서를 적으시오. (힌트: TIM3은 16비트이므로 ARR ≤ 65535)
적용 4. 20kHz PWM (모터 제어용)의 PSC/ARR 조합을 구하시오. Duty 해상도가 0.1% 이하가 되도록 ARR을 선택하시오.
분석 5. 아래 코드에서 HAL_Delay()와 TIM 인터럽트가 혼재되어 있습니다. 어떤 문제가 발생할 수 있는지 분석하고, 개선 방안을 제시하시오.

{
    HAL_UART_Transmit(&huart2, "tick\n", 5, 100);

다음 챕터 예고

현재 프로젝트는 v0.7 상태로, GPIO/EXTI, 로그, 아키텍처, UART DMA 링 버퍼, 그리고 TIM 주기 타이머 + PWM까지 구현되었습니다.

다음 Ch08. Advanced TIM 심화에서는 TIM1의 Complementary PWM과 Dead-time, Input Capture와 Output Compare, 그리고 버튼으로 제어하는 스톱워치 FSM을 구현하여 프로젝트를 v0.8로 업그레이드합니다.

Ch07에서 만든 tim_driver 인터페이스가 Ch08에서도 그대로 유지됩니다. Ch06에서 uart_driver의 DMA 전환을 체험했듯이, Ch08에서는 tim_driver의 내부가 Advanced 기능으로 확장되는 것을 체험합니다. Service 레이어 첫 등장도 Ch08에서 만나게 됩니다 — 기대하세요!
