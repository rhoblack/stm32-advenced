Ch08. Advanced TIM 심화

타이머의 숨겨진 능력을 깨워 Complementary PWM으로 양방향을 제어하고, FSM으로 스톱워치를 완성한다

누적 기능: Complementary PWM + Input Capture + Stopwatch FSM (stopwatch_service 추가)

§8.0 "스톱워치의 꿈 — 왜 Advanced TIM인가?"

여기까지의 여정을 돌아봅시다. Ch07에서 TIM2로 정밀 주기 타이머와 PWM을 완성했습니다. LED가 원하는 밝기로 빛나고, 500ms마다 정확히 콜백이 호출됩니다. v0.7의 tim_driver — 여러분이 직접 만든 첫 타이머 드라이버입니다.

그런데 현실의 임베디드 시스템은 이보다 훨씬 복잡합니다. 모터를 양방향으로 돌리려면 PWM 하나로는 부족합니다 — 상보 출력(Complementary PWM)이 필요합니다. 외부 신호의 주파수를 측정하려면 타이머를 "입력" 장치로 써야 합니다 — Input Capture입니다. 그리고 가장 중요한 것, Ch03에서 이론으로만 배웠던 FSM(유한 상태 머신)을 드디어 실제 코드로 구현합니다.

이 챕터가 끝나면:

  • TIM1(Advanced Control Timer)으로 Complementary PWM + Dead-time을 출력합니다.
  • Input Capture로 외부 신호의 주파수를 정밀 측정합니다.
  • Stopwatch FSM을 설계하고, Service 레이어에 구현합니다 — 아키텍처 v0.7 → v0.8.

특히 Service 레이어의 첫 등장이 중요합니다. 지금까지는 App이 Driver를 직접 호출했습니다. 이 챕터에서 처음으로 App → Service → Driver 3단 구조를 구현합니다. 이것이 Ch10(clock_service), Ch11(sensor_service), Ch13(ui_service)의 원형입니다.

§8.1 TIM1 — Advanced Control Timer의 세계

Ch07에서 우리는 TIM2(GP Timer)를 사용했습니다. 타이머 계층 표에서 한 줄이 남아 있었죠 — TIM1, Advanced Control Timer. "GP 기능 + Complementary PWM + Dead-time"이라고 적혀 있었습니다. 이제 그 한 줄의 의미를 파헤칩니다.

TIM1 vs TIM2~5 — 무엇이 다른가?

핵심 차이는 세 가지입니다:

항목GP Timer (TIM2~5)Advanced (TIM1)
버스APB1 (84MHz)APB2 (84MHz)
비트TIM2/5: 32bit, TIM3/4: 16bit16bit
PWM 채널CH1~CH4 (단방향)CH1~CH4 + CH1N~CH3N (상보 출력)
Dead-time없음BDTR 레지스터 (DTG 비트 필드)
Break 입력없음비상 정지 핀 (BKIN)
MOE 비트없음Main Output Enable — 안전 잠금

MOE는 Advanced Control Timer(TIM1)만의 안전 장치입니다. 고전력 모터를 제어할 때 실수로 양쪽이 동시 ON 되면 합선입니다. 그 순간에 모든 출력을 즉시 끌 수 있도록 MOE라는 마스터 스위치를 넣었습니다. 일반용 GP Timer(TIM2~5)는 이런 고전력 제어를 하지 않으니까 없습니다.

"TIM2는 만능 스톱워치, TIM1은 안전장치가 달린 산업용 스톱워치"라고 생각하세요. 기본 사용법(PSC/ARR/CCR, UEV, PWM)은 완전히 동일합니다. 차이는 안전 관련 기능이 추가된 것뿐입니다.

Complementary PWM — "시소(Seesaw)"

비유로 이해하기
시소를 떠올리세요. 한쪽이 올라가면 반대쪽은 반드시 내려갑니다. 절대로 양쪽이 동시에 위에 있을 수 없습니다. Complementary PWM이 바로 이것입니다 — CH1이 HIGH면 CH1N은 LOW, CH1이 LOW면 CH1N은 HIGH.
그런데 시소가 방향을 바꾸는 그 짧은 순간, 양쪽 다 중간 높이에 있습니다. 이 순간이 Dead-time — 양쪽 FET이 모두 꺼지는 안전 구간입니다.
※ 이 비유의 한계: 실제 시소는 기계적으로 연결되어 반드시 반대로 움직이지만, Complementary PWM은 하드웨어 설정으로 반전이 구현됩니다. 시소의 "중간 순간"은 자연스럽게 발생하지만, Dead-time은 엔지니어가 의도적으로 설정하는 보호 구간입니다.

왜 상보 출력이 필요할까요? 하프 브리지(Half-Bridge) 회로를 생각해 봅시다. 상단 FET과 하단 FET이 번갈아 켜지면서 모터를 양방향으로 구동합니다. 만약 소프트웨어로 두 PWM 채널의 타이밍을 맞추면, 인터럽트 지연으로 둘 다 동시에 켜지는 순간이 발생할 수 있습니다. 전원에서 GND로 직통 — shoot-through. 매우 큰 전류가 흘러 소자가 파괴됩니다.

Complementary PWM은 하드웨어가 자동으로 반전시키므로 이 위험이 없습니다. Dead-time은 "둘 다 꺼지는 안전 구간"을 하드웨어가 강제 삽입합니다.

Dead-time 설정 — "에어록(Airlock)" 안전 구간

에어록(가압 실린더) 원리와도 같습니다. 한쪽 문이 열릴 때 반대쪽 문이 동시에 열리면 기압이 빠져 위험합니다. 따라서 짧은 순간 양쪽 문 모두 닫히는(Dead-time) 안전 구간을 넣습니다. Complementary PWM의 Dead-time이 바로 이 역할입니다.

FET(트랜지스터)은 꺼지는 데 시간이 걸립니다. 수십~수백 나노초입니다. CH1이 OFF되는 순간 CH1N이 바로 ON되면, 짧은 순간 둘 다 ON — shoot-through입니다. Dead-time은 이 전환 구간에 "둘 다 OFF"를 삽입합니다.

STM32의 Dead-time은 BDTR(Break and Dead-Time Register)의 DTG 비트 필드로 설정합니다. DTG 인코딩은 비선형적이라 직접 계산하기 복잡합니다. 실무에서는 CubeMX의 Dead-time 슬라이더로 ns 단위로 설정하고, 오실로스코프로 확인합니다.

DTG 값 (예시)Dead-time84MHz 기준
DTG = 4242 × T_DTS500 ns
DTG = 8484 × T_DTS1000 ns (1 us)
DTG = 168(64+40) × 2 × T_DTS~2.5 us
💡 실무 팁
Dead-time 계산은 DTG 비트 필드의 구간별 공식 4가지가 있어 복잡합니다. 실무에서는 "CubeMX에서 원하는 ns 설정 → 오실로로 파형 확인"이 표준 절차입니다. "원리는 알되, 도구를 쓴다"가 올바른 자세입니다. DTG 세부 계산은 Reference Manual의 TIM1 BDTR 섹션을 참조하세요.

TIM1 핵심 HAL API

GP Timer와의 핵심 차이는 API 이름에 Ex와 N이 붙는 것입니다:

동작GP Timer (TIM2)Advanced (TIM1)
일반 PWM 시작HAL_TIM_PWM_Start()동일
보상 PWM 시작없음HAL_TIMEx_PWMN_Start()
Dead-time 설정없음CubeMX BDTR 또는 HAL_TIMEx_ConfigBreakDeadTime()
 
/* ch08_01_complementary_pwm.c — Complementary PWM 시작 핵심 코드 */

/* CH1 (주 출력) 시작 — PA8 */
HAL_TIM_PWM_Start(&htim1, TIM_CHANNEL_1);
LOG_I("TIM1 CH1 PWM 시작 (PA8)");

/* CH1N (보상 출력) 시작 — PB13 */
HAL_TIMEx_PWMN_Start(&htim1, TIM_CHANNEL_1);
LOG_I("TIM1 CH1N PWM 시작 (PB13)");

/* 듀티 50% 설정 */
uint32_t arr = __HAL_TIM_GET_AUTORELOAD(&htim1);
uint32_t ccr = (arr + 1) * 50 / 100;
__HAL_TIM_SET_COMPARE(&htim1, TIM_CHANNEL_1, ccr);
LOG_D("CPWM 듀티 50%% (CCR=%lu, ARR=%lu)", ccr, arr);
❓ 수강생 단골 질문
Q. Complementary PWM과 일반 PWM의 차이가 뭐예요?
A. 일반 PWM은 출력이 하나입니다. Complementary PWM은 출력이 두 개 — 주 채널(CH1)과 보상 채널(CH1N)이 항상 반대입니다. H-Bridge 회로에서 상단/하단 FET을 번갈아 제어할 때 필수입니다. 일반 PWM 2개를 소프트웨어로 반전시키면 타이밍 지연으로 shoot-through 위험이 있지만, Complementary PWM은 하드웨어가 자동 반전하므로 안전합니다.
❓ 수강생 단골 질문
Q. Dead-time은 왜 필요해요? 없으면 안 되나요?
A. FET은 꺼지는 데 시간이 걸립니다 (수십~수백 ns). CH1 OFF 순간 CH1N이 바로 ON되면 둘 다 ON → shoot-through → 소자 파괴. Dead-time은 "둘 다 OFF" 구간을 하드웨어가 강제 삽입합니다. 실무에서 Dead-time 설정 실수로 MOSFET을 태운 경험은 임베디드 엔지니어의 통과의례입니다.

Q. "Complementary PWM을 사용하는 이유와 Dead-time의 역할을 설명하세요."

§8.2 Complementary PWM 실습 — "시소를 만들자"

이론을 확인했으니 직접 파형을 만들어 봅시다. TIM1으로 1kHz Complementary PWM을 설정하고, Dead-time 1us를 적용합니다. 오실로스코프(또는 느린 주파수로 LED 2개)로 CH1과 CH1N이 반전되는 것을 확인합니다.

CubeMX 설정

  1. TIM1 활성화: Clock Source → Internal Clock
  2. Channel1: PWM Generation CH1
  3. Channel1N: PWM Generation CH1N
  4. PSC = 83 (84MHz / 84 = 1MHz 카운터)
  5. ARR = 999 (1MHz / 1000 = 1kHz PWM)
  6. CCR = 500 (듀티 50%)
  7. Dead-time = 84 (1us @ 84MHz)
  8. Break → Disable (이 실습에서 불필요)
  9. 핀 확인: PA8(CH1), PB13(CH1N)

코드

/* ch08_01_complementary_pwm.c — CPWM 실습 핵심 */

extern TIM_HandleTypeDef htim1;

void complementary_pwm_demo(void)
{
    /* CH1 (주 출력) 시작 — PA8 */
    HAL_TIM_PWM_Start(&htim1, TIM_CHANNEL_1);
    LOG_I("TIM1 CH1 PWM 시작 (PA8)");

    /* CH1N (보상 출력) 시작 — PB13 */
    HAL_TIMEx_PWMN_Start(&htim1, TIM_CHANNEL_1);
    LOG_I("TIM1 CH1N PWM 시작 (PB13)");

    /* 듀티 50% 설정 */
    uint32_t arr = __HAL_TIM_GET_AUTORELOAD(&htim1);
    uint32_t ccr = (arr + 1) * 50 / 100;
    __HAL_TIM_SET_COMPARE(&htim1, TIM_CHANNEL_1, ccr);
    LOG_D("CPWM 듀티 50%% (CCR=%lu, ARR=%lu)", ccr, arr);

    LOG_I("오실로스코프로 PA8/PB13 파형 확인하세요");
}

동작 확인

오실로스코프 확인 시:

  • PA8(CH1): 1kHz, 듀티 50%, HIGH 기준
  • PB13(CH1N): 동일 주파수, 반전된 파형
  • 전환 순간에 1us Dead-time 확인 (둘 다 LOW 구간)

오실로스코프 없는 경우: PSC=83999(100Hz), ARR=9999로 설정 후 LED 2개를 PA8/PB13에 연결하면 번갈아 깜빡이는 것을 육안으로 확인할 수 있습니다.

💡 실습이 안 될 때
가장 흔한 실수 두 가지: 1) HAL_TIMEx_PWMN_Start()를 빠뜨림 — CH1만 시작하면 CH1N은 출력되지 않습니다. 2) MOE(Main Output Enable) 비활성 — CubeMX에서 BDTR 설정에서 "Automatic Output State"를 Enable로 설정하세요. TIM1은 GP Timer와 달리 MOE 비트가 세트되어야 출력이 나옵니다.
📌 강사 꿀팁
오실로스코프가 있으면 CH1과 CH1N을 동시에 보여주세요. Dead-time 구간을 확대하면 "둘 다 LOW"인 짧은 구간이 보입니다. "이 1마이크로초가 수백만 원짜리 MOSFET을 살리는 구간입니다" — 학생들이 Dead-time의 가치를 체감합니다.

§8.3 Input Capture — "외부 신호를 찰칵 찍다"

Ch07에서 타이머는 출력 장치였습니다 — PWM을 만들어 외부로 내보냈습니다. 이제 관점을 180도 전환합니다. 타이머를 입력 장치로 사용하여 외부 신호의 주기와 주파수를 측정합니다. 이것이 Input Capture(입력 캡처)입니다.

비유로 이해하기
스톱워치의 랩(Lap) 버튼을 떠올리세요. 100m 달리기에서 선수가 지나갈 때마다 버튼을 누르면, 그 순간의 시간이 기록됩니다. Input Capture는 외부 신호의 엣지(edge)가 발생하는 순간, 카운터(CNT) 값을 CCR에 "찰칵!" 저장합니다. 두 번 "찰칵"한 값의 차이가 곧 주기입니다.
※ 이 비유의 한계: 실제 랩 버튼은 사람이 누르므로 정확도가 낮지만, Input Capture는 하드웨어가 나노초 단위로 정확하게 기록합니다.

Input Capture 원리

핵심 공식: 

/* Input Capture 주파수 계산 */
주기(카운트)  = Capture2 - Capture1
주기(초)      = (Capture2 - Capture1) / f_cnt
주파수(Hz)    = f_cnt / (Capture2 - Capture1)

/* 예: f_cnt = 1MHz (PSC=83), 차이 = 2500 카운트
 *   주기  = 2500 / 1,000,000 = 2.5ms
 *   주파수 = 1,000,000 / 2500 = 400 Hz
 */

측정 한계: 16비트 카운터(TIM3)에서 PSC=83(f_cnt = 1MHz)일 때, 최대 카운트 65535에 해당하는 주기는 약 65.5ms이므로 약 15Hz 이하의 주파수는 카운터 오버플로우로 측정 불가능할 수 있습니다. 더 낮은 주파수를 측정하려면 PSC를 높이거나 오버플로우 카운팅을 추가해야 합니다.

Input Capture 구현

/* ch08_02_input_capture.c — IC 주파수 측정 핵심 코드 */

extern TIM_HandleTypeDef htim3;

static volatile uint32_t s_capture_1st  = 0;
static volatile uint32_t s_capture_2nd  = 0;
static volatile uint8_t  s_capture_done = 0;
static volatile uint8_t  s_capture_idx  = 0;

/* IC 측정 시작 */
void input_capture_start(void)
{
    s_capture_idx  = 0;
    s_capture_done = 0;
    HAL_TIM_IC_Start_IT(&htim3, TIM_CHANNEL_1);
    LOG_I("Input Capture 시작 (TIM3 CH1, PA6)");
}

/* HAL IC 콜백 — Rising Edge마다 호출 */
void HAL_TIM_IC_CaptureCallback(TIM_HandleTypeDef *htim)
{
    if (htim->Instance != TIM3) return;

    uint32_t captured = HAL_TIM_ReadCapturedValue(htim, TIM_CHANNEL_1);
    if (s_capture_idx == 0) {
        s_capture_1st = captured;
        s_capture_idx = 1;
    } else {
        s_capture_2nd = captured;
        s_capture_done = 1;
        HAL_TIM_IC_Stop_IT(htim, TIM_CHANNEL_1);
    }
}

/* 메인 루프에서 결과 확인 */
void input_capture_poll(void)
{
    if (!s_capture_done) return;
    s_capture_done = 0;

    uint32_t diff = (s_capture_2nd >= s_capture_1st)
        ? s_capture_2nd - s_capture_1st
        : (0xFFFF - s_capture_1st) + s_capture_2nd + 1;

    LOG_I("주파수 = %lu Hz, 주기 = %lu us",
          1000000 / diff, diff);
}

Q. Input Capture로 뭘 측정할 수 있어요? 실무에서 쓰나요?
A. 외부 신호의 주파수, 주기, 펄스 폭을 측정합니다. 실무 사례: ① 자동차 엔진 RPM 측정 (크랭크 센서 펄스 주기) ② 초음파 센서 거리 측정 (에코 펄스 폭) ③ 적외선 리모컨 신호 디코딩 (펄스 패턴 분석). 우리 실습에서는 버튼 누르기 간격 측정으로 시작합니다.
❓ 수강생 단골 질문
Q. GPIO EXTI + HAL_GetTick()으로도 시간 측정할 수 있지 않나요?
A. 가능합니다. 하지만 정확도가 다릅니다. HAL_GetTick()은 1ms 해상도이고, EXTI 콜백 진입까지 인터럽트 지연이 있습니다. Input Capture는 하드웨어가 Rising Edge 순간 CNT 값을 직접 CCR에 저장하므로, 1 카운트 해상도(PSC=83이면 1us)로 정확합니다. 고주파 신호(kHz 이상)는 Input Capture만 가능합니다.

§8.4 Output Compare & One-Pulse — "알람 시계와 카메라 플래시"

타이머의 입출력 모드 "3형제"를 정리합시다. §8.1~8.2에서 PWM(출력)과 Input Capture(입력)를 배웠습니다. 나머지 두 가지 — Output Compare(출력 비교)와 One-Pulse Mode(원 펄스) — 를 개념 수준으로 살펴봅니다.

Output Compare — "알람 시계"

PWM과 비슷하지만 목적이 다릅니다. PWM은 "일정한 듀티의 반복 파형"을 만드는 것이 목적이고, Output Compare는 "카운터가 CCR에 도달하는 특정 시점"에 이벤트를 발생시키는 것이 목적입니다. 알람 시계가 설정한 시각에 "삐!" 하고 울리듯, 카운터가 CCR 값에 도달하면:

  • 핀을 Toggle/Set/Clear
  • 인터럽트 또는 DMA 트리거 발생
  • 매 주기마다 반복

Ch09(RTC)에서 알람 기능을 구현할 때 이 "특정 시점 이벤트" 개념이 다시 등장합니다.

One-Pulse Mode — "카메라 플래시"

트리거 한 번에 펄스 딱 하나만 나오고 자동 정지합니다. 반복 PWM과 달리 "한 번만 실행"이 필요한 상황에 사용합니다:

  • 카메라 플래시 트리거 (정확한 시간 동안 빛)
  • 서보 모터 위치 지정 (한 번의 펄스로 각도 설정)
  • 부저 단음 발생
/* ch08_03_one_pulse.c — One-Pulse 핵심 코드 */

extern TIM_HandleTypeDef htim4;

void one_pulse_start(void)
{
    /* One-Pulse 모드: 트리거 → 지연 → 펄스 1개 → 자동 정지 */
    HAL_TIM_OnePulse_Start(&htim4, TIM_CHANNEL_1);
    LOG_I("One-Pulse 대기 (TIM4 CH1, PB6)");
    LOG_D("  지연: 3ms, 펄스 폭: 7ms, 총 주기: 10ms");
}
❓ 수강생 단골 질문
Q. One-Pulse 모드는 언제 쓰나요?
A. 트리거 한 번에 펄스 딱 하나만 나오고 자동 정지합니다. 반복 PWM과 달리 "한 번만 실행"이 필요한 상황에 씁니다. 실무에서는 초음파 센서 트리거 펄스 생성, 서보 위치 제어, 특정 시간 후 GPIO 토글 등에 활용합니다.
❓ 수강생 단골 질문
Q. Input Capture와 Output Compare 뭐가 다른가요?
A. 입출력이 반대입니다. IC는 "외부 신호를 타이머가 재기" (신호 → MCU), OC는 "MCU가 신호를 타이머로 만들기" (MCU → 신호). 쉽게 기억하세요: "IC = 스톱워치로 측정", "OC = 신호 발생기".
📌 강사 꿀팁
§8.4는 개념 중심입니다. IC에 비해 OC와 One-Pulse는 활용 빈도가 낮으므로, "이런 모드가 있다" 수준으로 넘어가세요. 학생이 관심 있으면 Reference Manual 링크를 안내하고, 나머지 시간을 §8.5~8.7(FSM)에 투자하는 것이 효과적입니다.

§8.5 타이머 연계 — "부장과 대리의 팀워크"

지금까지 타이머는 각자 독립적으로 동작했습니다. TIM2는 TIM2의 PSC/ARR로, TIM1은 TIM1의 PSC/ARR로. 하지만 STM32 타이머들은 내부 연결선(ITR)으로 서로 연결되어 있어, 한 타이머의 이벤트가 다른 타이머를 제어할 수 있습니다. 이것이 Master-Slave(마스터-슬레이브) 타이머 연계입니다.

비유로 이해하기
부장과 대리를 떠올리세요. 부장(Master)이 "지금!" 하고 명령하면, 대리(Slave)가 자기 일을 시작합니다. 대리는 자기만의 업무 속도(PSC/ARR)로 일하지만, 시작/정지/리셋은 부장 명령에 따릅니다. 부장은 정기적으로(UEV마다) 명령을 내리고, 대리는 그때마다 자기 일을 수행합니다.
※ 이 비유의 한계: 실제 조직에서는 부장-대리가 양방향 소통하지만, 타이머 Master-Slave는 단방향(Master→Slave)입니다.

Master-Slave 동작 원리

Master 타이머는 TRGO(Trigger Output) 신호를 생성합니다. 이 신호는 MCU 내부 연결선(ITR)을 통해 Slave 타이머로 전달됩니다. 외부 핀이 필요 없습니다 — 칩 내부에서 직접 연결됩니다.

Slave 타이머는 TRGO를 받아서 세 가지 중 하나를 수행합니다:

Slave 모드TRGO 수신 시 동작용도
Trigger카운터 시작 (Enable)특정 시점에 타이머 시작
Reset카운터 0으로 리셋주기 동기화
GatedTRGO HIGH 동안만 카운트특정 구간만 측정
 
/* ch08_04_master_slave.c — 타이머 연계 핵심 코드 */

extern TIM_HandleTypeDef htim1;
extern TIM_HandleTypeDef htim2;

void master_slave_start(void)
{
    /* 순서 중요: Slave 먼저 대기, Master 나중에 시작 */

    /* Slave: TIM2 PWM 대기 (Trigger 수신 시 자동 시작) */
    HAL_TIM_PWM_Start(&htim2, TIM_CHANNEL_1);
    LOG_I("Slave(TIM2) PWM 대기");

    /* Master: TIM1 시작 → 1초마다 TRGO 출력 */
    HAL_TIM_Base_Start(&htim1);
    LOG_I("Master(TIM1) 시작 — 1초마다 TRGO");

    LOG_D("TIM1(1s UEV) → TRGO → ITR0 → TIM2(PWM)");
}

Master-Slave 연계의 가장 흔한 실무 사례는 ADC 정밀 샘플링입니다. 타이머가 정확한 간격으로 TRGO → ADC 시작을 트리거하면, 소프트웨어 지연 없이 정밀한 샘플링 주기를 보장합니다. 이 패턴은 Ch17(아키텍처 심화)에서 다시 만납니다.
🎯 면접 포인트
Q. "타이머 Master-Slave 모드를 설명하고, 사용 사례를 들어주세요."
A. 하나의 타이머(Master)가 TRGO 신호로 다른 타이머(Slave)를 트리거/게이팅/리셋. 내부 연결(ITR)로 핀 불필요. 사례: ADC 정밀 샘플링, 복수 PWM 동기화, 모터 제어 시퀀싱.

§8.6 Stopwatch FSM 설계 — "상태 전이표를 직접 그리다"

이 챕터의 핵심 성취 구간에 왔습니다. Ch03 §3.4에서 FSM의 4요소를 이론으로 배웠습니다 — 상태(State), 전이(Transition), 이벤트(Event), 액션(Action). Ch06에서 TX_IDLE/TX_BUSY를 암묵적으로 사용했지만, 명시적 FSM 설계 → 구현 → 검증은 지금이 처음입니다.

Ch03에서 자판기 비유를 기억하시나요? 동전을 넣으면 대기 → 선택가능, 버튼을 누르면 선택가능 → 배출. 스톱워치도 같은 구조입니다. 버튼을 누르면 IDLE → RUNNING, 다시 누르면 RUNNING → PAUSED. 같은 FSM 패턴, 다른 도메인입니다.

Step 1: 상태 정의

핸드폰 스톱워치를 관찰하면 세 가지 상태가 보입니다:

Step 3: 상태 전이표 — "빈 칸 채우기"

FSM 설계의 핵심은 상태 전이표입니다. 행은 현재 상태, 열은 이벤트. 모든 칸을 채워야 합니다 — 빈 칸이 곧 버그입니다.

❓ 수강생 단골 질문
Q. PAUSED에서 다시 RUNNING으로 갈 때 카운터는 어떻게 되나요?
A. 핵심은 누적 처리입니다. Pause 시 elapsed += tick_count로 현재까지 시간을 누적 저장하고, Resume 시 tick_count = 0으로 리셋한 뒤 타이머를 다시 시작합니다. 이전 누적값(elapsed)에 새로운 구간 틱이 더해지므로 시간이 이어집니다. 흔한 실수: Resume 시 elapsed를 0으로 초기화하면 이전 시간이 날아갑니다!
🔍 스스로 점검
  • 상태 전이표에서 "무시"가 적힌 칸이 왜 중요한지 설명할 수 있나요?
  • RUNNING → PAUSED → RUNNING 재개 시 누적 시간이 이어지는 원리를 코드로 설명할 수 있나요?
  • FSM의 4요소(상태, 전이, 이벤트, 액션)를 스톱워치에 대입하여 각각 나열할 수 있나요?

§8.7 Stopwatch FSM 구현 — "설계를 코드로 옮기다"

§8.6에서 설계를 완료했습니다. 상태 3개, 이벤트 2개, 전이 5개, 각 전이의 액션까지 명확합니다. 이제 코드로 옮깁니다. 그리고 이 코드는 Driver(tim_driver)가 아닌 Service(stopwatch_service)에 위치합니다.

왜 Service 레이어인가?

Ch03에서 배운 4계층 원칙을 복습합시다:

  • Driver = 하드웨어 추상화. "타이머를 시작/정지/설정" → tim_driver
  • Service = 비즈니스 로직. "언제 시작하고, 언제 정지하고, 상태를 어떻게 관리할까?" → stopwatch_service

"Driver에 FSM을 넣으면 안 되나요?" 안 됩니다. Driver에 비즈니스 로직을 넣으면, tim_driver가 스톱워치 전용이 되어 다른 용도(알람, 모터)로 재사용할 수 없습니다. Driver는 어떤 목적으로든 사용 가능한 범용 인터페이스를 유지해야 합니다.

/* ch08_05_stopwatch_service.h — Service 레이어 공개 API */

typedef enum {
    SW_STATE_IDLE    = 0,   /* 초기 상태, 00:00.000 */
    SW_STATE_RUNNING = 1,   /* 시간 측정 중 */
    SW_STATE_PAUSED  = 2    /* 일시 정지 */
} sw_state_t;

typedef enum {
    SW_EVT_BTN_SHORT = 0,   /* 버튼 짧게 누름 (<2초) */
    SW_EVT_BTN_LONG  = 1,   /* 버튼 길게 누름 (≥2초) */
    SW_EVT_TICK      = 2    /* 타이머 1ms 틱 (내부용) */
} sw_event_t;

typedef struct {
    uint32_t total_ms;       /* 총 경과 밀리초 */
    uint8_t  minutes;        /* 분 (0~59) */
    uint8_t  seconds;        /* 초 (0~59) */
    uint16_t milliseconds;   /* 밀리초 (0~999) */
} sw_time_t;

void       stopwatch_init(void);
void       stopwatch_handle_event(sw_event_t evt);
sw_state_t stopwatch_get_time(sw_time_t *time);
sw_state_t stopwatch_get_state(void);

stopwatch_service.c — FSM 구현 (switch-case 패턴)

/* ch08_05_stopwatch_service.c — FSM 핵심 구현 */

#include "stopwatch_service.h"
#include "tim_driver.h"
#include "log.h"

static volatile sw_state_t s_state      = SW_STATE_IDLE;
static volatile uint32_t   s_elapsed_ms = 0;
static volatile uint32_t   s_tick_count = 0;

static const char * const STATE_NAMES[] = {
    "IDLE", "RUNNING", "PAUSED"
};

/* 1ms 타이머 콜백 — RUNNING 상태에서만 카운트 */
static void stopwatch_tick_callback(void)
{
    if (s_state == SW_STATE_RUNNING) {
        s_tick_count++;
    }
}

void stopwatch_init(void)
{
    s_state      = SW_STATE_IDLE;
    s_elapsed_ms = 0;
    s_tick_count = 0;

    tim_set_period_ms(1);                     /* 1ms 주기 */
    tim_register_callback(stopwatch_tick_callback);
    LOG_I("stopwatch_service 초기화 완료");
}

void stopwatch_handle_event(sw_event_t evt)
{
    sw_state_t prev = s_state;

    switch (s_state) {

    case SW_STATE_IDLE:
        if (evt == SW_EVT_BTN_SHORT) {
            /* IDLE → RUNNING: 시작 */
            s_elapsed_ms = 0;
            s_tick_count = 0;
            tim_start();
            s_state = SW_STATE_RUNNING;
            LOG_I("스톱워치 시작!");
        }
        break;

    case SW_STATE_RUNNING:
        if (evt == SW_EVT_BTN_SHORT) {
            /* RUNNING → PAUSED: 일시정지 */
            tim_stop();
            s_elapsed_ms += s_tick_count;
            s_tick_count = 0;
            s_state = SW_STATE_PAUSED;
            LOG_I("일시정지 (%lu ms)", s_elapsed_ms);
        }
        else if (evt == SW_EVT_BTN_LONG) {
            /* RUNNING → IDLE: 리셋 */
            tim_stop();
            s_elapsed_ms = 0;
            s_tick_count = 0;
            s_state = SW_STATE_IDLE;
            LOG_I("리셋 (RUNNING → IDLE)");
        }
        break;

    case SW_STATE_PAUSED:
        if (evt == SW_EVT_BTN_SHORT) {
            /* PAUSED → RUNNING: 재개 (누적) */
            s_tick_count = 0;
            tim_start();
            s_state = SW_STATE_RUNNING;
            LOG_I("재개 (%lu ms부터)", s_elapsed_ms);
        }
        else if (evt == SW_EVT_BTN_LONG) {
            /* PAUSED → IDLE: 리셋 */
            s_elapsed_ms = 0;
            s_tick_count = 0;
            s_state = SW_STATE_IDLE;
            LOG_I("리셋 (00:00.000)");
        }
        break;
    }

    if (prev != s_state) {
        LOG_D("FSM: %s → %s",
              STATE_NAMES[prev], STATE_NAMES[s_state]);
    }
}

sw_state_t stopwatch_get_time(sw_time_t *time)
{
    uint32_t total = s_elapsed_ms + s_tick_count;
    if (time != NULL) {
        time->total_ms     = total;
        time->minutes      = (uint8_t)(total / 60000);
        time->seconds      = (uint8_t)((total % 60000) / 1000);
        time->milliseconds = (uint16_t)(total % 1000);
    }
    return s_state;
}

App 레이어에서 사용하기 — Button → FSM

/* main.c — App 레이어에서 Stopwatch 사용 */

#include "stopwatch_service.h"

/* 버튼 이벤트 플래그 (EXTI ISR에서 세트) */
static volatile uint8_t  g_btn_pressed  = 0;
static volatile uint32_t g_btn_press_ms = 0;

/* EXTI 콜백 — ISR 컨텍스트, 플래그만 세트 */
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin)
{
    if (GPIO_Pin == B1_Pin) {
        g_btn_pressed  = 1;
        g_btn_press_ms = HAL_GetTick();
    }
}

/* 메인 루프 */
int main(void)
{
    HAL_Init();
    SystemClock_Config();
    MX_GPIO_Init();
    MX_TIM2_Init();

    tim_driver_init();
    stopwatch_init();

    LOG_I("=== Stopwatch 시작 ===");

    while (1) {
        /* 버튼 이벤트 처리 — 메인 루프에서 FSM 호출 */
        /* ⚠️ 주의: 이 로직은 "개념 설명용" 스켈레톤입니다.
         * 실제 실무에서는 Button Press → Release 타이밍을 명확히 하기 위해
         * 별도의 상태 머신(예: IDLE → PRESSING → PRESSED → RELEASED)을 씁니다.
         * Ch10(모터) 이후에 실제 패턴을 배웁니다. */
        if (g_btn_pressed) {
            g_btn_pressed = 0;
            uint32_t duration = HAL_GetTick() - g_btn_press_ms;

            if (duration >= 2000) {
                stopwatch_handle_event(SW_EVT_BTN_LONG);
            } else {
                stopwatch_handle_event(SW_EVT_BTN_SHORT);
            }
        }

        /* 주기적 시간 표시 (예: 500ms마다) */
        static uint32_t last_print = 0;
        if (HAL_GetTick() - last_print >= 500) {
            last_print = HAL_GetTick();

            sw_time_t t;
            sw_state_t st = stopwatch_get_time(&t);
            if (st == SW_STATE_RUNNING) {
                LOG_I("%02d:%02d.%03d",
                      t.minutes, t.seconds, t.milliseconds);
            }
        }
    }
}
💡 실무 팁
ISR에서 FSM 전이를 직접 하지 마세요. EXTI 콜백에서 stopwatch_handle_event()를 호출하면, ISR 안에서 tim_start()/tim_stop()이 실행됩니다. 중첩 인터럽트 문제가 발생할 수 있습니다.

왜일까요? 만약 ISR 도중에 상태를 바꾸고, 동시에 메인 루프가 그 상태를 읽는다면? MCU가 값을 읽는 도중에 ISR이 값을 바꾸면 → 중간 상태 읽기 → 논리 에러. 따라서 ISR은 "깃발(flag)을 세운다"만 하고, 메인 루프가 "깃발을 보고 상태를 바꾼다"는 분리가 필수입니다. Ch01에서 배운 패턴을 따르세요: "인터럽트에서는 플래그만, 처리는 메인에서."
❓ 수강생 단골 질문
Q. 버튼 하나로 Start/Pause/Resume/Reset을 어떻게 구분하나요?
A. 두 가지 기법을 조합합니다: 1) FSM 상태에 따른 해석 — 같은 BTN_SHORT라도 IDLE에서는 Start, RUNNING에서는 Pause, PAUSED에서는 Resume입니다. FSM이 맥락을 제공합니다. 2) 길게 누르기 — BTN_LONG(2초 이상)은 Reset 전용입니다. 이것이 FSM의 힘입니다 — "현재 상태"가 입력의 의미를 결정합니다.
🎯 면접 포인트
Q. "FSM(유한 상태 머신)을 실무에서 어떻게 구현하셨나요?"
A. enum으로 상태 정의, switch-case로 전이 처리, 이벤트와 액션 분리. "인터럽트에서는 이벤트 플래그만 세트하고, 메인 루프에서 FSM 처리" 패턴. 상태 전이표를 먼저 작성하고, 모든 (상태, 이벤트) 조합을 검증한 후 코드로 옮김. 핵심: 설계 → 구현 순서를 지키면 전이 누락 버그를 예방할 수 있다.
📌 강사 꿀팁
FSM 구현 실습은 스켈레톤 코드를 제공하세요. 전체 구조(enum, switch-case 틀, init 함수)는 주고, 학생은 전이 로직(case 안의 if-else)만 채우는 방식입니다. 30분 안에 완성하려면 이 방법이 현실적입니다. 완성 후 "Button 누르면 시리얼 모니터에 시간이 올라가는" 것을 확인하면 큰 성취감을 느낍니다.

축하합니다! 지금까지 우리는 타이머만 다뤘다면, 이제 Service 레이어 첫 모듈을 완성했습니다. 이것이 v0.8의 핵심입니다. 앞으로 v0.9, v1.0, v1.1로 진행하면서 같은 방식으로 RTC, 모터, 센서, LCD 각각의 Service를 쌓게 됩니다.

/* Complementary PWM (TIM1) */
tim_status_t cpwm_start(void)
{
    HAL_TIM_PWM_Start(&htim1, TIM_CHANNEL_1);       /* CH1 */
    HAL_TIMEx_PWMN_Start(&htim1, TIM_CHANNEL_1);    /* CH1N */
    LOG_I("CPWM 시작 (PA8 + PB13)");
    return TIM_OK;
}

tim_status_t cpwm_set_duty(uint8_t percent)
{
    uint32_t arr = __HAL_TIM_GET_AUTORELOAD(&htim1);
    uint32_t ccr = (arr + 1) * percent / 100;
    __HAL_TIM_SET_COMPARE(&htim1, TIM_CHANNEL_1, ccr);
    LOG_D("CPWM 듀티: %d%%", percent);
    return TIM_OK;
}

/* Input Capture (TIM3) */
tim_status_t ic_start(void)
{
    HAL_TIM_IC_Start_IT(&htim3, TIM_CHANNEL_1);
    LOG_I("IC 시작 (PA6)");
    return TIM_OK;
}

uint32_t ic_get_frequency(void)
{
    /* 캡처 차이로 주파수 계산 */
    if (!s_ic_ready || s_ic_period_us == 0) return 0;
    return IC_CNT_FREQ_HZ / s_ic_period_us;
}

아키텍처 원칙 확인

  • 레이어 규칙 준수: stopwatch_service → tim_driver → HAL. 역방향 참조 없음.
  • 인터페이스 불변: 기존 tim_driver API(tim_start/stop, pwm_start/stop) 변경 없이 새 함수만 추가.
  • 관심사 분리: FSM 로직은 Service에, 타이머 제어는 Driver에. Service가 HAL을 직접 호출하지 않음.
  • ISR 안전: EXTI에서 플래그만 세트, FSM 전이는 메인 루프에서 처리.
💡 실무 팁
이 패턴(Service → Driver → HAL)은 이후 챕터에서 반복됩니다: Ch10: clock_service → rtc_driver → HAL_RTC Ch11: sensor_service → i2c_driver → HAL_I2C Ch13: ui_service → spi_driver → HAL_SPI 패턴을 한 번 익히면, 새로운 서비스를 만들 때 구조가 자연스럽게 나옵니다.

핵심 정리

  • TIM1(Advanced)은 GP Timer의 모든 기능 + Complementary PWM + Dead-time + Break + MOE를 가진다.
  • Complementary PWM은 CH1과 CH1N이 항상 반전된 파형을 출력하며, Dead-time은 전환 순간 "둘 다 OFF" 구간을 삽입하여 shoot-through를 방지한다.
  • Input Capture는 외부 신호의 Rising/Falling Edge 시점에 CNT 값을 CCR에 저장하여 주파수/주기를 정밀 측정한다.
  • Output Compare는 CNT == CCR 시점에 이벤트를 발생시키고, One-Pulse는 트리거 1회에 펄스 1개를 자동 생성한다.
  • Master-Slave 연계는 TRGO/ITR 내부 연결을 통해 타이머 간 동기화를 구현한다.
  • Stopwatch FSM은 IDLE/RUNNING/PAUSED 3개 상태로 구성되며, switch-case 패턴으로 Service 레이어에 구현한다.
  • Service 레이어는 Driver(하드웨어 추상화) 위에 비즈니스 로직(FSM)을 배치하여 관심사를 분리한다. Driver 재사용성 보장.

아키텍처 업데이트

이 챕터 완료 후 전체 아키텍처에 stopwatch_service(Service)가 추가되고, tim_driver(Driver)에 CPWM/IC 기능이 확장되었습니다.
