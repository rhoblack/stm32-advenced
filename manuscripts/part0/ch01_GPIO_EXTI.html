§1.1 첫 번째 불 켜기 — GPIO 출력과 LED 제어

프로그래밍을 배울 때 화면에 "Hello, World!"를 출력하듯, 임베디드 시스템에서의 첫 걸음은 LED를 켜는 것입니다. 하지만 단순히 "불이 켜졌다"에서 멈추면 안 됩니다. 왜 그 방식으로 연결해야 하는지, 왜 다른 방식은 안 되는지를 이해해야 실무에서 회로 설계자와 대화하고, 트러블슈팅을 할 수 있습니다. 이 절에서는 GPIO(General Purpose Input/Output, 범용 입출력) 출력 모드의 두 가지 핵심 방식인 Push-Pull과 Open-Drain의 동작 원리를 깊이 파헤칩니다.

GPIO 출력 모드란?

MCU(Micro Controller Unit, 마이크로컨트롤러)의 핀은 기본적으로 HIGH(3.3V) 또는 LOW(0V) 두 가지 전압 상태를 출력할 수 있습니다. STM32의 GPIO 출력 모드에는 크게 두 가지가 있습니다.

  • Push-Pull (밀고-당기기): MCU 내부에 HIGH를 만드는 PMOS와 LOW를 만드는 NMOS가 모두 들어 있어, 소프트웨어 명령 하나로 핀을 HIGH나 LOW로 능동적으로 구동할 수 있습니다.
  • Open-Drain (열린 배수구): 내부에 NMOS만 있어 LOW로만 능동 구동됩니다. HIGH 상태는 외부 풀업 저항이 VCC로 연결되어야 만들어집니다.
비유로 이해하기 — 수도꼭지 vs 배수구
Push-Pull은 양방향 수도꼭지와 같습니다. 레버를 한쪽으로 돌리면 물이 나오고(HIGH, 전류 공급), 반대로 돌리면 물을 빨아들입니다(LOW, 전류 흡수). MCU 혼자 능동적으로 양방향을 제어할 수 있습니다.

Open-Drain은 배수구만 있는 욕조와 같습니다. 배수구를 막으면 수압(외부 풀업)에 의해 물이 찹니다(HIGH). 배수구를 열면 물이 빠집니다(LOW). 물을 채우는 힘은 외부(풀업 저항)에서 나옵니다.
※ 비유의 한계: 실제 MOSFET은 수도꼭지처럼 서서히 열리지 않고 디지털로 켜지고 꺼집니다. 또한 Open-Drain에서 여러 핀이 하나의 라인을 공유할 수 있는 Wired-AND 특성은 이 비유만으로 설명하기 어렵습니다.

Push-Pull 내부 구조

Push-Pull 출력 드라이버의 핵심은 두 개의 MOSFET(Metal-Oxide-Semiconductor Field-Effect Transistor, 금속산화막 반도체 전계효과 트랜지스터)입니다. 소프트웨어가 HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET)을 호출하면, MCU 내부의 출력 레지스터(ODR, Output Data Register)의 해당 비트가 1로 설정됩니다. 이 신호가 PMOS(P채널 금속산화물 반도체 트랜지스터, 전원 쪽 스위치) 게이트에 전달되어 PMOS가 켜지고, VDD(전원 공급 전압, 3.3V)에서 핀으로 전류가 흐릅니다 — 이것이 "Push"입니다. 반대로 LOW를 출력하면 NMOS(N채널 금속산화물 반도체 트랜지스터, GND 쪽 스위치)가 켜져 핀에서 GND로 전류가 흡수됩니다 — 이것이 "Pull"입니다.

Open-Drain이 필요한 상황

Open-Drain은 주로 세 가지 상황에서 사용됩니다.

  1. 레벨 시프팅(Level Shifting): 3.3V MCU가 5V 장치를 구동할 때, 외부 풀업을 5V로 연결하면 Open-Drain으로 5V 신호를 만들 수 있습니다.
  2. Wired-AND 버스: I2C, SMBus 등 여러 장치가 하나의 신호선을 공유하는 버스 프로토콜. 어느 하나라도 LOW를 출력하면 버스 전체가 LOW가 됩니다.
  3. 오픈 콜렉터 드라이버와 인터페이스: 일부 센서나 구형 TTL 소자가 Open-Collector 출력을 사용합니다.
❓ 수강생 단골 질문
Q. Open-Drain 핀에서 HIGH를 출력하도록 설정하면 실제로 어떤 전압이 나오나요? 3.3V가 나오나요?
A. 아닙니다. Open-Drain에서 HIGH는 "아무것도 하지 않는다"는 의미입니다. 내부 NMOS가 꺼져서 핀이 전기적으로 떠 있는(floating) 상태가 됩니다. 이때 외부에 풀업 저항이 VCC에 연결되어 있으면 저항을 통해 전류가 흘러 핀이 VCC 수준으로 올라갑니다. 풀업이 없으면 핀은 부동 상태가 되어 노이즈에 취약해집니다. 따라서 Open-Drain을 사용할 때는 반드시 외부 풀업 저항이 필요합니다.

NUCLEO-F411RE의 LED 회로

NUCLEO-F411RE 보드에는 사용자 LED인 LD2(초록)가 PA5 핀에 연결되어 있습니다. PA5 → 저항(510Ω) → LED → GND 순서로 연결되어 있어, PA5가 HIGH(3.3V)일 때 LED가 켜집니다. 이것을 액티브 하이(Active High)라고 합니다.

이 회로에서는 MCU가 전류를 공급(source)해야 하므로 Push-Pull 모드가 적합합니다. 만약 Open-Drain을 사용한다면 HIGH 상태에서 내부 드라이버가 없어 LED가 제대로 켜지지 않습니다.

HAL API로 LED 제어하기

STM32 HAL(Hardware Abstraction Layer, 하드웨어 추상화 계층)은 GPIO를 제어하는 세 가지 핵심 함수를 제공합니다. 

/* gpio_led_basic.c — PA5 LED 기본 제어 예제 */
/* LOG는 Ch02에서 추가됩니다 */

#include "main.h"

/* PA5 = LD2 (액티브 하이: GPIO_PIN_SET = LED 켜짐) */

/**
 * @brief LED를 켜는 함수
 */
void led_on(void)
{
    /* GPIO_PIN_SET = 1 = HIGH = 3.3V 출력 → LED 켜짐 */
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET);
}

/**
 * @brief LED를 끄는 함수
 */
void led_off(void)
{
    /* GPIO_PIN_RESET = 0 = LOW = 0V 출력 → LED 꺼짐 */
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET);
}

/**
 * @brief LED 상태를 반전시키는 함수 (켜져 있으면 끄고, 꺼져 있으면 켜기)
 */
void led_toggle(void)
{
    HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);
}

/**
 * @brief 블링크 예제 — 500ms 간격으로 LED 점멸
 */
void led_blink_blocking(void)
{
    while (1)
    {
        led_toggle();
        HAL_Delay(500);    /* 500ms 대기 (블로킹) */
    }
}
💡 실무 팁
HAL_GPIO_WritePin의 세 번째 인자는 GPIO_PIN_SET(1) 또는 GPIO_PIN_RESET(0)입니다. 1이나 0을 직접 써도 컴파일은 되지만, HAL 매크로를 사용하면 가독성이 높아지고 코드 리뷰 시 의도가 명확해집니다. 팀 코드베이스에서는 항상 HAL 매크로를 사용하는 것을 권장합니다.

GPIO_PIN_5는 무엇인가?

STM32 HAL에서 GPIO_PIN_5는 단순히 숫자 5가 아닙니다. 이것은 비트마스크(bitmask)입니다. GPIO_PIN_5는 0x0020(2진수 0000 0000 0010 0000)으로 정의되어 있습니다. 5번째 비트(0부터 시작)가 1로 설정된 값입니다.

GPIO 포트(GPIOA)는 0번부터 15번까지 16개의 핀을 가집니다. HAL은 이 16개 핀을 16비트 마스크로 관리합니다. HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5 | GPIO_PIN_3, GPIO_PIN_SET)처럼 비트 OR 연산으로 여러 핀을 동시에 제어할 수도 있습니다.

§1.2 핀 설정의 해부학 — CubeMX로 GPIO 구성하기

LED를 켜는 코드 한 줄을 작성하기 전에, MCU는 해당 핀이 어떤 역할을 하는지 알아야 합니다. 핀을 입력으로 쓸지 출력으로 쓸지, 속도는 얼마로 할지, 초기 상태는 어떻게 할지 — 이 모든 설정이 초기화 코드에 담겨야 합니다. STM32CubeMX는 GUI(Graphical User Interface, 그래픽 사용자 인터페이스)로 이 설정을 손쉽게 생성해 주는 도구입니다. 이 절에서는 CubeMX가 생성한 코드를 한 줄씩 해부하여 "자동 생성된 코드도 정확히 이해하고 쓰는" 능력을 기릅니다.

RCC — 클록을 켜야 GPIO가 살아난다

STM32에서 GPIO를 사용하려면 먼저 해당 GPIO 포트에 클록(Clock, 동작 신호)을 공급해야 합니다. STM32는 전력 소모를 줄이기 위해 사용하지 않는 주변장치(Peripheral, 내부 장치)의 클록을 기본적으로 끕니다. RCC(Reset and Clock Control, 리셋 및 클록 제어) 레지스터에서 해당 포트의 클록을 활성화해야 합니다.

비유로 이해하기 — 사무실 전원 차단기
대형 빌딩에서 에너지 절약을 위해 사용하지 않는 층의 전기를 차단합니다. GPIOA를 사용하려면 "A층 차단기를 올려라" 같은 명령이 필요합니다 — 이것이 RCC 클록 활성화입니다. 클록이 없는 GPIO 레지스터에 값을 쓰면 그 값은 아무 효과가 없습니다.
※ 비유의 한계: 실제로는 전력이 완전히 차단되는 게 아니라 클록 신호만 게이팅(gating)됩니다. 레지스터 값은 유지될 수 있지만 동작은 하지 않습니다. 
/* CubeMX가 생성하는 MX_GPIO_Init() 함수 분석 */
/* LOG는 Ch02에서 추가됩니다 */

static void MX_GPIO_Init(void)
{
    GPIO_InitTypeDef GPIO_InitStruct = {0};    /* 초기화 구조체, 0으로 초기화 */

    /* ── 1단계: 클록 활성화 ─────────────────────────────── */
    /* GPIOA 포트 클록 켜기 — PA5 (LD2)를 사용하기 위해 필요 */
    __HAL_RCC_GPIOA_CLK_ENABLE();

    /* GPIOC 포트 클록 켜기 — PC13 (B1 버튼)을 사용하기 위해 필요 */
    __HAL_RCC_GPIOC_CLK_ENABLE();

    /* ── 2단계: PA5 = LD2 출력 설정 ────────────────────── */
    /* LED 초기 상태: 꺼짐 (GPIO_PIN_RESET) */
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET);

    GPIO_InitStruct.Pin   = GPIO_PIN_5;             /* 대상 핀: PA5 */
    GPIO_InitStruct.Mode  = GPIO_MODE_OUTPUT_PP;    /* 출력 모드, Push-Pull */
    GPIO_InitStruct.Pull  = GPIO_NOPULL;            /* 내부 풀업/풀다운 없음 */
    GPIO_InitStruct.Speed = GPIO_SPEED_FREQ_LOW;    /* 슬루율 최저 (LED는 빠를 필요 없음) */
    HAL_GPIO_Init(GPIOA, &GPIO_InitStruct);

    /* ── 3단계: PC13 = B1 버튼 입력 설정 ───────────────── */
    GPIO_InitStruct.Pin  = GPIO_PIN_13;             /* 대상 핀: PC13 */
    GPIO_InitStruct.Mode = GPIO_MODE_INPUT;         /* 입력 모드 */
    GPIO_InitStruct.Pull = GPIO_NOPULL;             /* 보드에 외부 풀업 있으므로 내부 불필요 */
    HAL_GPIO_Init(GPIOC, &GPIO_InitStruct);
    /* Speed 멤버는 입력 모드에서 무시됨 */
}

GPIO_InitTypeDef 구조체 해부

GPIO_InitTypeDef는 핀 하나(또는 여러 핀)의 동작 방식을 기술하는 구조체입니다. 멤버 하나하나가 실제 하드웨어 레지스터 비트에 대응합니다.

  • Pin: 비트마스크로 대상 핀을 지정합니다. GPIO_PIN_5 | GPIO_PIN_6처럼 여러 핀을 한 번에 설정할 수 있습니다.
  • Mode: 핀의 기능을 결정합니다. GPIO_MODE_OUTPUT_PP(Push-Pull 출력), GPIO_MODE_OUTPUT_OD(Open-Drain 출력), GPIO_MODE_INPUT(입력), GPIO_MODE_AF_PP(대체 기능, Alternate Function), GPIO_MODE_ANALOG(아날로그) 등이 있습니다.
  • Pull: 내부 풀업/풀다운 저항 설정. GPIO_PULLUP, GPIO_PULLDOWN, GPIO_NOPULL 중 선택합니다.
  • Speed: 출력 슬루율(slew rate)을 설정합니다. 고속 신호(SPI, SDIO 등)는 GPIO_SPEED_FREQ_VERY_HIGH를 사용합니다. LED처럼 느린 신호는 LOW로도 충분하고 EMI(전자기 간섭)를 줄일 수 있습니다.
  • Alternate: Mode가 AF(Alternate Function)일 때 어떤 주변장치 기능을 사용할지 지정합니다 (예: GPIO_AF7_USART1).

GPIO_Speed와 EMI의 관계

GPIO Speed를 높이면 출력 신호의 상승/하강 시간(rise/fall time)이 짧아집니다. 이는 빠른 통신에 유리하지만, 급격한 전류 변화는 라인 임피던스와 결합해 오버슈트(overshoot), 언더슈트(undershoot), 링잉(ringing) 현상을 유발하고 EMI(Electro-Magnetic Interference, 전자기 간섭)를 증가시킵니다. LED 같은 저속 신호에는 항상 GPIO_SPEED_FREQ_LOW를 사용하세요.

❓ 수강생 단골 질문
Q. GPIO_PIN_13과 숫자 리터럴 13은 다른가요?
A. 전혀 다릅니다. GPIO_PIN_13은 ((uint16_t)0x2000), 즉 2진수로 0010 0000 0000 0000인 비트마스크입니다. 도출 과정: 213 = 8192 = 0x2000 — 13번째 비트(0부터 시작)가 1인 값입니다. 반면 숫자 13은 0x000D = 0000 0000 0000 1101로, 0번, 2번, 3번 비트가 1입니다. HAL 함수에 13을 넣으면 전혀 다른 핀들(PA0, PA2, PA3)이 영향을 받아 예상치 못한 오동작이 발생합니다.
🔍 스스로 점검
  • __HAL_RCC_GPIOA_CLK_ENABLE()을 호출하지 않으면 어떤 일이 생기는지 설명할 수 있나요?
  • GPIO_InitTypeDef의 각 멤버(Pin, Mode, Pull, Speed)가 무엇을 제어하는지 외우지 말고 이해했나요?
  • Push-Pull 모드로 설정한 핀에 Open-Drain 장치를 연결하면 어떤 위험이 있는지 생각해 보세요.
📌 강사 꿀팁
__HAL_RCC_GPIOA_CLK_ENABLE() 없이 GPIO를 초기화하는 코드를 라이브 코딩으로 보여주고, LED가 전혀 반응하지 않는 상황을 연출하면 수강생이 클록 활성화의 중요성을 강하게 기억합니다. "처음에 다들 이 줄을 빼먹는다"고 미리 말해두면 실수에 대한 심리적 부담도 줄어듭니다.

§1.3 버튼 읽기 — GPIO 입력과 폴링

LED를 켜고 끄는 것은 MCU가 세상에 말을 거는 행위입니다. 버튼을 읽는 것은 세상이 MCU에게 말하는 행위입니다. 이 절에서는 GPIO 입력 모드의 동작 원리와, 가장 직관적인 버튼 읽기 방법인 폴링(Polling, 주기적 확인)을 다룹니다. 그리고 버튼 하드웨어의 물리적 특성에서 오는 디바운싱(debouncing) 문제를 이해하고 소프트웨어로 해결하는 방법을 배웁니다.

GPIO 입력 모드의 동작 원리

입력 모드에서 MCU는 핀 전압을 감지합니다. STM32의 GPIO 입력 경로는 핀 → 슈미트 트리거(Schmitt Trigger, 노이즈 제거용 히스테리시스 회로) → 입력 데이터 레지스터(IDR, Input Data Register) 순서로 이루어집니다. 소프트웨어에서 HAL_GPIO_ReadPin()을 호출하면 IDR의 해당 비트 값을 읽어옵니다.

NUCLEO B1 버튼의 특성 — 액티브 로우(Active Low)

NUCLEO-F411RE의 B1 버튼(파란색)은 PC13에 연결되어 있습니다. 보드 회로를 보면:

  • 버튼이 눌리지 않은 상태: PC13은 보드 내부 풀업 저항을 통해 VCC(3.3V)에 연결 → PC13 = HIGH = GPIO_PIN_SET
  • 버튼이 눌린 상태: PC13이 GND에 연결 → PC13 = LOW = GPIO_PIN_RESET

이처럼 버튼을 누를 때 LOW가 되는 방식을 액티브 로우(Active Low)라고 합니다. NUCLEO 보드 내부에 이미 외부 풀업 저항이 있으므로, CubeMX에서 Pull 설정을 GPIO_NOPULL로 해야 합니다. 내부 풀업까지 추가하면 신호 레벨에 영향을 줄 수 있습니다.

비유로 이해하기 — 초인종 배선
집 초인종이 눌리면 회로가 연결되어 전류가 흐르는 방식(액티브 HIGH)과, 평상시 전류가 흐르다가 초인종을 누르면 회로가 끊기는 방식(액티브 LOW)이 있습니다. NUCLEO B1은 두 번째 방식입니다 — 평소에는 풀업으로 HIGH, 누르면 GND로 연결되어 LOW.
※ 비유의 한계: 실제 초인종 회로는 220V AC를 다루므로 MCU GPIO와 직접 비교는 위험합니다. 그리고 MCU의 풀업 저항은 수십 kΩ으로 실제 전류가 매우 미약합니다(수십 μA 수준).
❓ 수강생 단골 질문
Q. PC13이 눌렸을 때 LOW가 된다면, 코드에서 버튼 눌림 판정을 어떻게 하나요? HIGH를 눌림으로 오해하지 않으려면?
A. HAL_GPIO_ReadPin(GPIOC, GPIO_PIN_13) == GPIO_PIN_RESET 조건이 참(true)일 때 버튼이 눌린 것입니다. 헷갈리면 매크로나 함수로 추상화하세요: #define BUTTON_PRESSED (HAL_GPIO_ReadPin(GPIOC, GPIO_PIN_13) == GPIO_PIN_RESET). 이렇게 하면 코드 본문에서 논리적 의미가 명확해집니다.

폴링 방식으로 버튼 읽기

폴링(Polling)은 메인 루프에서 주기적으로 입력 상태를 확인하는 가장 단순한 방법입니다.

비유로 이해하기 — 전자레인지 앞에 서 있기
음식을 넣고 전자레인지 앞에 서서 "다 됐나? 다 됐나?" 계속 확인하는 것이 폴링입니다. 음식이 다 되기 전까지 당신은 그 자리를 떠날 수 없습니다. CPU도 마찬가지로 버튼을 확인하는 동안 다른 일을 하지 못합니다.
※ 비유의 한계: MCU의 폴링은 루프 주기(예: 1ms)마다 확인하므로 항상 100% CPU를 소비하지는 않습니다. 하지만 다른 작업이 있다면 타이밍 정확성이 떨어질 수 있습니다. 
/* polling_button.c — 폴링 방식 버튼 → LED 제어 */
/* LOG는 Ch02에서 추가됩니다 */

#include "main.h"

/* B1: PC13, 액티브 로우 (눌리면 GPIO_PIN_RESET) */
/* LD2: PA5, 액티브 하이 (켜려면 GPIO_PIN_SET)    */

/**
 * @brief 버튼 눌림 여부 반환 (액티브 로우 처리 포함)
 * @return 1: 눌림, 0: 안 눌림
 */
uint8_t button_is_pressed(void)
{
    /* PC13이 LOW(RESET)면 버튼이 눌린 것 */
    return (HAL_GPIO_ReadPin(GPIOC, GPIO_PIN_13) == GPIO_PIN_RESET) ? 1 : 0;
}

/**
 * @brief 폴링 방식 메인 루프 — 버튼 누르는 동안 LED ON
 * 문제: 빠른 진동(바운싱)으로 인해 LED가 깜빡일 수 있음
 */
void polling_main_loop_v1(void)
{
    while (1)
    {
        if (button_is_pressed())
        {
            /* 버튼이 눌려 있는 동안 LED 켜기 */
            HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET);
        }
        else
        {
            /* 버튼이 안 눌려 있으면 LED 끄기 */
            HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_RESET);
        }
    }
}

디바운싱(Debouncing) — 버튼의 물리적 진동

기계식 버튼을 누르면 접점(contact)이 딱 한 번 닫히는 것이 아니라, 수 밀리초(ms) 동안 여러 차례 튀면서(bounce) 연결/해제를 반복합니다. MCU는 이 미세한 진동을 여러 번의 버튼 누름으로 인식합니다. 이것이 스위치 바운싱(switch bouncing)이고, 이를 제거하는 기술이 디바운싱(debouncing)입니다.

비유로 이해하기 — 스프링 달린 문
무거운 문을 빠르게 닫으면 완전히 닫히지 않고 약간 열렸다 닫혔다를 반복합니다. "문이 닫혔다"고 판단하려면 일정 시간(예: 50ms) 동안 지속적으로 닫혀 있을 때만 판단해야 합니다. 버튼 디바운싱도 같습니다 — 신호가 안정될 때까지 기다렸다가 판단합니다.
※ 비유의 한계: 실제 버튼 바운스 시간은 버튼 종류마다 다릅니다(1ms~20ms가 일반적). 스프링 문은 눈에 보이지만 버튼 바운스는 오실로스코프로만 볼 수 있습니다. 
/* debounce_polling.c — 소프트웨어 디바운싱 포함 폴링 */
/* LOG는 Ch02에서 추가됩니다 */

#include "main.h"

#define DEBOUNCE_TIME_MS    20U     /* 바운스 안정화 대기 시간 (ms) */

/**
 * @brief 디바운싱 포함 버튼 눌림 감지 (엣지 감지 방식)
 * @return 1: 이번 프레임에 버튼이 새로 눌림, 0: 아님
 */
uint8_t button_detect_press(void)
{
    static uint8_t  prev_state     = 0;     /* 이전 상태 */
    static uint32_t last_change_ms = 0;     /* 마지막 상태 변화 시각 */

    uint8_t  cur_raw   = (HAL_GPIO_ReadPin(GPIOC, GPIO_PIN_13) == GPIO_PIN_RESET) ? 1 : 0;
    uint32_t now_ms    = HAL_GetTick();     /* 부팅 후 경과 시간 (ms) */

    /* 상태가 변했고, 마지막 변화로부터 DEBOUNCE_TIME_MS 이상 지났으면 유효 */
    if (cur_raw != prev_state)
    {
        if ((now_ms - last_change_ms) >= DEBOUNCE_TIME_MS)
        {
            prev_state     = cur_raw;
            last_change_ms = now_ms;

            if (cur_raw == 1)
            {
                return 1;    /* 유효한 버튼 누름 감지 */
            }
        }
    }
    return 0;
}

/**
 * @brief 디바운싱 포함 폴링 메인 루프 — 버튼 누름마다 LED 토글
 */
void polling_main_loop_v2(void)
{
    while (1)
    {
        if (button_detect_press())
        {
            HAL_GPIO_TogglePin(GPIOA, GPIO_PIN_5);    /* LED 토글 */
        }
        /* 다른 작업 수행 가능 (단, HAL_Delay 없이) */
    }
}

§1.4 인터럽트로 반응하기 — EXTI와 NVIC 기초

💡 이 절에서 다루는 EXTI와 NVIC는 처음 접하면 낯설게 느껴지는 것이 정상입니다. 비유와 그림을 통해 차근차근 따라오시면 반드시 이해할 수 있습니다. 지금 당장 모든 세부 사항을 외울 필요는 없으며, 실습 코드를 직접 작성하면서 자연스럽게 익히게 됩니다.

폴링은 구현이 단순하지만 CPU가 계속 버튼을 확인해야 합니다. 버튼 이벤트가 드물게 발생하는 경우, 대부분의 폴링 사이클은 "아무 일 없음"을 확인하는 데 낭비됩니다. 더 심각한 문제는 메인 루프가 오래 걸리는 작업을 수행하는 동안 버튼 누름을 놓칠 수 있다는 점입니다. 이 문제를 해결하는 것이 EXTI(External Interrupt, 외부 인터럽트)입니다. 이 절에서는 EXTI의 동작 원리, 엣지 트리거 설정, 그리고 인터럽트 우선순위를 제어하는 NVIC의 개념을 배웁니다.

인터럽트란 무엇인가?

인터럽트(Interrupt, 인터럽트)는 하드웨어 이벤트가 발생했을 때 CPU에게 "지금 하던 일을 잠깐 멈추고 이것을 먼저 처리하라"고 알리는 신호입니다. CPU는 현재 실행 위치를 저장하고, ISR(Interrupt Service Routine, 인터럽트 서비스 루틴)이라는 특별한 함수를 실행한 뒤, 원래 위치로 돌아와 작업을 계속합니다.

비유로 이해하기 — 전자레인지 타이머 알람
폴링이 "전자레인지 앞에 서서 계속 확인"이라면, 인터럽트는 타이머 알람입니다. 음식을 넣고 타이머를 설정한 뒤 다른 일을 합니다. "띵!" 소리(인터럽트 신호)가 나면 하던 일을 잠깐 멈추고 음식을 꺼냅니다(ISR 실행). 그리고 원래 하던 일로 돌아갑니다. CPU는 이벤트가 없는 동안 다른 작업에 집중할 수 있습니다.
※ 비유의 한계: 실제 인터럽트는 "현재 명령어 완료 직후" 처리되어 수십 나노초 단위의 지연만 발생합니다. 전자레인지 알람처럼 사람이 반응하는 것보다 훨씬 즉각적입니다. 또한 인터럽트는 중첩될 수 있어 ISR 실행 중 더 높은 우선순위의 인터럽트가 또 발생할 수 있습니다.

EXTI — 외부 핀을 인터럽트 소스로

STM32의 EXTI(External Interrupt/Event Controller, 외부 인터럽트/이벤트 컨트롤러)는 GPIO 핀의 전압 변화를 인터럽트 요청으로 변환합니다. STM32F411은 EXTI 라인 0~15를 가지며, 각 라인은 해당 번호의 GPIO 핀에 매핑됩니다.

Q. PA0과 PB0 둘 다 EXTI0를 사용하면 어떻게 되나요? 구별이 안 되나요?
A. 안 됩니다. 같은 번호의 핀(PA0, PB0, PC0 등)은 모두 EXTI0 라인을 공유하며, 동시에 하나의 핀만 EXTI0로 사용할 수 있습니다. 어느 포트를 EXTI0에 연결할지는 SYSCFG(System Configuration, 시스템 설정) 레지스터의 EXTICR(External Interrupt Configuration Register)에서 설정합니다. CubeMX가 자동으로 처리해 주지만, PA3와 PB3를 동시에 EXTI로 쓰려 하면 충돌이 발생합니다. 이럴 때는 핀 번호를 다르게 설계해야 합니다 (예: PA3와 PB4).

엣지 트리거(Edge Trigger) — 언제 인터럽트가 발생하나?

EXTI는 신호의 어느 시점에 인터럽트를 발생시킬지 세 가지로 선택할 수 있습니다.

  • Rising Edge (상승 엣지): LOW→HIGH 변화 시 발생. 버튼 해제 시 발생(액티브 로우 버튼의 경우).
  • Falling Edge (하강 엣지): HIGH→LOW 변화 시 발생. 버튼 누름 시 발생(액티브 로우 버튼의 경우). ← B1에 적합
  • Both Edges (양쪽 엣지): 양방향 모두 발생. 로터리 엔코더, 쿼드러처 신호 처리에 사용.
비유로 이해하기 — 엘리베이터 문 센서
엘리베이터 문이 열릴 때(하강 → 상승, Rising Edge)만 감지하거나, 닫힐 때(상승 → 하강, Falling Edge)만 감지하거나, 둘 다(Both) 감지할 수 있습니다. 우리 버튼은 "누르는 순간"(Falling Edge)에만 반응하도록 설정합니다.
※ 비유의 한계: 엘리베이터 센서는 광학식이어서 아날로그 신호를 처리합니다. 디지털 EXTI는 명확한 전압 임계값(슈미트 트리거)으로 엣지를 감지합니다.

NVIC — 인터럽트 우선순위 관리

여러 인터럽트가 동시에 발생하면 어떤 것을 먼저 처리할까요? 이를 결정하는 것이 NVIC(Nested Vectored Interrupt Controller, 중첩 벡터 인터럽트 컨트롤러)입니다. "Nested(중첩)"라는 이름처럼, 높은 우선순위 인터럽트는 낮은 우선순위 ISR 실행 중에도 선점할 수 있습니다.

비유로 이해하기 — 응급실 환자 분류(Triage)
응급실에서는 환자를 도착 순서가 아닌 응급도(PreemptPriority)로 먼저 분류합니다. 같은 응급도의 환자가 여럿이면 그 안에서 순서(SubPriority)로 처리합니다. 심정지 환자(PreemptPriority 0)가 오면 골절 환자(PreemptPriority 2)를 치료하던 의사가 잠깐 멈추고 심정지 환자를 먼저 처리합니다 — 이것이 선점(Preemption)입니다.
※ 비유의 한계: NVIC 우선순위 숫자는 작을수록 높은 우선순위입니다(0이 가장 높음). 현실 응급도는 반드시 그렇지 않을 수 있습니다. 또한 STM32는 우선순위 그룹 설정에 따라 PreemptPriority와 SubPriority 비트 수가 달라집니다.

NVIC 우선순위 그룹

STM32의 NVIC는 4비트(0~15)로 우선순위를 표현합니다. 이 4비트를 PreemptPriority(선점 우선순위)와 SubPriority(부선점 우선순위)로 어떻게 나눌지를 결정하는 것이 우선순위 그룹(Priority Group)입니다. CubeMX는 기본적으로 NVIC_PRIORITYGROUP_4를 사용하는데, 이는 4비트 모두를 PreemptPriority에 할당합니다(SubPriority 없음). 즉 우선순위 값은 0~15로, 숫자가 작을수록 높은 우선순위입니다. 

/* EXTI + NVIC CubeMX 생성 코드 분석 */

static void MX_GPIO_Init(void)
{
    GPIO_InitTypeDef GPIO_InitStruct = {0};

    __HAL_RCC_GPIOC_CLK_ENABLE();

    /* PC13 = EXTI 인터럽트 입력 설정 */
    GPIO_InitStruct.Pin  = GPIO_PIN_13;
    GPIO_InitStruct.Mode = GPIO_MODE_IT_FALLING;    /* 하강 엣지(버튼 누름)에 인터럽트 */
    GPIO_InitStruct.Pull = GPIO_NOPULL;             /* 보드 내부 풀업 있음 */
    HAL_GPIO_Init(GPIOC, &GPIO_InitStruct);

    /* EXTI15_10_IRQn: EXTI10~15번 라인이 공유하는 인터럽트 핸들러 */
    /* PC13 = EXTI13 → EXTI15_10_IRQn을 사용 */
    HAL_NVIC_SetPriority(EXTI15_10_IRQn, 5, 0);    /* PreemptPriority=5, SubPriority=0 */
    /* SysTick = Priority 15(최저, HAL 기본값). EXTI는 5~10 범위 권장. 0은 긴급용 예약. */
    HAL_NVIC_EnableIRQ(EXTI15_10_IRQn);
}

{
    /* HAL이 플래그 확인 후 HAL_GPIO_EXTI_Callback을 호출 */

Q. HAL_GPIO_EXTI_Callback이 어떻게 자동으로 호출되나요? 직접 호출한 적이 없는데요.
A. HAL 라이브러리 내부에서 __weak 속성으로 선언된 빈 함수가 기본으로 존재합니다. 우리가 같은 이름으로 함수를 정의하면, 링커(linker)가 자동으로 우리 버전을 선택합니다(약한 심볼 오버라이딩). 즉, EXTI15_10_IRQHandler → HAL_GPIO_EXTI_IRQHandler(HAL 내부) → HAL_GPIO_EXTI_Callback(우리가 구현) 순서로 호출 체인이 형성됩니다.
지금은 "같은 이름의 함수를 만들면 자동으로 내 코드가 호출된다"는 것만 기억하면 충분합니다. 링커 동작 원리는 고급 과정에서 다룹니다.

Q: NVIC PreemptPriority와 SubPriority의 차이를 설명하세요.
핵심 답변 키워드: PreemptPriority — 선점 가능 여부 결정. 낮은 번호의 인터럽트가 높은 번호의 ISR을 중단시키고 선점 가능. SubPriority — 같은 PreemptPriority를 가진 인터럽트들 사이의 처리 순서만 결정, 선점 불가. CubeMX 기본: NVIC_PRIORITYGROUP_4 (PreemptPriority 4비트, SubPriority 없음). HAL 내부 함수(SysTick, HAL_Delay)는 우선순위 15(최저)를 사용 — 왜 그런지 설명할 수 있어야 함.

폴링 vs 인터럽트 — 실무 선택 기준

폴링과 인터럽트 각각의 특성을 아래와 같이 정리할 수 있습니다.

  • 폴링 — 사용 권장 상황: 빠른 프로토타이핑, 이벤트가 매우 자주 발생해 인터럽트 오버헤드가 더 큰 경우(예: 고속 ADC 데이터 연속 수집), RTOS 태스크 내에서 주기적 확인이 필요한 경우, 디버깅이 용이해야 하는 경우.
  • 인터럽트 — 사용 권장 상황: 이벤트가 드물고 빠른 반응성이 필요한 경우(버튼, 통신 수신), CPU가 다른 작업을 하면서 이벤트를 놓치면 안 되는 경우, 저전력 모드에서 슬립 중 깨어나야 하는 경우.

ISR 작성의 황금 법칙

ISR은 빠르게 실행되고 종료되어야 합니다. ISR이 오래 실행되면 같거나 낮은 우선순위의 다른 인터럽트가 처리되지 못하고 시스템 반응성이 떨어집니다. 다음 규칙을 반드시 지켜야 합니다.

  1. ISR에서 HAL_Delay() 사용 금지: HAL_Delay는 SysTick 인터럽트를 기다리는데, SysTick의 우선순위(15)가 현재 ISR보다 낮으면 SysTick 인터럽트가 발생하지 않아 무한 대기가 됩니다.
  2. ISR에서 printf() 등 블로킹 I/O 사용 금지: UART 송신 완료를 기다리는 동안 인터럽트 컨텍스트가 묶입니다.
  3. ISR과 메인 코드가 공유하는 변수는 volatile로 선언: 컴파일러가 해당 변수를 레지스터에 캐시하지 못하게 합니다.
  4. 공유 데이터 접근 시 원자성(Atomicity) 보장: ARM Cortex-M4에서 32비트 이하의 정렬된(aligned) 변수 읽기/쓰기는 원자적(atomic)입니다 — 이 교재에서 사용하는 volatile uint8_t g_button_flag나 volatile uint32_t 타입 변수는 원자성 문제가 없습니다. 단, uint64_t 등 64비트 이상 변수나 read-modify-write 연산(++, |=, &=)은 여러 명령어로 처리될 수 있어 ISR에 의해 중간에 끊길 수 있습니다. 이 경우 __disable_irq() / __enable_irq()로 임계 구간을 보호해야 합니다.
비유로 이해하기 — volatile과 공유 화이트보드
CPU는 자주 쓰는 변수를 "개인 메모지(레지스터 캐시)"에 복사해 빠르게 사용합니다. 하지만 ISR이 메모리의 변수를 바꾸면, CPU는 여전히 오래된 "개인 메모지" 값을 봅니다. volatile은 "이 변수는 개인 메모지 금지, 항상 공유 화이트보드(메모리)에서 직접 읽어라"는 지시입니다.
※ 비유의 한계: volatile은 메모리 가시성(visibility)만 보장하고 원자성(atomicity)은 보장하지 않습니다. 멀티코어 환경이라면 추가적인 메모리 배리어(memory barrier)가 필요합니다. STM32는 단일 코어이므로 단순 플래그 공유에는 volatile로 충분합니다.

Q: ISR과 메인 루프 사이에서 공유하는 변수에 volatile을 쓰는 이유를 설명하세요.
핵심 답변 키워드: 컴파일러 최적화 시 자주 읽는 변수를 CPU 레지스터에 캐시 → ISR이 메모리의 값을 바꿔도 메인 루프는 캐시된 오래된 값을 계속 읽음 → volatile 선언으로 "항상 메모리에서 직접 읽어라" 지시. 추가 포인트: volatile은 메모리 가시성(visibility)만 보장, 원자성(atomicity) 보장 아님. STM32(단일 코어 Cortex-M4)에서 32비트 이하 단순 플래그는 volatile로 충분.
이 질문은 중견 기업 STM32 포지션 면접에서 자주 출제됩니다.

Q. HAL_GetTick()이 약 49.7일 후에 0으로 오버플로우되면, 디바운싱 코드 (now - lastTick) >= 50이 오동작하지 않나요?
A. 오동작하지 않습니다. uint32_t는 약 49.7일 후 0으로 초기화되지만, now - lastTick 뺄셈은 unsigned 산술(모듈러 산술)로 처리됩니다. 예: now = 5, lastTick = 0xFFFFFFF0 → 5 - 0xFFFFFFF0 = 0x15 = 21 (올바른 21ms). 이처럼 오버플로우 전후에도 뺄셈 결과는 실제 경과 시간을 정확히 반영합니다. 이 패턴은 임베디드 타이머 처리에서 표준적으로 사용되는 안전한 방법입니다.
❓ 수강생 단골 질문
Q. ISR에서 HAL_Delay(10)을 쓰면 왜 안 되나요? 10ms 정도는 짧지 않나요?
A. 짧고 길고의 문제가 아닙니다. HAL_Delay는 내부적으로 SysTick 인터럽트(HAL 기본값: 우선순위 15, 최저)가 uwTick을 증가시키기를 기다립니다. EXTI ISR의 우선순위는 0~14 중 어떤 값이든 SysTick(15)보다 높으므로, EXTI ISR이 실행되는 동안 SysTick은 선점하지 못합니다. 따라서 ISR 내에서 HAL_Delay를 호출하면 uwTick이 절대 증가하지 않아 무한 루프에 빠집니다.
참고: ISR 내에서 HAL_GetTick()은 사용 가능하지만, ISR 실행 중 SysTick이 선점할 수 없으므로 최대 1ms의 오차가 발생할 수 있습니다. 50ms 기준 디바운싱에는 이 오차가 문제가 되지 않습니다.
ISR에서 지연이 필요하면 HAL_GetTick()으로 타임스탬프를 저장하고 메인 루프에서 확인하는 방식을 사용해야 합니다.

}

EXTI 콜백 구현 — 작성 파일 안내

HAL_GPIO_EXTI_Callback 함수는 main.c 파일 하단 또는 별도 gpio_handler.c 파일에 작성합니다. stm32f4xx_it.c에는 작성하지 마세요.
이유: stm32f4xx_it.c는 CubeMX가 재생성(Regenerate)할 때 덮어쓸 수 있습니다. HAL_GPIO_EXTI_Callback은 __weak 오버라이딩 방식이므로, 프로젝트 내 어디에 정의해도 링커가 자동으로 올바른 버전을 선택합니다. 

/* main.c 하단 또는 gpio_handler.c */
/* LOG는 Ch02에서 추가됩니다 */

#include "main.h"

/* 외부에서 선언된 공유 변수 */
extern volatile uint8_t  g_button_event;
extern volatile uint32_t g_last_exti_ms;

/**
 * @brief EXTI 콜백 — HAL이 IRQ 처리 후 자동 호출 (__weak 오버라이딩)
 * @param GPIO_Pin: 인터럽트가 발생한 핀 (비트마스크)
 *
 * !! ISR 황금 법칙 !!
 * 1. HAL_Delay() 사용 금지 — SysTick 대기 시 데드락 위험
 * 2. printf() 등 블로킹 I/O 금지
 * 3. 가능한 짧게 유지 (플래그 세트 후 즉시 리턴)
 */
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin)
{
    if (GPIO_Pin == GPIO_PIN_13)    /* PC13 (B1 버튼) 확인 */
    {
        uint32_t now_ms = HAL_GetTick();

        /* 타임스탬프 디바운싱: 마지막 인터럽트로부터 50ms 경과해야 유효 */
        if ((now_ms - g_last_exti_ms) >= DEBOUNCE_MS)
        {
            g_last_exti_ms = now_ms;    /* 타임스탬프 갱신 */
            g_button_event = 1;         /* 메인 루프에게 이벤트 알림 */
        }
        /* DEBOUNCE_MS 미경과 시: 바운싱으로 판단, 무시 */
    }
}

Step 3. 동작 확인

예상 결과: B1(파란 버튼)을 누를 때마다 LD2(초록 LED)가 켜짐 ↔ 꺼짐으로 토글됩니다. 버튼을 빠르게 여러 번 눌러도 각 누름이 정확히 1회 토글로 인식됩니다.

동작 확인 체크리스트:

  • [ ] 빌드 오류 없음 (Ctrl+B)
  • [ ] 플래시 다운로드 성공 (Run → Debug 또는 Run)
  • [ ] 버튼 1회 누름 → LED 상태 1회 변경
  • [ ] 버튼을 길게 누르고 있어도 LED가 계속 토글되지 않음 (엣지 트리거 확인)
  • [ ] 버튼을 매우 빠르게 연속 클릭 시 각 클릭이 정확히 인식됨 (디바운싱 확인)
  • [ ] 아무 버튼도 안 눌렀을 때 LED 상태 유지 (오동작 없음)

Step 4. 폴링 버전과 비교

참고용으로 폴링 방식의 메인 루프로 교체하여 동작을 비교해 보세요. 버튼 누르는 동안 LED가 켜지는 방식(레벨 감지)과 누름 순간에만 토글(엣지 감지)의 차이를 체험합니다. 그리고 HAL_Delay(1000)을 메인 루프에 추가했을 때 폴링은 버튼 반응이 느려지지만 인터럽트 방식은 영향을 받지 않는 것도 확인하세요.

💡 실습이 안 될 때 — 흔한 실수 목록
1. LED가 전혀 켜지지 않음: __HAL_RCC_GPIOA_CLK_ENABLE()이 호출되었는지 확인. MX_GPIO_Init()이 main()에서 호출되는지 확인.
2. 버튼 누름에 반응 없음: NVIC에서 EXTI15_10 인터럽트가 Enable되었는지 확인. HAL_NVIC_EnableIRQ(EXTI15_10_IRQn) 호출 확인.
3. 버튼 1회 누름에 LED가 여러 번 토글: 디바운싱 시간을 늘려보세요(50ms → 100ms). DEBOUNCE_MS 정의가 올바른지 확인.
4. 콜백이 호출되지 않음: HAL_GPIO_EXTI_Callback 함수명이 정확한지 확인(오타 주의). __weak 함수 오버라이딩은 이름이 완전히 일치해야 합니다.
5. g_button_event가 항상 0: volatile 키워드가 없으면 컴파일러 최적화로 변경 사항이 무시될 수 있습니다.
📌 강사 꿀팁
실습 중 가장 자주 발생하는 실수는 ① NVIC Enable 누락, ② volatile 누락, ③ DEBOUNCE_MS 미선언입니다. 실습 시작 전 "이 세 가지가 빠지면 동작 안 합니다"라고 미리 안내하면 실습 시간이 단축됩니다. 또한 extern volatile 선언을 별도 헤더에 넣는 구조를 보여주면 향후 멀티파일 구조 학습의 기초가 됩니다.

핵심 정리

GPIO 출력 모드

  • Push-Pull: 내부 PMOS+NMOS로 HIGH/LOW 양방향 능동 구동. LED, 일반 디지털 출력에 사용.
  • Open-Drain: 내부 NMOS만으로 LOW만 능동 구동. HIGH는 외부 풀업 필요. I2C, 레벨 시프팅에 사용.

GPIO 초기화

  • 사용 전 반드시 __HAL_RCC_GPIOx_CLK_ENABLE()로 클록 활성화 필요.
  • GPIO_InitTypeDef로 Pin(비트마스크), Mode, Pull, Speed를 설정 후 HAL_GPIO_Init() 호출.
  • GPIO_PIN_13은 비트마스크 0x2000이며 리터럴 13과 전혀 다름.

버튼 입력

  • NUCLEO B1(PC13): 액티브 로우 — 버튼 누름 시 GPIO_PIN_RESET.
  • 폴링: 메인 루프에서 주기적 확인. 구현 단순, 반응 지연 가능.
  • 디바운싱: 기계식 버튼의 접점 진동을 소프트웨어로 제거. 타임스탬프(HAL_GetTick()) 방식 권장.

EXTI 인터럽트

  • EXTI 라인 번호 = GPIO 핀 번호 (PA5, PB5 모두 EXTI5). 같은 번호는 동시 사용 불가.
  • PC13 → EXTI13 → EXTI15_10_IRQn (10~15번 라인 공유 핸들러).
  • 엣지 트리거: 버튼 누름(액티브 로우) = Falling Edge.
  • ISR 황금 법칙: HAL_Delay() 금지, 블로킹 I/O 금지, 가능한 짧게 유지.
  • 공유 변수는 반드시 volatile 선언.

NVIC

  • PreemptPriority: 선점 우선순위 (숫자 작을수록 높음, 0이 최고).
  • SubPriority: 같은 PreemptPriority 내 처리 순서 (선점 불가).
  • CubeMX 기본: NVIC_PRIORITYGROUP_4 (PreemptPriority 0~15, SubPriority 없음).

프로젝트 v0.1 완성 상태

  • B1 버튼 누름 감지: EXTI 인터럽트 + 50ms 타임스탬프 디바운싱
  • LD2 LED 토글: HAL_GPIO_TogglePin()
  • 메인 루프는 이벤트 플래그 확인만 — CPU 낭비 없음

연습문제

기억 1. Open-Drain 출력 모드로 설정된 핀에서 HAL_GPIO_WritePin(..., GPIO_PIN_SET)을 호출했을 때 핀의 전기적 상태로 올바른 것은?
  1. MCU 내부 PMOS가 켜져 핀이 3.3V로 강하게 구동된다.
  2. 내부 NMOS가 꺼져 핀이 외부 풀업 저항에 의해 VCC 수준이 된다.
  3. 내부 NMOS가 켜져 핀이 GND로 강하게 구동된다.
  4. 핀이 floating(부동) 상태가 되어 무작위 전압이 나타난다.
정답: (b) — Open-Drain에서 HIGH 설정은 NMOS를 끄는 것이며, 핀 전압은 외부 풀업 저항에 의해 결정됩니다.
