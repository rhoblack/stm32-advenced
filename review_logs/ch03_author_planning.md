## 1. 아키텍처 4계층 설명: HAL → Driver → Service → App

### 각 계층의 책임과 경계

| 계층 | 색상 | 책임 | 경계 규칙 |
|------|------|------|----------|
| **HAL (Hardware Abstraction Layer)** | #1A73E8 파랑 | ST 제공 라이브러리. 레지스터 직접 접근을 추상화하여 `HAL_GPIO_WritePin()` 같은 표준 API 제공 | 우리가 **수정하지 않는** 계층. CubeMX가 생성하고 ST가 유지보수 |
| **Driver** | #34A853 초록 | 특정 하드웨어 주변장치를 HAL 위에서 감싸는 모듈. 하드웨어 의존적이지만 비즈니스 로직 없음 | HAL만 호출. 상위 계층에 **하드웨어 독립적 인터페이스** 노출 |
| **Service** | #FBBC04 노랑 | 여러 Driver를 조합하여 의미 있는 기능 단위를 구성. 비즈니스 로직 포함 | Driver만 호출 (HAL 직접 호출 금지). 하드웨어를 모름 |
| **App** | #EA4335 빨강 | 최상위 애플리케이션 로직. main 루프, UI, CLI 등 사용자 상호작용 | Service만 호출. 시스템 진입점 |

### 핵심 원칙
- **단방향 의존**: App → Service → Driver → HAL (역방향 호출 절대 금지)
- **계층 건너뛰기 금지**: App이 HAL을 직접 호출하면 안 됨
- **콜백 예외**: 인터럽트 콜백은 HAL → Driver 방향이지만, 이는 **이벤트 통지**이므로 의존성 역전이 아님 (함수 포인터/weak 함수 패턴)

## 2. Ch01~02 코드 매핑

### 현재 v0.2 코드의 계층 배치

```
┌─────────────────────────────────────────┐
│ App 레이어 (빨강)                         │
│   main() — 초기화 순서 + 메인 루프        │
│   HAL_GPIO_EXTI_Callback() — 이벤트 처리  │
└─────────────┬───────────────────────────┘
              │ (현재는 Driver/Service 없이 직접 HAL 호출)
              ▼
┌─────────────────────────────────────────┐
│ Service 레이어 (노랑) — 아직 없음          │
│   (Ch10부터 clock_service 등 등장 예정)    │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│ Driver 레이어 (초록)                      │
│   log_print() — 로그 출력 드라이버        │
│   LOG_E/W/I/D 매크로 — 로그 인터페이스    │
│   (GPIO Driver는 아직 분리 안 됨)         │
└─────────────┬───────────────────────────┘
              ▼
┌─────────────────────────────────────────┐
│ HAL 레이어 (파랑) — ST 제공               │
│   HAL_GPIO_Init/WritePin/TogglePin      │
│   HAL_NVIC_SetPriority/EnableIRQ        │
│   HAL_Init, HAL_Delay, HAL_GetTick      │
│   SWO/ITM 하드웨어 접근                   │
└─────────────────────────────────────────┘
```

### 매핑 상세

| 현재 코드 요소 | 출처 | 현재 위치 | 이상적 위치 | 비고 |
|---------------|------|----------|------------|------|
| `HAL_GPIO_WritePin()` | ST HAL | HAL | HAL | 변경 없음 |
| `MX_GPIO_Init()` | CubeMX | HAL/Driver 경계 | Driver | CubeMX 생성 코드는 Driver 초기화로 볼 수 있음 |
| `HAL_GPIO_EXTI_Callback()` | 사용자 구현 | App | App (이벤트 핸들러) | 향후 Driver로 1차 수신 후 App에 전달하는 구조로 발전 |
| `log_print()` | ch02_06 | Driver | Driver | 횡단 관심사(Cross-cutting) — 모든 계층에서 사용 |
| `LOG_E/W/I/D` 매크로 | ch02_04 | 인터페이스 | 인터페이스 | 계층 무관 유틸리티 |
| `main()` | 사용자 구현 | App | App | 초기화 오케스트레이션 + 메인 루프 |
| `SystemClock_Config()` | CubeMX | HAL 설정 | HAL 설정 | 시스템 레벨, 계층 외부 |

### 현재 아키텍처의 문제점 (의도적)
- `main()`에서 `HAL_GPIO_WritePin()`을 직접 호출 → App이 HAL을 직접 참조
- GPIO Driver 모듈이 분리되지 않음 → Ch04부터 점진적으로 분리
- 이 문제점을 Ch03에서 **인식**하고, 향후 챕터에서 **해결**하는 것이 학습 포인트

## 4. 기술 용어 비유

### "레이어드 아키텍처" → 사무실 조직도 비유

> **회사 조직도로 이해하는 소프트웨어 레이어**
>
> 소프트웨어 아키텍처의 4계층은 회사의 조직 구조와 같습니다.
>
> | 계층 | 회사 비유 | 설명 |
> |------|----------|------|
> | **App** | **CEO / 경영진** | "무엇을 할지" 결정. 세부 실행은 모름 |
> | **Service** | **팀장 / 중간관리자** | 여러 실무자를 조합해서 업무 완수 |
> | **Driver** | **실무 담당자** | 특정 도구(하드웨어)를 다루는 전문가 |
> | **HAL** | **도구 / 장비** | 실제로 일을 수행하는 물리적 도구 |
>
> CEO가 복사기를 직접 고치지 않듯이, App 레이어는 HAL을 직접 호출하지 않습니다.
>
> **비유의 한계**: 회사에서는 CEO가 실무자에게 직접 지시할 수 있지만, 소프트웨어에서는 계층 건너뛰기가 엄격히 금지됩니다.

### "FSM(유한 상태 머신)" → 자판기 비유

> 자판기는 대표적인 FSM입니다.
> - **상태**: 대기 중, 금액 투입됨, 음료 선택됨, 배출 중
> - **이벤트**: 동전 투입, 버튼 누름, 음료 배출 완료
> - **전이**: "대기 중" + "동전 투입" → "금액 투입됨"
> - **액션**: 잔액 표시, 음료 배출, 거스름돈 반환

### "인터페이스" → 리모컨 비유

> TV 리모컨의 버튼은 인터페이스입니다.
> 사용자는 "전원", "볼륨+", "채널" 버튼만 알면 됩니다.
> TV 내부 회로가 어떻게 동작하는지 몰라도 됩니다.
> Driver가 상위 계층에 노출하는 함수 시그니처가 바로 이 "리모컨 버튼"입니다.

## 5. 실무 코드 예제: 4계층 아키텍처 함수 시그니처

### 현재 v0.2의 이상적 구조 (의사코드)

```c
/* ============================================
 * HAL 레이어 (파랑) — ST 제공, 수정하지 않음
 * ============================================ */
// stm32f4xx_hal_gpio.h (ST 제공)
HAL_StatusTypeDef HAL_GPIO_Init(GPIO_TypeDef *GPIOx, GPIO_InitTypeDef *GPIO_Init);
void HAL_GPIO_WritePin(GPIO_TypeDef *GPIOx, uint16_t GPIO_Pin, GPIO_PinState PinState);
void HAL_GPIO_TogglePin(GPIO_TypeDef *GPIOx, uint16_t GPIO_Pin);

/* ============================================
 * Driver 레이어 (초록) — 하드웨어 캡슐화
 * ============================================ */
// led_driver.h — LED 하드웨어를 추상화
typedef enum { LED_OFF = 0, LED_ON = 1 } led_state_t;

void led_driver_init(void);              /* GPIO 초기화 캡슐화 */
void led_driver_set(led_state_t state);  /* HAL_GPIO_WritePin 캡슐화 */
void led_driver_toggle(void);            /* HAL_GPIO_TogglePin 캡슐화 */

// button_driver.h — 버튼 하드웨어를 추상화 (디바운싱 포함)
typedef void (*button_callback_t)(void);

void button_driver_init(button_callback_t on_press);  /* EXTI + 디바운싱 */
uint32_t button_driver_get_press_count(void);

// logger.h — 로그 출력 (횡단 관심사)
void log_print(int level, const char *tag, ...);
// LOG_E / LOG_W / LOG_I / LOG_D 매크로

/* ============================================
 * Service 레이어 (노랑) — 아직 없음 (Ch10~부터)
 * ============================================ */
// 향후 예시:
// void clock_service_update(void);   /* RTC + 스텝모터 조합 */
// void sensor_service_read(void);    /* I2C + SHT31 조합 */

/* ============================================
 * App 레이어 (빨강) — 애플리케이션 로직
 * ============================================ */
// main.c
int main(void)
{
    /* 초기화 — 하위 계층부터 상위 계층 순서 */
    HAL_Init();                  /* HAL */
    SystemClock_Config();        /* HAL */
    led_driver_init();           /* Driver */
    button_driver_init(&on_button_press);  /* Driver */

    LOG_I("=== v0.3 시스템 시작 ===");

    while (1)
    {
        /* App 로직: Driver/Service API만 호출 */
    }
}

/* App 레이어의 이벤트 핸들러 */
static void on_button_press(void)
{
    led_driver_toggle();  /* Driver API 호출 (HAL 직접 호출 X) */
    LOG_I("버튼 → LED 토글");
}
```

## 6. 집필 시 주의사항

### Ch03 특수성
- Ch03은 **코드보다 개념 중심** 챕터이지만, "코드 없는 이론 설명" 금지 원칙 준수 필요
- 해결: 매 절마다 Ch01~02 코드를 예시로 활용하여 이론을 구체화
- FSM 절에서 버튼-LED를 FSM으로 재작성하는 코드 예제 포함

### Ch04 이후와의 연결
- Ch03에서 작성한 아키텍처 다이어그램이 Ch04부터 **매 챕터 도입부에 갱신**되는 "살아있는 문서"
- v0.3 아키텍처 다이어그램은 빈 칸이 많아야 함 → 학습자가 "앞으로 채워갈 것"에 대한 기대감 유발

### 감정 곡선 설계
1. **3.1**: "내 코드가 스파게티였다고?" → 위기감 + 호기심
2. **3.2**: "아, 이렇게 나누면 되는구나" → 체계적 이해
3. **3.3**: "내가 쓴 코드를 직접 분류해보니 재미있다" → 참여감
4. **3.4**: "FSM이 어렵게 느껴지는데..." → 약간의 불안
5. **3.5**: "다이어그램으로 그리니까 명확하다" → 도구 습득 성취감
6. **3.6**: "앞으로 이 그림에 하나씩 추가하는 거구나!" → 기대감
