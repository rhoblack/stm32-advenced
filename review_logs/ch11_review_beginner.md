### §11.1 I2C 프로토콜 개요 — ⭐⭐⭐⭐ (이해 가능)
- UART vs I2C 비교 명확 (2핀 vs TX/RX)
- 오픈드레인 개념: "풀업 저항" 설명 필요

  **문제점**: 오픈드레인이라는 용어만으로 이해하기 어려움.

  **현재 텍스트**:
  > "LOW를 만들 때만 능동적으로 드라이브합니다."

  **독자 입장**: "그럼 HIGH는 어떻게 만드는가?" → **풀업 저항이 없으면 설명 부족**

  **권장 개선**:
  ```
  오픈드레인은 일종의 "협력 시스템"입니다.
  마스터와 슬레이브가 동시에 같은 핀을 "잡아당길(pull)" 수 있는데,
  아무도 "밀어올리지(push)" 않으므로, 풀업 저항(보통 4.7kΩ)이
  자동으로 HIGH로 유지합니다.

  비유: 줄다리기에서 아무도 "끌지" 않으면 자동으로 HIGH 위치로 돌아감.
  ```

- START/STOP/ACK/NACK 시퀀스: **타이밍 다이어그램(SVG) 필수**
  - 현재 텍스트만으로는 "START는 SCL HIGH일 때 SDA HIGH→LOW"의 의미 모호
  - ch11_sec01_i2c_timing.svg 있으니 HTML에서 충분히 참조하는지 확인 필요

- **발견**: 리스트 아이템 6번 문제
  > "STOP 조건: SCL이 HIGH인 상태에서 SDA를 LOW → HIGH로 전환"

  읽는 순간 "어라, START와 반대네?" 인식 → 정확하나 설명 충분함.

### §11.2 클록 스트레칭 — ⭐⭐⭐⭐⭐ (탁월)
- "택시 신호등" 비유 훌륭함
- SCL/SDA "둘 다 놓아야 녹색" 개념이 오픈드레인을 보완적으로 설명
- 50ms 대기 구체적 수치 제시로 이해도 ↑
- 타임아웃 개념 자연스럽게 도입

### §11.4 SHT31 센서 규격 — ⭐⭐⭐ (기본 이해, 일부 어려움)
- **强점**: SHT31_ADDR=0x44, 측정 명령=0x2C 0x06 구체적
- **문제**: "응답 프레임" 설명이 추상적

  **현재**:
  > "[Temp_H, Temp_L, Temp_CRC, Humid_H, Humid_L, Humid_CRC] (6바이트)"

  **독자 입장**: "응답이 6바이트면, 마스터가 6바이트를 읽는 건가?" → 명확함

  그런데 **CRC 검증 방법**이 바로 이어지지 않음 (§11.5에서 CRC 소개)

  **권장**:
  ```
  응답 프레임은 온도(2바이트) + CRC(1바이트) + 습도(2바이트) + CRC(1바이트)로 구성됩니다.
  각 센서값 뒤에 CRC가 붙어서, 데이터 손상을 감지할 수 있습니다.
  (다음 섹션에서 CRC 검증법을 배웁니다)
  ```

- **온도/습도 계산식**:
  - 공식이 수학적으로 제시됨: "175.0f / 65535.0f * raw - 45.0f"
  - **의미 설명 부족**: "왜 175.0? 왜 45.0?" → 데이터시트 참조하라는 설명 필요

  **권장**:
  ```
  계산식은 SHT31 데이터시트에서 정의됩니다.
  온도: (175 × raw ÷ 65535) - 45
  → 예: raw=30000 → 온도 ≈ 32.9°C

  (자세한 유도는 데이터시트 참조)
  ```

### §11.5 CRC-8 검증 — ⭐⭐ (어려움, 필수 개선)
**가장 큰 문제점**.

- **현재 상황**:
  - §11.5 도입: "CRC(Cyclic Redundancy Check) — 데이터 무결성 검증"
  - 체크섬 vs CRC 차이 설명은 좋음
  - 그러나 **"다항식 0x31"이 뭔지 설명 없음**

- **독자 입장**:
  > "다항식? 뭐지? 이게 왜 필요한가?"

- **문제**: CRC 알고리즘 이해도 ↓↓↓
  ```c
  for (uint8_t j = 0; j < 8; j++) {
      if (crc & 0x80) {
          crc = (crc << 1) ^ 0x31;  // ← 이 0x31이 다항식?
      }
  }
  ```
  이 코드만으로는 **CRC가 어떻게 동작하는지 알 수 없음**.

- **권장 개선**:
  1. **개념 도입**:
     ```
     CRC-8은 "오류 감지 코드"입니다.
     센서가 6바이트 데이터를 보낼 때, 각 3바이트 단위로 CRC를 붙입니다.

     수신자는:
     1. 처음 2바이트로 CRC를 계산
     2. 센서가 보낸 CRC(3번째 바이트)와 비교
     3. 일치 → 데이터 정상, 불일치 → 데이터 손상
     ```

  2. **비유 추가**:
     ```
     비유: 시험지 답안지의 바코드처럼, CRC는 데이터의 "지문(fingerprint)"입니다.
     지문이 맞으면 "이건 내가 보낸 데이터" 확인할 수 있습니다.
     ```

  3. **다항식 설명** (옵션, 상자로 분리):
     ```
     [심화] 다항식 0x31이 뭔가요?
     - CRC 계산의 "마법의 숫자"로 생각하세요.
     - Sensirion(SHT31 제조사)이 정한 표준입니다.
     - 다항식이 달라지면 CRC 값도 달라집니다.
     - (수학적 유도는 CRC 전문서 참조)
     ```

  4. **테스트 예시** 추가:
     ```
     예: 온도 데이터 0xBE 0x00을 보내면,
     CRC 계산 결과는 0xAC입니다.
     센서가 0xAC를 함께 보내므로, 우리가 계산해서 비교하면 되는 것입니다.
     ```

## 개선이 필요한 부분 (구체적)

### 1️⃣ 오픈드레인(Open-Drain) 설명 강화
**현재** (§11.1):
> "물리적으로는 풀업 저항(pull-up resistor, 보통 4.7kΩ)을 통해 HIGH 상태를 유지"

**문제**: "HIGH 유지"만 있고, "LOW는 어떻게?"가 없음.

**개선**:
```
오픈드레인(Open-Drain):
- 핀을 "켜기(HIGH)" → 풀업 저항이 자동 처리
- 핀을 "끄기(LOW)" → 트랜지스터로 GND에 연결
- 마스터, 슬레이브 누구나 "끌 수 있음" = 협력 구조
```

### 2️⃣ CRC-8 섹션 재구성
**현재**: 알고리즘 코드만 제시
**개선**:
1. 개념 (1단락)
2. 비유 (1단락)
3. 동작 흐름 (도표 또는 다이어그램)
4. 코드 (3단계로 분해)
5. 테스트 예시

### 3️⃣ 섹션 연결 개선
- §11.3 끝: "다음 예제에서 HAL API를 사용하겠습니다" → §11.4로 명시적 연결

### 4️⃣ 용어 사전 추가
- 오픈드레인, 클럭 스트레칭, CRC, 다항식 등
- 각 용어 첫 등장 시 괄호로 한글 설명 필수

## 강점 (👍)

1. **비유가 효과적**
   - "택시 신호등" (클럭 스트레칭)
   - "교사와 학생 손 올리기" (동기식 통신)

2. **계층 분리가 명확**
   - Driver → Device Driver → Service 구조가 직관적

3. **Ch09 패턴 재활용**
   - FSM 구조가 같으므로, 새로 배울 것 감소
   - "이미 봤던 패턴이네" 인식 → 학습 불안 ↓

4. **코드 예제가 30줄 이하**
   - 따라 읽기 용이

5. **한국어 주석 충분**
   - 변수명, 로그 메시지 모두 명확
