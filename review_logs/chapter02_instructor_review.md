## 종합 강의 적합성 평가: ⭐⭐⭐⭐☆

전반적으로 강의 현장에서 바로 사용할 수 있는 수준의 원고다. SWO/ITM → RTT → 로그 레벨 설계 → 매크로 구현 → 컴파일 타임 필터링 → 통합의 흐름이 논리적이고, 현장 강사가 칠판/화면 전환 없이도 설명 순서를 따라가기 편하다. 강사 꿀팁 박스가 실제로 현장 유용성이 높은 내용을 담고 있어 높이 평가한다. 다만 **§2.2 RTT 설치 절차의 모호함**과 **4시간 시간 배분 명시 부재**, **ISR 내 LOG 경고의 불완전한 설명** 등이 현장 혼란을 유발할 수 있어 개선이 필요하다.

## Major 이슈

### [Major-01] §2.2 RTT 라이브러리 설치 절차 미설명

**위치**: §2.2 "NUCLEO-F411RE에서 RTT Viewer 연결" 섹션

**문제**: 원고는 `SEGGER_RTT.c`, `SEGGER_RTT.h`, `SEGGER_RTT_Conf.h`를 "프로젝트에 추가하라"고만 언급하고, 실제로 어디서 파일을 구하는지, CubeIDE에서 어떻게 추가하는지 절차가 없다. 강의 현장에서 수강생 절반이 "파일 어디서 받나요?" 질문을 하게 된다.

**현장 경험**: SEGGER J-Link 드라이버 설치 여부, Ozone 설치 여부에 따라 RTT 라이브러리 위치가 다르다. 초보자는 SEGGER 공식 사이트에서 직접 다운로드해야 한다는 사실조차 모른다.

**개선 방향**:
- SEGGER 공식 사이트 다운로드 경로 또는 J-Link 소프트웨어 패키지 내 위치 명시
- CubeIDE 프로젝트에 `.c/.h` 파일 추가 2-3단계 절차 추가 (또는 §2.6 실습에 통합)
- "이 챕터에서 RTT는 개념 이해 목적이며, 실습은 SWO만 진행한다"는 명시적 안내 필요

### [Major-02] §2.1 ISR 내 ITM_SendChar() 경고가 불완전

**위치**: §2.1 강사 꿀팁 박스 "ISR 내부에서 ITM_SendChar() 호출 주의"

**문제**: "꼭 필요하다면 LOG_D 대신 LOG_I 수준으로만 사용하세요"라는 조언이 혼란을 유발한다. LOG_D와 LOG_I의 차이는 **출력되는 정보의 양이 다를 뿐**, ISR 내 busy-wait 문제와는 무관하다. ITM_SendChar()는 레벨에 관계없이 동일하게 FIFO를 기다린다.

**현장 경험**: 이 내용을 그대로 읽은 수강생이 "LOG_I는 ISR에서 써도 안전하다"고 오해하는 사례를 실제로 목격했다. ISR에서는 레벨 불문하고 로그 출력 자체를 최소화해야 한다는 원칙을 명확히 해야 한다.

**개선 방향**:
"꼭 필요하다면 LOG_D 대신 LOG_I 수준으로만 사용하세요" → "꼭 필요하다면 로그 호출 횟수 자체를 최소화하세요. 레벨과 무관하게 ITM_SendChar()는 FIFO busy-wait을 발생시킵니다."로 수정 권장

---

### [Major-03] 예상 로그 출력에서 함수명 컬럼이 실제와 다름

**위치**: §실습 Step 4 "예상 로그 출력" 코드 블록

**문제**:
```
[  1523][INFO ][main.c:38][HAL_GPIO_EX ] 버튼 클릭 감지 @ 1523 ms
```
`__func__` 매크로는 `"HAL_GPIO_EXTI_Callback"` (전체 함수명)을 반환한다. `%-12s` 포맷이 12자로 잘라주는 것이 아니라 **최소 12자 좌정렬 패딩**을 의미하므로, 함수명이 12자보다 길면 전체가 출력된다. `HAL_GPIO_EXTI_Callback`은 22자이므로 실제 출력은 `HAL_GPIO_EXTI_Callback`으로 나온다.

**현장 경험**: 수강생이 예상 출력과 실제 출력이 다르면 "내가 뭔가 잘못 설정했나?" 혼란을 겪는다. 예제 출력은 실제 실행 결과와 정확히 일치해야 한다.

**개선 방향**: 예상 출력 예시를 `[HAL_GPIO_EXTI_Callback]`으로 수정하거나, logger.c의 `%-12s`를 `%.12s`(최대 12자 잘림)로 변경하고 그 의미를 설명하는 팁을 추가할 것.

## Minor 이슈

### [Minor-01] §2.1 콜 체인 설명에서 fwrite() 포함 여부 불확실

**위치**: §2.1 "_write() 오버라이드" 섹션 콜 체인 설명

**내용**: `printf() → vfprintf() → fwrite() → _write()` 콜 체인이 제시되어 있다. Newlib 구현에 따라 `fwrite()`를 거치지 않고 직접 `_write()`를 호출하는 경로도 있다. 현장에서 "fwrite가 뭐예요?"라는 질문이 반드시 나온다.

**제안**: 콜 체인을 `printf() → (Newlib 내부 처리) → _write() → ITM_SendChar()`로 단순화하거나, `fwrite()` 경유에 대해 "Newlib 구현 세부사항이며 컴파일러 버전에 따라 다를 수 있음" 주석 추가 권장.

### [Minor-03] §2.5 Makefile 예제의 플래그 형식 오류

**위치**: §2.5 "CubeIDE에서 LOG_LEVEL 설정하기" 섹션

**내용**:
```
/* make -DLOG_LEVEL=3 all      → INFO 레벨 빌드 */
```
GCC make의 전처리기 정의 전달은 `CFLAGS=-DLOG_LEVEL=3`이나 `EXTRA_CFLAGS`를 통해서 하는 것이 일반적이다. `make -DLOG_LEVEL=3`은 make의 `-D` 옵션(Makefile 변수 정의)으로 GCC 전처리기에 직접 전달되지 않는다. CubeMX 생성 Makefile에서는 `C_DEFS` 변수를 수정해야 한다.

**제안**: 해당 예제를 삭제하거나, CubeMX Makefile의 `C_DEFS` 수정 방법으로 교체. 또는 "정확한 방법은 프로젝트 Makefile 구조에 따라 다르므로 CubeIDE GUI 방식을 권장한다"는 안내 추가.

### [Minor-04] §2.2 RTT 섹션에 ST-LINK와 J-Link 혼용 시 RTT Viewer 연결 제약 미언급

**위치**: §2.2 "NUCLEO-F411RE에서 RTT Viewer 연결" 섹션

**내용**: NUCLEO 내장 ST-LINK v2를 SEGGER RTT Viewer에서 사용하려면 SEGGER가 제공하는 ST-LINK 지원 드라이버 설치가 별도로 필요하며, 모든 ST-LINK 펌웨어 버전에서 지원되지 않는다. "Connection: ST-LINK 선택"이라고 간단히 언급하고 넘어가기엔 현장 트러블슈팅 빈도가 높다.

**제안**: "ST-LINK를 RTT Viewer에 연결할 때는 SEGGER J-Link DLL for STLink 드라이버가 필요하며, 연결이 안 되면 ST-LINK를 J-Link 펌웨어로 업그레이드하는 방법도 있다(NUCLEO 한정, 무료)"는 한 줄 안내 추가.

**위치**: 연습문제 4번

## 수강생 Q&A 보완 제안

원고에 포함된 Q&A는 실제 현장에서 나오는 질문들이며 답변도 정확하다. 추가로 현장에서 자주 나오지만 원고에 없는 질문들을 보완할 것을 권장한다.

### 추가 권장 Q&A

**Q1. printf()를 쓰면 느리지 않나요? ISR에서도 써도 되나요?**
> A: printf() → _write() → ITM_SendChar()는 각 문자를 동기식으로 전송합니다. 긴 메시지를 ISR에서 출력하면 인터럽트 레이턴시가 늘어납니다. ISR 안에서는 플래그만 세우고, 메인 루프에서 출력하는 패턴을 권장합니다.
(현재 §2.1 팁 박스에 부분적으로 있으나 Q&A 형식으로도 보완 권장)

**Q2. logger.c의 `char buf[256]`이 스택에 할당되는데, 스택 오버플로우 위험은 없나요?**
> A: 256바이트 스택 할당은 Cortex-M4(일반적 스택 4~8KB)에서 문제없습니다. 단, 재귀 호출이나 ISR 중첩이 깊은 환경에서는 주의가 필요합니다. 이 이슈는 Ch10 스택 분석 챕터에서 다룹니다.

**Q3. Release 빌드에서 LOG_LEVEL=0으로 설정하면 logger.c의 log_print() 함수도 바이너리에서 사라지나요?**
> A: 매크로가 `((void)0)`으로 치환되면 log_print()를 호출하는 코드가 없어지고, 링커가 미참조 함수를 제거(--gc-sections 옵션)합니다. 단, `--gc-sections`가 활성화되어 있어야 합니다. CubeIDE 기본 설정은 활성화되어 있습니다.

## 면접 포인트 평가

원고에 포함된 두 개의 면접 포인트(`SWD와 SWO의 차이`, `가변 인자 매크로 경험`)를 현장 관점에서 평가한다.

### 면접 포인트 1 — "SWD와 SWO의 차이" (§2.2)
**현장 적합성**: ⭐⭐⭐⭐⭐ (매우 적합)
실제로 STM32 기반 임베디드 면접에서 빈출 질문이다. 제시된 모범 답안은 ITM/ETM 구조, 핀 수, 방향성, RTT와의 관계까지 포함하여 수준 높은 답변이다. 그대로 암기해도 충분히 실전에서 통한다.

### 면접 포인트 2 — "가변 인자 매크로 경험" (§2.4)
**현장 적합성**: ⭐⭐⭐⭐☆ (적합)
`##__VA_ARGS__`, `__FILE__`/`__LINE__`/`__func__`, 컴파일 타임 필터링 세 가지 키워드를 포함한 답변 구조가 좋다. 다만 "IAR이나 MSVC 호환성 문제를 알고 있는가" 질문이 이어서 나올 수 있으므로, `__VA_OPT__` 대안 언급이 이미 원고 팁 박스에 있어 연결이 자연스럽다.

### 추가 권장 면접 포인트

**"컴파일 타임 최적화와 런타임 최적화의 차이를 예를 들어 설명하세요"** — §2.5 내용이 완벽한 답변 재료다. 현장에서 빠르게 성장하는 주니어와 그렇지 않은 주니어를 가르는 질문이므로, 해당 섹션에 면접 포인트 박스를 하나 추가하는 것을 강력 권장한다.

**"배포 펌웨어에서 디버그 정보를 남기는 것의 보안 위험은?"** — §2.5 스스로 점검 항목에 언급되어 있으나 면접 포인트로 승격할 만하다.

## 잘 된 부분

1. **단계별 매크로 빌드업(§2.4)**: 1단계 → 2단계 → 3단계로 점진적으로 복잡도를 높이는 방식이 현장에서 매우 효과적이다. 완성 코드만 보여주고 "이해했죠?"라고 넘어가는 교재와 달리, 각 단계에서 무엇이 왜 추가되는지 명확하다.

2. **Before/After 코드 비교(§2.6)**: Ch01 코드에 로그를 추가하는 비교가 직관적이다. "코드가 달라진 이유"를 설명하지 않아도 수강생이 스스로 파악하게 유도하는 구조다.

3. **실습 트러블슈팅 체크리스트(§2.6 실습 팁)**: "로그가 전혀 안 나오는 경우", "글자가 깨지는 경우", "빌드 에러" 세 가지로 분류한 체크리스트가 현장에서 바로 활용 가능하다. SB12 점퍼부터 Core Clock까지 빠짐없이 포함되어 있다.

4. **§2.1 강사 꿀팁 박스 — 칠판 콜 체인 작도**: `printf → _write → ITM_SendChar` 콜 체인을 칠판에 먼저 그려두라는 제안이 매우 현실적이다. 필자가 실제 강의 경험에서 터득한 기법임이 느껴진다.

5. **컴파일 타임 필터링의 .map 파일 검증법(§2.5)**: "정말 플래시가 줄어드나?"를 직접 확인하는 방법을 제시함으로써 "믿으세요" 수준의 설명을 탈피했다. 수강생 신뢰도를 높이는 중요한 요소다.

6. **Ch01 연결 — "눈이 없습니다" 도입부**: §2.1 시작에서 "LED가 켜지면 성공이라고 했지만, 정말 한 번만 발생했을까요?"라는 질문이 수강생의 ch01 경험을 즉시 불러일으킨다. 챕터 전환 동기부여가 자연스럽다.

7. **다음 챕터 예고 — v0.3 예고 방식**: "이제 main.c 한 파일에 모든 코드가 쌓여가고..."라는 문제 제기로 Ch03 필요성을 자연스럽게 연결한다. 수강생이 스스로 "아, 정리가 필요하겠다"는 필요성을 느끼도록 유도하는 서술이 좋다.
