## Sub-agent 1: 기술 저자 관점

### MISRA-C 핵심 규칙 10선 (임베디드 STM32 실무 기준)

| 번호 | 규칙 | 카테고리 | STM32 적용 예 |
|------|------|---------|--------------|
| R1 | 정수 연산 결과는 명시적 캐스팅으로 타입 보장 | 필수 | `(uint16_t)(adc_val * 3300U / 4095U)` |
| R2 | `volatile` 한정자 — 공유 변수 필수 선언 | 필수 | ISR과 main 공유 플래그 |
| R3 | 함수 반환값 반드시 확인 | 필수 | `HAL_StatusTypeDef ret = HAL_UART_Transmit(...)` |
| R4 | 암시적 타입 변환 금지 — 부호 있는/없는 혼용 주의 | 필수 | `int8_t`와 `uint8_t` 연산 |
| R5 | 동적 메모리 할당 금지 (임베디드 환경) | 필수 | `malloc`/`free` 대신 정적 풀 |
| R6 | 매크로 인수는 괄호로 완전 감싸기 | 권고 | `#define MAX(a,b) ((a)>(b)?(a):(b))` |
| R7 | switch-case 마다 break/return/throw 명시 | 필수 | FSM 구현 시 fall-through 방지 |
| R8 | 단일 파일 내 함수 최대 60줄 (SRP 준수) | 권고 | 큰 초기화 함수 분리 |
| R9 | 재귀 함수 금지 (스택 오버플로우 방지) | 필수 | 트리 순회 → 반복문으로 대체 |
| R10 | 초기화되지 않은 변수 사용 금지 | 필수 | 선언 즉시 초기값 대입 |

### 리팩토링 Before/After 설계

**사례 1: 레이어 위반 — App이 HAL 직접 호출**
```c
// BEFORE: app_main.c에서 HAL 직접 호출 (레이어 위반)
HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, GPIO_PIN_SET); // LED 직접 제어

// AFTER: Driver 레이어를 통해 호출 (레이어 준수)
LED_SetState(LED_GREEN, LED_ON); // led_driver.h 인터페이스 사용
```

**사례 2: volatile 누락으로 인한 버그**
```c
// BEFORE: volatile 누락 — 컴파일러 최적화로 플래그 읽기 생략 가능
uint8_t dma_done = 0;  // ISR에서 설정하는 플래그
while (dma_done == 0) {}  // 컴파일러가 루프 제거 가능

// AFTER: volatile 선언으로 매번 메모리 참조 강제
volatile uint8_t dma_done = 0;
while (dma_done == 0) {}  // 올바르게 동작
```

**사례 3: HAL 반환값 무시**
```c
// BEFORE: 오류 무시
HAL_UART_Transmit(&huart2, buf, len, 100);

// AFTER: 오류 처리
HAL_StatusTypeDef ret = HAL_UART_Transmit(&huart2, buf, len, 100);
if (ret != HAL_OK) {
    LOG_E("UART 전송 실패: ret=%d", ret);
}
```

## Sub-agent 2: 교육 설계자 관점

### 학습 목표 5개 (블룸 분류체계 적용)

1. **[이해/분석]** MISRA-C 핵심 10개 규칙의 목적을 설명하고, 현재 프로젝트 코드에서 위반 사례를 찾을 수 있다.
2. **[적용/창조]** 레이어 의존성 위반 코드를 Before/After 형태로 리팩토링할 수 있다.
3. **[적용]** volatile 누락, 정수 오버플로우, 재진입 문제 등 공통 버그 패턴을 진단하고 수정할 수 있다.
4. **[분석]** .map 파일을 분석하여 스택/힙 사용량을 파악하고 `static` 활용으로 메모리를 최적화할 수 있다.
5. **[창조/평가]** HAL Mock 기법으로 단위 테스트를 작성하고, 코드 리뷰 체크리스트를 기반으로 전체 프로젝트를 v2.2로 인증할 수 있다.

## Sub-agent 3: 강사 관점

### 수강생 막힘 Top 5 + 해소 방법

1. **MISRA-C 규칙 피로감**: "왜 이렇게 엄격해야 하나?" → 자동차/의료 산업 사례로 필요성 체감
2. **volatile 이해**: "컴파일러 최적화가 왜 문제가 되나?" → 어셈블리 레벨 비교로 시각화
3. **단위 테스트 낯섦**: "임베디드에 테스트가 가능한가?" → HAL Mock 개념을 의존성 주입으로 설명
4. **.map 파일 해독**: "섹션이 너무 많다" → FLASH/RAM 두 가지만 집중 분석
5. **리팩토링 범위**: "어디까지 바꿔야 하나?" → Before/After 예시 3가지로 명확히 한정

- **중반부**: 리팩토링 실습 → "내가 짠 코드가 점점 깨끗해지는 경험"
