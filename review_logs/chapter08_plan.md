## 1. 학습 목표 (블룸 분류체계)

### Ch08 핵심 성격 분석

Ch08은 **Ch07(TIM 기초)의 심화 확장 챕터**입니다. Ch07에서 GP Timer의 기본(PSC/ARR/CCR, Base IT, Edge-Aligned PWM)을 익혔으므로, Ch08에서는 다음 세 가지 축으로 확장합니다:

1. **PWM 심화** — Complementary 출력, Dead-time, TIM1(Advanced Control Timer) 활용
2. **타이머 입력 기능** — Input Capture(외부 신호 주기/주파수 측정), Output Compare, One-Pulse
3. **FSM 첫 구현** — Ch03에서 개념만 배운 FSM을 Stopwatch로 실제 구현 (IDLE/RUNNING/PAUSED/RESET)

교육적으로 가장 중요한 전환점은 **FSM의 첫 실체화**입니다. Ch03에서 "상태/전이/이벤트/액션"을 이론으로 배웠고, Ch06에서 TX_IDLE/TX_BUSY를 암묵적으로 사용했지만, **명시적 FSM 설계 → 구현 → 검증**은 Ch08이 최초입니다. 이것이 Ch09(Alarm FSM), Ch10(Motor FSM), Ch13(Display FSM)의 기반이 됩니다.

### 최종 학습 목표 (5개)

```
LO-1 [이해]   TIM1(Advanced Control Timer)과 GP Timer(TIM2~5)의 차이를 설명하고,
              Complementary PWM 출력의 필요성(H-브리지 모터 드라이버, 하프 브리지 인버터)과
              Dead-time의 역할(shoot-through 방지)을 설명할 수 있다.

LO-2 [적용]   NUCLEO-F411RE에서 TIM1을 사용하여 Complementary PWM을 설정하고,
              Dead-time Generator(DTG)를 구성하여 안전한 PWM 출력을 생성할 수 있다.
              PA7/PB0/PB1 핀 충돌 문제를 식별하고 해결할 수 있다.

LO-3 [적용]   Input Capture 모드를 사용하여 외부 신호의 주기 또는 주파수를 측정하고,
              Output Compare 모드로 특정 시점에 이벤트를 발생시킬 수 있다.
              One-Pulse 모드의 동작 원리를 설명하고 설정할 수 있다.

LO-4 [분석]   타이머 연계(Trigger/Master-Slave) 구조를 분석하고, 한 타이머의 출력이
              다른 타이머의 시작/정지를 제어하는 시나리오를 설계할 수 있다.

LO-5 [창조]   Ch03에서 학습한 FSM 설계 원칙을 적용하여 Stopwatch FSM
              (IDLE → RUNNING → PAUSED → RESET)을 설계하고, Button 이벤트와
              TIM Base 인터럽트를 연동하여 Service 레이어에 구현할 수 있다.
```

### 블룸 수준 분포 평가

| 수준 | 개수 | 평가 |
|------|------|------|
| 이해 | 1 | TIM1 vs GP, Complementary PWM, Dead-time 개념 |
| 적용 | 2 | Complementary PWM 설정 + Input Capture/OC/One-Pulse 구현 |
| 분석 | 1 | 타이머 연계(Master-Slave) 구조 분석 |
| 창조 | 1 | Stopwatch FSM 설계 + Service 레이어 구현 — **Ch08 최종 도달점** |

> **판단:** Ch07이 "기억~평가"(5수준)였다면, Ch08은 "이해~창조"(4수준)로 **상위 블룸 수준에 집중**합니다. Ch07에서 기초를 확보했으므로 "기억" 수준은 불필요하고, 대신 **창조(FSM 설계+구현)**가 등장합니다. 이는 Part 2의 핵심인 FSM 패턴의 첫 실전이므로 매우 적절합니다.

## 2. Ch07과의 누적 성장 연결점 분석

### Ch07 → Ch08 확장 매핑

| Ch07 기초 | Ch08 심화 | 확장 방향 |
|-----------|-----------|-----------|
| GP Timer (TIM2~5) | **Advanced Control Timer (TIM1)** | 같은 타이머이지만 Complementary 출력, Dead-time, Break 기능 추가 |
| Edge-Aligned PWM (CH1 단독) | **Complementary PWM (CH1 + CH1N)** | 단일 채널 → 상보 채널 쌍, 상위/하위 스위치 동시 제어 |
| PSC/ARR/CCR 주기 계산 | **Dead-time 계산 (DTG 레지스터)** | 주기/듀티 계산 경험 → Dead-time 나노초 계산으로 확장 |
| UEV 인터럽트 → 콜백 | **Input Capture 인터럽트 → 주기 측정** | 타이머 출력(UEV) → 타이머 입력(외부 신호 캡처) 관점 전환 |
| `tim_driver.h` 인터페이스 | **`tim_driver.h` 확장 + `stopwatch_service.h` 신설** | Driver 확장 + Service 레이어 첫 구현 |
| Ch03 FSM 이론 | **Stopwatch FSM 실제 구현** | 이론 → 실전 (상태 전이표 → switch-case 코드) |
| Ch01 Button EXTI | **Button → Stopwatch FSM 이벤트** | GPIO 입력 → FSM 이벤트 소스로 재활용 |

### v0.7 → v0.8 코드 확장 포인트

```
v0.7 tim_driver.c/h:
  - tim_driver_init(), tim_start/stop()
  - tim_set_period_ms(), pwm_start/stop(), pwm_set_duty()
  - tim_register_callback()
  → 모두 TIM2 기반

v0.8 확장:
  ① tim_driver.c/h에 TIM1 관련 함수 추가:
     - cpwm_start()/cpwm_stop()          — Complementary PWM
     - cpwm_set_duty(uint8_t percent)     — Dead-time 포함
     - ic_start()/ic_get_frequency()      — Input Capture
  ② stopwatch_service.c/h 신설 (Service 레이어 첫 구현):
     - stopwatch_init()
     - stopwatch_handle_event(sw_event_t evt)  — START/STOP/RESET
     - stopwatch_get_time(uint32_t *ms)
     - 내부 FSM: IDLE → RUNNING → PAUSED → RESET → IDLE
```

### 아키텍처 레이어 변화

```
v0.7:  App ─────────────────→ [tim_driver] → HAL
                               (Driver만)

v0.8:  App → [stopwatch_service] → [tim_driver] → HAL
              (Service 레이어)      (Driver 확장)
       App → Button EXTI ──→ [stopwatch_service]
              (이벤트 소스)
```

> **핵심 전환**: Ch08에서 드디어 **Service 레이어가 실체화**됩니다. Ch07에서 점선 박스로 예고했던 것이 stopwatch_service로 구현됩니다. 이것은 Ch10(clock_service), Ch11(sensor_service), Ch13(ui_service)의 패턴 원형이 됩니다.

### 인지 부하 관리 전략

1. **§8.1이 최대 난관** — Complementary PWM + Dead-time + TIM1 핀 배치가 동시에 등장
   - **대응**: H-브리지 비유("위아래 스위치가 동시에 켜지면 단락!") + CubeMX 스크린샷 + 핀 충돌 표
   - Dead-time은 "안전 간격"으로 단순화: "위 스위치 끄고 → 잠깐 대기 → 아래 스위치 켜기"

2. **§8.2는 "카탈로그" 성격** — IC/OC/One-Pulse 각각이 독립적
   - **대응**: 3개를 "타이머의 입출력 모드 3형제"로 묶고, IC에 집중 (스톱워치 + 실용도 높음)
   - OC와 One-Pulse는 개념 + 간단 예제 수준, 심화는 생략

3. **§8.3은 추상도가 높음** — Master-Slave 개념
   - **대응**: "부장-대리 관계" 비유 + 시퀀스 다이어그램 + CubeMX 설정 데모 1개

4. **§8.4~8.5는 Ch08의 핵심 성취** — FSM 설계→구현
   - **대응**: §8.4에서 설계(상태 전이표, 시퀀스 다이어그램)를 확실히 한 후, §8.5에서 코드로 옮기는 2단계
   - Ch03 FSM 이론 복습 링크 제공 ("Ch03 §3.4 참조")

### 개념 순서: 쉬운 것 → 어려운 것

```

  ↓ Ch07 PWM 기억 환기

§8.3 타이머 연계 (추상도 상승 — 타이머끼리 대화한다)
  ↓ 하드웨어 → 소프트웨어 설계

  ↓ 설계 → 구현

```

### 핵심 주의 사항 4가지

1. **§8.1 Dead-time 계산을 과도하게 다루지 말 것** — DTG 비선형 인코딩은 aside 참조용. CubeMX ns 설정에 집중. "원리는 알되, 실무에서는 도구를 쓴다"

2. **§8.2에서 IC/OC/One-Pulse를 균등하게 다루지 말 것** — Input Capture에 집중(스톱워치 + 주파수 측정 활용도 높음). OC/One-Pulse는 "이런 것도 있다" 수준

3. **§8.4 FSM 설계가 Ch08의 진짜 핵심** — 상태 전이표를 학생이 직접 작성하는 시간(20분) 반드시 확보

4. **Service 레이어 분리의 "왜"를 Ch03 원칙으로 명확히** — "Driver에 FSM 넣으면 안 되나?" → "Driver=하드웨어 추상화만, 비즈니스 로직=Service"

# [교육전문강사] Ch08 기획 — 교육전문강사 의견서

## 1. 수강생 막힘 포인트 Top 5 및 사전 해소 방법

### 막힘 #1: Complementary PWM의 동작 원리 — "왜 출력이 두 개인가?"

**예상 반응**: "Ch07에서 PWM 하나로 LED 밝기 조절 잘 됐는데, 왜 갑자기 반대 신호가 필요한 거예요?"

**사전 해소**:
- **반드시 먼저 설명할 것**: 하프 브리지(Half-Bridge) 회로 개념. 상단 FET과 하단 FET이 번갈아 켜지는 구조를 그림으로 보여주고, "둘 다 동시에 켜지면 단락(shoot-through) → 소자 파괴"라는 위험을 강조.
- **순서**: ① 일반 PWM 복습(Ch07) → ② 모터/인버터에서 양방향 제어 필요성 → ③ 하프 브리지 회로 → ④ 상보 출력의 필요성 → ⑤ Dead-time의 존재 이유
- **핵심 문장**: "Complementary PWM은 '짝꿍 신호'입니다. 하나가 켜지면 다른 하나는 반드시 꺼져야 합니다. 그 사이 '둘 다 꺼지는 안전 구간'이 Dead-time입니다."

### 막힘 #2: Dead-time 설정값 계산 — "숫자가 어떻게 시간이 되는 거예요?"

**예상 반응**: TIM1의 BDTR(Break and Dead-Time Register)의 DTG 비트 필드 인코딩이 비선형적이라 혼란.

**사전 해소**:
- DTG 인코딩 규칙을 표로 정리 (DTG[7:5] 구간별 공식 4가지)
- **실무 접근법 우선**: "실무에서는 CubeMX의 Dead-time 슬라이더로 ns 단위 설정하고, 오실로스코프로 파형을 확인합니다. 레지스터 계산은 참고용입니다."
- 강의에서는 100ns~500ns 범위의 간단한 예제 1개만 직접 계산하고, 나머지는 CubeMX에 위임

### 막힘 #3: 입력 캡처(Input Capture) — "캡처가 뭘 캡처하는 건데요?"

**예상 반응**: Ch07에서는 타이머가 출력(PWM)을 만들었는데, 갑자기 외부 신호를 "잡는다"는 개념 전환이 어려움.

**사전 해소**:
- **비유 먼저**: "스톱워치의 랩(Lap) 버튼을 떠올리세요. 100m 달리기에서 선수가 지나갈 때마다 버튼을 누르면, 그 순간의 시간이 기록됩니다. Input Capture는 외부 신호의 엣지(edge)가 발생하는 순간, 카운터 값을 CCR에 '캡처(기록)'합니다."
- **순서**: ① PWM = 타이머→외부(출력) 복습 → ② Input Capture = 외부→타이머(입력) 개념 대비 → ③ "무엇을 캡처? → CNT 값" → ④ 두 캡처 값의 차이 = 주기/주파수 측정
- 실습에서 버튼 누르기 간격 측정 → 즉각 결과 확인으로 이해도 확보

### 막힘 #4: Stopwatch FSM 상태 전이 — "PAUSED에서 다시 RUNNING으로 갈 때 카운터는?"

**예상 반응**: FSM 자체는 Ch03에서 배웠지만, 실제 타이머 카운터와 FSM 상태를 동기화하는 것이 어려움. 특히 일시정지 후 재개 시 "경과 시간 누적"과 "카운터 리셋" 사이의 혼동.

**사전 해소**:
- **칠판에 상태 전이도 먼저 그리기**: IDLE → [Start] → RUNNING → [Pause] → PAUSED → [Resume] → RUNNING / [Reset] → IDLE
- **각 전이에서 "타이머에 하는 일"을 명시**: Start → tim_start + 누적시간 초기화 / Pause → tim_stop + 현재값 저장 / Resume → tim_start (누적 이어가기) / Reset → tim_stop + 누적시간=0
- **핵심**: "FSM은 '무엇을 할지' 결정하고, 타이머 드라이버는 '어떻게 할지' 실행합니다. PAUSED 상태에서 카운터를 멈추고, RUNNING 재진입 시 이전 누적값에 더합니다."

### 막힘 #5: TIM1(Advanced) vs TIM2(GP) — "왜 갑자기 다른 타이머를 써요?"

**예상 반응**: Ch07에서 TIM2만 썼는데, Ch08에서 TIM1이 등장하면 "설정이 다르다", "APB2 버스다", "MOE 비트가 뭐다" 등 혼란 가중.

**사전 해소**:
- Ch07 §7.1의 타이머 계층 표를 다시 띄우고, TIM1 행을 강조
- **핵심 차이 3가지만 강조**: ① Complementary 출력 가능 ② Dead-time 레지스터(BDTR) 존재 ③ MOE(Main Output Enable) 비트 — 안전 잠금
- "TIM2는 만능 스톱워치, TIM1은 안전장치가 달린 산업용 스톱워치입니다. 기본 사용법은 같지만, 안전 관련 설정이 추가된 것뿐입니다."
- HAL API 차이: `HAL_TIMEx_PWMN_Start()` (N = Complementary) 강조

## 3. Complementary PWM 비유 개발

### 비유: "시소(Seesaw) 놀이터"

**매핑**:
| 비유 요소 | 기술적 대응 | 설명 |
|-----------|------------|------|
| 시소의 한쪽 끝 | CH1 (주 출력) | 한쪽이 올라가면 → HIGH |
| 시소의 반대쪽 끝 | CH1N (보상 출력) | 반대쪽은 내려감 → LOW |
| 시소가 수평인 순간 | Dead-time | 양쪽 다 중간 → 둘 다 OFF 구간 |
| 시소 축(지렛대 받침) | 타이머 카운터 | 주기적으로 방향 전환 |
| 시소 위 아이의 무게 | 듀티비 | 무게 차이 → HIGH/LOW 비율 변화 |

**강의 멘트**: "시소를 떠올리세요. 한쪽이 올라가면 반대쪽은 반드시 내려갑니다. 절대로 양쪽이 동시에 위에 있을 수 없죠. Complementary PWM이 바로 이것입니다. 그런데 시소가 방향을 바꾸는 그 짧은 순간, 양쪽 다 중간 높이에 있습니다. 이 순간이 Dead-time — 양쪽 FET이 모두 꺼지는 안전 구간입니다."

**한계점 (반드시 명시)**:
- 실제 시소는 기계적으로 연결되어 있어 반드시 반대로 움직이지만, Complementary PWM은 소프트웨어/하드웨어 설정으로 반전이 구현됨
- 시소의 "중간 순간"은 자연스럽게 발생하지만, Dead-time은 엔지니어가 의도적으로 설정하는 보호 구간
- 시소 비유는 1상(single phase)만 설명 가능. 3상 인버터(모터 구동)는 시소 3개가 동기화된 것으로 확장 설명 필요 (이 챕터 범위 밖)

## 4. Stopwatch FSM 설명 전략

### 강의 풀이 순서

**Step 1: 실물 스톱워치 시연 (2분)**
- 핸드폰 스톱워치 앱을 화면에 띄우고 직접 시연
- "Start → 숫자 올라감 → Pause → 숫자 멈춤 → Resume → 이어서 올라감 → Reset → 0으로"
- "이 동작을 코드로 구현하려면 어떤 '상태'가 필요할까요?"

**Step 2: 상태 전이도 함께 그리기 (5분)**
- 학생들에게 상태 이름을 물어보며 칠판에 그림
- 4개 상태: IDLE / RUNNING / PAUSED / RESET(→즉시 IDLE 전이)
- 각 전이 화살표에 이벤트(버튼) + 액션(타이머 제어) 기재

**Step 3: 버튼 매핑 (3분)**
- NUCLEO의 파란 버튼(B1/PC13) → Start/Pause 토글
- 길게 누르기(2초) → Reset
- "버튼 하나로 3가지 동작을 어떻게 구분할까요?" → 짧게/길게 구분 로직 설명

**Step 4: 타이머와 FSM 동기화 코드 (10분)**
- 각 상태 전이에서 tim_driver API를 어떻게 호출하는지 코드로 보여주기
- **핵심 변수**: `elapsed_ms` (누적 경과 시간), `lap_start_tick` (현재 구간 시작점)

### 학생이 놓치기 쉬운 부분

1. **PAUSED → RUNNING 재개 시 누적 처리**
   - 흔한 실수: Resume 시 카운터를 0부터 시작 → 이전 시간 날아감
   - 올바른 처리: `elapsed_ms += (현재tick - lap_start_tick)` 누적 후, `lap_start_tick = 현재tick`으로 갱신

2. **IDLE 상태에서 Pause 버튼 무시**
   - FSM의 핵심: "현재 상태에서 허용되지 않는 이벤트는 무시"
   - 코드에서 `switch(state)` 안에 해당 이벤트 핸들러가 없으면 자연스럽게 무시됨을 보여주기

3. **인터럽트 컨텍스트에서 FSM 전이 금지**
   - 버튼 EXTI 콜백에서 직접 상태 전이하면 안 됨 → 플래그 세트만 하고 main loop에서 처리
   - Ch01의 EXTI 패턴 복습: "인터럽트에서는 플래그만, 처리는 메인에서"

4. **Reset 후 디스플레이 갱신**
   - Reset → IDLE 전이 시 화면에 "00:00.000" 표시 필요
   - 나중에 Ch13(LCD)에서 실제 화면 연결 예고

## 5. 수강생 단골 질문 예측 및 답변

### Q1. "Complementary PWM과 일반 PWM의 차이가 뭐예요?"

**강의 현장용 답변**:
"일반 PWM은 출력이 하나입니다. HIGH-LOW를 반복하죠. Complementary PWM은 출력이 **두 개** — 주 채널(CH1)과 보상 채널(CH1N)이 항상 반대입니다. CH1이 HIGH면 CH1N은 LOW. 왜 필요하냐면, 모터를 양방향으로 돌리는 H-Bridge 회로에서 상단 FET과 하단 FET을 번갈아 제어해야 하기 때문입니다. 일반 PWM으로 하면 두 채널의 타이밍을 소프트웨어로 맞춰야 해서 위험합니다. Complementary PWM은 하드웨어가 자동으로 반전시켜주니까 안전합니다."

### Q2. "Dead-time은 왜 필요해요? 없으면 안 되나요?"

**강의 현장용 답변**:
"FET(트랜지스터)은 꺼지는 데 시간이 걸립니다. 수십~수백 나노초. CH1이 OFF되는 순간 CH1N이 바로 ON되면, 짧은 순간 **둘 다 ON** → 전원에서 GND로 직통(shoot-through) → **매우 큰 전류** → 소자 파괴. Dead-time은 '둘 다 꺼지는 안전 구간'을 하드웨어가 강제로 삽입하는 것입니다. 없으면? 보드가 탑니다. 실무에서 Dead-time 설정 실수로 MOSFET 태운 경험담은 임베디드 엔지니어 통과의례입니다."

### Q3. "Input Capture로 뭘 측정할 수 있어요? 실무에서 쓰나요?"

**강의 현장용 답변**:
"외부 신호의 **주파수, 주기, 펄스 폭**을 측정합니다. 실무 사례: ① 자동차 엔진 RPM 측정 (크랭크 센서 펄스 주기) ② 초음파 센서 거리 측정 (에코 펄스 폭) ③ 적외선 리모컨 신호 디코딩 (펄스 패턴 분석). 우리 실습에서는 버튼 누르기 간격 측정 → 스톱워치의 기반 기술로 사용합니다."

### Q4. "One-Pulse 모드는 언제 쓰나요?"

**강의 현장용 답변**:
"트리거 한 번에 펄스 딱 하나만 나오고 자동 정지합니다. 예: ① 카메라 플래시 트리거 (정확한 시간 동안 빛) ② 서보 모터 위치 지정 (한 번의 펄스로 각도 설정) ③ 부저 단음 발생. 반복 PWM과 달리 '한 번만 실행'이 필요한 상황에 씁니다."

### Q5. "타이머 Master-Slave 연계가 뭐예요? 꼭 배워야 하나요?"

**강의 현장용 답변**:
"타이머 하나가 다른 타이머를 제어하는 것입니다. 예: TIM1이 1초마다 TIM2를 시작시킴. 실무에서 ADC 샘플링 타이밍을 정밀 제어할 때 자주 씁니다. 이 챕터에서는 개념과 기본 설정만 다루고, 깊은 활용은 Ch17(아키텍처 심화)에서 만납니다."

## 6. 면접 연결 포인트

### 면접 질문 1: "Complementary PWM을 사용하는 이유와 Dead-time의 역할을 설명하세요."
**핵심 답변 가이드**: H-Bridge/Half-Bridge에서 shoot-through 방지, FET 스위칭 지연 보상, 하드웨어 레벨 안전 보장. "소프트웨어로 타이밍을 맞추면 인터럽트 지연으로 동시 ON 위험 → 하드웨어 Dead-time이 필수"

### 면접 질문 2: "Input Capture와 Output Compare의 차이를 설명하세요."
**핵심 답변 가이드**: Input Capture = 외부 이벤트 시점의 CNT 값을 CCR에 저장(측정), Output Compare = CCR 값에 도달하면 출력 핀 변경(제어). 방향이 반대: 캡처는 in→timer, 비교는 timer→out.

### 면접 질문 3: "FSM(유한 상태 머신)을 실무에서 어떻게 구현하셨나요?"
**핵심 답변 가이드**: Stopwatch FSM 경험 — enum으로 상태 정의, switch-case로 전이 처리, 이벤트(버튼)와 액션(타이머 제어) 분리. "인터럽트에서는 이벤트 플래그만 세트하고, 메인 루프에서 FSM 처리" 패턴 강조.

### 면접 질문 4: "타이머 Master-Slave 모드를 설명하고, 사용 사례를 들어주세요."
**핵심 답변 가이드**: 하나의 타이머(Master)가 TRGO 신호로 다른 타이머(Slave)를 트리거/게이팅/리셋. 사례: ADC 정밀 샘플링 타이밍, 복수 PWM 동기화, 모터 제어 시퀀싱.

### 위험 요소 및 대응

1. **NUCLEO-F411RE에서 TIM1 핀 충돌**: PA7(TIM1_CH1N)이 SPI1_MOSI와 겹침. CubeMX에서 핀 배치를 미리 확인하고, 충돌 없는 핀 조합을 교재에 명시해야 함 (예: PA8/CH1, PB13/CH1N 또는 PA7/CH1N — SPI 미사용 시)
2. **오실로스코프 없는 환경**: Complementary PWM 파형을 눈으로 확인 못 하면 이해도 급감. 대안: LED 2개를 CH1/CH1N에 연결하여 번갈아 깜빡이는 것으로 시각화 (느린 주파수 설정 필요)
3. **Stopwatch 실습 시간 부족**: FSM + 버튼 디바운싱 + 타이머 제어를 30분 안에 완성하기 어려울 수 있음. 스켈레톤 코드(FSM 뼈대 + TODO 주석)를 미리 제공하고, 학생은 전이 로직만 채우는 방식 권장

## 종합 의견

Ch08은 Ch07 대비 **난이도가 크게 상승**하는 챕터입니다. 특히:
- GP 타이머 → Advanced 타이머 전환 (TIM1의 추가 레지스터)
- 단방향 PWM → 양방향 Complementary PWM (하프 브리지 회로 이해 필요)
- 단순 타이머 사용 → FSM 기반 상태 관리 (소프트웨어 설계 역량)

**성공 핵심**: "왜 필요한가"를 먼저 납득시키는 것. Complementary PWM이 왜 필요한지(shoot-through 방지), Dead-time이 왜 필요한지(FET 스위칭 지연), FSM이 왜 필요한지(복잡한 상태 관리)를 실물/비유로 먼저 설명하고, 코드는 그 다음에 보여줘야 합니다.

스톱워치 FSM은 이 교재의 **첫 번째 본격 FSM 구현**이므로, Ch03에서 배운 개념이 실제 코드로 구현되는 "아하!" 순간을 만들어주는 것이 이 챕터의 교육적 하이라이트입니다.
