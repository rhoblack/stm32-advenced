# Ch07. TIM 기초 (General Purpose Timer) — 교육 설계 분석

**작성자:** 교육 설계자 (Instructional Designer)
**작성일:** 2026-03-17
**Phase:** 1 (기획 회의)
**프로젝트 버전:** v0.6 → v0.7
**누적 기능:** 정밀 주기 타이머 + PWM 출력
**상태:** Phase 1 기획 완료

## 1. 학습 목표 (블룸 분류체계)

### Ch07 핵심 성격 분석

Ch07은 **Part 2(타이머 + RTC + 스텝 모터)의 시작 챕터**입니다. Part 1(통신 + DMA)에서 Part 2로의 전환이므로, 학습자 입장에서의 인지 전환:

- Part 1: "데이터를 주고받는 방법(UART)과 그것을 효율적으로 하는 방법(DMA)을 배웠다"
- **Part 2: "이제 '시간'을 제어한다 — 정확한 주기, 정확한 펄스, 정확한 시각"**
- Ch07: "타이머의 기본 원리를 이해하고, 주기적 이벤트와 PWM을 직접 생성한다"

Ch07의 교육적 핵심은 **세 가지**입니다:
1. **타이머 하드웨어 이해** — PSC/ARR/CCR 레지스터가 어떻게 주기와 듀티를 결정하는지
2. **시간 계산 능력** — 클럭 주파수, 프리스케일러, 자동 리로드 값으로 원하는 주기를 역산
3. **Service 레이어 첫 도입** — Driver(tim_driver)가 Service 레이어에 시간 기반 서비스를 제공하는 구조

추가로, **PWM(Pulse Width Modulation)**이라는 새로운 출력 방식이 등장합니다. 이것은 Ch08(Advanced TIM)의 Complementary PWM, Ch10(스텝 모터) 제어의 기반이 되므로, **Edge-Aligned PWM의 원리를 확실히 잡는 것**이 중요합니다.

### 최종 학습 목표 (5개)

```
LO-1 [기억]   STM32F411의 타이머 계층(Basic / General Purpose / Advanced Control)을
              열거하고, 각 계층의 대표 타이머(TIM6-7 / TIM2-5 / TIM1)와 주요 차이를
              설명할 수 있다.

LO-2 [이해]   PSC(Prescaler), ARR(Auto-Reload Register), CCR(Capture/Compare Register)의
              역할과 관계를 설명하고, "카운터가 0에서 ARR까지 세는 동안 한 주기"라는
              동작 원리를 타이밍 다이어그램으로 표현할 수 있다.

LO-3 [적용]   원하는 주기(ms)와 듀티비(%)가 주어졌을 때, 시스템 클럭(100MHz)으로부터
              PSC와 ARR 값을 계산하고, CubeMX에서 TIM을 설정하여 주기적 인터럽트와
              PWM 출력을 구현할 수 있다.

LO-4 [분석]   TIM Base 인터럽트(UEV) 기반 주기 실행과 SysTick 기반 HAL_Delay()의
              차이를 정확도, CPU 점유율, 다중 작업 관점에서 비교 분석할 수 있다.

LO-5 [평가]   주어진 시나리오(LED 깜빡임 1Hz, 부저 음계 출력, 서보 제어 50Hz)에 대해
              적절한 타이머 종류(Basic/GP/Advanced)와 모드(Base/PWM/OC)를 선택하고
              그 근거를 제시할 수 있다.
```

### 블룸 수준 분포 평가

| 수준 | 개수 | 평가 |
|------|------|------|
| 기억 | 1 | 타이머 계층 분류 — STM32 타이머 전체 지도의 첫걸음 |
| 이해 | 1 | PSC/ARR/CCR 동작 원리 — 레지스터 수준 이해 |
| 적용 | 1 | 주기/듀티 계산 + CubeMX 설정 — **핵심 실습** |
| 분석 | 1 | TIM vs SysTick 비교 — 기존 지식과의 차별화 |
| 평가 | 1 | 시나리오별 타이머 선택 — Ch07 최종 도달점 |

> **판단:** 기억 1 + 이해 1 + 적용 1 + 분석 1 + 평가 1 = 총 5개, 블룸 5수준 커버. Ch06(평가 2개)에서 **적용 비중이 상대적으로 높아짐**. 이는 Ch07이 Part 2의 첫 챕터로서 새로운 하드웨어(타이머)의 **기초 조작 능력 확보**가 우선이기 때문입니다. "창조" 수준은 Ch08(Stopwatch FSM 설계)에서 다룹니다.

## 3. 인지 부하 분석

### 핵심 신개념 분석

| # | 신개념 | 인지 부하 | 선행 지식 | 비고 |
|---|--------|-----------|-----------|------|
| 1 | **타이머 3계층 분류 (Basic/GP/Advanced)** | **낮음** — 분류 체계 암기, 개념적 복잡도 낮음 | 없음 (완전 신규) | 표로 정리하면 빠르게 습득 |
| 2 | **PSC/ARR 레지스터와 주기 계산 공식** | **중상** — 수학적 사고 필요, T = (PSC+1)(ARR+1)/f_clk | 기본 수학 (나눗셈) | **Ch07의 최대 난관** |
| 3 | **UEV(Update Event)와 TIM Base 인터럽트** | **중간** — Ch01 EXTI 인터럽트 패턴과 유사, 콜백 구조 동일 | Ch01 NVIC, Ch04 콜백 패턴 | EXTI→TIM 확장 |
| 4 | **PWM 개념 + CCR 레지스터** | **중간** — 새로운 출력 방식이지만, PSC/ARR 이해 후 자연스러운 확장 | §7.2 PSC/ARR | §7.2 위에 얹어지는 개념 |
| 5 | **Edge-Aligned PWM 모드** | **낮음** — Center-Aligned와 비교 없이 기본만 다룸 | §7.4 PWM 개념 | Ch08에서 심화 |

**총 핵심 신개념: 5개** — 권장 한도(3개)를 2개 초과

### 권장 처리 방안: 개념 계층화 + 섹션 분리

```
[개념 A] 타이머 분류 + PSC/ARR         → "타이머란 무엇인가" (§7.1~7.2)
[개념 B] UEV + TIM Base 인터럽트       → "타이머로 주기 만들기" (§7.3)
[개념 C] PWM + CCR + Edge-Aligned      → "타이머로 파형 만들기" (§7.4)
```

**압축 전략:**
1. **타이머 3계층**은 독립 신개념이 아닌 "배경 지식" — §7.1에서 표 하나로 정리, 외울 필요 없이 참조용
2. **PSC/ARR**은 하나의 개념 쌍 — "분주기 + 카운터 상한"을 함께 다루면 1개 개념
3. **UEV**는 Ch01 EXTI의 TIM 버전 — "인터럽트 소스만 다르고 콜백 패턴 동일"로 기존 지식 활용
4. **CCR + Edge-Aligned**는 PWM의 하위 속성 — 독립 개념이 아닌 PWM의 구성 요소
5. 결과적으로 **절당 신개념 2~3개** 수준으로 관리 가능

## 4. Ch06과의 자연스러운 연결 포인트

### Part 전환 관리 (Part 1 → Part 2)

Ch07은 **새로운 Part의 시작**입니다. Part 간 전환은 학습자에게 "리셋" 느낌을 줄 수 있으므로, **연속성을 강조**하면서도 **새로운 시작의 신선함**을 동시에 전달해야 합니다.

| 연결 포인트 | 위치 | 설명 |
|-------------|------|------|
| **아키텍처 연속성** | §7.0 도입 | "v0.6에서 UART+DMA가 Driver 레이어에 자리잡았습니다. 이제 TIM Driver를 같은 레이어에 추가합니다" — 동일한 아키텍처 다이어그램에 새 블록 추가 |
| **인터페이스 설계 패턴 재사용** | §7.5 | Ch04에서 `UART_Send()` 인터페이스를 설계한 것처럼, `TIM_SetPeriod_ms()` / `PWM_SetDuty()`를 설계 — "같은 패턴, 다른 하드웨어" |
| **인터럽트 콜백 패턴 재사용** | §7.3 | Ch01 EXTI → Ch04 UART IT → **Ch07 TIM IT** — "HAL이 제공하는 콜백 패턴은 항상 동일" |
| **DMA 경험 활용** | §7.0 | "Part 1에서 DMA로 CPU를 해방시켰죠? Part 2에서는 타이머로 시간을 해방시킵니다 — HAL_Delay() 대신 타이머 인터럽트로" |
| **로그 시스템 지속** | 전체 | 모든 코드에 LOG_D/I/W/E 적용 — Ch02 인프라 계속 활용 |

### 핵심 전환 메시지 (§7.0 도입부)

> "Part 1에서 우리는 **데이터의 흐름**을 다뤘습니다 — UART로 보내고, DMA로 효율화했죠. Part 2에서는 **시간의 흐름**을 다룹니다. 정확한 1초, 정확한 50Hz, 정확한 시각. 스마트 시계의 '시계' 부분이 드디어 시작됩니다."

## 5. Ch08(Advanced TIM)로의 복선 제안

Ch07에서 Ch08로의 전환은 **같은 하드웨어(TIM)의 기초→심화** 관계입니다. Ch07 내에서 자연스럽게 "이건 나중에 더 깊이 다룰 거예요"라는 복선을 심어야 합니다.

| 복선 | 위치 | 방법 | Ch08 연결 |
|------|------|------|-----------|
| **TIM1(Advanced Control)** | §7.1 | 타이머 3계층 표에서 "TIM1은 모터 제어용 특수 기능 보유 → Ch08에서" | Ch08 Complementary PWM |
| **Center-Aligned PWM** | §7.4 | "Edge-Aligned만 다룹니다. Center-Aligned는 대칭 파형이 필요할 때 — Ch08에서" | Ch08 PWM 고급 |
| **Input Capture** | §7.1 또는 §7.6 | "타이머는 출력만 하는 게 아닙니다. 외부 신호의 주기를 측정할 수도 있죠 — Ch08에서" | Ch08 Input Capture |
| **스톱워치 FSM** | §7.6 | "TIM Base 인터럽트로 1ms 틱을 만들었죠? 이것으로 스톱워치를 만들 수 있습니다 — Ch08에서 FSM으로 설계" | Ch08 Stopwatch FSM |
| **Dead-time** | §7.4 aside | "PWM으로 모터를 구동할 때, 위아래 스위치가 동시에 켜지면 단락(shoot-through)! 이를 방지하는 Dead-time — Ch08에서" | Ch08 Dead-time |

> **복선 원칙**: "못하게 막는 것"이 아닌 "궁금증을 유발하는 것". 기능의 존재만 언급하고, 설명은 Ch08로 미룹니다. "왜 그게 필요하지?"라는 질문이 자연스럽게 생기도록 시나리오 제시.

| ✅ Ch04 UART | Driver 인터페이스 설계 패턴 | 동일 패턴으로 TIM 인터페이스 설계 |
| ✅ Ch06 DMA+링버퍼 | DMA로 CPU 해방 경험 | "DMA는 데이터 해방, TIM은 시간 해방" 대비 |

### 새롭게 강조 필요한 선행 지식

| 개념 | 왜 필요한가 | 어디서 강조할 것인가 |
|------|-------------|---------------------|
| **클럭 트리 (APB1/APB2 버스 클럭)** | PSC 계산의 입력값이 타이머별로 다름 (APB1: 50MHz, APB2: 100MHz) | §7.2 도입부에서 "타이머의 심장박동 = 버스 클럭" 설명 + CubeMX Clock Configuration 스크린샷 |
| **주파수↔주기 변환 (f = 1/T)** | 주기 계산의 기본 수학 | §7.2에서 "1Hz = 1초 주기, 1kHz = 1ms 주기" 변환표 제공 |
| **HAL_Delay()의 한계** | TIM 인터럽트의 장점을 대비시키기 위해 | §7.3에서 "HAL_Delay()는 CPU를 멈추지만, TIM은 CPU를 자유롭게" 비교 |

## 8. 비유 및 실생활 예시 체계

| 개념 | 비유 | 비유 세계관 연속성 |
|------|------|-------------------|
| 타이머 카운터 | **메트로놈** — 일정 속도로 째깍째깍 세다가 설정 값(ARR)에 도달하면 "띵!" 울리고 다시 처음부터. 메트로놈 속도 = PSC로 조절 | 음악 — 직관적, 청각적 |
| PSC (Prescaler) | **기어비** — 자동차 기어를 바꾸면 바퀴 회전 속도가 달라지듯, PSC를 바꾸면 카운터 속도가 달라짐. 높은 기어(큰 PSC) = 느린 카운터 | 자동차 — Ch10 스텝모터 기어비와 연결 |
| ARR (Auto-Reload) | **알람 시계의 설정 시각** — "100까지 세면 울려라". ARR = 100이면 0→100까지 센 후 이벤트 발생 | 시계 — 스마트 시계 프로젝트와 연결 |
| UEV (Update Event) | **메트로놈의 "띵!" 소리** — ARR까지 다 세면 발생하는 이벤트. 이 소리에 맞춰 일(LED 토글)을 하면 정확한 리듬 | 메트로놈 비유 연장 |
| PWM | **수도꼭지** — 1초 주기로 "0.7초 열고 0.3초 닫기"를 반복하면, 수도꼭지는 70% 열린 것과 비슷한 수량. 열린 시간 비율 = 듀티비 | 일상 수도 — 직관적 |
| CCR | **수도꼭지의 타이머 설정** — "카운터가 CCR에 도달하면 잠그고, ARR에 도달하면 다시 열어라". CCR/ARR = 듀티비 | 수도꼭지 비유 확장 |
| TIM vs SysTick | **알람 시계 vs 모래시계** — 알람 시계(TIM)는 설정 시각에 자동으로 울리고 내 할 일 방해 안 함. 모래시계(HAL_Delay)는 다 떨어질 때까지 지켜보고 있어야 함 | 시계 프로젝트 연결 |

## 9. 막힘 포인트 예측 (5개)

| # | 막힘 포인트 | 발생 절 | 원인 | 선제 대응 |
|---|-------------|---------|------|-----------|
| 1 | **"PSC와 ARR 값을 어떻게 정하지?"** — 주기 공식 역산 혼란 | §7.2 | T = (PSC+1)(ARR+1)/f_clk 공식에서 PSC/ARR 두 변수를 동시에 결정해야 함 | **2단계 전략 제시**: "① PSC로 카운터 클럭을 1kHz나 1MHz 같은 깔끔한 값으로 만들기 → ② ARR로 원하는 주기 설정". 계산 예제 3개 (1초, 500ms, 20ms) 반복 |
| 2 | **"왜 PSC+1, ARR+1인가? (0부터 세니까)"** | §7.2 | 레지스터 값과 실제 분주 비율의 off-by-one 차이 | FAQ 박스: "레지스터에 99를 넣으면 0~99까지 100번 센다 → 100분주. 레지스터 값 = 원하는 값 - 1" |
| 3 | **"TIM 인터럽트 콜백이 어느 함수?"** — HAL 콜백 이름 혼동 | §7.3 | HAL_TIM_PeriodElapsedCallback vs HAL_TIM_PWM_PulseFinishedCallback 등 다양한 콜백 | §7.3에서 **콜백 이름 매핑 표** 제공: "Base → PeriodElapsed, PWM → PulseFinished, IC → CaptureCallback" |
| 4 | **"PWM 듀티를 바꿔도 LED가 안 변해요"** — CCR 업데이트 시점 | §7.4 | __HAL_TIM_SET_COMPARE() 호출 후 즉시 반영 vs Preload 레지스터 사용 여부 | §7.4에서 **Preload 개념 간단 설명**: "ARR Preload Enable → 다음 주기부터 반영, 글리치 방지" + CubeMX 체크박스 안내 |
| 5 | **"APB1이 50MHz인데 왜 TIM 클럭은 100MHz?"** — 클럭 트리 혼동 | §7.2 | APB1 프리스케일러가 /2일 때 타이머 클럭은 x2 되는 STM32 특수 규칙 | §7.2 aside 박스: "APB1 PSC ≠ 1이면 TIM 클럭 = APB1 x 2. CubeMX Clock Configuration에서 확인!" + SVG 클럭 트리 |

### 막힘 포인트 심각도 순위

1. **#1 PSC/ARR 역산** (가장 빈번) — 실습 시 직접 겪음, 수식 기반
2. **#5 클럭 트리** (가장 혼란) — STM32 고유 규칙, 데이터시트 없이는 추론 불가
3. **#2 off-by-one** (실수 빈발) — 계산은 맞는데 결과가 1배 차이
4. **#4 CCR 반영 시점** (디버깅 난이도) — "코드는 맞는데 안 돼요" 유형
5. **#3 콜백 이름** (단순 참조) — 표 하나면 해결

## 13. Ch07의 아키텍처적 특별 사항: Service 레이어 첫 등장 예고

### v0.7의 아키텍처 업데이트

```
v0.6:  App → [UART Driver] → HAL        (Driver 레이어만)
v0.7:  App → [TIM Driver] → HAL         (Driver 레이어에 TIM 추가)
       App → [Service?] → TIM Driver    (Service 레이어 첫 준비)
```

TABLE_OF_CONTENTS.md에서 v0.7은 "**GP TIM + Service 레이어 추가**"로 명시되어 있습니다. 그러나 Ch07에서 본격적인 Service 레이어 구현은 과부하입니다. 제안:

1. **§7.5에서 Service 레이어 "예고"** — "tim_driver가 제공하는 TIM_SetPeriod_ms()를 상위에서 어떻게 쓸까? Ch08에서 Stopwatch Service, Ch10에서 Clock Service가 이 API를 호출합니다"
2. **아키텍처 SVG에 Service 레이어를 점선 박스로 표시** — "아직 구현 전이지만, 여기에 들어갈 예정" 표시
3. **실제 Service 구현은 Ch08(Stopwatch)에서** — FSM과 함께 첫 Service 레이어 완성

> **교육 설계 관점**: 한 챕터에서 "새 하드웨어 원리 + 새 레이어 도입"을 동시에 하면 인지 부하 과다. Ch07에서 Driver 확보, Ch08에서 Service 구축이 자연스러운 2단계 분리입니다.

### 핵심 주의 사항 4가지

1. **§7.2 주기 계산이 Ch07의 유일한 난관** — PSC/ARR 역산을 "2단계 전략"(PSC로 깔끔한 클럭 만들기 → ARR로 주기 설정)으로 체계화. 계산 예제 3개 반복이 핵심.

2. **클럭 트리(APB1/APB2 x2 규칙)를 반드시 CubeMX 스크린샷과 함께 설명** — 이 규칙을 모르면 계산 결과가 2배 차이. FAQ 또는 aside 박스로 명확히.

3. **PWM 실습(§7.4)이 Ch07의 최대 성취감 포인트** — LED가 부드럽게 밝아지는 시각적 피드백이 강력. 반드시 충분한 실습 시간(30분) 확보.

4. **Service 레이어는 "예고"만, 구현은 Ch08로** — v0.7에서 Service 레이어를 본격 구현하면 인지 부하 과다. 아키텍처 SVG에 점선 박스로 표시하고, Ch08 Stopwatch Service에서 첫 구현.
