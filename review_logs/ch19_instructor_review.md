## 비유 검증

### 비유 1: "MISRA-C = 도로교통법"
- 기술 매핑: MISRA-C는 규칙 위반이 즉각 사고로 이어지지 않듯, 컴파일 오류가 아닌 잠재적 버그 유발 → 정확 ✅
- 한계 명시: "MISRA-C 위반 자체는 컴파일 오류가 아닙니다" → 명시됨 ✅
- 임베디드 오해 방지: "정적 분석 도구 필요" 안내로 오해 예방 ✅

### 비유 2: "HAL Mock = 비행 시뮬레이터"
- 기술 매핑: 동일 인터페이스(조종간 = HAL 함수 시그니처), 다른 구현(엔진 없음 = 하드웨어 없음) → 정확 ✅
- 한계 명시: "타이밍, DMA 지연, 하드웨어 특성은 검증 불가" → 명시됨 ✅
- 임베디드 오해 방지: "Mock 테스트는 로직 검증에 초점" → 명시됨 ✅

### 비유 3: "레이어 위반 = 홀 직원이 창고에서 직접 재료 꺼내기"
- 기술 매핑: 레이어 우회 → 창고 위치 변경 시 홀 직원도 수정 필요 → 변경에 취약 → 정확 ✅
- 한계 명시: "레이어 위반은 컴파일 오류가 아닌 설계 오류" → 명시됨 ✅

**모든 비유 검증 통과** ✅

## 수강생 막힘 포인트 예측 및 해소 평가

### 막힘 1: "MISRA-C가 현실에서 얼마나 중요한가?"
- **예상 반응**: "이걸 다 지켜야 하나? 너무 엄격하지 않나?"
- **현재 해소**: "자동차/의료 산업 사례" 언급 ✅
- **추가 제안**: 강사 꿀팁에 "STM32CubeIDE -Wall -Wextra로 40% 탐지" 안내 ✅

### 막힘 2: "volatile을 어디에 붙여야 하는지 헷갈린다"
- **예상 반응**: "ISR에서 설정하는 모든 변수에 붙이면 되나?"
- **현재 해소**: FAQ 박스 "ISR과 main이 공유하는 변수에만 선택적으로" ✅ 완벽

### 막힘 3: ".map 파일이 너무 복잡하다"
- **예상 반응**: "섹션이 너무 많다, 어디를 봐야 하나?"
- **현재 해소**: "Memory Configuration, .bss 큰 버퍼, 미사용 함수" 3가지로 한정 ✅
- **강사 꿀팁**: Build Analyzer 추가 안내 ✅

### 막힘 4: "HAL Mock 테스트 환경 구성이 어렵다"
- **현재 해소**: 실습 "실습이 안 될 때" 에서 MinGW 설치 명령어 제공 ✅
- 코드 리뷰 체크리스트만으로도 v2.2 80% 달성 가능 안내 → 좌절 방지 ✅

### 막힘 5: "면접 20문항이 너무 많다"
- **현재 해소**: 카테고리별(DMA/RTC/FSM/아키텍처/인터럽트) 분류 + 핵심 키워드 강조 ✅
