## 1. 학습 목표 설계 (블룸 분류체계)

### Ch04 핵심 성격 분석

Ch04는 **Driver 레이어가 처음 등장하는 챕터**입니다. Ch03에서 4계층 아키텍처를 "설계도"로 배웠다면, Ch04에서는 Driver 레이어에 **첫 번째 실제 컴포넌트**를 구현합니다. 이 전환이 학습자에게 주는 의미:

- Ch01~02: HAL 함수를 직접 호출 (계층 의식 없이)
- Ch03: "아, 계층이 있구나" (설계도만 작성)
- **Ch04: "직접 Driver를 만들어보니 계층 분리가 이런 거구나"** (첫 체험)

### 최종 학습 목표 (6개)

```
LO-1 [기억]   UART 프레임의 구성 요소(Start bit, Data bits, Parity, Stop bit)와
              Baud Rate의 정의를 열거할 수 있다.
LO-2 [이해]   폴링 방식 송수신과 인터럽트 방식 수신의 차이를 설명하고,
              인터럽트 수신이 필요한 상황을 근거를 들어 설명할 수 있다.
LO-3 [적용]   CubeMX에서 USART2를 설정하고, HAL_UART_Transmit()으로
              PC 터미널에 문자열을 전송할 수 있다.
LO-4 [적용]   printf()를 UART로 리다이렉트하여 기존 LOG_I 매크로의
              출력 경로를 SWO에서 UART로 전환할 수 있다.
LO-5 [분석]   App → UART Driver → HAL → Hardware 시퀀스 다이어그램을 작성하고,
              각 호출이 어느 레이어에 속하는지 판별할 수 있다.
LO-6 [적용]   UART_Send() 인터페이스를 설계하여 Driver 레이어 컴포넌트를
              아키텍처 다이어그램 v0.4에 추가할 수 있다.
```

### 블룸 수준 분포 평가

| 수준 | 개수 | 평가 |
|------|------|------|
| 기억 | 1 | 프레임 구조는 암기 필수 — 최소 1개 필요 |
| 이해 | 1 | 폴링 vs 인터럽트 비교는 개념적 이해 핵심 |
| 적용 | 3 | **Ch04는 "첫 Driver 구현"이므로 적용 수준이 다수인 것이 적절** |
| 분석 | 1 | 시퀀스 다이어그램 작성은 Ch03 읽기→Ch04 쓰기 전환 |

> **판단:** 기억 1 + 이해 1 + 적용 3 + 분석 1 = 총 6개. Ch04는 실습 중심 챕터이므로 적용 수준이 50%를 차지하는 것이 적절합니다. 평가/창조 수준은 Ch04 수준에서는 과도하므로 포함하지 않습니다.

## 2. 절 구조 및 교육 흐름 설계

### 권장 절 구조 (6개 절)

```
[§4.1] 아키텍처 위치 & 인터페이스 설계 (Ch04부터 필수 도입부)
  → Driver 레이어에 uart_driver 배치
  → UART_Send() API 시그니처 설계
  → 시퀀스 다이어그램: App → UART Driver → HAL → HW
  → 아키텍처 다이어그램 v0.3 → v0.4 업데이트 미리보기

[§4.2] UART, 시리얼 통신의 기본 언어
  → "편지 봉투" 비유로 UART 프레임 구조 설명
  → Start bit, Data bits(8), Parity(None), Stop bit(1)
  → Baud Rate = "대화 속도 약속" (115200 = 초당 115,200 비트)
  → NUCLEO-F411RE USART2 ↔ ST-LINK ↔ PC USB 경로 설명

[§4.3] 첫 번째 UART 송신 — 폴링 방식
  → CubeMX USART2 설정 실습 (115200-8-N-1)
  → HAL_UART_Transmit() 폴링 송신
  → 터미널 프로그램(Tera Term/PuTTY)에서 "Hello UART!" 확인
  → 폴링의 한계: "전화 통화 중 다른 일을 못 하는 상황"

[§4.4] 인터럽트 기반 수신 — 1바이트 재등록 패턴
  → "비서가 전화를 받아 메모해두는 패턴"
  → HAL_UART_Receive_IT() 1바이트 등록
  → HAL_UART_RxCpltCallback() 에서 재등록
  → 수신 데이터를 배열에 저장하는 기초 패턴
  → ⚠️ 링 버퍼는 Ch06에서! 여기서는 단순 배열만

[§4.5] printf 리다이렉트 — _write() 오버라이드
  → Ch02 SWO printf vs Ch04 UART printf 비교
  → _write() 함수 오버라이드 구현
  → LOG_I 매크로가 UART로 출력되는 것 확인
  → "로그 백엔드 교체" — Ch02 아키텍처의 유연성 체험

[§4.6] UART Driver 완성 & 아키텍처 v0.4
  → uart_driver.c/h 파일 분리 (Driver 레이어)
  → UART_Send() 인터페이스 구현
  → 아키텍처 다이어그램 v0.4 완성
  → 연습문제 + Ch05(DMA) 예고
```

### 교육 흐름 패턴

```
§4.1: [설계 먼저] → "무엇을 만들지" 큰 그림 제시 (지도 펼치기)
§4.2: [개념]      → UART 프레임 / Baud Rate 이론
§4.3: [실습 1]    → 폴링 송신 — 즉각적 성공 경험 (Hello World 효과)
§4.4: [개념+실습]  → 인터럽트 수신 — 난이도 상승, 핵심 도전
§4.5: [연결]      → printf 리다이렉트 — Ch02 로그 시스템과 연결
§4.6: [통합]      → Driver 분리 — Ch03 아키텍처 이론의 실현
```

> **핵심 설계 의도:** §4.1에서 "설계도를 먼저 보여주고" → §4.2~4.5에서 "하나씩 구현" → §4.6에서 "설계도대로 정리"하는 **설계-구현-검증** 사이클. Ch03에서 배운 아키텍처 사고를 **최초로 실천**하는 경험.

## 3. 인지 부하 분석

### 핵심 신개념 분석

| # | 신개념 | 인지 부하 | 선행 지식 | 비고 |
|---|--------|-----------|-----------|------|
| 1 | **UART 프레임 구조** | **중간** — 비트 수준 구조는 처음이지만, "편지 봉투" 비유로 직관화 가능 | 없음 (독립) | 필수 — 이론적 토대 |
| 2 | **인터럽트 기반 수신 + 콜백 재등록** | **높음** — 비동기 패턴 + HAL 콜백 구조 + 재등록 루프가 복합적 | Ch01 EXTI 인터럽트 경험 | 핵심 난관 |
| 3 | **printf 리다이렉트 (_write 오버라이드)** | **중간** — 시스템 함수 오버라이드 개념은 새롭지만, 코드 자체는 단순 | Ch02 SWO printf | Ch02 연결점 |
| 4 | **Driver 레이어 분리 (인터페이스 설계)** | **중간** — Ch03에서 개념을 배웠으므로, "실천"은 인지 부하 ↓ | Ch03 4계층 아키텍처 | Ch03 연결점 |

### 인지 부하 판정

**총 핵심 신개념: 4개** — 권장 한도(3개)를 1개 초과

### 권장 처리 방안: 개념 병합으로 3개로 압축

```
[개념 A] UART 프레임 + Baud Rate        → 하나의 "통신 규약" 개념으로 통합 (§4.2)
[개념 B] 인터럽트 수신 + 콜백 재등록     → 핵심 신개념, 충분한 시간 배분 (§4.4)
[개념 C] Driver 레이어 분리              → Ch03 복습 성격, printf는 이 안에 포함 (§4.5~4.6)
```

**압축 전략:**
1. **UART 프레임과 Baud Rate**는 하나의 "통신 약속" 개념으로 묶음 → 별개의 신개념이 아닌 하나의 주제
2. **printf 리다이렉트**는 독립 신개념이 아니라 "Driver 레이어의 유용성"을 체감하는 **적용 사례**로 위치 재조정
3. **인터럽트 수신**에 가장 많은 인지 자원을 집중 — 이것이 Ch04의 진정한 도전

### 절별 인지 부하 배분

| 절 | 신개념 수 | 인지 부하 | 감정 곡선 |
|----|-----------|-----------|-----------|
| §4.1 아키텍처 위치 | 0 | **낮음** | 안심 — "큰 그림을 먼저 보여줌" |
| §4.2 UART 프레임/Baud | 1 | **중간** | 호기심 — "통신 규약이 이렇게 단순하다고?" |
| §4.3 폴링 송신 | 0 | **낮음** | 성취감 — "Hello UART! 출력 성공!" |
| §4.4 인터럽트 수신 | 1 | **높음** | 약간의 불안 → 이해 — "콜백 안에서 재등록하는 이유를 알겠다" |
| §4.5 printf 리다이렉트 | 0.5 | **중간** | 연결 — "Ch02에서 만든 로그가 UART로도 나온다!" |
| §4.6 Driver 완성 | 0.5 | **중간** | 성취감 — "내가 설계한 Driver가 아키텍처에 올라갔다" |

> **감정 곡선:** 안심 → 호기심 → 성취 → **불안(§4.4)** → 연결 → **큰 성취(§4.6)**
> 불안 지점(§4.4)을 챕터 중반에 배치하고, 양쪽에 성취감 구간을 둠으로써 학습 동기를 유지합니다.

## 4. 누적 성장 연결 분석

### Ch01~Ch03 → Ch04 연결 (되돌아보기)

| 연결점 | 선행 챕터 | Ch04 활용 | 자연스러움 |
|--------|-----------|-----------|-----------|
| EXTI 인터럽트 경험 | Ch01 | "Ch01에서 버튼 인터럽트를 썼듯이, UART도 인터럽트로 수신합니다" | **매우 자연** — 동일한 NVIC/콜백 패턴 |
| SWO printf 리다이렉트 | Ch02 | "SWO 대신 UART로 printf 경로를 바꿉니다" | **매우 자연** — 출력 경로만 변경 |
| LOG_I/D/W/E 매크로 | Ch02 | 모든 코드 예제에서 LOG_D로 UART 동작 확인 | **필수 연결** — Ch02 표준 인프라 |
| 4계층 아키텍처 | Ch03 | Driver 레이어에 uart_driver 배치, 인터페이스 설계 | **핵심 연결** — Ch03 설계도에 첫 블록 추가 |
| 시퀀스 다이어그램 읽기 | Ch03 | Ch04에서 시퀀스 다이어그램을 **직접 작성** (읽기→쓰기 전환) | **자연스러운 발전** |
| 아키텍처 다이어그램 v0.3 | Ch03 | v0.3 → v0.4로 업데이트 (Driver 블록 추가) | **누적 성장의 핵심 메커니즘** |

### Ch04 핵심 깨달음 예측

**"아, Driver 레이어를 만든다는 게 이런 거구나!"**

이 깨달음이 자연스러우려면:
1. §4.1에서 **설계도(v0.3)를 먼저 보여주고** "여기 Driver 영역이 비어 있죠? 오늘 채웁니다"
2. §4.3~4.5에서는 **HAL 함수를 직접 호출하는 코드**를 먼저 작성 (아직 Driver 분리 안 함)
3. §4.6에서 **"이 코드를 Driver로 분리하면?"** 질문 → 리팩토링 체험
4. 이 순서가 **"왜 Driver가 필요한지"**를 체감하게 만듦 (문제 먼저 → 해결 나중)

### Ch04 → Ch05~Ch06 연결 (내다보기)

| 예고 포인트 | 실현 챕터 | 예고 문구 제안 |
|-------------|-----------|---------------|
| "폴링 송신의 CPU 낭비" | Ch05 DMA | "Ch05에서 DMA를 배우면, CPU가 전송을 기다리지 않아도 됩니다" |
| "단순 배열 수신의 한계" | Ch06 링 버퍼 | "수신 데이터가 빨리 오면 배열이 넘칩니다. Ch06에서 링 버퍼로 해결합니다" |
| "UART_Send() 인터페이스" | Ch06 DMA 업그레이드 | "같은 UART_Send() 인터페이스 뒤에서 DMA로 바꿀 수 있습니다 — 이것이 Driver 분리의 힘" |

> **특히 세 번째 예고가 중요합니다.** Ch06에서 UART_Send() 내부만 DMA로 교체하고 상위 레이어 코드는 변경하지 않는 경험은 **Ch03 의존 방향 원칙의 가장 강력한 실증**이 됩니다.

## 8. 비유 및 실생활 예시 제안

| 개념 | 비유 | 근거 |
|------|------|------|
| UART 프레임 | **편지 봉투** — 봉투 앞면(Start bit)을 열면 편지지(Data 8bit)가 있고, 맞춤법 확인 도장(Parity)을 찍고, 봉투 뒷면을 봉함(Stop bit) | 순차적 구조가 물리적 봉투와 1:1 대응 |
| Baud Rate | **대화 속도 약속** — 전화 통화 시 "천천히 말해주세요" vs "빨리 말해도 돼요". 양쪽이 같은 속도로 약속해야 대화가 됨 | 통신의 "동기화" 필요성을 직관적으로 전달 |
| 폴링 vs 인터럽트 | **전화 기다리기** — 폴링 = 전화기 앞에서 울릴 때까지 대기 (아무것도 못 함). 인터럽트 = 벨이 울리면 받겠다고 해두고 다른 일 수행 | Ch01 GPIO 폴링/인터럽트 비교와 동일 패턴 — "또 이 비유!" 감으로 이전 학습 강화 |
| 콜백 재등록 | **비서에게 매번 부탁** — "다음 전화도 받아주세요"를 매번 말해야 함. 한 번 받고 끝나면 다음 전화를 놓침 | HAL_UART_Receive_IT 1회성 등록 → 재등록 필요성의 핵심 |
| Driver 분리 | **식당 주방장** — 손님(App)이 "스테이크 하나요"라고만 하면, 주방장(Driver)이 알아서 가스레인지(HAL)를 조작. 손님이 가스레인지 조작법을 알 필요 없음 | Ch03 레스토랑 비유의 **연속** — 같은 비유 세계관 유지 |
| printf 리다이렉트 | **우편함 교체** — 같은 편지(printf 호출)를 보내도, 우편함(SWO vs UART)을 바꾸면 다른 경로로 배달됨 | Ch02 로그 백엔드 교체 개념과 직접 연결 |

## 9. 종합 평가

| 항목 | 평가 | 비고 |
|------|------|------|
| 학습 목표 | **적절** (6개, 블룸 4수준) | 적용 수준 3개 — 실습 중심 챕터에 부합 |
| 절 구조 | **적절** (6개 절, 설계-구현-검증 사이클) | §4.1 아키텍처 도입부 필수 포함 |
| 인지 부하 | **관리 가능** (4→3개 압축) | §4.4 인터럽트 수신이 최대 난관 — 충분한 시간 배분 |
| 누적 성장 | **우수** (6개 연결점) | Ch01 인터럽트 + Ch02 로그 + Ch03 아키텍처 모두 활용 |
| Ch05 연결 | **자연스러움** | "폴링의 한계 → DMA로 해결" 예고 |
| 4시간 분량 | **적절** | 이론 46% : 실습 50% : 휴식 4% |
| 감정 곡선 | **적절** | 안심→호기심→성취→불안→연결→큰성취 |

### 핵심 주의 사항 3가지

1. **§4.4 인터럽트 수신이 Ch04의 승부처** — 시간과 비유를 충분히 투입. 시퀀스 다이어그램으로 재등록 루프를 시각화.
2. **링 버퍼는 절대 Ch04에서 다루지 않음** — Ch06으로 명확히 이관. "단순 배열이면 충분합니다. 더 좋은 방법은 Ch06에서!" 한 줄 예고만.
3. **§4.6 Driver 분리의 동기 부여가 핵심** — "Ch06에서 내부만 DMA로 바꿔도 App 코드는 안 바뀝니다"라는 미래 이점을 반드시 제시.
