## 각 절 상세 설계

### 3.1 왜 아키텍처가 필요한가? (25분, ~2500자)

**도입 전략: "3분 공포 체험"** (강사 권고 채택)

1. **공포 체험 (5분)**: v0.2의 `main.c`를 화면에 띄운 후, "여기에 UART, DMA, TIM, SPI, I2C, LCD, 스텝모터 코드를 전부 넣으면?"
   - 미리 준비한 500줄 "스파게티 main.c" 스크롤 시연
   - 질문: "이 코드에서 SHT31 센서 읽기가 안 될 때 어디부터 보시겠습니까?"

2. **폴링 확장성 문제 (5분)**: (강사 권고 채택)
   ```c
   while (1) {
       button_poll();        // 10ms 주기
       uart_check_rx();      // 수신 체크
       sensor_read_poll();   // 80ms 블로킹!
       lcd_refresh();        // 40ms 소요
       stepper_step();       // 1ms 정밀 타이밍 필요
       // → sensor_read_poll() 80ms 동안 stepper가 멈춤!
   }
   ```
   킬러 문장: "시침이 떨리는 시계를 만들고 싶으시면 이 구조로 가세요."

3. **전환 (5분)**: "아키텍처는 이론이 아니라, 야근하지 않기 위한 기술입니다."
   정리된 4계층 구조 미리보기 → 3.2로 자연스럽게 연결

4. **Before/After 비교 다이어그램** (10분): 스파게티 vs 레이어드 SVG

**포함 요소**:
- aside(faq): "Ch01~02 코드는 잘 동작하는데 왜 바꿔야 하나요?"
- aside(instructor-tip): "스파게티 코드 시연에서 수강생 반응이 없으면 '이 코드 리뷰해 주세요'라고 요청해 보세요"

### 3.2 레이어드 아키텍처 4계층과 의존 원칙 (65분, ~4000자)

**핵심 변경: 레이어 정의 + 의존 원칙을 하나의 절로 통합**

이유: 교육설계자 분석에서 "의존 원칙은 레이어 개념에 종속적이며, 레이어 이해 후에는 직관적"으로 평가. 별도 절로 분리하면 오히려 흐름이 끊김.

**비유: 레스토랑 주방** (강사 권고 채택, 사무실 조직도 대체)

| 레이어 | 레스토랑 대응 | 책임 | 경계 규칙 |
|--------|-------------|------|----------|
| **HAL** (파랑 #1A73E8) | 가스레인지, 오븐, 냉장고 | 하드웨어 직접 제어. ST 제공, 수정 불가 | 기기 교체(가스→인덕션)해도 요리법 동일 |
| **Driver** (초록 #34A853) | 요리사 (단위 조리 담당) | HAL 위에서 주변장치 캡슐화 | HAL만 호출. "볶기/끓이기" 단위 동작 제공 |
| **Service** (노랑 #FBBC04) | 메뉴 관리자 (코스 구성) | 여러 Driver 조합, 비즈니스 로직 | Driver만 호출 (HAL 직접 접근 금지) |
| **App** (빨강 #EA4335) | 홀 매니저 (고객 응대) | 주문 접수, 최종 결정 | Service만 호출 |

**의존 원칙 설명** (같은 절 내에서 자연스럽게 이어짐):
- "홀 매니저가 가스레인지를 직접 만지면?" → 단방향 의존 원칙
- "주문서 = API 함수 시그니처" → 인터페이스 개념
- "가스레인지를 인덕션으로 교체해도 메뉴 동일" → HAL 교체 가능성

**비유의 한계 (반드시 명시)**:
- 실제 레스토랑: 요리사가 "재료 떨어졌다"고 홀에 통보 → 소프트웨어: 콜백/이벤트로 처리 (의존성 역전이 아닌 이벤트 기반 통지)
- 동시성(인터럽트/DMA)은 이 비유로 설명 불가

**실습 (20분)**: 인터페이스 함수 시그니처 설계 미니 실습
```c
// led_driver.h — 수강생이 직접 시그니처 작성
void led_driver_init(void);
void led_driver_set(led_state_t state);
void led_driver_toggle(void);
```

**메타코그니션 체크**: "Ch01의 `HAL_GPIO_WritePin()`은 어느 레이어? — HAL입니다. 우리는 이미 HAL을 쓰고 있었습니다."

**포함 요소**:
- aside(tip): "Driver 네이밍 규칙: `[주변장치]_driver.c/h`"
- aside(interview): "레이어드 아키텍처의 장점 — 변경 영향 최소화, 버그 격리, 병렬 개발"
- aside(faq): "로그 시스템(LOG_I)은 모든 계층에서 쓰는데, 어느 레이어에 넣나요?" → 횡단 관심사 개념 맛보기

### 3.3 Ch01~02 코드를 레이어에 배치하기 (45분, ~3000자)

**핵심 활동: 워크시트 형태의 분류 실습** (교육설계자 권고)

패턴: "질문 → 사고 → 답 → 해설"

| 코드 요소 | 질문 | 정답 | 해설 |
|-----------|------|------|------|
| `HAL_GPIO_Init()` | "이 함수는 ST가 만들었나, 우리가 만들었나?" | HAL | ST 제공 함수, 레지스터 직접 접근 추상화 |
| `MX_GPIO_Init()` | "CubeMX가 생성한 초기화 코드는?" | HAL/Driver 경계 | CubeMX 코드는 HAL 설정이지만 프로젝트별 구성 포함 |
| `HAL_GPIO_EXTI_Callback()` | "콜백은 어느 레이어?" | App (이벤트 핸들러) | 토론 유도: Driver vs App 경계. "정답이 하나가 아닐 수 있다" |
| `log_print()` | "로그 출력 구현체는?" | Driver | SWO/RTT 하드웨어 접근 |
| `LOG_I()` 매크로 | "매크로 인터페이스는?" | 횡단 관심사 | 모든 계층에서 사용, 특정 계층에 속하지 않음 |
| `main()` | "메인 루프와 초기화 순서는?" | App | 최상위 오케스트레이션 |

**의존 방향 위반 발견 실습**:
```c
// v0.2 main.c에서 발견되는 위반
void HAL_GPIO_EXTI_Callback(uint16_t GPIO_Pin)
{
    // App 레이어에서 HAL 함수를 직접 호출! (계층 건너뛰기)
    HAL_GPIO_WritePin(GPIOA, GPIO_PIN_5, ...);  // ← 위반!
    // 이상적: led_driver_toggle();  ← Driver API 호출
}
```
"이 위반은 Ch01에서 의도적으로 만들었습니다. Ch04부터 점진적으로 수정합니다."

**산출물**: Ch01~02 코드를 4계층에 매핑한 다이어그램 (수강생이 직접 그림/채움)

**포함 요소**:
- aside(metacognition): "EXTI 콜백을 Driver/App 중 어디에 넣을지 고민되셨나요? 이 고민 자체가 아키텍처 설계입니다."
- aside(instructor-tip): "위반 발견 실습에서 수강생이 '위반 아니지 않나요?'라고 하면, '맞아요, 아직 Driver가 없으니 직접 호출할 수밖에 없었죠. Ch04에서 해결합니다'로 응대"

### 3.4 FSM(유한 상태 머신) 맛보기 (35분, ~2000자)

**핵심 변경: "맛보기 수준"으로 축소** (교육설계자 방안 A 채택)

- 깊이: 개념 도입 + 기존 코드 재해석까지만
- 직접 FSM 코드 작성은 Ch08(스톱워치)에서 처음 수행
- 상태 전이 다이어그램 "읽기"까지만, "작성"은 Ch08부터

**비유: 자판기** (검증 완료, 임베디드 교재 고전 비유)
- 상태: 대기 → 동전 투입됨 → 음료 선택됨 → 배출 중
- 이벤트: 동전 투입, 버튼 누름, 배출 완료
- 전이: "대기" + "동전 투입" → "금전 투입됨"
- 액션: 잔액 표시, 음료 배출, 거스름돈 반환

**Ch01 디바운싱 → FSM 재해석** (강사 핵심 권고):
```
[IDLE] --버튼 누름(엣지 감지)--> [PRESSED]
[PRESSED] --50ms 경과--> [DEBOUNCED] → LED 토글, 카운트 증가
[PRESSED] --50ms 미만 재트리거--> [PRESSED] (무시)
[DEBOUNCED] --> [IDLE]
```
메타코그니션: "Ch01에서 `if (now - last >= 50ms)` 패턴이 이미 2-상태 FSM이었습니다."

**레스토랑 비유 연결**: 주문 상태(접수 → 조리중 → 서빙 → 완료) = 상태 머신

**포함 요소**:
- aside(interview): "FSM을 실무에서 어디에 적용했는가?" → "버튼 디바운싱, UART TX/RX 상태, 모터 제어, UI 화면 전환"
- aside(tip): "FSM은 if-else 체인의 진화형입니다. 상태가 3개 이상이면 FSM 도입을 고려하세요"
- aside(faq): "FSM이 어려운데 당장 코드로 작성해야 하나요?" → "아닙니다. Ch08에서 처음 작성합니다. 지금은 개념만 이해하면 됩니다."

### 3.5 프로젝트 아키텍처 v0.3 설계 (50분, ~3500자)

**핵심 활동: 아키텍처 설계도 직접 작성 (최종 산출물)**

**시퀀스 다이어그램 읽기 (10분)** — 별도 절에서 통합 이동:
- Ch01 버튼 → EXTI → 콜백 → LED 토글 시퀀스를 SVG로 제시
- "이것이 시퀀스 다이어그램입니다. Ch04부터 매 챕터에서 직접 작성하게 됩니다."
- 읽기만, 작성은 Ch04부터

**아키텍처 설계도 v0.3 작성 실습 (30분)**:
1. 빈 4계층 다이어그램 템플릿 제공
2. Ch01~02 컴포넌트를 올바른 레이어에 배치
3. **미래 컴포넌트를 회색으로 미리 표시** (교육설계자 권고):
   - Driver: `uart_driver (Ch04)`, `tim_driver (Ch07)`, `rtc_driver (Ch09)`, `stepper_driver (Ch10)`, `sht31_driver (Ch11)`, `ili9341_driver (Ch12)` — 회색
   - Service: `clock_service (Ch10)`, `sensor_service (Ch11)`, `gfx_service (Ch13)` — 회색
   - App: `cli_app (Ch15)`, `protocol_app (Ch16)` — 회색
4. "매 챕터에서 회색 블록이 하나씩 색칠됩니다"

**정리 + 면접 포인트 (10분)**:

면접 포인트 박스:
```
"레이어드 아키텍처의 장점을 실제 경험으로 설명하세요"
키워드: ① 변경 영향 최소화 ② 인터페이스 기반 교체 ③ 버그 격리 ④ 병렬 개발
```

디버깅 계층 판단 가이드:
| 증상 | 의심 레이어 |
|------|-----------|

| 전송은 되지만 데이터 깨짐 | Driver |
| 데이터 정상이지만 타이밍 틀림 | Service |
| 화면 표시 값 이상 | App |

**Ch04 예고**:

## 교육설계자/강사 질문에 대한 저자 답변

### Q1. 레스토랑 주방 비유로 변경할 수 있는가?
**가능합니다.** 레스토랑 비유가 사무실 조직도보다 단방향 의존을 자연스럽게 설명하고, FSM(주문 상태)과도 연결됩니다. 인터페이스("주문서")도 직관적입니다. 비유의 한계(역방향 통보, 동시성)는 본문에 명시하겠습니다.

### Q2. FSM/시퀀스를 맛보기로 축소해도 학습 목표 달성 가능한가?
**가능합니다.** 조건:
- FSM LO-4는 "해석" 수준으로 제한 (작성은 Ch08)
- 시퀀스 다이어그램은 LO에서 제외하고 3.5절에서 "읽기만" 소개
- Ch04 도입부의 "아키텍처 위치" 섹션은 레이어 다이어그램 + 인터페이스 시그니처만으로 작성 가능 (시퀀스 다이어그램은 Ch04에서 처음 작성)

### Q3. 신개념 4개를 3개로 통합할 방법이 있는가?
**있습니다.** "레이어드 아키텍처"와 "의존 방향 원칙"은 동전의 양면이므로 3.2절에서 하나의 개념 블록으로 통합합니다. 결과:
1. 레이어드 아키텍처 + 의존 원칙 (3.2 — 통합)
2. FSM 기초 (3.4 — 맛보기)
3. 시퀀스 다이어그램 (3.5에 흡수 — 읽기만)

실질적 핵심 신개념은 **2개** (레이어+FSM), 보조 1개 (시퀀스 읽기)로 인지 부하 한도 내입니다.
