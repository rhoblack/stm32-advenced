### 막힘 포인트 상세 대응

**#1 "왜 DMA가 UART에 필요한가?"** (★★★★★ 반드시 나옴)
- 발생 시점: §6.0 도입부
- 킬러 문장: "Ch04 인터럽트는 편의점 알바입니다 — 손님(바이트) 한 명 올 때마다 '어서오세요!' Ch06 DMA는 무인 택배함입니다 — 쌓여있다 한꺼번에 처리합니다"
- Ch05 연결: "Ch05에서 DMA가 CPU 대신 메모리를 복사했죠? 이제 그 DMA가 UART 데이터를 RAM에 자동으로 채워줍니다"

**#2 링 버퍼 모듈로 연산** (★★★★★ 개념적 허들 최대)
- 발생 시점: §6.2
- 5단계 분산 전략:
  1. 먼저 일반 배열로 FIFO 구현 ("배열 끝에 도달하면?")
  2. "배열을 동그랗게 구부린다" → 원형 개념 시각화
  3. `% SIZE`가 "자동으로 처음으로 돌아가는 마법"임을 시연
  4. head/tail 포인터를 보드에서 실시간 로그로 추적
  5. "축하합니다, 이것이 운영체제의 기본 자료구조입니다"

**#3 DMA와 링 버퍼 역할 경계** (★★★★☆)
- 발생 시점: §6.3
- 핵심 구분표:

| 역할 | 누가 | 언제 |
|------|------|------|
| UART 레지스터 → RAM 복사 | DMA (하드웨어) | 바이트 도착 즉시 |
| head 포인터 갱신 | DMA 콜백 (소프트웨어) | 전송 완료/반완료 시 |
| tail 포인터 갱신 + 데이터 읽기 | 메인 루프 (소프트웨어) | 폴링으로 주기적 |

**#4 오버런 처리** (★★★★☆)
- 발생 시점: §6.2~6.3
- 시각적 시연: 링 버퍼 크기 8로 축소 → PC에서 10바이트 연속 전송 → 2바이트 손실 확인
- "실무에서는 버퍼 크기를 (보레이트 × 최대 처리 지연) / 8로 계산합니다"

**#5 DMA 콜백 컨텍스트** (★★★☆☆)
- 발생 시점: §6.3~6.5
- 실습: 콜백 안에서 `HAL_Delay(100)` 넣으면? → 시스템 멈춤 시연 → "인터럽트 안에서 블로킹 금지" 체험

### 면접 모범 답변 상세

**Q1 모범 답변**: "DMA를 Circular 모드로 설정하여 UART RX 데이터를 링 버퍼에 자동 저장합니다. Half-Transfer와 Transfer-Complete 콜백으로 수신 위치를 추적하고, 메인 루프에서 tail 포인터로 데이터를 소비합니다. 오버런 방지를 위해 버퍼 크기는 (보레이트/10) × 최대 처리 지연시간으로 산정하며, IDLE Line 감지를 조합하면 가변 길이 패킷도 처리할 수 있습니다."

**Q3 모범 답변**: "DMA 콜백은 DMA ISR 내에서 실행되므로 (1) 실행 시간을 최소화해야 합니다 — 플래그 설정이나 포인터 갱신 정도만 수행. (2) HAL_Delay(), printf() 등 블로킹 함수 호출 금지. (3) 공유 변수는 volatile 선언 필요. (4) 다른 인터럽트와의 우선순위 관계를 고려한 Critical Section 설계가 필요합니다."

### 절별 상세 실행 가이드

#### §6.0 도입 (10분)
- **오프닝**: "Ch05에서 DMA로 메모리 복사 속도를 10배 올렸죠? 오늘은 그 DMA를 UART에 장착합니다"
- **라이브 데모**: Ch05 성능 데이터 (CPU vs DMA) 그래프 재시연 (1분)
- **아키텍처 위치**: v0.6 다이어그램에서 uart_driver.c의 DMA 연결 표시 (2분)
- **기대감 조성**: "오늘 수업이 끝나면, 여러분 보드는 데이터를 받으면서 동시에 다른 일을 합니다" (1분)
- **목표 선포**: "v0.4(인터럽트) → v0.6(DMA) 업그레이드. API는 바꾸지 않고, 엔진만 교체합니다" (1분)

#### §6.1 UART DMA 아키텍처 (30분)
- **F411RE DMA 매핑표**: USART2_RX = DMA1_Stream5_Channel4, USART2_TX = DMA1_Stream6_Channel4
- **SVG 시퀀스**: PC → UART RX → DMA → RAM(링 버퍼) → 메인 루프에서 처리
- **HAL API 소개**: `HAL_UART_Receive_DMA()`, `HAL_UARTEx_ReceiveToIdle_DMA()`, `HAL_UART_Transmit_DMA()`
- **IDLE Line 감지**: "고정 길이가 아닌 가변 길이 데이터를 받으려면?" → IDLE Line이 답
- **라이브 시연**: CubeMX에서 USART2 DMA 설정 화면 보여주기 (5분)

#### §6.2a 링 버퍼 개념 (20분)
- **비유 1**: "카세트 테이프 — 끝까지 가면 처음으로 되감아서 덮어씀"
- **시각화**: SVG로 head/tail 포인터 이동 애니메이션 설명
- **핵심 연산 3가지**: write(1줄), read(1줄), available(1줄)
- **오버플로우**: "head가 tail을 한 바퀴 돌아 추월하면?" → 데이터 손실
- **비유 한계 명시**: "카세트는 기계적, 링 버퍼는 RAM — 덮어쓰면 복구 불가"

#### §6.2b 링 버퍼 구현 (20분)
- **단계 1**: 구조체 정의 (`buffer[], head, tail, size`)
- **단계 2**: `ring_buffer_write()` — `buf[head] = data; head = (head+1) % size`
- **단계 3**: `ring_buffer_read()` — `data = buf[tail]; tail = (tail+1) % size`
- **단계 4**: `ring_buffer_available()` — `(head - tail + size) % size`
- **라이브**: ch06_01_ring_buffer.c 실행 + LOG_D로 head/tail 값 실시간 추적

#### §6.3 RX DMA 구현 (30분)
- **초기화**: `HAL_UARTEx_ReceiveToIdle_DMA(&huart2, rx_dma_buf, DMA_BUF_SIZE)`
- **콜백**: `HAL_UARTEx_RxEventCallback()` → 수신 바이트 수 계산 → 링 버퍼에 push
- **메인 루프**: `while(ring_buffer_available() > 0) { byte = ring_buffer_read(); }`
- **라이브 데모**: PC 키보드 입력 → 보드 수신 + LOG_I 출력 (즉각적 성취감!)
- **핵심**: "DMA가 하드웨어 버퍼를 채우고, 콜백이 링 버퍼로 옮기고, 메인 루프가 소비한다"

#### §6.4 TX DMA 구현 (30분)
- **TX 큐**: 송신할 데이터를 TX 링 버퍼에 먼저 대기
- **비블로킹 송신**: `uart_send_data()` 호출 → 즉시 반환 (DMA가 백그라운드 전송)
- **TX Complete 콜백**: 다음 chunk 전송 또는 TX_IDLE 상태 전환
- **printf 리다이렉트**: `_write()` → `HAL_UART_Transmit_DMA()` → stdout 리다이렉트
- **라이브**: `printf("Hello DMA\r\n")` → 보드에서 수신 확인 (블로킹 없음!)

#### §6.5 통합 드라이버 (20분)
- **FSM**: `TX_IDLE` → `TX_BUSY` → `TX_IDLE` / `RX_RECEIVING` (항시 활성)
- **race condition 방지**: `__disable_irq()` / `__enable_irq()` 또는 BASEPRI 사용
- **API 유지**: `uart_send_data()`, `uart_receive_data()`, `uart_available()` — Ch04와 동일 시그니처!
- **핵심 메시지**: "인터페이스(API)는 동일, 구현(엔진)만 DMA로 교체 — 이것이 레이어드 아키텍처의 힘"

#### §6.6 종합 실습 (40분)
- **미션**: 양방향 DMA 통신 — PC에서 명령 수신 → 보드에서 응답 전송
- **CPU 점유율 측정**: Ch04 폴링 vs Ch04 인터럽트 vs Ch06 DMA 비교
- **축소안 1** (시간 부족): RX DMA만 완성 + TX는 완성 코드 배포
- **축소안 2** (심각한 시간 부족): ch06_04 완성 코드 배포 + 실행만 확인
- **확장안** (시간 충분): 에러 처리(ORE, FE) + 재초기화 루틴 추가

#### §6.7 정리 (15분)
- **아키텍처 v0.6**: Driver 레이어에 DMA 모드 추가, Ch02 로그 시스템 비동기화
- **면접 포인트 3개**: DMA+링 버퍼, 오버런 처리, IDLE Line 감지
- **Ch07 예고**: "UART로 시간을 주고받을 수 있게 되었으니, 이제 '정확한 시간'을 만들 차례 — Timer입니다"

## 5. 비유 검증

### 비유 1: 링 버퍼 = "카세트 테이프"

| 링 버퍼 개념 | 카세트 대응 | 정확도 | 한계 |
|------------|-----------|--------|------|
| 고정 크기 배열 | 테이프 길이 고정 | ★★★★★ | — |
| head 포인터 (쓰기 위치) | 녹음 헤드 위치 | ★★★★★ | — |
| tail 포인터 (읽기 위치) | 재생 헤드 위치 | ★★★★☆ | 카세트는 녹음/재생 동시 불가 |
| 모듈로 연산 (순환) | 자동 되감기 | ★★★★☆ | 카세트는 수동 되감기 |
| 오버플로우 (head가 tail 추월) | 아직 안 들은 노래를 덮어씀 | ★★★★★ | — |
| 데이터 손실 시 복구 불가 | 덮어쓴 녹음은 복구 불가 | ★★★★★ | — |

**활용 시점**: §6.2a 도입부, 링 버퍼 개념 첫 설명
**반드시 명시할 한계**: "카세트는 녹음과 재생을 동시에 못 하지만, 링 버퍼는 DMA(쓰기)와 CPU(읽기)가 동시에 접근합니다. 이것이 race condition의 원인입니다."

### 비유 2: 순환 DMA = "음악 재생 목록 반복(Repeat All)"

| DMA 개념 | 재생 목록 대응 | 정확도 | 한계 |
|----------|-------------|--------|------|
| Circular Mode | Repeat All 설정 | ★★★★★ | — |
| DMA 버퍼 시작 주소 | 첫 번째 곡 | ★★★★★ | — |
| 전송 완료 → 자동 재시작 | 마지막 곡 후 처음부터 | ★★★★★ | — |
| Half-Transfer 콜백 | "절반 재생 완료" 알림 | ★★★★☆ | 음악 앱에는 없는 기능 |
| 도중 읽기 (tail 이동) | 현재 재생 중인 곡 확인 | ★★★★☆ | — |

**활용 시점**: §6.1 Circular DMA 모드 설명 시
**반드시 명시할 한계**: "음악 앱은 'Half-Transfer' 알림이 없지만, DMA는 절반 전송 완료 시 인터럽트를 걸어 더블 버퍼링 효과를 줄 수 있습니다."

### 비유 3: DMA+링 버퍼 = "자동 우편함 + 우체부"

| 개념 | 우편 대응 | 정확도 | 한계 |
|------|----------|--------|------|
| DMA 전송 | 우체부가 편지를 우편함에 넣음 | ★★★★★ | — |
| 링 버퍼 | 회전식 우편함 (칸이 순환) | ★★★★☆ | 실제 우편함은 순환 안 함 |
| CPU 읽기 | 집주인이 우편함에서 꺼냄 | ★★★★★ | — |
| 오버런 | 우편함 꽉 차서 편지가 바닥에 떨어짐 | ★★★★★ | — |
| IDLE 감지 | 우체부가 "오늘 배달 끝!" 신호 | ★★★★☆ | — |

**활용 시점**: §6.3 DMA+링 버퍼 결합 설명 시
**반드시 명시할 한계**: "실제 우편함은 순환하지 않지만, 임베디드에서는 메모리가 제한적이므로 같은 공간을 재활용해야 합니다."

## 8. Ch05 → Ch06 연결 포인트

### Ch05 학습 자산 재활용

| Ch05 개념 | Ch06 적용 | 연결 문장 |
|-----------|----------|----------|
| DMA Mem-to-Mem | UART RX DMA (Periph-to-Mem) | "Ch05에서는 메모리→메모리였죠? 이제 UART→메모리입니다" |
| Circular Mode 개념 | UART RX Circular DMA | "Ch05 Bonus 과제에서 맛봤던 Circular가 여기서 진가를 발휘합니다" |
| 전송 완료 콜백 | RX/TX Complete 콜백 | "콜백 패턴은 동일합니다. 다만 UART 특화 콜백이 추가됩니다" |
| Stream/Channel 매핑 | USART2 RX/TX 매핑 | "Ch05에서 '아무 Stream'이었던 게 이제 '정해진 Stream'입니다" |
| CPU 점유율 비교 | 폴링 vs 인터럽트 vs DMA 비교 | "Ch05에서 측정한 것처럼, 이번에도 3가지 방식을 비교합니다" |

### Ch03 아키텍처 연속성

- **v0.4 (Ch04)**: uart_driver.c — 폴링/인터럽트 기반
- **v0.5 (Ch05)**: DMA 인프라 추가 (횡단 관심사)
- **v0.6 (Ch06)**: uart_driver.c DMA 모드 전환 + ring_buffer.c 추가
- **핵심 메시지**: "API(인터페이스)는 Ch04와 동일합니다. 내부 구현만 DMA로 교체됩니다. 이것이 Ch03에서 배운 '레이어드 아키텍처의 캡슐화' 원칙입니다"

### Ch02 로그 시스템 업그레이드

- Ch02 로그: SWO/ITM 기반 동기식
- Ch06 이후: 링 버퍼 기반 비동기 로그 출력 가능
- **연결**: "Ch02에서 만든 LOG_D/LOG_I가 이제 DMA를 타고 UART로 나갑니다. 로그 출력 때문에 시스템이 멈추지 않습니다"

## 최종 평가

- **강의 현장성**: ⭐⭐⭐⭐⭐ — Ch05 DMA 기반 위에 UART 실전 적용, 단계별 빌드업 우수
- **면접 적중율**: 높음 — DMA+링 버퍼, 오버런 처리, IDLE Line 감지는 임베디드 면접 핵심 3대장
- **불안 관리**: ⭐⭐⭐⭐⭐ — §6.2 링 버퍼가 불안 최고조이나, 5단계 분산 + 즉시 실습으로 해소
- **누적 성장 체감**: ⭐⭐⭐⭐⭐ — v0.4→v0.6 업그레이드, API 유지+엔진 교체 = 아키텍처 원칙 체감

### 가장 중요한 3가지

1. **링 버퍼를 '두렵지 않게' 가르치기** — 모듈로 연산이 낯선 학생들에게 카세트 비유 + 한 줄씩 코딩 + 실시간 로그로 3중 지원. §6.2에서 불안을 느끼지만 §6.3에서 "이래서 만들었구나"로 전환되어야 함.

2. **Ch04 API 유지를 강조하기** — "인터페이스는 동일, 구현만 교체"가 Ch03 아키텍처 원칙의 실전 증명. 학생이 "아, 레이어드 아키텍처가 이런 거구나"라고 체감하는 순간이 Ch06의 진짜 학습 목표.

3. **DMA와 링 버퍼의 역할 경계를 명확히** — DMA가 "자동으로 링 버퍼에 저장"한다는 오해를 사전 차단. "DMA는 하드웨어 버퍼에 쓰고, 콜백이 링 버퍼로 옮기고, 메인 루프가 소비한다"는 3단계를 반복 강조.
