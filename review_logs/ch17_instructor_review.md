## 긍정적 평가

### 1. 강의 현장에서 바로 쓸 수 있는 수준

1절 타임라인과 3절 테이블 FSM 비교 표는 강의 슬라이드로 그대로 활용 가능합니다.
특히 switch-case vs 테이블 비교(그림 17-4)는 판서보다 훨씬 효과적입니다.

### 2. 수강생 단골 질문 예측 및 해소

DIP vs ISP 구분(2절 FAQ), guard 함수 처리 로직(3절 aside.tip),
HFSM 부모 exit/entry 순서(4절 FAQ)가 현장에서 실제로 자주 나오는 질문들입니다.
본문에 사전 배치된 점이 매우 좋습니다.

### 3. 면접 연결 포인트

면접 포인트 aside가 각 절에 배치되어 있고, 특히 6절의 SOLID 면접 포인트는
"임베디드 C에서도 함수 포인터+구조체로 구현 가능"이라는 핵심을 잘 짚었습니다.

| 시간 | 내용 | 적합성 |
|------|------|--------|
| 0~30분 | 도입: v1→v2.0 타임라인 판서 + 학습자가 직접 채우기 | ✅ 참여 유도 |
| 30~60분 | 1절: 아키텍처 진화 리뷰 | ✅ 복습 + 인식 전환 |
| 60~90분 | 2절: 의존성 위반 패턴 | ✅ Before/After 라이브 코딩 |
| 90~120분 | 3절: 테이블 FSM 설명 | ✅ 구조 이해 |
| 120~150분 | 실습: switch-case → 테이블 FSM 리팩토링 | ✅ 성공 경험 |
| 150~180분 | 4절: HFSM 설명 | ⚠ 시간 압박 가능 |
| 180~210분 | 5절: 시나리오 다이어그램 (팀별 발표) | ✅ |

HFSM을 선택적 심화 내용으로 제시하거나, 핵심 3가지 메커니즘만 30분 내에 설명하는 방식 권고.

## 추가 강의 포인트 제안

### 제안 1: 실습 전 "오염 코드" 시연

3절 실습 전에 강사가 의도적으로 switch-case 코드에 새 상태를 추가하는 과정을 라이브로 보여줍니다.
"ALARM_VIEW 추가 시 switch-case에서 몇 군데를 수정해야 하는가?"를 체험시킨 뒤
테이블 방식으로 전환하면 OCP의 가치를 몸으로 느낄 수 있습니다.

### 제안 2: 5절 시나리오 팀 발표

3개 시나리오를 팀별로 나눠 "이 시나리오에서 버그가 생긴다면 어느 컴포넌트가 원인인가?"를 논의하게 합니다.
예: 알람이 울리는데 LCD가 바뀌지 않는 경우 → alarm_service → ui_service 호출 부분 점검.
