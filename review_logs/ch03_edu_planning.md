| LO-2 | HAL/Driver/Service/App 경계에서 어떤 코드가 어디에 속하는지 판단할 수 있다 | 분석 | **적절** — 핵심 역량. 단, "판단"은 블룸 4수준(분석)에 해당하므로 이해→분석 사이에 적용 수준 목표가 필요 |

### 최종 권장 목표 체계 (5개)

```
LO-1 [이해]   4계층(HAL/Driver/Service/App) 아키텍처의 정의와 계층별 책임을 설명할 수 있다
LO-2 [적용]   Ch01~02 코드를 4계층 다이어그램에 올바르게 배치할 수 있다
LO-3 [분석]   코드가 어느 레이어에 속하는지 판단하고, 의존 방향 위반을 식별할 수 있다
LO-4 [분석]   FSM의 상태/전이/이벤트/액션 4요소를 식별하고 상태 전이 다이어그램을 해석할 수 있다
LO-5 [창조]   프로젝트 아키텍처 설계도 v0.3을 작성할 수 있다
```

> **판단 근거:** 기존 3개는 기억~적용에 집중되어 있어, Ch03의 핵심 산출물인 "아키텍처 설계도 작성"에 해당하는 창조 수준이 빠져 있었음. FSM 관련 목표도 별도 분리하여 인지 부하를 명확히 관리.

## 2. 학습 흐름 평가

### 제안된 흐름 검토

| 단계 | 제안 내용 | 평가 |
|------|-----------|------|
| 도입 | "왜 아키텍처가 필요한가?" (Ch01~02의 혼란스러운 코드 예시) | **우수** — 문제 인식(Problem-First) 접근은 동기 부여에 매우 효과적 |
| 개념 | 4계층의 정의와 상향식 의존 원칙 | **수정 필요** — "상향식 의존"이라는 표현이 혼란 가능. "상위→하위 단방향 의존"으로 명확화 |
| 예시 | 기존 GPIO/UART/Log 코드 분석 | **수정 필요** — Ch03 시점에서 UART 코드는 아직 없음(Ch04). GPIO/Log 코드만 사용해야 함 |
| 실습 | Ch01~02 코드를 직접 4계층에 재배치 | **적절** — 구체적이고 자기 코드 활용 |

### 권장 학습 흐름 (수정안)

```
[도입] "왜 아키텍처가 필요한가?"
  → Ch01~02 코드를 main.c 하나에 모두 넣은 "스파게티 버전" 제시
  → "기능이 10개로 늘어나면?" 질문으로 문제 체감

[개념 1] 레이어드 아키텍처 4계층
  → 건물 층 비유: 1층(HAL-기초공사) → 2층(Driver-배관/전기) → 3층(Service-인테리어) → 4층(App-입주자)
  → 각 층의 책임 정의 + "위층은 아래층을 사용하지만, 아래층은 위층을 모른다"

[개념 2] 의존 방향 원칙
  → 잘못된 의존(App→HAL 직접 호출) vs 올바른 의존(App→Service→Driver→HAL) 비교
  → 위반 시 어떤 문제가 생기는지 구체적 시나리오

[예시/실습 1] Ch01~02 코드 레이어 배치
  → HAL_GPIO_Init() = HAL, GPIO_LED_Toggle() = Driver, EXTI_Callback = ?, logger.c = ?
  → 수강생이 직접 분류하는 워크시트 형태

[개념 3] FSM 기초 (상태/전이/이벤트/액션)
  → 자판기 비유: 대기→동전투입→선택→배출→대기
  → Ch01 버튼 디바운싱을 FSM으로 재해석

[개념 4] 시퀀스 다이어그램 기초
  → 카카오톡 단체방 메시지 전달 비유
  → Ch01 EXTI 인터럽트 흐름을 시퀀스 다이어그램으로 표현

[실습 2] 프로젝트 아키텍처 설계도 v0.3 작성
  → 빈 4계층 다이어그램 템플릿에 Ch01~02 컴포넌트 배치
  → "이후 챕터마다 여기에 블록을 추가합니다" 예고

[정리] 핵심 요약 + 면접 포인트 + Ch04 예고
  → "다음 챕터에서 UART를 구현하면, 이 다이어그램의 Driver 레이어에 uart_driver가 추가됩니다"
```

### 흐름 변경 근거

1. **UART 코드 예시 제거** — Ch04 선행 지식 의존 방지. Ch01~02 코드만으로 충분히 레이어 배치 실습 가능
2. **FSM과 시퀀스 다이어그램을 별도 개념 블록으로 분리** — 인지 부하 관리 (아래 §3 참조)
3. **실습을 2회로 분산** — 개념 → 즉시 적용 → 다음 개념 → 즉시 적용 패턴이 장기 기억 형성에 효과적

## 3. 인지 부하 관리

### 핵심 신개념 분석

| # | 신개념 | 인지 부하 | 선행 지식 요구 |
|---|--------|-----------|---------------|
| 1 | **레이어드 아키텍처 4계층** | **높음** — 추상적 개념, 코드에 직접 대응하는 물리적 실체가 없음 | C 프로젝트 구조 이해 (Ch01~02로 확보) |
| 2 | **의존 방향 원칙** | **중간** — 개념 1에 종속. 개념 1 이해 후에는 직관적 | 레이어 개념 이해 |
| 3 | **FSM 기초** | **높음** — 새로운 사고 모델. 순차 실행에 익숙한 개발자에게 패러다임 전환 | 없음 (독립적) |
| 4 | **시퀀스 다이어그램** | **낮음** — 시각적 표현법으로, 개념보다는 표기법에 가까움 | 함수 호출 순서 이해 |

### 인지 부하 판정

**총 핵심 신개념: 4개** — 권장 한도(3개)를 1개 초과

### 권장 처리 방안

**방안 A (권장): FSM과 시퀀스 다이어그램을 "맛보기 수준"으로 축소**
- 레이어드 아키텍처 + 의존 원칙 = **핵심** (전체 분량의 60%)
- FSM = **기초 개념만** 소개, 자판기 비유 + Ch01 버튼 FSM 재해석 1개 (20%)
- 시퀀스 다이어그램 = **읽는 법만** 소개, 직접 작성은 Ch04부터 (10%)
- 아키텍처 설계도 작성 실습 (10%)

**방안 B (비권장): FSM/시퀀스를 Ch04로 완전 이관**
- 장점: 인지 부하 최소화
- 단점: Ch04 도입부에서 "아키텍처 위치" 섹션 작성 시 FSM/시퀀스 선행 지식 부재. TABLE_OF_CONTENTS.md에 Ch03 핵심 주제로 이미 명시되어 있어 구조 변경 비용 큼

> **최종 권장: 방안 A** — FSM은 "개념 도입"만, 시퀀스는 "읽기"만. Ch08(Stopwatch FSM)과 Ch17(FSM 심화)에서 단계적 심화하는 스캐폴딩 설계.

## 4. 누적 성장 연결 분석

### Ch01~02 → Ch03 연결 (되돌아보기)

| 연결점 | 구체적 활용 | 효과 |
|--------|------------|------|
| Ch01 `HAL_GPIO_Init()` | HAL 레이어 대표 사례 | "이미 써봤던 함수가 HAL 레이어였다" — 친숙함 기반 이해 |
| Ch01 `HAL_GPIO_EXTI_Callback()` | 콜백은 어느 레이어? (Driver vs App 경계 토론) | 비판적 사고 촉진, 정답이 하나가 아닐 수 있음을 경험 |
| Ch01 버튼 디바운싱 | FSM 재해석 (IDLE→PRESSED→DEBOUNCE→RELEASED) | "이미 FSM을 쓰고 있었다" — 아하 모먼트 |
| Ch02 `logger.c/h` | 횡단 관심사(cross-cutting concern) 맛보기 | "로그는 모든 레이어에서 쓰이는데, 어느 레이어에 넣지?" — 설계 고민 체험 |
| Ch02 `Log_SetBackend()` | 인터페이스 추상화 실례 | "백엔드를 바꿔도 LOG_I() 호출 코드는 안 바뀐다" — 인터페이스 분리의 가치 |

### 재해석 경험 충분성 평가: **충분** (5개 연결점)

단, 다음 조건 필수:
- §3.4에서 **직접 손으로 분류하는 워크시트** 활동 포함
- 단순 설명이 아니라 "이 함수는 어느 레이어일까?" **질문→사고→답→해설** 패턴 적용

### Ch03 → Ch04 연결 (내다보기)

| 예고 포인트 | Ch04 실현 | 자연스러움 평가 |
|-------------|-----------|----------------|
| "Driver 레이어에 다음 블록이 추가됩니다" | `uart_driver.c/h` 생성 | **매우 자연스러움** — 빈 칸을 채우는 느낌 |
| "인터페이스 설계 섹션을 매 챕터 도입부에 작성합니다" | `UART_Send(const char *buf, uint16_t len)` | **자연스러움** — Ch03에서 "인터페이스"를 배웠으니 직접 설계 |
| "시퀀스 다이어그램으로 호출 흐름을 그립니다" | App → UART Driver → HAL → HW | **자연스러움** — Ch03에서 읽는 법을 배웠으니 Ch04에서 쓰기로 전환 |
| "FSM으로 상태를 관리합니다" | Ch04에서는 아직 FSM 불필요 → Ch06(TX_IDLE/TX_BUSY) | **적절한 거리감** — 배운 개념이 바로 다음이 아닌 2~3챕터 뒤에 등장해도 OK |

### Ch04 깨달음 예측: **"아, 이게 Driver 레이어구나"**

이 깨달음이 자연스러우려면 Ch03에서 다음 장치가 필요:
1. **빈 아키텍처 다이어그램**에 "UART Driver (Ch04)", "TIM Driver (Ch07)" 등 **미래 컴포넌트 이름을 회색으로 미리 표시** — 앞으로 채워나갈 로드맵 시각화
2. Ch03 마지막에 **"다음 챕터 예고"** 문장: "Ch04에서 UART를 구현하면, 여러분의 아키텍처 다이어그램 Driver 레이어에 첫 번째 통신 블록이 추가됩니다"

## 6. 추가 권장 사항

### 6.1 비유/실생활 예시 제안

| 개념 | 비유 | 이유 |
|------|------|------|
| 레이어드 아키텍처 | **아파트 건물** — 1층 기초(HAL), 2층 배관/전기(Driver), 3층 인테리어(Service), 4층 입주자(App). 입주자가 직접 기초 콘크리트를 건드리면 건물이 위험 | 건물의 층은 물리적으로 분리되어 직관적. "위층은 아래층에 의존하지만 아래층은 위층을 모른다"가 자연스러움 |
| 의존 방향 위반 | **식당 주방** — 손님(App)이 직접 가스레인지(HAL)를 만지면 위험. 반드시 주방장(Driver)을 통해 요리(Service)를 주문 | 위반 시 "위험"이라는 감정적 연결이 규칙 기억에 도움 |
| FSM | **자동판매기** — 대기→동전투입→음료선택→배출→대기. 각 상태에서 가능한 행동이 제한됨 | 임베디드 교재의 고전적 비유, 수강생 대부분 즉시 이해 |
| 시퀀스 다이어그램 | **카카오톡 단체방** — 누가 누구에게 언제 메시지를 보냈는지 시간순 정렬 | 일상적 경험에서 바로 연결 |

### 6.2 연습문제 설계 (블룸 5수준)

| # | 수준 | 문제 내용 |
|---|------|-----------|
| 1 | 기억 | 4계층의 이름과 각 계층의 한 줄 정의를 작성하시오 |
| 2 | 이해 | `HAL_GPIO_WritePin()`, `LED_Toggle()`, `Log_Init()`, `main()` 루프를 각 레이어에 매핑하시오 |
| 3 | 적용 | 주어진 스파게티 코드를 4개 파일(hal, driver, service, app)로 분리하시오 |
| 4 | 분석 | 제시된 코드에서 레이어 의존 방향 위반 2개를 찾고 수정 방향을 제안하시오 |
| 5 | 평가/창조 | 온습도 센서(SHT31) + LCD 디스플레이 시스템의 아키텍처 설계도를 4계층으로 설계하시오 (Ch11~13 예습 성격) |

### 6.3 위험 요소 및 대비

| 위험 | 대비 |
|------|------|
| "아키텍처"라는 단어 자체의 심리적 부담 | §3.1에서 "여러분은 이미 아키텍처를 쓰고 있습니다" 안심 문구 |
| 추상적 개념에 대한 "이걸 왜 배우나" 의문 | 스파게티 코드 → 정리된 코드 Before/After 즉시 비교 |
| FSM 개념이 레이어드 아키텍처와 혼재 | 물리적으로 휴식을 배치하여 두 개념 블록을 분리 |
| "코드를 안 짜는 챕터"에 대한 지루함 | 실습 3회 분산 + 워크시트 활동으로 능동 학습 보장 |
