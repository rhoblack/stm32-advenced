## 비유 검증

### 비유 1: 레스토랑 (1절, GFX 추상화)
- **기술적 매핑**: 주방설비(LCD Driver) → 요리사(GFX Service) → 웨이터(UI App) ✅
- **한계 명시**: "완전히 독립적이지는 않지만 의존 방향이 단방향" ✅
- **오해 유발 가능성**: "요리사가 오븐을 신경 안 쓴다"는 표현이 지나치게 단순화될 수 있음
  → **권장 보완**: "요리사는 오븐의 구체적 모델은 모르지만, 오븐 API(버튼 이름)는 알아야 한다"고 추가
- **임베디드 맥락 적합성**: 적합 ✅

### 비유 2: 모눈종이 도장 (2절, 비트맵 폰트)
- **기술적 매핑**: 8×16 칸 = 8×16 픽셀 그리드 ✅
- **한계 명시**: "실제로는 비트 연산으로 처리" ✅
- **오해 유발 가능성**: "도장을 찍는다"는 표현이 DMA burst 전송과 혼동될 수 있음
  → 단순 교육 맥락에서는 허용 범위
- **평가**: ✅ 적합

### 비유 3: 신문 편집 (3절, 레이아웃)
- **기술적 매핑**: 편집 격자 = 좌표 상수 ✅
- **한계 명시**: "LCD는 픽셀 단위" ✅
- **오해 유발 가능성**: 없음 ✅
- **평가**: ✅ 적합

### 비유 4: 지하철 노선도 (4절, FSM 테이블)
- **기술적 매핑**: 현재역+방향 = 현재상태+이벤트 ✅
- **한계 명시**: "단일 활성 상태만 가짐" ✅
- **오해 유발 가능성**: "동시에 여러 열차" vs "단일 상태" 차이가 처음에 혼란될 수 있음
  → 한계 명시가 있으므로 OK
- **평가**: ✅ 적합

## 수강생 단골 질문 추가 제안

현재 포함된 FAQ 외에 다음 질문도 추가 권장:

1. **"UI_Init()을 호출하는데 ILI9341_Init()은 또 따로 호출해야 하나요?"**
   - 답: UI_Init → GFX_Init → ILI9341_Init 순서로 자동 호출되므로 별도 호출 불필요.
   - 위치: 1절 또는 실습 step에 추가 권장

2. **"scale=3으로 그리면 느리지 않나요?"**
   - 답: scale 배율로 그린 각 픽셀은 GFX_FillRect를 통해 DMA로 처리되므로 scale=1 대비 9배 전송량이나,
     GFX_DrawChar 호출 자체는 빠릅니다. 핵심 데이터만 scale 큰 것을 사용하는 것이 권장.

## 코드 강의 흐름 평가

### 4절 FSM 코드 순서 평가

현재 순서:
1. FSM 전이 테이블 정의 → 2. `UI_HandleEvent()` 구현 → 3. 진입/부분 갱신 액션

**권장 강의 순서** (교재와 다를 수 있음):
1. 먼저 동작하는 간단한 버전 (if-else 기반) → 문제점 인식
2. 테이블 기반으로 리팩토링
3. 함수 포인터 도입 설명

현재 교재는 완성된 코드를 바로 제시하는 방식 → 코딩 스타일로 충분하나
강의에서는 "리팩토링 과정" 시연을 추가하면 이해도 향상 가능.
