**프로젝트 버전:** v0.5 → v0.6

## 1. 학습 목표 재검토 (블룸 분류체계)

### Ch06 핵심 성격 분석

Ch06는 **Ch04(UART 기초)와 Ch05(DMA 원리)가 실전에서 만나는 통합 챕터**입니다. 학습자 입장에서의 인지 전환:

- Ch04: "Driver를 만들어서 UART를 폴링/인터럽트로 제어해봤다"
- Ch05: "DMA라는 하드웨어 가속기가 있고, Mem-to-Mem으로 성능을 느꼈다"
- **Ch06: "이제 그 DMA를 Ch04의 uart_driver에 적용한다 — 인터페이스는 그대로인데 내부만 바뀐다!"**

이 챕터의 교육적 핵심은 **두 가지**입니다:
1. **DMA의 실전 적용** — Mem-to-Mem이 아닌 Peripheral DMA로의 확장
2. **레이어드 아키텍처의 진가 체험** — Driver 내부 구현만 교체하고 App 코드 변경 없음 (Ch03 설계 원칙의 가장 강력한 실증)

추가로, **링 버퍼(Ring Buffer)**라는 새로운 자료구조가 등장합니다. 이것은 단순 배열과 달리 순환 구조를 가지며, DMA Circular Mode와 자연스럽게 결합됩니다. 링 버퍼는 Ch15(CLI)에서도 재사용되므로, **범용 모듈로 설계하는 경험**이 중요합니다.

### 최종 학습 목표 (6개)

```
LO-1 [기억]   UART DMA 수신/송신 프로세스의 핵심 단계(DMA Stream 할당, 콜백 등록,
              IDLE Line 감지)를 열거하고, HAL API 이름을 대응시킬 수 있다.
LO-2 [이해]   Ch04 인터럽트 드라이버와 Ch06 DMA 드라이버의 인터페이스가 동일한 이유를
              레이어드 아키텍처 관점에서 설명할 수 있다.
LO-3 [적용]   Ch05의 Mem-to-Mem DMA 경험을 확장하여, CubeMX에서 UART RX/TX DMA를
              설정하고 Circular Mode 수신을 구현할 수 있다.
LO-4 [분석]   링 버퍼의 head/tail 인덱스 관계를 분석하고, 오버플로우가 발생하는
              조건과 데이터 유실 시나리오를 정확히 설명할 수 있다.
LO-5 [평가]   폴링/인터럽트/DMA 세 가지 UART 구현 중, 주어진 시나리오(데이터량,
              실시간성, CPU 여유)에 따라 최적 방식을 선택하고 근거를 제시할 수 있다.
LO-6 [평가]   양방향 DMA(RX + TX 동시 운영) 흐름을 분석하고, 잠재적 문제점
              (우선순위 충돌, 버퍼 경합)에 대한 개선 방안을 제안할 수 있다.
```

> **기술 저자 원안과의 차이**: LO-6을 "양방향 DMA 설계(창조)"에서 "양방향 DMA 분석 + 개선 제안(평가)"으로 하향 조정했습니다. Ch06은 Part 1의 마무리 챕터이지 설계 프로젝트 챕터가 아닙니다. 창조 수준은 Ch10(v1.0 마일스톤)이나 Ch17(아키텍처 심화)에서 더 적합합니다.

### 블룸 수준 분포 평가

| 수준 | 개수 | 평가 |
|------|------|------|
| 기억 | 1 | DMA + UART 조합 API 용어 — 최소 1개 필요 |
| 이해 | 1 | 아키텍처 관점 이해 — Ch03 연결의 핵심 |
| 적용 | 1 | CubeMX + Circular DMA 구현 — 실습 핵심 |
| 분석 | 1 | 링 버퍼 오버플로우 분석 — Ch06 고유 사고력 |
| 평가 | 2 | 3-way 최적 선택 + 양방향 분석 — Ch06 최종 도달점 |

> **판단:** 기억 1 + 이해 1 + 적용 1 + 분석 1 + 평가 2 = 총 6개. Ch05(분석+평가 각 1개)에서 **평가 비중이 증가**합니다. 이는 Ch06이 Part 1의 마지막 챕터로서, 폴링→인터럽트→DMA 전체 스펙트럼에 대한 **종합 판단력**을 요구하기 때문입니다.

## 2. 섹션별 인지 부하 분석

### 핵심 신개념 분석

| # | 신개념 | 인지 부하 | 선행 지식 | 비고 |
|---|--------|-----------|-----------|------|
| 1 | **Peripheral DMA (Periph-to-Mem, Mem-to-Periph)** | **중간** — Ch05 Mem-to-Mem에서 방향만 변경, 패턴 동일 | Ch05 DMA 컨트롤러 구조 | Ch05→Ch06 핵심 확장 |
| 2 | **링 버퍼 자료구조 (head/tail, 모듈로 연산)** | **높음** — 순환 인덱싱 + 오버플로우 감지가 동시 등장 | C 배열, `%` 연산자 | **Ch06의 최대 난관** |
| 3 | **IDLE Line 감지 + HAL_UARTEx_ReceiveToIdle_DMA** | **중상** — 가변 길이 수신의 새로운 패턴, HAL Ex API | Ch04 UART 콜백 | 실무 핵심 패턴 |
| 4 | **TX DMA 연쇄 송신 (큐 기반)** | **중상** — TX 완료 콜백에서 다음 전송 트리거 | Ch05 DMA 콜백 | 선택적 심화 가능 |
| 5 | **FSM: TX_IDLE / TX_BUSY / RX_RECEIVING** | **낮음** — Ch03 FSM 기초의 적용, 상태 3개로 단순 | Ch03 FSM 개념 | 아키텍처 연결 |

**총 핵심 신개념: 5개** — 권장 한도(3개)를 2개 초과

### 권장 처리 방안: 개념 병합 + 섹션 분할

```
[개념 A] Peripheral DMA + IDLE Line    → "UART에 DMA 연결하기" (§6.1~6.3)
[개념 B] 링 버퍼 자료구조              → 독립 1개 (§6.2, 분할하여 인지 부하 분산)
[개념 C] TX DMA 연쇄 + FSM            → "송수신 통합" (§6.4~6.5)
```

**압축 전략:**
1. **Peripheral DMA**는 Ch05 Mem-to-Mem의 자연스러운 확장 — 독립 신개념이 아닌 "방향만 바꾼 응용"
2. **링 버퍼**는 개념과 구현을 2개 하위 섹션으로 분할 — 개념 먼저, 코드는 다음에
3. **TX 연쇄 송신의 "중단점 처리"**는 선택적 심화로 분리 — 시간 부족 시 스킵 가능
4. **FSM**은 Ch03에서 배운 것의 적용이므로 독립 신개념으로 계산하지 않음

## 5. 막힘 포인트 예측 (5개)

| # | 막힘 포인트 | 발생 절 | 원인 | 선제 대응 |
|---|-------------|---------|------|-----------|
| 1 | **"링 버퍼가 왜 필요한가?"** — 단순 배열로 안 되나? | §6.2a | DMA Circular Mode가 연속으로 데이터를 쓰는데, 배열 끝에 도달하면 처음부터 덮어쓴다는 것을 직감적으로 이해하지 못함 | §6.2a 도입부에서 **택배 보관함 비유**: "보관함 10칸 → 10번까지 채우면 1번부터 다시 사용. 비운 칸만 새 택배 가능." + "단순 배열은 끝에 가면 멈추지만, 링 버퍼는 계속 돌아간다" SVG 비교 |
| 2 | **"(head + 1) % SIZE 연산이 왜 이렇게 되는가?"** | §6.2b | 모듈로 연산의 순환 특성을 코드로 처음 접하면 직관이 부족 | **시계 비유**: "11시 다음은 0시(12%12=0). head가 SIZE-1 다음은 0(SIZE%SIZE=0)." + 인덱스 추적 표 (head: 0→1→2→...→SIZE-1→0) |
| 3 | **"DMA가 링 버퍼에 자동으로 쓰는가?"** — 데이터 흐름 혼동 | §6.3 | DMA Circular가 하드웨어적으로 버퍼를 순환 기록한다는 것과, 소프트웨어 링 버퍼의 head/tail 관리가 별개임을 혼동 | §6.3에서 **두 가지 순환의 구분** 명시: "DMA가 하드웨어적으로 순환 기록(write) → 소프트웨어가 head/tail로 읽기(read) 위치 추적" + 시퀀스 다이어그램으로 데이터 흐름 시각화 |
| 4 | **"오버런(Overrun)은 언제 생기는가?"** — 링 버퍼 가득 참 vs DMA 부족 | §6.2b~6.3 | 두 가지 다른 종류의 오버런이 존재: (a) 링 버퍼 head가 tail을 따라잡음 (b) UART 하드웨어 오버런(ORE). 혼동 가능 | FAQ 박스에서 **두 가지 오버런 명확 구분**: "① 소프트웨어 오버런: head가 tail 추월 → 데이터 덮어쓰기" "② 하드웨어 오버런(ORE): DR 레지스터를 안 읽으면 새 데이터 유실" → DMA 사용 시 ②는 자동 해소, ①만 관리 |
| 5 | **"IDLE Line이 뭔가?"** — UART 프레임 종료 감지 개념 | §6.3 | Ch04에서 1바이트 인터럽트 재등록 패턴만 배웠으므로, "가변 길이 수신 종료"를 감지하는 새로운 메커니즘이 낯설 | §6.3 도입부에서 **전화 통화 비유**: "상대방이 말을 멈추면(IDLE) '아, 한 문장 끝났구나' 인식. UART도 데이터 후 일정 시간 신호 없으면 IDLE 이벤트 발생." + 타임라인 SVG로 IDLE Line 시점 표시 |

### 막힘 포인트 심각도 순위

1. **#2 모듈로 연산** (가장 빈번) — 코드 수준의 혼란, 실습 중 직접 겪음
2. **#3 DMA↔링버퍼 관계** (가장 개념적) — 하드웨어/소프트웨어 경계 혼동
3. **#1 링 버퍼 필요성** (동기 부족) — "왜?"가 해소되지 않으면 이후 학습 동기 저하
4. **#4 오버런 구분** (혼동) — 용어 동일, 의미 상이
5. **#5 IDLE Line** (새 개념) — 비유로 빠르게 해소 가능

## 10. 비유 및 실생활 예시 체계

| 개념 | 비유 | 비유 세계관 연속성 |
|------|------|-------------------|
| 링 버퍼 | **택배 보관함** — 10칸짜리 보관함. 10번까지 채우면 1번부터 재사용. 비운 칸만 새 택배 수령 가능. 다 차면 새 택배 반송(오버플로우) | Ch05 "택배 경비원" 비유의 확장 — 경비원(DMA)이 보관함(링 버퍼)에 택배 넣어둠 |
| head/tail | **보관함 번호표** — head = "다음에 넣을 칸 번호", tail = "다음에 꺼낼 칸 번호". 두 번호가 같으면 비어있음, head+1이 tail이면 가득 참 | 택배 보관함 비유의 구체화 |
| 모듈로 연산 | **시계 바늘** — 11시 다음은 12시가 아닌 0시(12%12=0). 배열도 마지막 칸 다음은 첫 칸(SIZE%SIZE=0) | 일상 경험과 직결 |
| IDLE Line | **전화 통화 침묵** — 상대방이 말 멈추면 "아, 한 문장 끝났구나" 인식. UART도 데이터 후 1프레임 시간 동안 High 유지되면 IDLE | 직관적, 비기술적 |
| DMA Circular | **회전 초밥 벨트** — Ch05에서 소개한 비유 그대로 적용. "접시 놓는 곳 = DMA write, 접시 집는 곳 = SW read" | Ch05 비유 재사용 — 일관성 |
| TX 연쇄 송신 | **우편함 묶음 발송** — 편지 3통을 우편함에 넣으면, 우체부가 1통 배달 완료 후 다음 1통 가져감. 우편함 비면 대기 | 일상 우편 시스템 |
| 인터페이스 불변 | **자동차 핸들** — 엔진을 가솔린→전기로 바꿔도 핸들 조작법은 동일. Driver 내부를 인터럽트→DMA로 바꿔도 UART_Send() 호출법은 동일 | Ch03 레이어드 아키텍처의 핵심 가치를 자동차에 대응 |

## 11. Ch05 → Ch06 → Ch07 역할 분리 확인

| 항목 | Ch05 | Ch06 | Ch07 |
|------|------|------|------|
| 핵심 주제 | DMA 하드웨어 원리 | UART DMA 실전 통합 | GP Timer + Service 레이어 |
| DMA 모드 | Normal (Mem-to-Mem) | **Circular** (Periph DMA) | Timer Update + DMA Burst (참조만) |
| 자료구조 | 없음 | **링 버퍼 구현** | 없음 |
| UART | 없음 | DMA UART 드라이버 완성 | 없음 (UART는 v0.6 유지) |
| 아키텍처 | v0.4 → v0.5 (DMA 추가) | v0.5 → v0.6 (DMA 통합) | v0.6 → v0.7 (Timer 추가) |
| Part | Part 1 중반 | **Part 1 마무리** | Part 2 시작 |

> **Ch06의 위치**: Part 1(통신 기초 + DMA)의 **마무리 챕터**. Ch04~Ch06의 3개 챕터를 통해 "폴링 → 인터럽트 → DMA" 진화를 완전히 체험합니다. Ch07부터는 새로운 Part 2(타이머)로 전환하므로, Ch06에서 Part 1의 성취를 충분히 느끼게 하는 것이 중요합니다.

### 핵심 주의 사항 4가지

1. **§6.2 링 버퍼가 Ch06의 최대 난관** — 반드시 개념(6.2a)과 구현(6.2b)을 분리하고, 비유(택배 보관함)→의사코드→C코드 3단계로 진행. 모듈로 연산은 시계 바늘 비유로 사전 해소.

2. **"인터페이스 불변" 체험이 Ch06의 아키텍처적 핵심** — §6.5에서 "App 코드 한 줄도 안 바꿨는데 DMA로 동작합니다"를 반드시 보여줘야 함. 이것이 Ch03 레이어드 아키텍처의 가장 강력한 실증이며, 학습자가 "아키텍처를 왜 배웠는지" 체감하는 순간.

3. **Part 1 마무리 회고를 충분히** — §6.6에서 v0.3→v0.6 아키텍처 다이어그램 4장을 나란히 보여주고 "여러분이 만든 시스템"이라는 메시지 전달. Part 2(타이머)로의 전환 예고.

4. **축소안을 항상 준비** — 링 버퍼 구현(§6.2b)과 TX 연쇄 송신(§6.4)이 시간 압박 시 축소 대상. 핵심은 §6.3 RX DMA와 §6.5 통합 체험.
