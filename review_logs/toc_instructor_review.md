- **검토 대상**: `TABLE_OF_CONTENTS.md` — STM32 고급 실무교육 목차 (16챕터, 64시간)
- **검토 일자**: 2026-03-16

## 🔴 Critical 이슈 (강의 진행이 불가능한 문제)

### C-01. Ch04 DMA 이론 챕터의 실습 부재로 4시간 완성 불가
- **문제**: Ch04는 "DMA 아키텍처와 동작 원리"를 4시간 안에 이론+실습으로 다루겠다고 명시하지만, 누적 기능이 "DMA 개념 이해, 다음 챕터 적용 준비"에 그친다. 즉 수강생이 4시간 동안 손으로 뭔가를 완성했다는 성취감이 없다.
- **현장 경험**: 이론만 4시간이면 오후 강의에서 수강생 집중도가 급격히 떨어진다. DMA Mem-to-Mem 실습을 포함해야 강의가 살아난다.
- **해결 방안**: Ch04 누적 기능에 "Mem-to-Mem DMA로 대용량 버퍼 복사 성능 측정" 실습을 명시적으로 추가할 것.

### C-02. Ch08 스텝 모터가 "스마트 시계" 컨셉과 하드웨어 충돌 위험
- **문제**: 스텝 모터(ULN2003) + NUCLEO-F411RE 조합은 추가 전원, 납땜, 드라이버 보드가 필요하다. 현장에서 수강생 20명이 동시에 하드웨어를 연결할 경우, 결선 오류로 보드 손상이나 전원 공급 문제가 빈번하게 발생한다.
- **현장 경험**: 스텝 모터 실습은 반드시 사전 배선된 실습 키트나 미리 검증된 회로도와 함께 배포돼야 한다. 목차에 하드웨어 준비 사항이 전혀 명시되어 있지 않아 강의 당일 셋업 지연이 발생할 수 있다.
- **해결 방안**: Ch08 앞에 "하드웨어 준비물 및 사전 배선 확인" 항목을 반드시 추가하거나, 스텝 모터 대신 서보 모터(PWM 직결 가능)로 교체하는 것을 검토할 것.

## 🟡 Major 이슈 (수강생 이탈 위험, 분량 불균형)

### M-01. Ch10 (SPI DMA + ILI9341 TFT LCD)은 4시간으로 소화 불가
- **문제**: ILI9341 드라이버 설계는 단독으로도 한 챕터 분량이다. 여기에 SPI DMA 초기화, 3레이어 드라이버 설계(LCD → GFX → App), 폰트 렌더링, 레이아웃 설계까지 4시간에 모두 넣으면 강의가 아니라 데모 관람이 된다.
- **현장 경험**: SPI+DMA 입문에서 수강생이 자주 막히는 포인트: CS핀 타이밍, D/C핀 제어 시점, DMA 전송 완료 콜백 내 다음 청크 전송 처리. 이 세 가지만 해결해도 4시간이 빠듯하다.
- **해결 방안**: Ch10을 두 챕터로 분리 — "Ch10. SPI DMA 기초 + ILI9341 초기화" / "Ch10-B. GFX 레이어 + 화면 렌더링 완성".

### M-02. Ch13 (Python PC 대시보드)는 수강생 이탈 위험이 매우 높음
- **문제**: 임베디드 수강생 중 Python GUI(PyQt5/tkinter) 경험자 비율은 현장 경험상 10~20% 수준이다. 프레임 구조 설계 + pyserial 스레딩 + GUI 실시간 렌더링을 4시간에 다루면 C/Python 이중 학습 부담으로 이탈이 발생한다.
- **해결 방안**: Python 코드를 완성본으로 제공하고, MCU 측 프레임 설계와 체크섬 계산에 집중하는 방향으로 챕터 목표를 재조정할 것. GUI는 시연 수준으로 축소.

### M-03. Ch14 소프트웨어 아키텍처 설계론 위치가 너무 늦음
- **문제**: FSM과 레이어드 아키텍처 개념은 Ch01 GPIO에서 "FSM 예고"로 시작하고 Ch08 스텝 모터에서 FSM이 본격 등장한다. 그런데 FSM의 공식 설명이 Ch14에서야 나온다. 수강생은 Ch08~Ch13 동안 FSM을 개념 정립 없이 사용하게 된다.
- **현장 경험**: FSM 없이 스텝 모터 상태관리(IDLE/MOVING/HOMING) 코드를 설명하면 수강생이 "왜 이렇게 코드를 짜느냐"를 이해하지 못한다. FSM 이론은 Ch06 이전에 별도 섹션(30~60분)으로 먼저 다루거나, Ch08 앞으로 이동해야 한다.
- **해결 방안**: Ch14의 FSM/아키텍처 설계 내용을 Part 0 또는 Part 2 시작 전 "설계 기초" 미니챕터(1~2시간)로 앞당길 것.

### M-04. Ch02 로그 시스템에 링 버퍼 구현이 포함되어 있어 Ch01 대비 난이도 급등
- **문제**: Ch01은 GPIO/EXTI 기초인데, Ch02에서 바로 링 버퍼 기반 비동기 로그 출력을 다룬다. 링 버퍼는 Ch05 UART DMA에서 다시 나오는데, Ch02 단계의 수강생(중급이지만 HAL 기본 수준)에게 링 버퍼를 직접 구현하게 하면 2챕터 만에 이탈이 발생할 수 있다.
- **해결 방안**: Ch02에서는 링 버퍼를 완성본 라이브러리로 제공하고, 링 버퍼 직접 구현은 Ch05로 이동하는 것이 자연스럽다.

### M-05. Ch15, Ch16에 프로젝트 버전 번호가 없음
- **문제**: Ch01~Ch13은 모두 `v0.x`, `v1.x` 형태로 누적 성장형 구조를 명확히 보여주는데, Ch15와 Ch16에는 버전 번호가 없다. 수강생 입장에서 "이 챕터가 프로젝트와 연결되는가, 아니면 독립 강의인가"가 불명확하다.
- **해결 방안**: Ch15 → `v2.1 (디버깅 고도화)`, Ch16 → `v2.2 (리팩토링 완성본)` 형태로 버전 명시.

## 🟢 Minor 이슈 (보완하면 좋은 것)

### m-01. CubeMX 설정 화면 스크린샷 포인트 미명시
- Ch04, Ch07, Ch09, Ch10에서 CubeMX 설정이 등장하지만, 어떤 설정 화면에서 수강생이 자주 실수하는지 목차에 힌트가 없다. 실제 강의에서 "Clock Configuration" 탭의 APB 프리스케일러 설정과 DMA Request 매핑은 반드시 시각 자료가 필요하다.
- **제안**: 각 챕터 핵심 주제에 "(CubeMX 실습 포인트: XXX)" 태그 추가.

### m-02. 전압/전류 관련 하드웨어 안전 주의사항 누락
- 3.3V 시스템인 NUCLEO-F411RE에 5V 입력을 연결하는 실수는 현장에서 매 기수마다 발생한다. I2C, SPI 센서의 레벨 시프터 필요 여부가 목차에 전혀 언급되지 않는다.
- **제안**: Part 3 도입부에 "하드웨어 레벨 매칭 체크리스트" 1페이지 추가 권장.

### m-03. 면접 연결 포인트 항목 부재
- 에이전트 역할 정의(expert_instructor.md)에 "면접 연결 포인트 추가 제안"이 명시되어 있으나, 현재 목차에 면접 관련 항목이 전혀 없다. Ch16 마무리에 "실무 면접 예상 질문 20선" 섹션을 추가하면 수강생 만족도가 크게 올라간다.

### m-04. RTOS 완전 누락
- 64시간 과정에서 FreeRTOS를 아예 다루지 않는 것은 실무 관점에서 아쉽다. 모든 챕터를 슈퍼루프(Super Loop) 기반으로 유지하는 설계 의도라면 목차 도입부에 "이 과정은 Bare-metal 중심 설계이며 RTOS는 후속 과정에서 다룬다"는 명시가 필요하다. 없으면 수강생이 "왜 RTOS 안 배워요?"라는 질문을 반드시 한다.

### m-05. Ch11 (I2C DMA 심화)의 독립성 부족
- "UART/SPI/I2C DMA 동시 운용 아키텍처"는 훌륭한 주제이나, Ch05(UART DMA) + Ch10(SPI DMA) + Ch11(I2C DMA)를 합쳐 복습하는 형태여서 새로운 내용이 적다. 4시간 분량으로 채우기 어려울 수 있다.
- **제안**: Ch11에 "DMA 충돌 해결(버스 아비트레이션, 우선순위 재설정)" 또는 "HAL_DMA_Abort 패턴"을 추가.

### m-06. 수강생 사전 지식 확인 챕터 부재
- 대상 독자가 "C언어 숙련, HAL 기본 사용 경험"이지만, 이를 검증하는 Ch00 또는 사전 점검 퀴즈가 없다. HAL 경험의 편차가 크면 Ch01부터 진행 속도가 갈린다.
- **제안**: 강의 시작 전 30분 사전 점검 세션(기본 GPIO 코드 읽기, CubeMX 프로젝트 생성) 또는 사전 과제를 명시.

## ✅ 잘 된 점

### 1. 누적 성장형 프로젝트 방식의 일관성
v0.1 → v2.2까지 매 챕터마다 버전이 올라가며 완성형 시스템을 향해 나아가는 구조는 강의 동기 부여 측면에서 매우 효과적이다. 수강생이 "내가 지금 무엇을 만들고 있는가"를 항상 인지할 수 있다. 현장에서 가장 수강생 만족도가 높은 방식이다.

### 2. Part 구분의 명확성
GPIO/UART/DMA/타이머/센서/CLI/아키텍처/품질 순서가 "하드웨어 → 통신 → 제어 → 시스템화 → 품질 향상"이라는 임베디드 개발 실무 흐름과 잘 일치한다. 수강생이 커리큘럼 전체 구조를 직관적으로 이해할 수 있다.

### 3. Ch02 로그 시스템의 조기 배치
많은 STM32 교육 과정이 디버깅 도구를 마지막에 다루는데, 이 목차는 Ch02에서 로그 시스템을 먼저 구축하고 이후 모든 챕터에 적용하는 구조이다. 이는 매우 탁월한 선택으로, 수강생이 Ch03 이후부터 스스로 디버깅할 수 있는 역량을 갖추게 된다.

### 4. HAL API와 레지스터 수준의 균형
각 챕터가 HAL API 사용법에만 치우치지 않고 내부 레지스터 구조(DMA 스트림/채널, RTC BCD 레지스터, EXTI 엣지 트리거 등)를 함께 다루는 점이 실무형 강의에 적합하다. "HAL만 써보고 원리는 모르는" 수강생 양산을 방지한다.

### 5. CLI + PC 대시보드 조합 (Ch12 + Ch13)
단순히 MCU에서 데이터를 찍는 것을 넘어 양방향 명령 제어(CLI)와 시각화(대시보드)까지 연결하는 구성은 실무 포트폴리오로 직결된다. 취업을 앞둔 수강생에게 강력한 차별화 포인트가 된다.

### 6. HardFault 핸들러 해석 포함 (Ch15)
HardFault 분석은 STM32 실무에서 가장 자주 만나는 벽인데, 이를 스택 덤프 분석 수준까지 다루는 커리큘럼은 드물다. 강의 후 수강생의 실무 투입 시 생존율을 크게 높여주는 핵심 주제이다.

### 7. SHT31 CRC 검증 포함 (Ch09)
단순 데이터 읽기를 넘어 CRC 검증까지 포함한 것은 실무 드라이버 설계 관점에서 매우 적절하다. "센서 값이 이상한데 이유를 모르겠어요"라는 현장 질문의 원인 중 상당수가 CRC 미검증이다.
