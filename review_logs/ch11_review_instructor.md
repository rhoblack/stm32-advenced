## 비유 검증 (5개 비유 평가)

### 1️⃣ "교사와 학생의 손 올리기" (동기식 통신)

**비유 텍스트** (§11.1):
> "I2C는 교사와 학생의 손 올리기입니다. 학생(슬레이브)이 준비되면 손을 들고, 교사(마스터)가 '좋아, 지금 말해'라고 신호(클럭)를 보냅니다."

**검증**:
- ✅ **기술 정확성**: UART(비동기)와 I2C(동기) 구분 명확
- ✅ **직관성**: 학생들이 학교 경험과 직접 연결
- ⚠️ **한계점 명시**: 있음 ("오픈드레인 특성 제외")
- ✅ **강의장 적용**: "지금 손 들어볼래?" 라이브 데모 가능

**강사 평가**: ⭐⭐⭐⭐⭐ (완벽)

**강의실 팁**:
```
라이브 데모: 학생 2-3명을 불러서 "손 올리기" 시연
- 교사(강사)가 "이제 말해" 신호 → 학생 손 올림
- 오픈드레인: "아무도 손 내렸으면 LOW, 다 내렸으면 HIGH"
```

---

### 2️⃣ "택시 신호등 대기" (클록 스트레칭)

**비유 텍스트** (§11.2):
> "택시를 호출했습니다(START). 택시 기사(마스터)가 신호등을 켭니다... 하지만 급격한 회전이 필요하면, 기사가 신호등을 '빨강'으로 유지합니다."

**검증**:
- ✅ **기술 정확성**: SCL LOW = 버스 점유, 높은 정확도
- ✅ **직관성**: "신호등"은 실생활 경험과 일치
- ✅ **구체성**: "승객(슬레이브)도 빨강으로 누른다" → 협력 구조 명확
- ⚠️ **한계점**: "실제 교통 신호와 다릅니다" 명시되어 좋음

**강사 평가**: ⭐⭐⭐⭐⭐ (탁월)

**강의실 팁**:
```
화이트보드 그림: SCL/SDA 타이밍 다이어그램
- SCL: 신호등(일정 리듬)
- SDA: 데이터(신호등 리듬에 맞춰 변환)
- "둘 다 LOW 상태" = 대기 상태 (클록 스트레칭)
```

### 3️⃣ "줄다리기" (오픈드레인 풀업)

**강사 평가**: 아직 미완성, 강화 필요 ⭐⭐⭐

**강의실 팁**:
```
매우 효과적 비유. 꼭 추가하세요:
- "누가 끌어야 LOW?"
- "아무도 안 끌면?"
- "여럿이 동시에 끌 수 있나?" (YES, 협력)

라이브: 줄(실)로 풀업 효과 시연 가능
```

## 수강생 막힘 포인트 (Top 5 예상)

### 1️⃣ **"I2C 풀업 저항이 왜 필요한가?"**
**난이도**: ⭐⭐⭐ (중간, 이해 필수)

**예상 질문**:
> "UART는 풀업이 필요 없는데, I2C는 왜 필요해요?"

**선제 대응**:
```
강의에 추가 (§11.1):
"UART는 TX/RX가 명확히 분리되어 있고(방향성),
I2C는 SDA/SCL을 마스터와 슬레이브가 동시에 사용합니다(방향 없음).
마스터가 HIGH를 강제할 수 없으므로, 풀업 저항이 필요합니다."

라이브 데모:
- 풀업 저항 없이 센서 연결 → "I2C 통신 실패"
- 풀업 저항 추가 → "통신 성공!"
```

---

### 2️⃣ **"CRC-8 다항식 0x31은 왜 이 숫자인가?"**
**난이도**: ⭐⭐⭐⭐⭐ (어려움, 필수 회피)

**예상 질문**:
> "다항식이 0x31인 이유가 뭐예요? 0x30은 안 되나요?"

**선제 대응**:
```
강의에 추가 (§11.5):
"다항식은 CRC 알고리즘의 '설정값'입니다.
제조사(Sensirion)가 정한 SHT31 표준이 0x31이므로,
우리는 그냥 따릅니다.

마치 'C언어의 표준 라이브러리'처럼,
다항식도 표준이 정해지면 그것을 사용하는 것입니다."

💡 핵심: "당신이 설정하는 게 아니라, 센서 제조사가 정한 것입니다."
```

---

### 3️⃣ **"50ms 대기 동안 메인 루프가 먹통되지 않나요?"**
**난이도**: ⭐⭐⭐⭐ (중간~상, 설계 이해 필수)

**예상 질문**:
> "sht31_read_temp_humid()에서 HAL_Delay(50)을 호출하면,
> 메인 루프가 50ms 동안 멈추지 않나요?"

**선제 대응**:
```
강의에 강조 (§11.6 vs §11.7):
"좋은 질문입니다!

❌ 블로킹 패턴 (§11.6):
sht31_read_temp_humid() → 내부에서 HAL_Delay(50)
→ 메인 루프 50ms 멈춤 (프로토타입용)

✅ 비블로킹 패턴 (§11.7 FSM):
Sensor Service가 상태를 관리 (MEASURING → 50ms 경과 체크)
→ 메인 루프는 100ms 주기로 process() 호출
→ 50ms 대기는 FSM 상태에 분산되어 보이지 않음!

실무에서는 항상 비블로킹 패턴을 사용합니다!"

라이브 데모:
- 블로킹 버전: "LED 점멸이 20ms 간격인데... 처음엔 정지 후 점멸"
- 비블로킹 버전: "LED가 끊김 없이 계속 점멸"
```

---

### 4️⃣ **"온도 계산식이 왜 이렇게 복잡한가?"**
**난이도**: ⭐⭐⭐ (중간)

**예상 질문**:
> "(175.0f / 65535.0f) * raw - 45.0f 이 공식은 뭐예요?"

**선제 대응**:
```
강의에 추가 (§11.4):
"이 공식은 SHT31 센서가 내부적으로 사용하는 것입니다.
센서가 측정한 온도를 0~65535 범위의 숫자(raw)로 보내면,
우리가 섭씨 온도로 변환해야 합니다.

공식은 데이터시트에서 제공되므로,
'당신이 유도할 필요는 없습니다.'
복사+붙여넣기만 하면 됩니다!"

💡 핵심: "데이터시트는 당신의 친구입니다. 공식이 있으면 그냥 써요."
```

---

### 5️⃣ **"AlarmA와 AlarmB가 동시에 발동하면?"**
**난이도**: ⭐⭐⭐⭐ (상)

**예상 질문**:
> "시각 알람(AlarmA)과 온도 알람(AlarmB)이 동시에 발동하면 어떻게 되나요?"

**선제 대응**:
```
강의에 있음 (§11.8 FAQ) — 충분 ✅

추가 강화 (선택):
"우선순위를 NVIC에서 설정합니다. 높은 숫자 = 낮은 우선순위"

라이브: 실제 NVIC 설정 화면 보여주기
```
