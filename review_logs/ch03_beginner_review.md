# Ch03 초급자 이해도 평가

**리뷰어**: 초급자 독자 (중급 임베디드 개발자 관점)
**대상**: `manuscripts/part0/chapter03.html`
**날짜**: 2026-03-17

---

## 섹션별 이해도

### §3.1 도입 — "3분 공포 체험": ⭐⭐⭐⭐⭐

스파게티 코드 예시가 매우 효과적이다. `sht31_read()`의 20ms 블로킹이 스텝모터 1ms 타이밍을 망치는 구체적 시나리오가 "시침이 떨리는 시계"라는 생생한 결과로 이어져서, 아키텍처 필요성이 즉각 와닿는다. "센서 값이 틀려요" 버그 리포트 상황에서 3,000줄 전체를 수색해야 한다는 설명도 실무 경험이 있는 중급 개발자에게 강한 공감을 불러일으킨다.

Ch01~02 코드가 200줄이라 괜찮지만 Ch19까지 가면 3,000줄이 된다는 점진적 확장 논리가 설득력 있다. FAQ "잘 동작하는데 왜 구조를 바꿔야 하나요?"가 정확히 내 의문을 짚어준다.

### §3.2 개념 — "4계층 정의": ⭐⭐⭐⭐⭐

레스토랑 주방 비유가 탁월하다. 홀 매니저(App), 메뉴 관리자(Service), 요리사(Driver), 주방 기기(HAL) 매핑이 직관적이다. "홀 매니저가 가스레인지를 직접 만지지 않듯" 이라는 한 줄이 단방향 의존 원칙의 핵심을 완벽하게 전달한다.

4계층 각각의 정의가 명확하고, 실제 파일명(`uart_driver.c`, `clock_service.c`)과 함께 제시되어 추상적이지 않다. `led_driver.h` 인터페이스 예시 코드가 구체적이어서 "인터페이스 = 주문서" 비유가 코드 수준에서 와닿는다.

횡단 관심사(LOG_I) 설명에서 "화재 경보기" 비유도 적절하다.

비유의 한계를 명시한 점(콜백/이벤트 메커니즘 언급)도 좋다 — 비유에 과도하게 의존하지 않게 해준다.

### §3.3 워크시트 — "코드를 계층에 배치하기": ⭐⭐⭐⭐

분류 워크시트가 능동적 학습을 유도한다. "내 답" 칸을 먼저 채운 후 해설을 확인하라는 지시가 명확하다. `MX_GPIO_Init()`의 경계선 애매함을 다루면서 "고민하는 과정 자체가 아키텍처 설계"라는 메시지가 좋다.

의존 방향 위반 찾기 실습이 특히 효과적이다. v0.2 코드에서 `HAL_GPIO_EXTI_Callback()` → `HAL_GPIO_WritePin()` 직접 호출이 왜 위반인지 설명하고, `led_driver_toggle()`로 수정해야 한다는 방향을 제시한다. "아직 Driver가 없으니 의도적 위반"이라는 설명이 Ch04로의 자연스러운 동기를 만든다.

### §3.4 FSM 맛보기: ⭐⭐⭐⭐

자판기 비유가 FSM 개념을 친숙하게 소개한다. 상태/이벤트/전이/액션 4요소가 자판기 예시로 명확하게 매핑된다. "금액 투입됨 상태가 아닌데 음료 버튼을 눌러도 아무 일도 안 일어나는 이유"라는 설명이 FSM의 핵심을 일상 경험으로 설명한다.

Ch01 디바운싱을 FSM으로 재해석하는 부분이 "아하!" 순간을 준다. 이미 작성한 코드가 사실은 FSM이었다는 깨달음이 개념을 친근하게 만든다.

"지금은 개념만 이해하면 된다, 코드 작성은 Ch08에서"라는 FAQ가 불안감을 해소한다.

### §3.5 실습 — "v0.3 설계도 작성": ⭐⭐⭐⭐

실습 지시사항이 Step 1~4로 명확하게 구조화되어 있다. 시퀀스 다이어그램 읽는 방법 소개도 적절하다. 디버깅 계층 판단 가이드 표가 실무적으로 유용하다.

"Service 레이어에 뭘 넣어야 할지 모르겠다" → "빈 칸으로 두세요"라는 팁이 불안감을 해소한다.

## 막힘 포인트

1. **§3.3 `HAL_GPIO_EXTI_Callback()`의 레이어 분류 근거**: "HAL에서 호출하니까 HAL 아닌가?"에 대한 해설은 있지만, "함수의 구현 내용이 App 로직이므로 App"이라는 판단 기준이 처음 접하는 독자에게 다소 혼란스러울 수 있다. `__weak` 오버라이드 메커니즘에 대한 이해가 전제되어 있는 느낌이다. Ch01에서 `__weak`를 다뤘다면 괜찮지만, 명시적으로 "Ch01에서 배운 `__weak` 오버라이드를 떠올려 보세요"와 같은 연결 문구가 있으면 더 좋겠다.

2. **§3.4 FSM 상태 전이 다이어그램 읽기**: 다이어그램 읽는 방법(원 = 상태, 화살표 = 전이 등)은 설명하지만, 실제 SVG 다이어그램을 아직 확인할 수 없어 다이어그램 자체의 품질은 평가하기 어렵다. 텍스트 설명만으로는 IDLE → PRESSED → DEBOUNCED → IDLE 흐름이 이해되지만, "PRESSED에서 50ms 미만 재트리거 시 어떻게 되는가?"에 대한 전이가 다이어그램에 포함되어 있는지 확인 필요.

3. **§3.5 시퀀스 다이어그램이 ASCII art로 표현됨**: CLAUDE.md에서 "ASCII art 절대 금지, SVG만 사용"이라는 규칙이 있는데, §3.5의 시퀀스 다이어그램이 `<pre><code>` 블록으로 ASCII art 형태로 작성되어 있다. SVG로 교체해야 한다.

4. **§3.2 `button_driver.h`의 콜백 패턴**: `typedef void (*button_callback_t)(void);`가 C 함수 포인터 문법인데, 중급 개발자라면 알겠지만 간단한 설명("함수 포인터로 버튼 눌림 시 호출할 함수를 등록합니다") 한 줄이 있으면 더 친절하겠다.

## 개선 제안

1. **§3.3 `__weak` 연결 문구 추가**: `HAL_GPIO_EXTI_Callback()` 해설에 "Ch01에서 배운 `__weak` 오버라이드 패턴을 기억하세요"와 같은 이전 챕터 참조를 추가하면 연결이 자연스러워진다.

2. **§3.5 시퀀스 다이어그램 SVG 전환**: 현재 ASCII art 형태의 시퀀스 다이어그램을 SVG로 교체해야 한다. 프로젝트 규칙(다이어그램은 SVG만 사용)에도 부합하고, 시각적 가독성도 높아진다.

3. **§3.4 FSM 재진입 전이 명시**: "PRESSED 상태에서 50ms 미만에 다시 EXTI가 발생하면?" → "무시(self-loop 또는 전이 없음)"를 텍스트로 한 줄 추가하면 FSM의 "허용되지 않는 이벤트는 무시" 원칙이 더 명확해진다.

4. **§3.2 함수 포인터 한 줄 설명 추가**: `button_callback_t` 타입 정의에 "함수 포인터로, 버튼이 눌렸을 때 호출될 함수를 미리 등록하는 패턴입니다"라는 주석 또는 본문 설명 추가.

5. **연습문제 난이도 표시 강화**: 연습문제의 `기억/이해/적용/분석/창조` 레벨 표시가 좋지만, 1~3번은 본문 내용으로 충분히 풀 수 있고, 4~5번은 약간의 추론이 필요하다는 안내가 있으면 학습자의 기대치 설정에 도움이 된다.

## 전체 평가

### ⭐⭐⭐⭐ — 전체적으로 충분히 이해 가능하며, 소수의 보강으로 완성도를 높일 수 있다

**강점**:
- 스파게티 코드 "공포 체험"이 아키텍처 필요성을 강력하게 동기부여한다
- 레스토랑 비유가 4계층 개념을 즉각 이해하게 만든다
- Ch01~02 코드를 직접 분류하는 워크시트가 능동적 학습을 유도한다
- FSM을 기존 디바운싱 코드로 재해석하는 부분이 "이미 알고 있었다"는 자신감을 준다
- 감정 곡선(공포 → 비유로 안심 → 직접 분류 → 성취)이 자연스럽게 설계되어 있다
- FAQ, 면접 포인트, 강사 꿀팁이 적절히 배치되어 다양한 관점에서 이해를 보강한다

**보강 필요**:
- ASCII art 시퀀스 다이어그램 → SVG 전환 (규칙 위반)
- `__weak` 오버라이드 관련 이전 챕터 참조 명시
- FSM PRESSED 상태에서의 재진입 전이 명시

**최소 승인 기준 ⭐⭐⭐ 충족 여부**: 충족. 모든 섹션이 ⭐⭐⭐⭐ 이상이며, 중급 임베디드 개발자가 Ch01~02 지식만으로 이해하는 데 무리가 없다.
