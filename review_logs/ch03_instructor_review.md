## 기획 원칙 준수도

### 1. "3분 공포 체험"
**평가: ★★★★☆**

**잘된 점**:
- 도입부(§3.1)에서 "3분만 투자해서 미래를 미리 엿보겠습니다"로 시작하여 기획 의도를 충실히 따름
- 스파게티 코드 예시가 `while(1)` 루프 안에 센서/LCD/모터를 혼재시켜 블로킹 문제를 생생하게 보여줌
- "시침이 떨리는 시계를 만들고 싶으신 분은 이 구조로 진행하시면 됩니다" — 기획 단계의 킬러 문장이 그대로 반영됨
- "어디부터 보시겠습니까?" 질문 → "3,000줄 전체를 수색해야 합니다" 답변 — 수강생에게 공포를 체감시키는 흐름이 자연스러움

**개선 필요**:
- 본문 내 코드 블록은 약 25줄로 적절하나, `code_examples/ch03_01_spaghetti_demo.c`(별도 파일)는 160줄 이상으로 기획 시 권고한 "50줄 스크롤 시연"보다 김. 강의 시연 시 처음 50줄만 보여주고 "이게 160줄 더 있습니다"로 전환하는 가이드가 본문이나 강사 팁에 없음
- "3분"이라는 시간 프레이밍이 본문에는 있지만, 실제 분량은 §3.1 전체가 25분 설계. 강의 현장에서 도입부만 정확히 3분으로 끊는 타이밍 가이드가 필요함

**권고**: instructor-tip에 "스파게티 코드 시연은 본문 코드(25줄)만 화면에 띄우고, 별도 파일(160줄)은 스크롤만 빠르게 넘기세요. '이게 아직 절반도 안 된 겁니다'가 핵심 멘트"를 추가

### 2. "레스토랑 주방" 비유
**평가: ★★★★★**

**잘된 점**:
- 기획 단계에서 강력 권장한 비유가 충실히 구현됨
- 홀 매니저(App)/메뉴 관리자(Service)/요리사(Driver)/주방 기기(HAL) 매핑이 명확
- "홀 매니저가 가스레인지를 직접 만지지 않듯, App 레이어는 HAL을 직접 호출하지 않습니다" — 단방향 의존 원칙의 핵심이 한 문장에 압축됨
- 비유의 한계가 명시적으로 기술됨: "실제 레스토랑에서는 요리사가 '재료 떨어졌다'고 홀에 통보합니다" → 콜백/이벤트 메커니즘 설명
- 횡단 관심사를 "화재 경보기"로 설명한 것은 기획에 없던 추가 비유인데, 매우 효과적
- SVG(`ch03_sec02_restaurant_layers.svg`)에서 이모지(🎩📋👨‍🍳🔥)를 사용하여 시각적 구분이 뛰어남

**개선 필요**:
- 동시성(인터럽트/DMA) 한계는 기획에서 명시 요청했으나 본문에 반영되지 않음. "이 비유는 100% 정확하지 않습니다" 문구 중 동시성 관련 한계도 추가 필요

### 4. "이미 아키텍처를 쓰고 있다" 안심 장치
**평가: ★★★★☆**

**잘된 점**:
- §3.2 metacognition 박스: "Ch01의 `HAL_GPIO_WritePin()`은 어느 레이어인가? — HAL입니다. 우리는 이미 HAL을 쓰고 있었습니다."
- §3.4: "Ch01 디바운싱 = 이미 FSM이었다" — `if (now - last >= 50ms)` 패턴이 2-상태 FSM임을 재해석
- §3.2 FAQ 박스: LOG_I()를 횡단 관심사로 재해석 — Ch02의 logger.c가 아키텍처적 역할을 하고 있었음을 보여줌

**개선 필요**:
- 기획 단계에서 제안한 "Ch01 디바운싱 = FSM"과 "logger.c = 횡단 관심사" 두 가지 안심 장치가 모두 있으나, **하나의 통합 메시지로 묶는 구간이 없음**. §3.5 핵심 정리 또는 도입부에서 "여러분은 Ch01~02에서 이미 HAL 호출, 횡단 관심사(로거), FSM(디바운싱)을 사용했습니다. 오늘은 이것에 이름을 붙이는 시간입니다"와 같은 종합 안심 메시지가 있으면 더 효과적

### 코드 예제 강의성
**평가: ★★★★☆**

**1. 스파게티 코드 예시 (§3.1)**:
- 본문 내 코드(25줄)가 핵심만 보여줘서 효과적
- `sht31_read()` 20ms + LCD 40ms + 모터 1ms 타이밍 충돌이 구체적 수치로 제시되어 설득력 있음
- 강의 현장에서 쓸 말: "여기 20ms 블로킹이 보이시죠? 이 동안 모터가 멈춥니다. 시침이 1초에 한 번 떨리는 시계가 만들어지는 겁니다."
- **좋음**: 실제 강의에서 바로 활용 가능

**2. 인터페이스 .h 설계 (§3.2)**:
- `led_driver.h`와 `button_driver.h` 두 가지 예시가 코드와 주석으로 깔끔하게 작성됨
- "App은 이 3개 함수만 알면 된다. GPIOA, GPIO_PIN_5를 모른다" 주석이 인터페이스 개념을 코드 레벨에서 확인시켜줌
- "디바운싱은 Driver 내부에서 처리한다" — Ch01에서 App에 있던 디바운싱 로직이 Driver로 내려가는 것을 암시하여 아키텍처 진화 동기 형성
- **좋음**: 수강생이 "아, .h 파일이 주문서구나"를 체감

**3. v0.2 계층 위반 분석 (§3.3)**:
- `HAL_GPIO_WritePin()` 직접 호출이 "App → HAL 계층 건너뛰기 위반"으로 명확히 표시됨
- 이상적 코드(`led_driver_toggle()`)가 바로 아래에 주석으로 제시되어 Before/After 대비가 뚜렷
- "이 위반은 의도적입니다. Ch04부터 해결됩니다" — 불안 없이 다음 챕터로 동기 연결
- **좋음**: 위반을 발견하고 해결 방향을 아는 것까지가 이 실습의 목표이며, 잘 달성됨

**개선 필요**:
- 코드 예제에 `LOG_D`/`LOG_I` 사용이 본문 내 코드에서는 보이지 않음. 스파게티 코드 예시에 `LOG_I("Main", "Running...")`이 한 줄이라도 있으면 Ch02와의 연결이 강화됨
