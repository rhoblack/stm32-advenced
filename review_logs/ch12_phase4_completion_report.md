#### Issue #2: LCD Data Volume Not Explained ✅ COMPLETE

**문제**:
- "LCD 화면은 초당 수십만 바이트를 송수신해야 합니다" (정성적)
- **왜** 그렇게 많은 데이터가 필요한가? → 설명 없음

**해결책** - 상세 계산 및 I2C vs SPI 비교:

```html
💡 LCD 화면이 필요한 데이터 계산

픽셀 데이터량:
- 해상도: 320 × 240 = 76,800픽셀
- 색상: RGB565 = 2바이트/픽셀
- 한 프레임: 76,800 × 2 = 153,600 바이트

I2C (100 kbps):
- 초당 처리: 100 kbps ÷ 8 = 12,500 바이트/초
- 한 화면: 153,600 ÷ 12,500 = 12초
- fps: 0.08 fps (매우 느림, 사용 불가)

SPI (10 Mbps):
- 초당 처리: 10 Mbps ÷ 8 = 1,250,000 바이트/초
- 한 화면: 153,600 ÷ 1,250,000 = 0.12초
- fps: 8 fps (부드러운 화면)

결론: 부드러운 화면을 위해 SPI는 필수입니다.
```

**추가 개선**:
- I2C vs SPI 비유 개선:
  ```
  이전: "우편 배송 vs 직업 전화" (낯섦)
  이후: "편도 1차선(ACK 대기) vs 양방향 4차선(CS 지정)"
  ```

**소요 시간**: 20분 (예상) → ✅ 완료
**검증**: ✅ PASS (계산 정확함, 153,600 바이트 확인)

#### Issue #3: D/C Signal Persistency Clarification ✅ COMPLETE

**문제**:
- "D/C를 매 바이트마다 바꿔야 하나? 아니면 한 번 설정 후 유지?"
- 불명확 → 학생 혼동

**해결책** - 상세 FAQ 추가:

```html
Q. "D/C 신호를 매 바이트마다 바꿔야 하나요?"

A. 아니오. D/C를 한 번 설정하면, 그 상태에서 여러 바이트를
   전송합니다. D/C가 명령어에서 데이터로 바뀔 때만 신호를 변경합니다.

예시:
1. D/C = 0으로 설정
2. 명령어 0xCB 전송 (1바이트)
3. D/C = 1로 변경
4. 파라미터 0x39, 0x2C, 0x00, ... 전송 (여러 바이트)

이유: 매 바이트마다 GPIO를 제어하면 GPIO 제어 속도가 병목이 되어
      전체 전송이 느려집니다.
```

**소요 시간**: 10분 (예상) → ✅ 완료
**검증**: ✅ PASS (코드 구현 일치)

#### Issue #4: Sleep Out 120ms Timing Justification ✅ COMPLETE

**문제**:
- "최소 120ms 대기" (이유 없음)
- 학생: "왜 120ms? 100ms는 안 돼?"

**해결책** - 기술 정당화:

```html
💡 Sleep Out 명령어 후 120ms 딜레이의 의미

Sleep Out (0x11) 명령어는 LCD를 슬립 모드에서 깨웁니다.
그러나 LCD 칩 내부의 전압(특히 back-light 드라이버 전압)이
안정화되는 데 최소 100~120ms가 필요합니다.

이 딜레이를 무시하면:
- 화면이 켜지지 않음 (또는 깜빡임)
- 색상이 이상함 (감마 교정 적용 안 됨)
- 밝기가 불안정함
- 일부 픽셀이 표시 안 됨

따라서 이 120ms는 선택이 아니라 필수입니다.
데이터시트 Section "Power Supply Sequencing" 참조.
```

**소요 시간**: 10분 (예상) → ✅ 완료
**검증**: ✅ PASS (ILI9341 데이터시트 확인)

#### Issue #5: I2C vs SPI Metaphor Improvement ✅ COMPLETE

**문제**:
- 기존 비유 "우편 배송 vs 직업 전화" → 학생에게 낯섦

**해결책** - 도로 비유로 명확화:

```html
I2C: 편도 1차선 도로처럼 마스터와 슬레이브가 같은 신호선(SDA)으로
     데이터를 주고받습니다. 데이터를 보낼 때마다 상대방의 응답(ACK)을
     기다려야 하므로 느립니다.

SPI: 양방향 고속도로처럼 마스터와 슬레이브가 각각 독립적인 신호선
     (MOSI, MISO)을 가지며, 신호등(ACK) 대신 Chip Select(CS)로
     상대를 지정합니다. 따라서 빠르고 간단합니다.
```

**소요 시간**: 15분 (예상) → ✅ 완료
**검증**: ✅ PASS (SPI 프로토콜 정확함)

#### Issue #6: Init Command Purpose Explanation ✅ COMPLETE

**문제**:
- 표에 "Power Control A = 전압 설정"만 있음
- "이게 뭔가? 왜 필요한가?" → 설명 부족

**해결책** - 15개 명령어 상세 설명:

```html
1. Power Control A (0xCB): AVDD/GVDD 설정 → 백라이트 밝기에 영향
2. Power Control B (0xCF): VRH/VREG1 설정 → 화면 안정성
3. Power Control 1 (0xC0): VREG1OP 설정 → 화질 보정
4. Power Control 2 (0xC1): SAP 설정 → 출력 드라이버 강도
5. VCOMH Regulator (0xC5): VCM 설정 → 흑백 명암 대비 (이게 잘못되면 화면 밝거나 어두움)
6. VCOMH Voltage (0xC7): VCOMH 값 → 감마 곡선 조정
7. Memory Access Control (0x36): 화면 방향, RGB/BGR 순서 설정
8. Pixel Format Set (0x3A): 0x55 = 16비트(RGB565) 우리는 이것만 사용
9. Frame Rate Control (0xB1): DIVA/RTNA 설정 → 화면 갱신 빈도 (보통 60Hz)
10. Display Function Control (0xB6): 주사선, gate 타이밍 → 320×240 최적값
11. Gamma Set (0x26): 0x01 = Gamma 1 기본 → 밝기 인지 선형성
12. Positive Gamma Correction (0xE0): 밝은 부분 톤 조정
13. Negative Gamma Correction (0xE1): 어두운 부분 톤 조정 (양수 곡선 보수)
14. Sleep Out (0x11): LCD 웨이크업 → 뒤에 반드시 120ms 딜레이
15. Display On (0x29): 최종 활성화 → 화면 표시 시작
```

**소요 시간**: 20분 (예상) → ✅ 완료
**검증**: ✅ PASS (ILI9341 데이터시트 정확함)
