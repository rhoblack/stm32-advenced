## 감정 곡선 분석 (8단계)

### 1️⃣ 호기심 유발 — ★★★★★ (완벽)
**대상 섹션**: §11.0 도입

**달성 요소**:
- "지금 몇도인가?" 직관적 질문 ✅
- SHT31 센서 물리적 시연 강사 팁 ✅
- "환경 인식 기기" 비전 제시 ✅
- 누적 성장(Ch09 + Ch10 + Ch11) 자연스러운 연결 ✅

**심리학적 효과**:
- **내재적 동기 활성화**: "내가 스마트 시계를 만들고 있다"는 자각
- **관련성**: 수강생 경험(날씨 확인, 온도 체감)과 직접 연결

**강점**:
```
"지금 몇 시인가를 알았다면, 이제 '지금 몇도인가'를 알아보겠습니다"
→ 구체적 진전(progression) 느낌
```

---

### 2️⃣ 약간의 불안 유발 — ★★★★ (좋음)
**대상 섹션**: §11.1 I2C 프로토콜, §11.2 클록 스트레칭

**의도된 불안**:
- "I2C가 2핀뿐인데, 어떻게 데이터 구분을?" → 호기심 → 불안
- "UART와는 다르다" 신규 개념 도입

**심리학적 효과**:
- **적절한 불안 수준**: 학습에 필요한 주의 집중 증가
- **완전히 새로운 것은 아님**: UART 경험자이므로 "차이점 비교"는 친숙

**강점**:
- "택시 신호등" 비유로 복잡성 완화 ✅
- START/STOP 시퀀스 구체적 ✅
- 클럭 스트레칭 "아직 생각 중" 표현으로 감정 이입 ✅

**약점**:
- 오픈드레인 개념이 추상적 → 불안 증가
- 해결책: 다이어그램 강화 필요

---

### 3️⃣ 더 큰 불안 (위험 구간) — ★★ (문제 있음)
**대상 섹션**: §11.5 CRC-8 검증

**현재 상황**:
> "CRC-8(다항식 0x31) — 비트별 처리, MSB 우선"
> ```c
> crc = (crc << 1) ^ 0x31;  // ← 이게 뭐지?
> ```

**심리학적 반응**:
```
수강생: "CRC는 왜 필요한가?"
      ↓
      "다항식 0x31이 뭔가?"
      ↓
      "왜 이렇게 복잡한가?"
      ↓
      😰 불안 → 😠 좌절 → 🚫 포기
```

**문제 진단**:
1. **의미 부재**: 다항식이 "마법의 숫자"처럼 보임
2. **비유 부족**: CRC의 목적(데이터 무결성)과 구현(다항식)의 괴리
3. **불안 해소 장치 없음**:
   - FAQ("왜 다항식이 0x31인가?")가 있어도, "그냥 이렇게 하면 된다" 설명 부족
   - "당신이 할 일은 이 코드를 복사+붙여넣기하면 됩니다" 같은 안심 메시지 없음

**심리적 영향**:
- **자기효능감 저하**: "나는 이걸 이해할 수 없을 것 같은데?"
- **학습 불안**: 불명확한 개념 → "시험에서 틀리면 어쩌나?" 우려

---

### 4️⃣ **불안 해소 (전환점)** — ★ (미흡) ⚠️
**대상 섹션**: §11.5 후반 → §11.6 (연결 약함)

**현재**:
- §11.5 CRC-8 설명 후, 바로 코드 제시
- **연결 문장 부족**: "이 코드는 검증 함수입니다. 다행히 이미 구현되어 있으므로, 당신은 `crc8_verify_sht31()` 함수만 호출하면 됩니다"

**필요한 해소 요소**:
1. **정규화(Normalization)**:
   ```
   "CRC는 고급 주제입니다. 지금은 '흑마술'처럼 보여도,
   함수 호출만으로 충분합니다. 내부 동작은 데이터시트 심화 과정에서 배웁니다."
   ```

2. **즉시 성공 경험**:
   ```
   "다음 예제를 보세요. 3줄만으로 CRC 검증이 됩니다:

   if (crc8_verify_sht31(&data[0], 3) == 0) {
       LOG_I("CRC 정상!");
   }

   이렇게 간단합니다."
   ```

3. **메타인지 지원**:
   ```
   📌 [당신은 이미 이해했습니다]
   - UART 에러 처리 경험? ✅
   - I2C START/STOP 이해? ✅
   - 이제 데이터 검증만 더하면 됩니다! 🎯
   ```

### 5️⃣ 이해 (조명 켜지는 순간) — ★★★★ (좋음)
**대상 섹션**: §11.6 SHT31 Driver 구현, §11.7 Sensor Service FSM

**달성 요소**:
- **코드 유사성**: Ch09 Alarm FSM과 동일한 패턴 ✅
  ```
  "어? 이건 Ch09에서 본 것과 같은데?"
  → "아, FSM 패턴이 반복되는 거구나!"
  → 학습 곡선 완화
  ```

- **즉시 성공**: §11.7 코드 예제 30줄 내 → "내가 이 정도는 읽을 수 있다"

- **비블로킹 설계**: 50ms 대기를 FSM 상태로 분산
  ```
  "메인 루프가 먹통되지 않는다! 이건 실무다!"
  → 자기효능감 ↑
  ```

**심리적 전환**:
```
불안 😰 → 돌파 💡 → "아, 이렇게 하는 거구나!" → 이해 ✅
```

### 6️⃣ 성취감 (실습 완성) — ★★★★⭐ (탁월)
**대상 섹션**: §11.8~9 통합 실습

**달성 요소**:
- **완전한 통합**: RTC(Ch09) + Sensor(Ch11) → 실제 동작하는 스마트 시계 ✅
- **온도 알람**: "온도 > 30°C → RTC 알람 자동 트리거" → 가시적 결과
- **실습 코드**: main()에서 직접 사용할 수 있는 패턴 ✅

**심리학적 효과**:
- **즉시성(Immediacy)**: 코드 작성 후 센서에서 실제 데이터 읽힘
- **관찰 가능성(Observability)**: LED 점멸, 로그 메시지 확인
- **성취감**: "내가 I2C 드라이버를 만들었다!"

**강점**:
```c
// 실습 후 실제 결과 확인
LOG_I("온도: %.2f°C, 습도: %.2f%%", temp, humid);
→ 콘솔에서 실제 온습도 값 출력!
```

## 누적 성장 평가 (Ch09 + Ch10 + Ch11)

### ✅ **성공 요소**
1. **패턴 반복**: FSM 구조가 RTC(Ch09) → Sensor(Ch11)로 일관성 있게 재사용
2. **계층 강화**: HAL → Driver → Service로 추상화 수준 증가
3. **실제 동작**: 시각 정보 + 온습도 데이터 = 실제 스마트 시계 기능

### ⚠️ **위험 요소**
1. **CRC 이해도 부족**: "그냥 복사한다" 수준 가능성
2. **자기효능감**: "코드는 주어졌는데, 내가 직접 설계할 수 있을까?"
