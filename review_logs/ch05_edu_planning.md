# Ch05. DMA 아키텍처와 동작 원리 — 교육 설계 분석

**작성자:** 교육 설계자 (Instructional Designer)
**작성일:** 2026-03-17
**Phase:** 1 (기획 회의)
**프로젝트 버전:** v0.5
**누적 기능:** Mem-to-Mem DMA 버퍼 복사 성능 측정 (폴링 대비)

## 1. 학습 목표 설계 (블룸 분류체계)

### Ch05 핵심 성격 분석

Ch05는 **HAL 레이어 하위의 횡단 관심사(Cross-cutting Concern)**가 처음 등장하는 챕터입니다. Ch04에서 Driver 레이어를 처음 만들었다면, Ch05에서는 **모든 Driver가 공통으로 활용하는 하드웨어 가속 메커니즘**을 학습합니다. 이 전환이 학습자에게 주는 의미:

- Ch01~02: CPU가 모든 일을 직접 처리 (폴링, 직접 쓰기)
- Ch03: "계층이 있구나" (설계도)
- Ch04: "Driver를 만들어보니 인터럽트로 CPU 부담을 줄일 수 있구나"
- **Ch05: "CPU가 아예 관여하지 않고 하드웨어가 데이터를 옮길 수 있다고?"** (패러다임 전환)

### 최종 학습 목표 (6개)

```
LO-1 [기억]   DMA 컨트롤러의 3가지 핵심 구성 요소(Stream, Channel, Priority)를
              열거하고, F411RE의 DMA1/DMA2 차이를 설명할 수 있다.
LO-2 [이해]   폴링, 인터럽트, DMA 세 가지 데이터 전송 방식의 CPU 관여도 차이를
              다이어그램으로 설명하고, 각 방식이 적합한 상황을 근거를 들어 설명할 수 있다.
LO-3 [적용]   CubeMX에서 Mem-to-Mem DMA를 설정하고, HAL_DMA_Start_IT()로
              메모리 간 데이터 복사를 구현할 수 있다.
LO-4 [적용]   DWT 사이클 카운터를 활용하여 폴링(memcpy) 대비 DMA의 성능을
              측정하고 결과를 LOG_I로 출력할 수 있다.
LO-5 [분석]   성능 측정 결과를 CPU 활용도 관점에서 분석하고, "DMA가 항상
              빠른 것은 아닌 이유"를 버스 경합과 설정 오버헤드로 설명할 수 있다.
LO-6 [평가]   주어진 시나리오(소량 데이터 vs 대량 데이터, 주기적 vs 1회성)에서
              폴링/인터럽트/DMA 중 적합한 전송 방식을 판단하고 근거를 제시할 수 있다.
```

### 블룸 수준 분포 평가

| 수준 | 개수 | 평가 |
|------|------|------|
| 기억 | 1 | DMA 구조 용어는 암기 필수 — 최소 1개 필요 |
| 이해 | 1 | 3가지 전송 방식 비교는 개념적 이해 핵심 |
| 적용 | 2 | CubeMX 설정 + 성능 측정 코드 — 실습 중심 |
| 분석 | 1 | 성능 결과 해석 — Ch05의 핵심 사고력 |
| 평가 | 1 | "언제 DMA를 쓸까" 판단 — Ch05 최종 도달점 |

> **판단:** 기억 1 + 이해 1 + 적용 2 + 분석 1 + 평가 1 = 총 6개. Ch04(적용 다수)에서 한 단계 올라, **분석과 평가가 포함**됩니다. DMA는 "쓸 줄 아는 것"보다 **"언제 쓸지 판단하는 것"**이 실무에서 더 중요하기 때문입니다.

## 2. 절 구조 및 교육 흐름 설계

### 권장 절 구조 (6개 절)

```
[§5.1] 아키텍처 위치 & DMA의 역할 (Ch04부터 필수 도입부)
  → DMA = "횡단 관심사" — 모든 Driver가 활용하는 하드웨어 가속기
  → 4계층 다이어그램에서 DMA 위치: HAL 레이어 하위, Driver 레이어와 연결
  → v0.4 → v0.5 아키텍처 갱신 미리보기
  → "Ch06에서 UART Driver 내부를 DMA로 교체합니다" 예고

[§5.2] DMA가 필요한 이유 — 폴링 vs 인터럽트 vs DMA
  → "택배 수령" 비유 확장:
    - 폴링 = 편의점 앞에서 택배 기다리기 (Ch04 복습)
    - 인터럽트 = 벨 울리면 받으러 가기 (Ch04 복습)
    - DMA = 경비원이 대신 받아서 집 앞에 놔두기 (신개념!)
  → CPU 타임라인 3-way 비교 SVG
  → "1MB 복사하면 CPU는 뭘 하나?" 질문으로 동기 유발

[§5.3] DMA 컨트롤러 구조 — Stream, Channel, Priority
  → F411RE DMA1/DMA2 블록 다이어그램 (SVG)
  → Stream = "전용 운반 통로", Channel = "목적지 선택 스위치"
  → DMA_SxCR 레지스터 핵심 필드 (방향, 크기, 증가 모드)
  → CubeMX에서 DMA 설정 화면 설명
  → Priority 설정: Very High / High / Medium / Low

[§5.4] Mem-to-Mem DMA 실습 — 성능 측정
  → CubeMX Mem-to-Mem DMA 설정 (DMA2만 지원!)
  → 폴링(memcpy) vs DMA 구현: 동일 데이터 복사
  → DWT 사이클 카운터로 정밀 측정
  → "예측 투표" → 실행 → 결과 확인 → 분석
  → ⚠️ "작은 데이터에서는 DMA가 더 느릴 수 있다" — 설정 오버헤드

[§5.5] DMA 심화 — 콜백, 우선순위 역전, Circular 프리뷰
  → DMA 완료 콜백(XferCpltCallback), 반완료 콜백(XferHalfCpltCallback)
  → 우선순위 역전(Priority Inversion): "고속도로에서 느린 차가 앞을 막는 상황"
  → Critical Section 개념 소개 (__disable_irq / __enable_irq)
  → Circular Mode 프리뷰: "Ch06 UART DMA에서 본격 사용"
  → Double Buffering 개념만 언급 (Ch09 Timer DMA에서 상세)

[§5.6] 아키텍처 v0.5 완성 & 정리
  → v0.4 → v0.5 아키텍처 갱신 (DMA 횡단 요소 추가)
  → "언제 DMA를 쓸까?" 판단 기준 정리표
  → 연습문제 (블룸 4수준) + 면접 포인트
  → Ch06 예고: "이 DMA를 UART Driver에 적용하면?"
```

### 교육 흐름 패턴

```
§5.1: [설계 먼저] → DMA의 아키텍처 위치 제시 (지도 펼치기)
§5.2: [동기 유발] → "왜 DMA가 필요한가?" 3-way 비교로 필요성 체감
§5.3: [개념]     → DMA 하드웨어 구조 학습 (신개념 집중 구간)
§5.4: [실습]     → Mem-to-Mem 구현 + 성능 측정 (핵심 체험)
§5.5: [심화]     → 콜백 + 우선순위 + Circular 프리뷰
§5.6: [통합]     → 아키텍처 갱신 + 판단 기준 정리
```

> **핵심 설계 의도:** §5.2에서 "왜?"를 충분히 느끼게 한 후 → §5.3에서 "어떻게?"를 배우고 → §5.4에서 "직접 측정"으로 체감 → §5.5에서 "주의사항"을 인식 → §5.6에서 "판단 기준"으로 종합. **"측정 → 분석 → 판단"** 사이클이 Ch05의 교육적 핵심입니다.

## 3. 인지 부하 분석

### 핵심 신개념 분석

| # | 신개념 | 인지 부하 | 선행 지식 | 비고 |
|---|--------|-----------|-----------|------|
| 1 | **DMA 컨트롤러 구조 (Stream/Channel/Priority)** | **높음** — 하드웨어 블록 다이어그램, 레지스터 개념이 동시에 등장 | 없음 (독립) | 핵심 — 충분한 시간 필요 |
| 2 | **Mem-to-Mem 전송 + 성능 측정** | **중간** — DMA API 호출 자체는 단순하지만, DWT 카운터가 추가 | Ch02 LOG 매크로 활용 | 실습 중심 |
| 3 | **DMA 콜백 (완료/반완료)** | **중간** — Ch04 UART 콜백 패턴과 유사, 구조적으로 동일 | Ch04 HAL_UART_RxCpltCallback | Ch04 연결점 |
| 4 | **Priority Inversion + Critical Section** | **높음** — RTOS 없이 설명하기 까다로운 개념 | Ch01 NVIC Priority | 심화 — 개념 수준만 |
| 5 | **Circular Mode** | **낮음** — "프리뷰"로 개념만, 상세는 Ch06 | 없음 | Ch06 예고 |

### 인지 부하 판정

**총 핵심 신개념: 5개** — 권장 한도(3개)를 2개 초과

### 권장 처리 방안: 개념 병합 + 축소로 3개로 압축

```
[개념 A] DMA 컨트롤러 구조 + Mem-to-Mem    → "DMA 하드웨어와 첫 실습" (§5.3~5.4)
[개념 B] DMA 콜백 메커니즘                  → Ch04 콜백 패턴의 "DMA 버전" (§5.5 전반)
[개념 C] 우선순위 역전 + Circular 프리뷰    → "DMA 사용 시 주의사항" 묶음 (§5.5 후반)
```

**압축 전략:**
1. **DMA 구조와 Mem-to-Mem**은 "구조를 배우고 바로 써보는" 하나의 학습 단위로 통합 — 이론과 실습이 분리되지 않게
2. **DMA 콜백**은 Ch04 UART 콜백의 자연스러운 확장 — 독립 신개념이 아닌 "같은 패턴의 DMA 적용"
3. **Priority Inversion과 Circular Mode**는 "주의사항 모음"으로 축소 — 깊이는 Ch06/Ch09에서 보충
4. **DWT 사이클 카운터**는 독립 신개념이 아닌 "성능 측정 도구"로 위치 재조정 (LOG_D 수준의 유틸리티)

### 절별 인지 부하 배분

| 절 | 신개념 수 | 인지 부하 | 감정 곡선 |
|----|-----------|-----------|-----------|
| §5.1 아키텍처 위치 | 0 | **낮음** | 안심 — "DMA가 어디에 위치하는지 큰 그림 확인" |
| §5.2 폴링/인터럽트/DMA 비교 | 0.5 | **중간** | 호기심 — "CPU가 아예 안 관여한다고?" |
| §5.3 DMA 구조 | 1 | **높음** | 약간의 불안 — "Stream? Channel? 복잡한데..." |
| §5.4 Mem-to-Mem 실습 | 0.5 | **중간→낮음** | **성취감** — "와, 실제로 DMA가 더 빠르다!" |
| §5.5 콜백 + 심화 | 1 | **중간** | 연결 — "Ch04 콜백 패턴이 여기서도!" |
| §5.6 정리 + 판단 기준 | 0 | **낮음** | 큰 성취 — "언제 DMA를 쓸지 스스로 판단할 수 있다" |

> **감정 곡선:** 안심 → 호기심 → **불안(§5.3)** → **성취(§5.4)** → 연결 → 큰 성취
> 불안 지점(§5.3)을 챕터 중반에 배치하고, **바로 다음 §5.4에서 실습 성공**으로 해소합니다. §5.3과 §5.4 사이에 휴식을 배치하여 인지적 리셋을 제공합니다.

### Ch05 핵심 깨달음 예측

**"CPU가 안 해도 되는 일을 CPU에게 시키고 있었구나!"**

이 깨달음이 자연스러우려면:
1. §5.2에서 **Ch04 폴링의 CPU 타임라인을 다시 보여주고** "이 빨간 구간 동안 CPU는 아무것도 못 합니다"
2. §5.3에서 **DMA 하드웨어가 별도로 존재한다는 사실** 전달 — "CPU 옆에 '운반 전문 직원'이 있었습니다"
3. §5.4에서 **직접 측정**하여 차이를 눈으로 확인 — 예측 vs 실제의 갭이 학습 동기
4. §5.5에서 **"그래도 만능은 아니다"** — 균형 잡힌 판단력 형성

| 예고 포인트 | 실현 챕터 | 예고 문구 제안 |
|-------------|-----------|---------------|
| "Mem-to-Mem 말고 주변장치에도?" | Ch06 UART DMA | "Ch06에서 UART_Send() 내부를 DMA로 바꿉니다 — 인터페이스는 그대로!" |
| "Circular Mode가 왜 필요한가?" | Ch06 링 버퍼 | "UART 수신처럼 끊임없이 들어오는 데이터는 Circular Mode로 받습니다" |

## 5. 성능 실습 전략 (Ch05 고유 설계)

### "측정 → 분석 → 판단" 사이클

Ch05는 다른 챕터와 달리 **성능 측정이 핵심 학습 활동**입니다. 단순히 "DMA를 써보자"가 아닌 **"정말 빠른지 측정해보자"**가 교육적 핵심입니다.

### 실습 설계 (4단계)

```
[1단계] 사전 예측 (5분)
  → "1024바이트를 복사할 때, DMA가 memcpy보다 몇 배 빠를까?"
  → 학생 예측: 2배? 10배? 100배?
  → "정답은 실습 후에!"

[2단계] 폴링 기준 측정 (15분)
  → memcpy로 1024바이트 복사
  → DWT 사이클 카운터로 사이클 수 측정
  → LOG_I("memcpy: %lu cycles", elapsed);

[3단계] DMA 측정 (20분)
  → HAL_DMA_Start_IT()로 동일 데이터 복사
  → DMA 완료 콜백에서 사이클 수 기록
  → LOG_I("DMA: %lu cycles", elapsed);

[4단계] 결과 분석 + 반전 (15분)
  → "예측과 비교해봅시다"
  → 반전 1: "작은 데이터(16바이트)에서는 DMA가 더 느립니다" → 설정 오버헤드
  → 반전 2: "DMA 중 CPU는 자유 — 진짜 이득은 '속도'가 아니라 'CPU 해방'"
  → 판단 기준 정리: 데이터 크기 임계값, CPU 활용도, 주기성
```

### 측정 데이터 예시 (F411RE 100MHz 기준, 사전 준비)

| 데이터 크기 | memcpy (cycles) | DMA (cycles) | DMA 이점 |
|-------------|-----------------|--------------|----------|
| 16 bytes | ~50 | ~200 | **memcpy 4배 빠름** (설정 오버헤드) |
| 256 bytes | ~800 | ~400 | DMA 2배 빠름 |
| 1024 bytes | ~3,200 | ~1,200 | DMA 2.7배 빠름 |
| 4096 bytes | ~12,800 | ~4,500 | DMA 2.8배 빠름 + **CPU 자유** |

> **교육적 핵심:** 단순한 "DMA가 빠르다"가 아닌, **"언제 빠르고 언제 아닌지"**를 직접 확인. 이것이 LO-5(분석)와 LO-6(평가)의 달성 경로입니다.

## 8. 비유 및 실생활 예시 제안

| 개념 | 비유 | 근거 |
|------|------|------|
| DMA 컨트롤러 | **택배 경비원** — 폴링=내가 편의점에서 기다림, 인터럽트=벨 울리면 받으러 감, DMA=경비원이 대신 받아서 집 앞에 놓아둠 (나는 자유) | Ch04 "택배 기다리기" 비유의 **자연스러운 확장** — 동일 비유 세계관 유지 |
| Stream과 Channel | **고속도로 차선과 IC(나들목)** — Stream = 차선(동시에 여러 차선 사용 가능), Channel = 나들목(어디로 갈지 선택) | 물리적 구조를 도로에 대응 — 직관적 |
| DMA Priority | **응급실 트리아지** — Very High = 심정지, High = 골절, Medium = 찰과상. 높은 우선순위가 먼저 처리 | Ch01 NVIC Priority 비유와 동일 패턴 |
| Mem-to-Mem | **이사 업체** — 내가 직접 짐 나르기(memcpy) vs 이사 업체에게 맡기기(DMA). 이사 업체 부르는 데 시간이 걸리지만(설정 오버헤드), 짐이 많으면 압도적으로 빠름 | 설정 오버헤드의 존재를 자연스럽게 전달 |
| Priority Inversion | **고속도로에서 느린 차가 추월 차선 점유** — 높은 우선순위(빠른 차)가 낮은 우선순위(느린 차)에 의해 막히는 상황 | 일상에서 흔히 경험하는 답답함과 연결 |
| Circular Mode | **회전 초밥 벨트** — 접시가 끝까지 가면 다시 처음으로 돌아옴. 끊임없이 데이터가 흐르는 상황에 적합 | 순환 버퍼의 물리적 이미지를 즉시 떠올릴 수 있음 |
| DMA 완료 콜백 | **이사 업체 전화** — "짐 다 옮겼습니다" 전화가 오면 확인하고 다음 일 진행. Ch04 UART 콜백과 동일 패턴 | Ch04 "비서 전화 받기" 비유와 연속성 |

## 10. 종합 평가

| 항목 | 평가 | 비고 |
|------|------|------|
| 학습 목표 | **적절** (6개, 블룸 5수준) | 분석+평가 포함 — "판단력" 챕터에 부합 |
| 절 구조 | **적절** (6개 절, 동기→이론→측정→심화→통합) | §5.1 아키텍처 도입부 필수 포함 |
| 인지 부하 | **관리 가능** (5→3개 압축) | §5.3 DMA 구조가 최대 난관 → 휴식 후 §5.4 실습으로 해소 |
| 누적 성장 | **우수** (6개 연결점) | Ch01 NVIC + Ch02 LOG + Ch03 아키텍처 + Ch04 콜백 모두 활용 |
| 성능 실습 | **Ch05 고유 강점** | 예측→측정→반전→판단 사이클 — 분석적 사고 촉진 |
| Ch06 연결 | **매우 자연스러움** | "Mem-to-Mem → Periph DMA" 확장, uart_driver.c 업그레이드 |
| 4시간 분량 | **적절** | 이론 53% : 실습 43% : 휴식 4% |
| 감정 곡선 | **적절** | 안심→호기심→불안→**성취(측정)**→연결→큰성취 |

### 핵심 주의 사항 4가지

1. **§5.3 DMA 구조가 Ch05의 최대 난관** — Stream/Channel 개념을 고속도로 비유와 SVG로 충분히 시각화. CubeMX가 자동 선택해준다는 점 강조로 암기 부담 감소.
2. **§5.4 성능 측정이 Ch05의 핵심 체험** — 단순 "DMA 코드 작성"이 아닌 "예측 → 측정 → 반전 → 판단" 사이클 반드시 포함. 작은 데이터에서 DMA가 느린 것을 보여주는 것이 교육적 가치.
3. **Circular Mode와 Double Buffering은 프리뷰만** — 상세 구현은 Ch06(Circular + 링 버퍼)과 Ch09(Double Buffering + Timer)로 명확히 이관. "맛보기"로 호기심만 유발.
4. **Ch06 연결이 Ch05의 존재 이유** — "이 DMA 개념을 Ch06에서 uart_driver.c에 적용합니다"라는 메시지를 §5.1, §5.6에서 반복. DMA를 독립 주제가 아닌 **Ch06 실전을 위한 준비**로 포지셔닝.

## 11. Ch04와의 역할 분리 명확화

| 항목 | Ch04 | Ch05 | Ch06 |
|------|------|------|------|
| 핵심 주제 | UART 통신 기초 | DMA 하드웨어 원리 | UART + DMA 실전 통합 |
| 데이터 전송 | 폴링 + 인터럽트 | Mem-to-Mem (주변장치 X) | Periph-to-Mem, Mem-to-Periph |
| DMA 모드 | 없음 | Normal (1회 전송) | Circular + IDLE Line |
| 링 버퍼 | 없음 (단순 배열) | 없음 | **직접 구현** |
| CPU 비교 | 폴링 vs 인터럽트 | 폴링 vs DMA (사이클 측정) | 인터럽트 UART vs DMA UART |
| 아키텍처 | v0.3 → v0.4 (Driver 추가) | v0.4 → v0.5 (DMA 추가) | v0.5 → v0.6 (DMA 통합) |

> **Ch05는 "DMA 하드웨어를 이해하는 챕터"이고, Ch06은 "DMA를 실전에 적용하는 챕터"입니다.** Ch05에서 Periph DMA를 다루지 않음으로써, Ch06의 "아하!" 순간을 보존합니다.
