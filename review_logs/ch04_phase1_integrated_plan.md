## 통합 기획 방향 (3명 의견 종합)

### Phase 1 합의 사항

**세 팀의 공통 강조 포인트:**
1. **SWO vs UART 혼란 선제 해소** (교육강사 ★★★★★) → 도입부에 비교 다이어그램 반드시 포함
2. **§4.4 인터럽트 1바이트 재등록이 승부처** (모두 일치) → 충분한 시간과 비유 투입
3. **Ch03 아키텍처 첫 실전 검증** (모두 강조) → Driver 분리 + v0.3→v0.4 갱신이 숨은 핵심
4. **감정 곡선**: 안심(아키텍처 도입) → 호기심(UART 프레임) → 성취(폴링 송신) → 불안(인터럽트) → 연결(printf) → 큰 성취(Driver 완성)

**기획안 차이 조정:**
- **절 구조**: 기술 저자(5절) vs 교육설계자(6절) → **6절 채택** (§4.1 아키텍처 도입부 필수)
- **신개념 압축**: 4개 → 3개 (UART 프레임+Baud Rate 통합)
- **시간 배분**: 이론 46% + 실습 50% + 휴식 4% 채택

## 학습 목표 (6개, 블룸 분류)

1. **[기억]** UART 프레임의 구성 요소(Start, Data, Parity, Stop)와 Baud Rate의 정의를 열거할 수 있다
2. **[이해]** SWO(Ch02)와 UART(Ch04)의 차이를 설명하고, 각각의 용도를 구분할 수 있다
3. **[이해]** 폴링 방식 송수신과 인터럽트 방식 수신의 차이를 CPU 활용 관점에서 설명할 수 있다
4. **[적용]** CubeMX에서 USART2를 설정하고, HAL_UART_Transmit으로 PC에 문자열을 전송할 수 있다
5. **[적용]** HAL_UART_Receive_IT 1바이트 재등록 패턴으로 인터럽트 기반 수신을 구현할 수 있다
6. **[적용]** uart_driver.c/h를 Driver 레이어 컴포넌트로 설계하고 아키텍처 v0.4를 갱신할 수 있다

## 핵심 메시지 3가지

### 1️⃣ "이미 했는데 왜 또?" 혼란 해소
> "Ch02에서 우리는 SWO를 통해 디버거에게만 '말'했습니다(청진기). Ch04에서는 UART를 통해 누구에게나 '말'합니다(목소리). 범위가 달라집니다."

### 2️⃣ "왜 인터럽트인가?" 체감
> "폴링으로는 UART 수신을 기다리느라 LED 토글을 못 합니다. 인터럽트를 쓰면 UART 데이터가 올 때만 콜백이 실행되고, 평상시에는 main이 계속 돕니다."

### 3️⃣ "아키텍처가 실제로 동작한다"는 깨달음
> "Ch03에서 그린 회색 UART Driver 블록이 실제 C 코드(`uart_driver.c/h`)가 되었습니다. `UART_Send()` 인터페이스만 지키면, 내부를 폴링 → 인터럽트 → DMA로 바꿀 수 있습니다. 이것이 왜 레이어를 나누는 이유입니다."
