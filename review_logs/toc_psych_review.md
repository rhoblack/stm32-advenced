## 상세 검토 결과

---

### 🔴 Critical 이슈 (심리적 안전성을 크게 해치는 구간)

#### [Critical-1] Ch04 → Ch05 연속 DMA 집중 투하 (Part 1 후반)

- **문제**: Ch04에서 DMA 아키텍처를 개념 중심으로 학습한 직후, Ch05에서 UART DMA + 링 버퍼 구현이 즉시 요구된다.
- **심리적 위험**: DMA는 임베디드 학습자가 가장 높은 인지 부하를 호소하는 주제 중 하나다. "개념 챕터(Ch04) → 즉시 실전 적용(Ch05)"의 간격이 너무 짧아, Ch04에서 완전히 이해되지 않은 상태에서 Ch05를 만나면 자기효능감이 급격히 하락할 수 있다.
- **특히**: 링 버퍼(Ring Buffer) 구현이 Ch05에 처음 등장하는데, 이는 DMA 이해와 별개로 자료구조 구현 능력도 요구한다. 두 가지 고난이도 개념이 한 챕터에 압축되어 있어 과부하 위험이 크다.
- **권고**: Ch04와 Ch05 사이에 "DMA Mem-to-Mem 단순 실습" 미니 세션을 삽입하거나, 링 버퍼를 Ch02 로그 시스템 구축 시점에 미리 소개하여 Ch05 인지 부하를 분산시킬 것.

---

#### [Critical-2] Ch09 → Ch10 → Ch11 연속 심화 (Part 3 전체)

- **문제**: I2C 심화(Ch09) → SPI DMA + ILI9341 TFT LCD 드라이버 설계(Ch10) → I2C DMA 심화(Ch11) 가 연속 배치되어 있다.
- **심리적 위험**: Ch10은 이 커리큘럼 전체에서 **난이도가 가장 높은 챕터**다. SPI DMA 통신, LCD 드라이버 레이어 설계(LCD → GFX → App), 폰트/색상 렌더링까지 하나의 챕터에 집중되어 있다. 이것이 Ch09의 I2C 심화 직후에 배치되고, Ch11에서 다시 I2C DMA 심화로 이어지는 구조는 "불안 구간이 3챕터 연속"으로 이어지는 형태다.
- **특히**: Ch10에서 v1.0 마일스톤(디지털 시계 + 온습도 + 스톱워치 화면)이 달성되지만, 이 마일스톤에 도달하기 위해 처리해야 할 기술 스택이 지나치게 많아 "완성 전 좌절"을 경험할 가능성이 높다.
- **권고**: Ch10을 두 챕터로 분리 고려 — Ch10a: SPI DMA 기초 + LCD 초기화/기본 출력, Ch10b: GFX 레이어 설계 + 전체 UI 완성. 또는 Ch11(I2C DMA)을 Part 4 이후로 이동하여 호흡 공간 확보.

---

### 🟡 Major 이슈 (불안 유발 가능성)

#### [Major-1] Ch02 로그 시스템 — 도입 시점의 추상성

- **문제**: Ch01에서 GPIO/LED/버튼의 구체적 성공 경험 후, Ch02에서 곧바로 SWO/ITM printf 리다이렉트, SEGGER RTT, 링 버퍼 기반 비동기 로그 출력을 다룬다.
- **심리적 위험**: 학습자는 "아직 뭔가 만들지도 않았는데 로그 시스템부터?"라는 목적 불명확 불안을 경험할 수 있다. 로그 시스템의 가치는 이후 챕터에서 버그를 만날 때 비로소 실감되므로, 초반 동기 연결이 약하다.
- **권고**: Ch02 도입부에 "이 챕터가 없으면 Ch09~Ch11의 SPI/I2C DMA 디버깅이 얼마나 고통스러운지"를 미리 체험 가능한 짧은 시나리오(버그 재현 데모)를 제시하여 학습 필요성을 내면화시킬 것.

---

#### [Major-2] Ch08 스텝 모터 — Part 2의 난이도 점프

- **문제**: Ch06(타이머 심화) → Ch07(RTC) 까지는 개념의 누적이 자연스럽지만, Ch08에서 스텝 모터 제어 + 가속/감속 프로파일 + FSM 상태 관리 + RTC 각도 변환 알고리즘이 동시에 등장한다.
- **심리적 위험**: 임베디드 개발자라도 모터 제어는 별도 도메인 지식이 필요하다. "RTC 시각 → 침 각도 변환 알고리즘"은 수학적 계산이 포함되어 있어, 소프트웨어 배경이 강한 학습자에게는 예상 외의 어려움이 된다.
- **권고**: Ch08 시작부에 "단계별 체크포인트" 제시 — (1) 모터 단순 정방향 회전 확인, (2) FSM 상태 전환 확인, (3) RTC 연동 순으로 소성공 경험을 명시적으로 분리.

---

#### [Major-3] Ch14 v2.0 전체 리팩토링 — 역행 심리

- **문제**: Ch14에서 v2.0으로 전체 코드를 재구조화(FSM/레이어 기반 리팩토링)한다. 이는 Ch01~Ch13까지 쌓은 코드를 "다시 고치는" 작업이다.
- **심리적 위험**: "내가 지금까지 잘못 만든 건가?"라는 자기효능감 저하 위험이 있다. 리팩토링은 성장의 증거이지만, 학습자는 이를 부정으로 인식할 수 있다. 특히 13챕터 분량의 코드를 재구조화하는 작업 범위는 심리적 부담이 크다.
- **권고**: Ch14 도입부 메시지를 "이제 여러분의 코드는 리팩토링할 가치가 있을 만큼 성장했습니다"라는 긍정적 프레이밍으로 명시. 리팩토링 전/후 비교를 통해 성취감이 즉시 확인되도록 설계.

---

#### [Major-4] Ch13 Python 대시보드 — 언어 전환 부담

- **문제**: Ch12까지 전부 C/임베디드 문맥이었으나, Ch13에서 갑자기 Python + pyserial + PyQt5/tkinter + 멀티스레드 수신 스레드가 요구된다.
- **심리적 위험**: 중급 임베디드 개발자 중 Python GUI에 익숙하지 않은 학습자에게는 "언어 맥락 전환 비용"이 발생한다. PyQt5/tkinter 중 선택지가 열려 있는 것도 결정 부담이 된다.
- **권고**: Python 코드는 최대한 복사-붙여넣기 수준의 완성 템플릿을 제공하고, 학습 포인트를 "프레임 구조 이해"와 "MCU-PC 통신 아키텍처"에 집중시켜 Python 자체 학습 부담을 최소화할 것.

---

### 🟢 Minor 이슈 (동기 보강 제안)

#### [Minor-1] Part 1 (Ch03~05) 성취 가시성 부족

- Ch03~05는 UART→DMA로 진화하는 과정인데, 각 챕터 완료 시 학습자가 눈으로 체감할 수 있는 "성능 전후 비교" 결과물이 목차 상에 명시되지 않았다.
- **제안**: 각 챕터 완료 시 "CPU 점유율 비교 그래프" 또는 "전송 속도 측정값"을 실습 결과물로 포함하여 DMA 적용 효과를 숫자로 확인하게 하면 성취감이 구체화된다.

---

#### [Minor-2] Ch15 디버깅 챕터 — 배치 적절성

- Ch15는 디버깅 스킬 심화인데, 프로젝트 버전 번호가 명시되어 있지 않다. 누적 성장형 방식에서 이 챕터의 산출물이 무엇인지 불명확하다.
- **제안**: "Ch15 완료 = 전체 프로젝트의 특정 버그를 HardFault 분석으로 해결하는 실습 시나리오 1건 완료"처럼 구체적 완료 기준을 제시하여 막연한 학습이 되지 않도록 할 것.

---

#### [Minor-3] Part 0~1 전환부 — 학습 지도 부재

- Ch01~02가 Part 0(기반 구축), Ch03~05가 Part 1(통신 기초)로 나뉘어 있지만, 파트 전환 시 "지금까지 배운 것 + 앞으로 배울 것 + 완성될 프로젝트 모습"을 보여주는 로드맵 페이지가 없다.
- **제안**: 각 Part 시작부에 "현재 프로젝트 상태(무엇이 동작 중인가) + 이번 파트 종료 시 달라지는 점"을 한 눈에 보여주는 시각적 미니 로드맵을 삽입할 것.

### ✅ 잘 된 점

1. **누적 성장형 버전 관리 (v0.1 → v2.0)**: 매 챕터마다 프로젝트 버전이 명확히 올라가는 구조는 학습자가 자신의 성장을 숫자로 확인할 수 있어 자기효능감 유지에 매우 효과적이다. 이 방식은 심리적 안전성의 핵심 기제다.

2. **Ch01 진입 설계**: 가장 친숙한 GPIO/LED/버튼으로 시작하는 것은 "첫 실습 반드시 성공" 원칙에 부합한다. 중급자 대상임에도 기초 확인에서 출발하는 설계가 자신감 기반을 형성한다.

3. **Ch02 로그 시스템의 전략적 위치**: 이후 모든 챕터의 표준 도구가 되는 로그 시스템을 초반에 배치한 것은 장기적으로 디버깅 좌절을 예방하는 현명한 구조다. 단, 이 의도가 학습자에게 명확히 전달되어야 효과가 있다.

4. **v1.0 마일스톤 (Ch10)**: "디지털 시각 + 온습도 + 스톱워치 화면 표시"라는 가시적이고 완성도 높은 중간 결과물이 존재한다. 화면에 동작하는 시스템을 보는 것은 강력한 성취 경험이다.

5. **Part 4 (CLI + 대시보드)의 배치**: 하드 기술 심화(Part 3) 이후 CLI와 PC 대시보드처럼 "내가 만든 시스템과 대화하는" 챕터를 배치한 것은 감정 회복 곡선을 고려한 좋은 설계다.

6. **Ch16 마무리 구조**: 코드 리뷰 체크리스트 + 전체 프로젝트 마무리 회고를 마지막에 배치한 것은 학습 여정의 클로저(closure)를 제공한다. 회고는 메타인지를 촉진하고 자기효능감을 강화하는 증거 기반 실천이다.

7. **FSM 예고 (Ch01)**: Ch01에서 FSM을 예고하고 이후 챕터에서 점진적으로 완성하는 구조는 인지적 앵커링(cognitive anchoring) 효과를 활용한 설계다.
