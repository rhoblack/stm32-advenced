#### 테이블 기반 FSM vs switch-case FSM 비교 코드 설계

**switch-case 방식 (기존 Ch13)**:
- 모든 상태/이벤트 조합이 중첩 switch-case로 구성
- 상태 추가 시 switch-case 전체 수정 필요
- 코드 라인 수: 상태×이벤트 수에 비례하여 선형 증가

**테이블 기반 방식 (v2.0 리팩토링)**:
- `fsm_transition_t` 구조체 배열 (상태, 이벤트, 조건함수, 액션함수, 다음상태)
- 상태 추가 = 테이블에 행 추가만으로 완결
- `fsm_run()` 엔진 함수 1개로 모든 FSM 처리

#### 3종 시나리오 시퀀스 다이어그램 기획

1. **알람 시나리오**: RTC AlarmA IRQ → Alarm FSM(TRIGGERED) → CLI 알람 메시지 → Motor 알람 동작 → LCD 알람 표시 → UART 대시보드 전송
2. **스톱워치 시나리오**: Button EXTI → Stopwatch FSM(RUNNING→PAUSED) → LCD 갱신 → UART 전송
3. **대시보드 갱신 시나리오**: TIM2 1초 UEV → RTC_GetTime → SHT31 Read → Protocol Frame 생성 → UART DMA 전송 → PC pyserial 수신

## Sub-agent 2 결과: 교육 설계자 관점

### 학습 목표 5개 ("~할 수 있다" 형태)

1. **[분석]** Ch03에서 v2.0까지 레이어드 아키텍처의 진화 과정을 단계별로 설명하고, 각 챕터에서 아키텍처가 어떻게 확장되었는지 추적할 수 있다.

2. **[평가]** 기존 코드에서 레이어 의존성 위반(App → HAL 직접 접근, 역방향 의존 등)을 식별하고, 올바른 인터페이스 계층을 설계하여 수정할 수 있다.

3. **[창조]** switch-case 방식의 FSM을 테이블 기반 FSM으로 리팩토링하고, DISPLAY FSM에 새 상태를 추가할 수 있다.

4. **[이해]** 계층적 FSM(HFSM)의 개념과 진입/진출 액션, 부모 상태 위임 메커니즘을 설명하고, 간단한 2단계 HFSM을 설계할 수 있다.

5. **[분석]** 전체 시스템의 알람/스톱워치/대시보드 갱신 3종 시나리오를 시퀀스 다이어그램으로 추적하며, SRP·DIP·ISP 설계 원칙이 실제 코드에 어떻게 적용되었는지 검증할 수 있다.

### Part 5 위상 분석

- **위치**: 전체 19챕터 중 Ch17 = 교재의 마지막 심화 통합 챕터
- **학습 상태**: 수강생은 Ch01~Ch16을 통해 HAL→Driver→Service→App 전 레이어를 직접 구현한 상태
- **역할**: 지식의 수직 통합(Vertical Integration) — 개별 챕터 지식을 아키텍처 관점에서 재조망
- **감정 설계**:
  - 도입: "내가 만든 16개 챕터 코드를 아키텍처 전문가처럼 리뷰한다" → 강한 호기심 + 자부심
  - 중반: HFSM·DIP 개념 → 약간의 불안 (단, 이미 만든 코드를 보며 설명하므로 안전)
  - 후반: "내 코드가 실무 아키텍처 패턴을 따른다"는 성취감

### 인지 부하 관리

- 1절: 순수 리뷰(기억/이해) → 부하 낮음
- 2절: 의존성 위반 탐색(분석) → 중간
- 3절: 테이블 FSM 구현(적용/창조) → 중간~높음
- 4절: HFSM 개념(이해/적용) → 높음 → 코드 예제로 반드시 완화
- 5절: 시퀀스 분석(분석) → 중간 (이미 만든 기능 복습)
- 6절: Pinout + 원칙 점검(기억/평가) → 낮음~중간

## Sub-agent 3 결과: 강사 관점

### 수강생 막힘 Top 5 + 해소 방법

**막힘 1: "HFSM에서 상위 상태의 전이가 하위 상태에 영향을 주는 방식이 이해가 안 된다"**
- 해소: 계층 조직도 비유 — 팀장이 자리를 비우면 팀 전체가 자동으로 '대기' 상태로 전환되는 것처럼, 부모 상태 전이는 자식 상태 전체에 적용된다
- 코드에서 `parent_state` 포인터로 위임 흐름을 추적하게 유도

**막힘 2: "의존성 역전(DIP)과 인터페이스 분리(ISP)가 결국 같은 말 아닌가?"**
- 해소: 구체적 예시 — DIP는 "App이 sht31_driver.h를 직접 include하지 말고 sensor_service.h를 include해라", ISP는 "sensor_service.h에 SHT31에만 필요한 CRC 함수를 노출하지 마라"
- 두 원칙의 적용 대상이 다름을 코드 diff로 시각화

**막힘 3: "테이블 FSM에서 조건 함수(guard function)가 NULL인 경우와 아닌 경우의 처리 로직"**
- 해소: fsm_run() 엔진 코드에서 `if (guard == NULL || guard())` 패턴 강조
- "guard가 NULL이면 무조건 전이, 있으면 조건 통과 시에만 전이" 원칙 판서

**막힘 4: "아키텍처 진화 타임라인에서 각 버전의 경계가 모호하다"**
- 해소: 각 버전을 Git 커밋처럼 설명 — "v1.0은 마일스톤 ①로, 스마트 시계 핵심 기능이 처음으로 동작하는 시점"
- SVG 진화 다이어그램에서 마일스톤 강조 표시

**막힘 5: "단일 책임 원칙(SRP)을 지키면 파일이 너무 많아지지 않나?"**
- 해소: "파일이 많아지는 것은 구조가 명확하다는 증거" — 스파게티 1000줄 vs 50줄 파일 20개 비교
- 임베디드에서 링커 스크립트로 최적화하면 런타임 비용 없음 강조

### 강의 현장 활용 포인트

- Ch17 시작 전 칠판에 "v1 → v2.0 진화" 타임라인 판서 후 학습자가 직접 채우게 함
- DISPLAY FSM 리팩토링 실습은 반드시 라이브 코딩으로 진행 (수강생이 기존 코드 알기 때문)
- 시퀀스 다이어그램 3종은 팀별로 나눠 발표하게 하면 자연스러운 복습 효과
